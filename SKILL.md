# doc-chain

> 用链式文档依赖网络约束 AI 行为——每份文档声明上游来源，变更必须全链路同步。修改上游必须同步下游，修改下游不得超出上游定义。文档是项目的唯一事实来源。

执行任何文档相关步骤前，先读取项目 AGENTS.md 了解项目信息，再检查项目内是否已有对应的文档。

- **有** → 按已有文档执行，同时检查上下游影响
- **没有** → 用本 skill 的模板生成文档

---

## 0. 参考流程（非强制）

以下是一种常见的步骤衔接方式，供项目参考：

```
[调研] → [PRD] → [交互设计] → [UI 设计] → [技术方案] → [测试]
          ↑                                          │
          └──── 发现上游问题需回改 ─────────────────────┘
```

步骤顺序由项目根据阶段决定，skill 不强制。每个步骤产出的文档数量（按模块、按端或其他方式拆分）由项目需求决定。

**模板分工**：

- `steps/` — 定义单个模块/步骤的文档结构和写作要求

## 2. 文档组织与关联

本 skill 的核心价值是帮助建立**可追溯的文档依赖网络**。各步骤按自己的最佳粒度自由拆分（PRD 按模块、交互按页面组/流程、UI 按页面、技术按服务），文档之间通过文件头部的"上游文档"表格建立引用关系。目录结构由项目根据团队习惯自定，唯一要求是**每份文档的文件头部必须声明上游输入**。

### 2.1 组织方式

目录结构不设限制。项目可按以下维度组织文档：

- **按模块聚合**：同一模块的所有步骤文档放在一起
- **按步骤聚合**：同一职责的所有模块文档放在一起
- **按产品线或端拆分**：不同产品线或端的文档分开存放

以上可任意组合。

### 2.2 文档关联原则

每份文档必须通过文件头部的**上游文档**表格，声明：
1. **引用了哪些上游文档**（如 PRD、全局规范、交互设计）
2. **引用了上游的哪些内容**（如功能点 `F001-F005`、页面 `P001-P003`）

**变更时的强制约束**：

| 场景 | 必须执行的操作 | 红线 |
|------|--------------|------|
| **修改上游文档** | 1. **分层扫描下游**：先扫描直接下游（如改 PRD → 扫交互+技术），再按需扫描间接下游。使用 `doc-audit.py --scan-downstream` 自动化扫描<br>2. 列出完整影响范围，按优先级排序（P0 功能点 > P1 > P2）<br>3. 询问用户是否级联修改<br>4. **用户确认** → 同步修改所有下游文档中引用的相关内容<br>5. **用户拒绝** → 在受影响下游文档的上游表格中标注 `[待同步]`，并记录冲突 | **禁止**不同步也不标注就留下上下游不一致 |
| **修改下游文档** | 1. 读取本文档引用的所有上游文档<br>2. 逐条比对：修改内容是否在上游定义范围内<br>3. **未超出** → 直接修改<br>4. **超出** → **立即停止修改**，先修改上游文档后再继续 | **禁止**擅自扩展上游定义 |

**范围判定标准**：
- 功能点：修改后的功能描述与上游 PRD 的功能编号对应关系不变
- 页面：修改后的页面范围在上游 PRD 页面清单之内
- 状态/术语/错误码：修改后的值在上游全局规范定义的枚举/规则之内

## 3. 执行方式

### 3.1 规范来源优先级

1. **项目已有文档**：项目目录内的现有文档
2. **Skill 模板**：`references/steps/` + `references/global/`

> 项目已有文档覆盖 Skill 模板。文档未定义的部分，参照 Skill 模板。

### 3.2 通用行为约束（适用于所有步骤）

以下约束适用于生成文档、修改文档、实现代码、修改代码的所有步骤：

- **执行前做 gap 分析，确认后再输出**：加载模板和上游后，先比对用户输入/修改要求与模板要求的章节结构，识别缺失信息（功能边界、异常场景、权限规则、状态流转等）。将缺失项列成问题清单询问用户，用户确认或明确忽略后再继续。禁止直接基于模糊需求输出完整文档/代码，或在修改范围不明确时直接动手。
- **不要脑补，不要过度复杂化**：严格基于已确认的上游文档和用户输入生成或修改内容。未定义的功能点、规则、字段、状态值、接口、配置项禁止自行补充。同时避免过度设计——不在用户要求之外增加不必要的复杂度、层级或依赖。
- **理性独立，不迎合，遇冲突/模糊必停**：基于专业判断分析用户需求，发现不合理、有风险、与最佳实践冲突或与上下游定义矛盾的地方，必须明确提出质疑和建议，禁止为了迎合用户而回避问题或输出明知有缺陷的方案。遇到需求不明确、规范冲突、技术不可行等情况，禁止用默认假设继续，必须暂停并询问用户。
- **缺少前置定义禁止继续**：当前步骤依赖的前置规范（文档/接口/配置/全局规范）缺失时，必须暂停并询问用户是否补全。用户拒绝或延后 → **立即停止**，禁止以默认假设或 skill 模板代替实际规范继续后续步骤。

### 3.3 文档生成与修改的通用流程

执行任何文档相关步骤前，按以下顺序建立上下文并执行：

#### 阶段 1：前置检查

| 顺序 | 动作 | 说明 |
|------|------|------|
| 1 | 读取 AGENTS.md | 获取项目信息（文档组织方式、项目背景）。**如 AGENTS.md 未定义文档组织方式，执行自动推断**（见下方「文档组织方式自动推断」） |
| 2 | 搜索现有文档 | 在项目目录内搜索当前步骤的现有文档（通过文件名特征或文件头部的 upstream-document 表格识别）。搜索结果不明确时，询问用户确认 |
| 3 | 确定模式 | **未找到** → 进入「新建模式」<br>**已找到** → 进入「修改模式」 |

#### 阶段 2：加载上下文（模式分支）

**新建模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4a | 加载全局规范 | **先查找项目目录**内是否有当前步骤类型对应的全局规范。<br>• **有** → 加载项目全局规范，作为生成约束；**同步加载该全局规范上游文档中声明的依赖**（如交互全局规范需加载 PRD 全局规范）。<br>• **无** → 加载 skill 中的全局规范模板 `references/global/*-global.md`；**同步加载该全局规范模板头部 upstream-document 中声明的上游全局规范模板**；询问用户是否基于该模板生成项目全局规范。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 5a | 解析上游文档 | 根据用户描述的任务范围推理需要哪些上游输入，或按用户明确指定加载。**上游不固定写死在步骤模板中** |
| 6a | 加载上游文档 | 在项目目录内搜索并读取步骤 5a 确定的上游文档 |
| 7a | 加载步骤模板 | 读取 `references/steps/*-step.md` 作为文档骨架，**同步加载对应示例文件 `references/examples/*-example.md` 作为参照** |

**修改模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4b | 加载现有文档 | 读取找到的现有文档内容 |
| 5b | 加载全局规范 | **先查找项目目录**内对应类型的全局规范，检查是否有更新；**同步加载该全局规范上游文档中声明的依赖**（如交互全局规范需加载 PRD 全局规范）。如无项目版本，则加载 skill 全局规范模板 `references/global/*-global.md`，**同步加载该模板头部声明的上游全局规范模板**，询问用户是否生成项目全局规范。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 6b | 加载上游文档 | 根据现有文档头部的 upstream-document 表逐项读取，检查是否有更新 |
| 7b | 分析修改要求 | 明确用户要求修改的具体范围和内容；**同步加载对应示例文件作为参照** |

#### 阶段 3：影响评估、执行与检查

| 顺序 | 动作 | 模式 | 说明 |
|------|------|------|------|
| 8 | 上下游影响扫描 | 仅修改模式 | 按 §2.2 分层扫描本次修改对下游文档的影响范围。如当前修改的是全局规范，需分层扫描所有下游全局规范和步骤文档的引用 |
| 9 | 生成/执行修改 | 两种模式 | 新建模式：按模板结构输出完整文档<br>修改模式：按用户要求执行具体修改 |
| 10 | 完整性与一致性检查 | 两种模式 | **必须由独立只读 subagent 执行**，主会话不得自行检查自己生成的文档。**subagent 权限：只读，禁止修改任何文件。**<br><br>审计分三层执行，**脚本做机械核对，模型做语义判断**：<br><br>**【修改模式】增量聚焦路径**<br>1. **生成 diff**：用 Shell `diff` 工具对比修改前后<br>2. **分析变更范围**：k2.6 读 diff，确定影响的功能点/页面/章节<br>3. **加载受影响内容**：只加载变更部分 + 受影响的上游/下游引用（不加载全文）。**上下文预算管理**：单次 subagent 加载的文档总 token 估算不超过 12k（约 6k 中文字）。超出时，按优先级分片加载：当前文档 > 直接上游 > 直接下游 > 间接上下游<br>4. **执行机械检查脚本**：subagent 调用 `references/tools/doc-audit.py` 执行机械检查，输出结构化 JSON。用法示例：`python doc-audit.py <doc_path> --type prd --delta F001,F003 --upstream up1.md up2.md`。脚本覆盖所有标记为 `[脚本]` 的检查项（编号连续性、表格格式、引用有效性、错误码格式、接口一致性等）<br>5. **语义判断**：k2.6 对受影响部分执行标记为 `[模型]` 的检查项（业务规则语义一致性、异常场景覆盖、状态流转匹配等）<br><br>**【新建模式】全量路径**<br>1. **加载全文** + 上游文档 + 对应步骤模板。若全文超出上下文预算，按**分层审计**策略：先审计当前文档机械合规性（脚本），再审计语义一致性（模型分段处理），最后审计跨文档链路（只加载关键引用段落而非全文）<br>2. **执行机械检查脚本**：调用 `doc-audit.py` 覆盖全文<br>3. **语义判断**：k2.6 对全文执行 `[模型]` 检查项。超出预算时，按模块/章节拆分多次 subagent 调用<br><br>**subagent 返回结果**：<br>• **脚本输出 + 模型判断均无异常** → 继续下一步<br>• **脚本发现机械性问题** → 直接输出问题清单（无需用户确认，机械性问题有明确修正标准），主会话修正后重新启动 subagent 从第4步执行<br>• **模型发现语义不一致** → 输出问题清单 + 影响范围 + 建议修正方案，等待用户确认后修正并重新启动 subagent |
| 11 | 写入/更新文档头部 | 两种模式 | 新建模式：在文件头部写入实际加载的上游文档表<br>修改模式：如上游文档有更新或引用范围有变化，同步更新 upstream-document 表 |

**文档组织方式自动推断**

当 AGENTS.md 未定义文档组织方式时，AI 通过以下启发式规则自动推断，**不停止流程**：

1. **搜索常见文档目录**：检查项目根目录是否存在 `docs/`、`doc/`、`specs/`、`documents/`、`design/` 等目录
2. **搜索文件特征**：在项目目录内搜索文件名匹配 `PRD-*.md`、`TECH-*.md`、`交互设计-*.md`、`UI设计-*.md`、`测试报告-*.md` 的文件
3. **推断逻辑**：
   - 发现 `docs/prd/`、`docs/tech/` 等子目录 → 按步骤聚合
   - 发现 `docs/订单模块/`、`docs/用户模块/` 等子目录 → 按模块聚合
   - 未发现任何文档目录或特征文件 → 询问用户期望的组织方式（只问一次，不阻塞），同时默认使用项目根目录下的 `docs/` 作为文档根目录
4. **告知用户**：自动推断完成后，向用户说明推断结果，如"未在 AGENTS.md 中找到文档组织方式，已自动推断为按步骤聚合：docs/prd/、docs/tech/、docs/interaction/"

**关键规则**：
- `references/global/*-global.md` 是 skill 提供的**全局规范写作指南（meta）**，不是项目全局规范本身。生成项目全局规范时，必须严格区分两类内容：
  - **元内容（仅供 AI 参考，禁止输出到项目文档）**：文件中的 `§1 章节结构`、`§2 各章节写作要求`、`§3 内容边界`、`§4 与模块文档的关系`、`§5 检查清单` 等段落。这些内容属于 Skill 内部写作指南，说明"全局规范应该怎么写"，严禁复制到项目文档中。
  - **实际规范内容（必须填充并输出）**：根据用户提供的项目信息，按照模板对应的章节编号和标题，输出具体的产品概述、模块划分表、术语定义、状态值枚举、错误码规则、技术栈选型、工程结构等实际内容。
  - **判定标准**：生成的项目全局规范中**不得出现**"章节结构"、"写作要求"、"内容边界"、"检查清单"、"格式："、"表格："、"树形结构"等指导性文字。每个章节必须是可直接被下游文档引用的**实际定义**，而非"告诉 AI 怎么写"的说明。
- 没有项目全局规范时，必须先基于该模板生成项目版本，禁止以 skill 模板代替项目全局规范继续后续步骤。
- 步骤模板 §1 中的 `upstream-document` 表格仅为**常见依赖示例**，实际生成时必须根据项目上下文动态判断，禁止机械套用。
- 如关键上游文档缺失，暂停当前步骤，提示用户先完成上游。
- 全局规范缺失时**先询问用户**是否基于模板生成项目版本。用户确认 → 生成后继续；**用户拒绝/延后时**：
  - 若当前任务是**生成/修改全局规范本身** → 可继续，因为目标就是创建全局规范
  - 若当前任务是**生成/修改模块级下游文档** → 如用户明确指定了任务范围且提供了足够上下文，可继续一次，但必须在生成文档的 upstream-document 表格中标注 `[全局规范待补全]`，并提醒用户在模块文档定稿前补全全局规范
- 生成的文档头部必须包含真实的 `upstream-document` 表，反映本次实际加载的所有上游。

### 3.4 回改触发与重新验证（阶段 4）

在执行过程中发现当前文档的上游定义存在缺陷、遗漏或与需求矛盾时，触发回改：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 12 | 暂停当前步骤 | 立即停止当前文档的生成/修改，向用户报告冲突点 |
| 13 | 定位回改范围 | 明确需要修改的上游文档名称、章节、具体条目 |
| 14 | 评估下游影响 | 按 §2.2 分层扫描该上游文档的下游引用。使用 `doc-audit.py --scan-downstream <docs_dir>` 自动化列出受影响文档清单，按直接/间接下游分层展示 |
| 15 | 用户确认 | 向用户呈现：回改内容 + 影响范围 + 建议方案。用户确认后执行上游修改 |
| 16 | 级联修改 | 同步修改所有受影响下游文档中引用的相关内容，或按 §2.2 标注 `[待同步]` |
| 17 | 重新验证 | 上游及下游全部修改完成后，回到阶段 3 第 10 步执行完整性与一致性检查，确认无异常后再继续。如回改涉及已落地代码，修正后还需触发阶段 5 代码落地验证 |

**关键规则**：
- 禁止在发现上游缺陷时擅自绕过或修改下游来"适配"
- 回改完成后必须重新执行一致性检查，不能假设修改正确
- 涉及多个上游文档的回改，按依赖顺序从下往上逐层处理

### 3.5 阶段 5：代码落地验证（文档→代码的闭环）

技术方案文档生成并通过检查（阶段 3）后，代码实现完成即触发本阶段。本阶段是文档与代码之间的最后一道锁，防止"文档完美、代码走样"。

#### 触发条件

- 用户明确告知"代码已实现"或"请检查代码与文档是否一致"
- 主会话检测到代码文件发生变更且对应技术方案已存在

#### 执行流程

| 顺序 | 动作 | 说明 |
|------|------|------|
| 18 | 定位代码文件 | 按项目技术全局规范 §2.2 的工程结构，定位当前服务/模块对应的代码文件（controller、service、entity/ORM、DAO、路由配置等） |
| 19 | 启动只读 subagent | **独立只读 subagent**，重新加载：当前技术方案文档、上游 PRD 文档、已定位的代码文件 |
| 20 | 执行五项对比 | subagent 按 §3.5.1 检查维度逐项对比文档定义与代码实现 |
| 21 | 产出审计报告 | 生成《文档-代码一致性审计报告》（参见 `references/steps/code-audit-report.md`） |
| 22 | 判定结果 | **全部通过** → 阶段 5 完成，继续后续步骤<br>**存在差异** → subagent 输出差异清单，主会话向用户呈现，等待用户决定改代码或改文档 |
| 23 | 修正后复验 | 修正完成后**重新启动只读 subagent**从第 19 步执行，禁止主会话自行复判 |

#### 检查维度

| 维度 | 文档侧 | 代码侧 | 判定标准 |
|------|--------|--------|---------|
| **接口一致性** | 技术方案 §6 接口设计（URL、方法、请求参数、响应结构、错误码） | controller/API 路由/方法签名/参数对象/响应对象 | 路径、方法、字段名、类型、必填性一致 |
| **数据模型一致性** | 技术方案 §5 表结构（字段、类型、约束、审计字段） | entity/ORM/建表语句/迁移脚本 | 字段名、类型、约束、审计字段一致 |
| **错误码一致性** | PRD §7 错误码定义 | 代码中抛出的异常/返回的错误码 | 错误码编码、触发场景一致，无新增未定义错误码 |
| **权限一致性** | PRD §6 权限规则 | 代码中的权限注解/拦截器/角色判断 | 权限层级（公开/登录/角色）一致 |
| **业务规则一致性** | 技术方案 §7 核心流程 | 代码中的核心业务流程实现 | 流程步骤、状态流转、触发条件一致 |

**关键规则**：
- subagent **只读**，禁止修改任何代码或文档
- 发现差异时必须明确标注：差异项、文档原文、代码原文、建议修正方向（改代码/改文档）
- 代码中的字段命名可能因语言规范与文档中的 snake_case/camelCase 不同，此类风格差异不视为不一致，但业务含义必须一致
- 代码中合理的内部辅助字段/方法（不暴露给外部）不纳入一致性检查范围

### 3.6 通用一致性硬规则（跨模板适用）

以下规则在所有步骤文档的生成和修改中强制适用，不随项目上下文变化：

**编号连续性**
- 功能编号 `Fxxx`、页面编号 `Pxxx` 在**同一模块/文档内**必须连续无跳号。跨模块允许跳号（遵循全局规范 §2.2 的编号段分配）
- 规则编号 `Rxxx`、用例编号 `TC-Fxxx` / `TC-Exxx` 在**同一文档内**必须连续无跳号
- 禁止同一编号在不同文档中定义不同含义
- 编号一旦分配即永久占用，删除功能/页面后编号留空不回收

**双向引用完整性**
- PRD §2 的功能点必须在 §4/§5/§6/§7/§9 中至少被一个章节引用
- 交互设计中的每个页面必须在「与 PRD 对应」子节中明确标注来源
- 技术方案 §10 接口清单必须与 §6 逐个接口一一对应（数量、路径、方法）
- 测试报告 §3 覆盖检查中的每个验收标准必须在 §1 或 §2 中有对应用例

**术语一致性**
- 同一术语在同一文档中必须使用完全相同的名称（禁止「用户/客户/会员」混用）
- 术语首次出现时标注来源（项目全局规范或本模块定义）
- 发现上下游术语定义冲突 → 立即停止，按 §3.4 回改流程处理

**全局规范交叉对齐**
- PRD 全局规范 §2.2 的模块划分，必须在交互全局规范 §2.1.1 Tab 栏和技术全局规范 §2.2.2 模块与代码包映射中有对应定义（模块数量、名称、职责边界一致）
- PRD 全局规范 §2.4 的全局术语表，必须在技术全局规范 §2.4.3 通用字段命名约定和交互全局规范 §2.1.1 Tab 栏名称中有对应使用
- PRD 全局规范 §2.5 的全局状态值，必须在技术全局规范 §2.4 公共表定义中有对应字段类型
- 交互全局规范 §2.4 的通用状态机（加载中/成功/失败/空数据/禁用），必须在 UI 全局规范 §2.1.2 语义色和 §2.5 组件状态中有对应视觉定义
- 禁止出现 PRD 全局规范有定义但其他全局规范无对应的情况，发现缺失 → 触发回改流程

**常见遗漏项（生成后必须逐条确认）**
- [ ] 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术）
- [ ] 所有异常场景都对应了错误码或业务规则
- [ ] 所有状态流转都有明确的触发条件和目标状态
- [ ] 所有页面都有空状态和错误状态的定义
- [ ] 所有接口都标注了对应的功能点和页面编号
- [ ] 所有表结构都包含审计字段（符合技术全局规范）

### 3.7 上下文预算管理与分片策略

当项目文档数量增加时，单次 AI 会话的上下文窗口可能无法容纳全部上下游文档。本 skill 通过以下策略保证可扩展性：

**上下文预算定义**
- 单次 subagent 调用加载的文档总 token 估算上限：**12k**（约 6k 中文字）
- 主会话加载的文档总 token 估算上限：**20k**
- 超出预算时，必须采用分片策略，禁止一次性硬塞

**分片加载优先级**
1. **当前文档**（全文或变更 diff）
2. **直接上游文档**（只加载被引用的章节/段落，非全文）
3. **直接下游文档**（只加载 upstream-document 表格 + 受影响段落）
4. **间接上下游**（按需加载，通常只在全局规范变更时涉及）

**分层扫描下游（解决 §2.2 的上下文瓶颈）**
- **第一层**：扫描直接下游（如 PRD 的直接下游是交互设计 + 技术方案）
- **第二层**：扫描间接下游（如交互设计的下游是 UI 设计，技术方案的下游是测试报告）
- 每层扫描独立启动 subagent，避免单个子任务过载
- 使用工具：`python references/tools/doc-audit.py <上游文档> --scan-downstream <文档目录>`

**增量审计（解决 §3.3 阶段 3 的上下文瓶颈）**
- 修改模式：只加载变更 diff + 受影响的功能点/页面/章节
- 使用 `--delta` 参数精确限定审计范围：`python doc-audit.py <doc> --type prd --delta F001,F002,P001`
- 脚本自动过滤未变更的编号和章节，减少无关检查

**全文超出预算时的拆分策略**
- 按章节拆分：先审计 §1-§5，再审计 §6-§10
- 按模块拆分：多模块 PRD 时，每次审计一个模块的功能点组
- 跨文档链路检查可独立进行：专门启动一个 subagent 只检查编号引用关系，不加载文档正文
