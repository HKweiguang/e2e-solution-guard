# 测试模板

> 定位：验证功能是否符合需求约定，包括功能测试、异常测试、性能测试、安全测试、兼容性测试。确保每个验收标准都有用例覆盖、每个异常场景都有测试路径、每个风险点都有验证手段。

---

## §1 文档信息

```markdown
# TEST-v{主版本号}-{测试集名}

**覆盖功能点**：{功能编号}, {功能编号}, ...
**覆盖页面**：{页面编号}, {页面编号}, ...
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v{主版本号}-{模块名} | 验收依据 | §3 功能需求、§6 业务规则、§8 验收标准 |
| 交互设计-{页面组名}（单业务端）或 交互设计-{业务端}-{页面组名}（多业务端） | 场景依据 | {页面编号范围}, 页面流程、状态机、异常场景 |
| UI设计-{页面组名}（单业务端）或 UI设计-{业务端}-{设备}-{页面组名}（多业务端） | 视觉依据 | HTML 原型、状态视觉映射、响应式断点 |
| 技术方案-{服务名}（一个测试集可能覆盖多个服务，可有多行） | 实现依据 | §3 数据模型、§4 接口设计、§5 状态机设计、§6 核心流程、§7 异常处理、§8 性能与扩展性、§9 高可用设计、§10 安全设计 |
| 项目 PRD-顶层定义 | 规范继承 | 编码规则、错误码格式 |
```

## §2 章节结构

测试产物以**功能点**或**测试维度**为主线组织。

| 章节 | 说明 |
|------|------|
| §1 功能测试用例 | 正向流程测试（按功能点分组） |
| §2 异常测试用例 | 边界值、异常场景、错误码触发测试 |
| §3 性能测试 | 响应时间、并发、容量、稳定性 |
| §4 安全测试 | 鉴权、越权、输入校验、数据安全 |
| §5 兼容性测试 | 多端、多浏览器、多分辨率、数据兼容 |
| §6 覆盖检查报告 | 验收标准与用例的双向映射 |
| §7 回归测试策略 | 回归范围、自动化计划、准入准出条件 |

## §3 测试不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 产品需求描述 | PRD 已写 | PRD §3 |
| 具体压测脚本/工具配置 | 超出测试策略范围 | 压测平台/CI配置 |
| 安全漏洞扫描详细报告 | 超出测试策略范围 | 安全审计报告 |
| 具体Bug修复方案 | 超出测试范围 | 技术方案/任务单 |
| 代码实现细节 | 超出测试产物范围 | 代码仓库 / 任务单 |
| 性能优化方案 | 技术方案已写 | 技术方案 §8 |

**越界检测**：若测试产物中出现类似"用户需要能够创建工单"的产品需求描述，或"点击XX按钮后跳转到XX页面"的交互描述，或"使用XX颜色表示错误状态"的视觉描述，或"调用XX接口时加Redis缓存"的技术实现描述，应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 功能测试用例

**内容**：按功能点分组的正向流程测试。每个用例必须关联 PRD §3 的一个功能点和 PRD §8 的一条验收标准。**§1 必须覆盖 PRD §3 中所有标记为 P0 的功能点的核心路径，每个 P0 功能点至少有一个对应测试用例。**每个技术方案 §4 接口汇总表中的接口，应在 §1 功能测试或 §2 异常测试中有至少一个对应用例。

**格式建议**：按功能点分组，每组一个表格。列：用例编号 | 关联故事 | 对应验收标准 | 引用规则 | 测试步骤 | 前置条件 | 预期结果 | 优先级。对应功能点通过表外分组声明（如"**功能点**：TICKET-DEFAULT-001"），不在表格中重复列示。

**字段规范**：
- 用例编号：继承项目测试规范或项目 PRD-顶层定义的编码规则，如无定义则询问用户补充
- 对应功能点：PRD §3 的功能编号（如 `TICKET-DEFAULT-001`）
- 关联故事：PRD §2 的用户故事编号（如 `STORY-TICKET-DEFAULT-001`），必填
- 对应验收标准：PRD 验收标准编号（如 `ACC-TICKET-DEFAULT-001`）
- 引用规则：本用例验证的业务规则编号（如 `RULE-TICKET-DEFAULT-001`），如涉及多个用逗号分隔
- 优先级：P0（核心，阻塞上线）/ P1（重要，影响体验）/ P2（一般，边缘场景）

> 填写示例见 `references/examples/test-examples.md` §4.1 功能测试用例

> 常见陷阱见 `references/examples/test-examples.md` §4.1 功能测试用例
---

### 4.2 异常测试用例

**内容**：边界值测试、异常场景测试、错误码触发测试。必须覆盖技术方案 §7 异常处理中定义的所有异常场景和 PRD §7 错误处理中定义的所有错误码。

**格式建议**：按异常类型分组（业务异常/系统异常/第三方异常），每组一个表格。列：用例编号 | 异常场景 | 触发条件 | 测试步骤 | 预期结果 | 对应错误码 | 优先级。异常类型通过表外分组声明（如"**异常类型**：业务异常"），不在表格中重复列示。

**边界**：至少覆盖以下场景：
- 参数非法（空值、超长、格式错误、非法字符）
- 权限不足（未登录、无角色、越权访问他人数据）
- 数据不存在（查询已删除/未创建的资源）
- 网络异常（超时、断网、弱网）
- 并发冲突（重复提交、资源竞争）
- 第三方故障（下游服务不可用）

> 填写示例见 `references/examples/test-examples.md` §4.2 异常测试用例

> 常见陷阱见 `references/examples/test-examples.md` §4.2 异常测试用例
---

### 4.3 性能测试

**内容**：响应时间、并发能力、容量、稳定性验证。基于技术方案 §8 性能与扩展性中定义的性能目标设计测试。**PRD §4 中声明的每个性能指标，必须在 §3 性能测试中有至少一个对应测试项；如有遗漏，须在 §6 覆盖检查报告中标注未覆盖原因。**

**格式建议**：表格。列：测试项 | 性能目标（引用） | 测试场景 | 测试数据量 | 通过标准 | 优先级。

**边界**：不写具体压测工具配置和脚本。只写测试策略、场景和通过标准。

> 填写示例见 `references/examples/test-examples.md` §4.3 性能测试

> 常见陷阱见 `references/examples/test-examples.md` §4.3 性能测试
---

### 4.4 安全测试

**内容**：鉴权、越权、输入校验、数据安全、接口限流验证。**技术方案 §10 中的每个安全措施必须在 §4 安全测试中有至少一个对应验证项；如某项安全措施无法测试，须在 §6 中标注原因。**基于技术方案 §10 安全设计中的安全措施，以及技术方案 §9 高可用设计中的限流策略设计测试。

**格式建议**：表格。列：测试项 | 安全要求（引用） | 测试步骤 | 预期结果 | 优先级。

> 填写示例见 `references/examples/test-examples.md` §4.4 安全测试

> 常见陷阱见 `references/examples/test-examples.md` §4.4 安全测试
---

### 4.5 兼容性测试

**内容**：多端、多浏览器、多分辨率、数据兼容、接口兼容验证。**响应式断点必须与 UI-顶层定义/交互设计 §4.9 声明的断点一致，不得遗漏。**

**格式建议**：表格。列：测试项 | 测试范围 | 测试步骤 | 预期结果 | 优先级。

> 填写示例见 `references/examples/test-examples.md` §4.5 兼容性测试

> 常见陷阱见 `references/examples/test-examples.md` §4.5 兼容性测试
---

### 4.6 覆盖检查报告

**内容**：测试完成后，输出验收标准与用例的双向映射关系。确保 PRD 中定义的每条验收标准都有至少一个用例覆盖。

**格式建议**：表格。列：验收标准编号 | 验收标准描述 | 覆盖用例编号 | 状态 | 未覆盖原因（如有）。

**状态定义**：
- **已覆盖**：有对应用例
- **部分覆盖**：验收标准涉及多个场景，部分有用例
- **未覆盖**：无对应用例

**边界**：未覆盖项必须标注原因（如"需第三方配合，后续补充"、"验收标准已废弃"），并经用户确认。

> 填写示例见 `references/examples/test-examples.md` §4.6 覆盖检查报告

> 常见陷阱见 `references/examples/test-examples.md` §4.6 覆盖检查报告

### 4.7 回归测试策略

**内容**：定义每次变更后的回归范围、自动化计划、准入准出条件。

**格式建议**：分三个子节。

#### 4.7.1 回归范围

**内容**：哪些用例必须在每次代码变更后重新执行。

**格式建议**：表格或列表。列：变更类型 | 回归用例范围 | 说明。

> 填写示例见 `references/examples/test-examples.md` §4.7 回归测试策略

> 填写示例见 `references/examples/test-examples.md` §4.7 回归测试策略

> 填写示例见 `references/examples/test-examples.md` §4.7 回归测试策略

> 常见陷阱见 `references/examples/test-examples.md` §4.7 回归测试策略

#### 4.7.2 自动化计划

**内容**：哪些测试用例已自动化、自动化工具/框架、执行频率。
**格式建议**：表格。列：用例编号 | 自动化状态 | 工具/框架 | 执行触发条件。

#### 4.7.3 准入准出条件

**内容**：测试完成的准入标准（什么条件下可以开始测试）和准出标准（什么条件下可以认为测试完成）。
**格式建议**：列表。准入条件包含：环境就绪、数据准备、代码冻结。准出条件包含：用例执行率、缺陷修复率、阻塞性差异数量。

## §5 格式规范

### 5.1 用例编号

用例编号继承项目 PRD-顶层定义 §6 编码规则。
- 统一格式：`TC-{模块缩写}-{业务端标识}-{序号}`（如 `TC-ORDER-DEFAULT-001`、`TC-ORDER-CONSUMER-001`）
- 异常用例格式：`TC-ERR-{业务端标识}-{序号}`（如 `TC-ERR-DEFAULT-001`、`TC-ERR-CONSUMER-001`）
- 模块缩写和业务端标识与 PRD 功能点编号保持一致。详见项目 PRD-顶层定义 §6 — 跨产物编号体系总览。

### 5.2 优先级定义

- **P0**：核心功能，阻塞上线。对应 PRD 中标注为"必须有"的功能。
- **P1**：重要功能，影响用户体验。对应 PRD 中标注为"应该有"的功能。
- **P2**：边缘场景/加分项。对应 PRD 中标注为"可以有"的功能。

### 5.3 测试数据

测试用例中涉及的具体数据（如用户ID、订单号）应使用描述性占位符（如"{有效用户ID}"），不硬编码真实数据。

### 5.4 错误码映射

异常测试用例中的错误码必须与 PRD 错误处理章节和技术方案 §7 异常处理建立完整双向映射。

### 5.5 字段名一致性

测试用例中涉及的字段名、参数名必须与 PRD §5 数据模型和技术方案 §3 表结构的命名保持一致，禁止自创字段名。

### 5.6 顶层定义映射检查

测试产物完成后，必须检查上游顶层定义在本产物及关联顶层定义中的完整映射。列出"已验证的顶层定义项"和"缺失项"，发现缺失标记为阻塞性差异。

## §6 检查清单

> 按 **6 维度 × 3 层级** 外加 **2 个横向维度** 组织（共 8 个子节）：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试的完整追溯链

### 6.1 结构维度

- [ ] **L1** `[脚本]` §1–§7 章节齐全
- [ ] **L1** `[脚本]` §1 每个用例包含前置条件/测试步骤/预期结果
- [ ] **L1** `[脚本]` §1 每个用例关联 PRD 的一条用户故事编号
- [ ] **L1** `[脚本]` §1 每个用例关联 PRD 的一条验收标准
- [ ] **L1** `[脚本]` §1 每个用例如涉及业务规则，标注引用的 RULE-xxx 编号
- [ ] **L1** `[模型]` §1 覆盖所有 P0 功能点核心路径
- [ ] **L1** `[脚本]` §2 每个异常场景对应 PRD/技术方案错误码
- [ ] **L1** `[脚本]` §3 每个性能测试项有明确通过标准
- [ ] **L2** `[模型]` 测试 §5 兼容性测试用例覆盖 UI-顶层定义/交互设计 §4.9 声明的所有响应式断点
- [ ] **L1** `[脚本]` §6 覆盖检查报告列出所有 PRD 验收标准

- [ ] **L2** `[脚本]` 用例编号格式符合规范，连续无重复
- [ ] **L2** `[脚本]` 测试覆盖的功能点在上游 PRD §3 范围内

（测试产物无独立状态定义，状态相关检查已在交互/UI/技术方案中覆盖）

- [ ] **L2** `[模型]` §2 异常测试覆盖参数非法/权限不足/数据不存在/网络异常/并发冲突/第三方故障

### 6.8 版本与边界

- [ ] **L2** `[脚本]` 版本号与上游 PRD/技术方案版本号一致
