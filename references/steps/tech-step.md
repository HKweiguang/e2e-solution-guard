# 技术方案模板

> 定位：定义产品怎么实现，包括表结构、接口设计、缓存策略、安全设计。

---

## §1 文件头部格式

```markdown
# TECH-v{主版本号}-{服务名/技术域名}

**覆盖功能点**：F{xxx}, F{yyy}, ...
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v{主版本号}-{模块名} | 需求输入 | F{xxx}, F{yyy} |
| 交互设计-{页面组名} | 设计输入 | P{xxx}, P{yyy} |
| 项目技术-顶层定义 | 规范继承 | 技术栈、公共表、接口约定 |
```

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 关联文档 | 相关文档链接 |
| §2 服务概述 | 技术目标、范围、关键决策 |
| §3 技术实现要点 | 决策项、选型、理由、备选方案 |
| §4 依赖关系 | 服务依赖谁、谁依赖服务 |
| §5 数据模型 | 表结构、建表 SQL、设计说明 |
| §6 接口设计 | HTTP API 接口定义 |
| §7 核心流程 | 关键业务流程的技术实现 |
| §8 异常处理 | 服务专属的异常场景及处理 |
| §9 性能与安全 | 性能优化点和安全设计 |
| §10 接口清单 | 全部接口汇总 |

## §3 各章节格式要求

### 3.1 关联文档

**内容**：对应 PRD、全局技术规范、全局架构文档的链接。
**格式**：表格，列：文档 | 链接。

### 3.2 服务概述

**内容**：服务技术目标、覆盖范围、不覆盖范围、关键技术决策摘要。
**格式**：自由文本。

### 3.3 技术实现要点

**内容**：关键技术决策，每个决策包含决策项、选型、理由、备选方案。
**格式**：表格，列：决策编号 | 决策项 | 选型 | 理由 | 备选方案。

### 3.4 依赖关系

**内容**：服务依赖谁、谁依赖服务。
**格式**：表格，列：依赖项 | 类型 | 说明。

### 3.5 数据模型

**内容**：服务的表结构、建表 SQL、设计说明。
**格式**：每个表一个子节，包含表格（字段 | 类型 | 约束 | 对应 PRD 字段 | 说明）+ SQL + 设计说明。
**规范**：
- 所有业务表必须包含审计字段（参见项目技术-顶层定义 2.4）
- 所有字段必须加 COMMENT
**边界**：不写具体业务逻辑。

### 3.6 接口设计

**内容**：HTTP API 接口定义。
**格式**：接口汇总表 + 每个接口的详细定义（URL、功能、请求参数、响应结构、错误码）。
**规范**：
- HTTP 方法继承项目技术-顶层定义 2.5
- 服务接口汇总表中直接写出完整路径
**边界**：不写业务逻辑。

### 3.7 核心流程

**内容**：关键业务流程的技术实现步骤。
**格式**：时序图或流程图。
**边界**：纯 CRUD 单表操作可不画。

### 3.8 异常处理

**内容**：服务专属的异常场景及技术处理方案。
**格式**：表格，列：异常编号 | 场景 | 触发条件 | 技术处理 | 用户提示。

### 3.9 性能与安全

**内容**：性能优化点和安全设计。
**格式**：两个子节，表格形式。
**边界**：不写全局安全规则（已在顶层定义中定义）。

### 3.10 接口清单

**内容**：全部接口汇总。
**格式**：表格，列：接口 | 方法 | 路径 | 对应功能点。

## §4 技术方案不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 产品需求描述 | PRD 已写 | PRD §2 |
| 用户价值说明 | PRD 已写 | PRD §1 |
| 全局架构 | 超出服务范围 | 架构设计文档 |
| 全局 ER 图 | 超出服务范围 | 数据库设计文档 |
| 数据库选型理由 | 顶层决策 | 技术-顶层定义 2.1 |

---

## §5 格式规范

### 5.1 SQL 规范

- 注释不写 `0:xxx 1:xxx`
- 布尔字段注释写语义描述

### 5.2 HTTP 规范

- HTTP 方法继承项目技术-顶层定义 2.5

### 5.3 错误码映射

**必须**与 PRD §7 建立完整双向映射。

### 5.4 缓存设计

表格：缓存数据 | 缓存类型 | Key 格式 | TTL | 失效策略。

## §6 检查清单

### 一致性检验

- [ ] **结构一致性** `[脚本]`：§1–§10 章节齐全
- [ ] **范围一致性** `[模型]`：所有功能点在上游 PRD 功能清单内；所有接口对应的上游页面/功能点已标注
- [ ] **格式一致性** `[脚本]`：表结构字段类型符合项目技术-顶层定义；接口 HTTP 方法和响应结构符合项目技术-顶层定义；错误码与 PRD §7 建立完整双向映射
- [ ] **语义一致性** `[模型]`：表结构字段与上游 PRD 数据模型一致；接口字段与上游 PRD/交互设计定义一致；权限规则与上游 PRD 权限定义一致
- [ ] **内部自洽性** `[脚本]`：§10 接口清单与 §6 接口设计的接口数量、路径、方法完全一致；§5 每张表的字段在 §6 接口的请求/响应参数中有对应；§8 异常处理中的每个异常场景在 §6 至少一个接口的错误码中有映射；缓存设计（§5.4）中的 Key 命名符合技术-顶层定义
- [ ] **跨文档链路一致性** `[模型]`（如已有下游文档）：§5 的表结构与下游数据库迁移脚本一致；§6 的接口变更不会破坏下游测试用例的测试步骤

### 完整性检验

- [ ] §5 每张表包含字段、类型、约束、说明 `[脚本]`

- [ ] §6 每个接口对应 PRD §2 的一个功能点 `[脚本]`
- [ ] §8 的每个异常场景对应 PRD §7 的一个错误码或模块特定异常 `[脚本]`

- [ ] §3.9 及 §5.4 缓存策略覆盖所有需要缓存的数据 `[模型]`
- [ ] §3.9 安全设计覆盖权限、数据安全、性能基线 `[模型]`
- [ ] **文档-代码一致性**（代码实现后）：接口、数据模型、错误码、权限、业务规则五项已通过只读 subagent 审计，审计报告无差异
