# 技术方案模板

> 定位：定义模块/服务怎么实现，以"决策→模型→接口→流程→质量"为主线组织技术设计。确保每个技术决策都有理由、每个数据模型都有出处、每个接口都有边界、每个异常都有处理。

---

## §1 文档信息

```markdown
# TECH-v{主版本号}-{服务名/技术域名}

**覆盖功能点**：{功能编号列表A}（来自 PRD：{模块名A}）；{功能编号列表B}（来自 PRD：{模块名B}）【一个服务可能覆盖多个 PRD 模块】
**覆盖页面**：{页面编号列表A}（来自 PRD/交互设计：{模块名A}）；{页面编号列表B}（来自 PRD/交互设计：{模块名B}）【一个服务可能支撑多个模块的页面】
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v{主版本号}-{模块名}（一个服务可能支撑多个模块，可有多行） | 需求输入 | {功能编号范围}, {页面编号范围} |
| 交互设计-{页面组名}（单业务端）或 交互设计-{业务端}-{页面组名}（多业务端） | 设计输入 | 页面流程、状态机、异常场景 |
| UI设计-{页面组名}（单业务端）或 UI设计-{业务端}-{设备}-{页面组名}（多业务端） | 设计输入 | HTML 原型、组件样式、状态视觉映射 |
| 项目技术-顶层定义 | 规范继承 | 技术栈（§1）、工程结构（§2）、公共模块（§3）、公共表定义（§4）、全局接口约定（§5）、全局安全规范（§6）、基础设施（§7） |
```

## §2 章节结构

技术方案产物以**服务/模块**为主线组织。

| 章节 | 说明 |
|------|------|
| §1 技术决策 | 关键决策项、选型、理由、备选方案（ADR 格式） |
| §2 依赖关系 | 服务依赖谁、谁依赖服务 |
| §3 数据模型 | 表结构、索引、ER 图、容量评估 |
| §4 接口设计 | HTTP API 接口定义（含版本、兼容性） |
| §5 状态机设计 | 业务状态枚举、转换条件、状态图 |
| §6 核心流程 | 关键业务流程的技术实现（时序图/流程图） |
| §7 异常处理 | 异常分类、处理策略、降级方案、错误码映射 |
| §8 性能与扩展性 | QPS/RT 目标、缓存策略、容量规划 |
| §9 高可用设计 | 熔断、降级、限流、重试、超时 |
| §10 安全设计 | 鉴权、数据安全、输入校验、防攻击 |
| §11 监控与日志 | 监控指标、日志规范、告警阈值 |
| §12 灰度与回滚 | 灰度策略、上线顺序、回滚方案 |
| §13 风险评估 | 风险点、待跟进问题、缓解措施 |

## §3 技术方案不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 产品需求描述 | PRD 已写 | PRD §3 |
| 用户价值说明 | PRD 已写 | PRD §1（背景与目标） |
| 交互设计（页面结构、状态机、页面流转） | 行为规则不属于技术实现 | `interaction-step.md` |
| 视觉设计（色彩、字体、间距、组件样式） | 视觉规范不属于技术实现 | `ui-step.md`（HTML 原型） |
| 全局架构 | 超出服务范围 | 项目技术-顶层定义 §2 工程结构（模块划分与依赖关系） |
| 全局 ER 图 | 超出服务范围 | 数据库设计产物 |
| 数据库选型理由 | 顶层决策 | 项目技术-顶层定义 §1 技术栈 |
| 全局安全规则 | 顶层决策 | 项目技术-顶层定义 §6 全局安全规范 |
| 监控平台选型 | 顶层决策 | 项目技术-顶层定义 §7 基础设施 |

**越界检测**：若技术方案中出现类似"用户点击XX按钮后感到满意"的产品价值描述，或"页面跳转到XX"的交互描述，或"使用主色#1890ff"的视觉描述，应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 技术决策

**内容**：本模块的关键技术决策，每个决策包含决策项、最终选型、理由、备选方案及后果。采用 ADR（架构决策记录）格式。

**格式建议**：表格。列：决策编号 | 决策项 | 选型 | 理由 | 备选方案 | 后果。

> 备选方案必须列出至少一个，并说明弃用原因。后果分正面/负面/风险三类。

**格式建议**：表格。列：依赖项 | 类型（被依赖/依赖）| 通信方式 | 说明 | 不可用时的影响 | 降级策略。

---

### 4.3 数据模型

**内容**：模块的表结构、索引设计、ER 图、容量评估。

**格式建议**：每个表一个子节，包含：
1. 表信息（表名、用途、数据量级预估）
2. 字段定义表（字段 | 类型 | 约束 | 对应 PRD 字段 | 说明）
3. 索引设计表（索引名 | 类型 | 字段 | 说明）
4. 建表 SQL（可选）
5. 设计说明

**规范**：
- 所有业务表必须包含项目技术-顶层定义 §4 公共表定义 — 审计字段
- 所有字段必须加 COMMENT
- 字段命名遵循项目技术-顶层定义 §4 公共表定义 — 通用字段命名约定
- 索引命名遵循项目技术-顶层定义 §4 公共表定义 — 索引设计规范

**边界**：不写具体业务逻辑。不写全局 ER 图（超出模块范围）。

> 填写示例见 `references/examples/tech-examples.md` §4.3 数据模型

> 常见陷阱见 `references/examples/tech-examples.md` §4.3 数据模型
---

### 4.4 接口设计

**内容**：HTTP API 接口定义。包含接口汇总表 + 每个接口的详细定义。

**格式建议**：

1. **接口汇总表（⚠️ 强制：必须放在 §4.4 最开头，禁止放在接口详细定义之后）**：

   | 接口编号 | 接口名 | 方法 | 路径 | 对应功能点 | 关联故事 | 引用规则 | 覆盖验收项 | 权限 | 版本 |
   |---------|--------|------|------|-----------|---------|---------|-----------|------|------|

   > **接口编号规范**：继承 PRD 功能编号的模块缩写和业务端标识，追加 `-API-` 后缀。
> - 统一格式：`{模块缩写}-{业务端标识}-API-{序号}`（如 `TICKET-DEFAULT-API-001`、`TICKET-CONSUMER-API-001`）
>
> 顺序号按接口汇总表中的排列顺序从 001 开始递增。模块缩写、业务端标识与 PRD 功能点编号保持一致。详见项目 PRD-顶层定义 §6 — 编号引用契约。
   >
   > **引用列填写规范**：
   > - **对应功能点**：PRD §3 功能编号（如 `TICKET-DEFAULT-001`），必填
   > - **关联故事**：PRD §2 故事编号（如 `STORY-TICKET-DEFAULT-001`），必填
   > - **引用规则**：本接口涉及的业务规则编号（如 `RULE-TICKET-DEFAULT-001`），多个用逗号分隔
   > - **覆盖验收项**：本接口响应验证的验收项编号（如 `ACC-TICKET-DEFAULT-001`），多个用逗号分隔
   > - **权限**：引用 PRD-顶层定义 §2 全局功能范围 — 权限级别定义。填写格式：`公开` / `登录` / `{权限级别标识}`（如 `ADMIN`、`SUPER_ADMIN`）

2. **每个接口的详细定义**，包含：
   - 接口基本信息（URL、方法、功能描述、对应功能点、关联故事）
   - 请求参数（参数名 | 类型 | 必填 | 说明 | 校验规则 | 引用规则编号）
   - 响应结构（成功响应 + 失败响应示例）
   - 错误码（错误码 | 触发场景 | 前端提示）

**规范**：
- HTTP 方法继承项目技术-顶层定义 §5 全局接口约定 — HTTP 方法规范
- 响应结构继承项目技术-顶层定义 §5 全局接口约定 — 统一响应结构
- 接口版本继承项目技术-顶层定义 §5 全局接口约定 — 版本管理与兼容性
- 错误码必须与 PRD §7（错误处理）建立双向映射
- **接口响应字段应与 UI 设计展示字段建立映射，确保前后端数据一致**

**边界**：不写业务逻辑实现细节。

> 填写示例见 `references/examples/tech-examples.md` §4.4 接口设计

> 常见陷阱见 `references/examples/tech-examples.md` §4.4 接口设计

## §5 格式规范

### 5.1 SQL 规范

- 注释不写 `0:xxx 1:xxx`
- 布尔字段注释写语义描述
- 所有字段必须加 COMMENT
- 索引必须在表结构中显性标注

### 5.2 HTTP 规范

- HTTP 方法继承项目技术-顶层定义 §5 全局接口约定 — HTTP 方法规范
- 响应结构继承项目技术-顶层定义 §5 全局接口约定 — 统一响应结构
- 接口版本继承项目技术-顶层定义 §5 全局接口约定 — 版本管理与兼容性

### 5.3 错误码映射

**必须**与 PRD 错误处理章节建立完整双向映射。

### 5.4 缓存设计

缓存策略必须包含：缓存对象、缓存类型、Key 格式、TTL、失效策略、一致性保障。

## §6 检查清单

> 按 **6 维度 × 3 层级** 组织：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试的完整追溯链

- [ ] **L1** `[脚本]` §1–§13 章节齐全
