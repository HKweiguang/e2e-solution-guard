# 技术方案模板

> 定位：定义模块/服务怎么实现，以"决策→模型→接口→流程→质量"为主线组织技术设计。确保每个技术决策都有理由、每个数据模型都有出处、每个接口都有边界、每个异常都有处理。

---

## §1 文档信息

```markdown
# TECH-v{主版本号}-{服务名/技术域名}

**覆盖功能点**：{功能编号列表A}（来自 PRD：{模块名A}）；{功能编号列表B}（来自 PRD：{模块名B}）【一个服务可能覆盖多个 PRD 模块】
**覆盖页面**：{页面编号列表A}（来自 PRD/交互设计：{模块名A}）；{页面编号列表B}（来自 PRD/交互设计：{模块名B}）【一个服务可能支撑多个模块的页面】
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v{主版本号}-{模块名}（一个服务可能支撑多个模块，可有多行） | 需求输入 | {功能编号范围}, {页面编号范围} |
| 交互设计-{页面组名}（单角色）或 交互设计-{角色}-{页面组名}（多角色） | 设计输入 | 页面流程、状态机、异常场景 |
| UI设计-{页面组名}（单角色）或 UI设计-{角色}-{设备}-{页面组名}（多角色） | 设计输入 | HTML 原型、组件样式、状态视觉映射 |
| 项目技术-顶层定义 | 规范继承 | 技术栈（§1）、工程结构（§2）、公共模块（§3）、公共表定义（§4）、全局接口约定（§5）、全局安全规范（§6）、基础设施（§7） |
```

## §2 章节结构

技术方案产物以**服务/模块**为主线组织。

| 章节 | 说明 |
|------|------|
| §1 技术决策 | 关键决策项、选型、理由、备选方案（ADR 格式） |
| §2 依赖关系 | 服务依赖谁、谁依赖服务 |
| §3 数据模型 | 表结构、索引、ER 图、容量评估 |
| §4 接口设计 | HTTP API 接口定义（含版本、兼容性） |
| §5 状态机设计 | 业务状态枚举、转换条件、状态图 |
| §6 核心流程 | 关键业务流程的技术实现（时序图/流程图） |
| §7 异常处理 | 异常分类、处理策略、降级方案、错误码映射 |
| §8 性能与扩展性 | QPS/RT 目标、缓存策略、容量规划 |
| §9 高可用设计 | 熔断、降级、限流、重试、超时 |
| §10 安全设计 | 鉴权、数据安全、输入校验、防攻击 |
| §11 监控与日志 | 监控指标、日志规范、告警阈值 |
| §12 灰度与回滚 | 灰度策略、上线顺序、回滚方案 |
| §13 接口清单 | 全部接口汇总 |
| §14 风险评估 | 风险点、待跟进问题、缓解措施 |

## §3 技术方案不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 产品需求描述 | PRD 已写 | PRD §3 |
| 用户价值说明 | PRD 已写 | PRD §1（背景与目标） |
| 交互设计（页面结构、状态机、页面流转） | 行为规则不属于技术实现 | `interaction-step.md` |
| 视觉设计（色彩、字体、间距、组件样式） | 视觉规范不属于技术实现 | `ui-step.md`（HTML 原型） |
| 全局架构 | 超出服务范围 | 项目技术-顶层定义 §2 工程结构（模块划分与依赖关系） |
| 全局 ER 图 | 超出服务范围 | 数据库设计产物 |
| 数据库选型理由 | 顶层决策 | 项目技术-顶层定义 §1 技术栈 |
| 全局安全规则 | 顶层决策 | 项目技术-顶层定义 §6 全局安全规范 |
| 监控平台选型 | 顶层决策 | 项目技术-顶层定义 §7 基础设施 |

**越界检测**：若技术方案中出现类似"用户点击XX按钮后感到满意"的产品价值描述，或"页面跳转到XX"的交互描述，或"使用主色#1890ff"的视觉描述，应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 技术决策

**内容**：本模块的关键技术决策，每个决策包含决策项、最终选型、理由、备选方案及后果。采用 ADR（架构决策记录）格式。

**格式建议**：表格。列：决策编号 | 决策项 | 选型 | 理由 | 备选方案 | 后果。

> 备选方案必须列出至少一个，并说明弃用原因。后果分正面/负面/风险三类。

**填写示例**：

| 决策编号 | 决策项 | 选型 | 理由 | 备选方案 | 后果 |
|---------|--------|------|------|---------|------|
| TDR-001 | 工单数据存储 | MySQL + 分表 | 关系型数据，事务强一致；预计 1 年数据量 1000 万，按用户 ID 分 64 表 | MongoDB（文档灵活但事务弱）| 正面：查询性能稳定；负面：分表增加运维复杂度；风险：分片键选择不当导致数据倾斜 |
| TDR-002 | 前端框架/组件库 | Vue 3 + Element Plus | 团队熟悉 Vue，Element Plus 组件丰富；与 UI 设计状态类（.loading/.disabled/.empty）有对应实现 | React + Ant Design（学习成本高）| 正面：开发效率高；负面：组件定制需覆盖默认样式；风险：组件库升级导致样式断裂 |
| TDR-003 | 工单状态变更通知 | RabbitMQ | 可靠投递，支持延迟队列；团队熟悉 | Kafka（吞吐高但延迟队列需二次开发）| 正面：技术栈统一；负面：单点瓶颈需集群化；风险：消息堆积影响延迟 |

**常见陷阱**：
- ❌ 只写选型不写理由——评审时无法判断合理性
- ❌ 不写备选方案——显得决策过程不严谨
- ❌ 后果只写正面不写负面/风险——导致上线后问题无人预料

---

### 4.2 依赖关系

**内容**：服务依赖谁、谁依赖服务。

**格式建议**：表格。列：依赖项 | 类型（被依赖/依赖）| 通信方式 | 说明 | 不可用时的影响 | 降级策略。

**填写示例**：

| 依赖项 | 类型 | 通信方式 | 说明 | 不可用时的影响 | 降级策略 |
|--------|------|---------|------|--------------|---------|
| MySQL | 被依赖 | JDBC | 工单数据持久化 | 服务不可用 | 无（核心依赖） |
| Redis | 被依赖 | Redis 客户端 | 热点数据缓存 | 性能下降 | 直接查数据库 |
| 用户服务 | 依赖 | HTTP | 查询用户信息 | 无法创建工单 | 返回"系统繁忙，请稍后重试" |
| 通知服务 | 依赖 | RabbitMQ | 发送状态变更通知 | 通知延迟 | 写入本地队列，服务恢复后补偿发送 |

**常见陷阱**：
- ❌ 只列出依赖，不说明不可用时的影响——导致故障时无预案
- ❌ 缺少降级策略——依赖故障时服务雪崩

---

### 4.3 数据模型

**内容**：模块的表结构、索引设计、ER 图、容量评估。

**格式建议**：每个表一个子节，包含：
1. 表信息（表名、用途、数据量级预估）
2. 字段定义表（字段 | 类型 | 约束 | 对应 PRD 字段 | 说明）
3. 索引设计表（索引名 | 类型 | 字段 | 说明）
4. 建表 SQL（可选）
5. 设计说明

**规范**：
- 所有业务表必须包含项目技术-顶层定义 §4.2 审计字段
- 所有字段必须加 COMMENT
- 字段命名遵循项目技术-顶层定义 §4.3 通用字段命名约定
- 索引命名遵循项目技术-顶层定义 §4.4 索引设计规范

**边界**：不写具体业务逻辑。不写全局 ER 图（超出模块范围）。

**填写示例**（工单表）：

**表名**：`ticket`（工单表）
**用途**：存储工单基本信息
**数据量级**：预计 1 年 1000 万条，按 `user_id` 分 64 表

**字段定义**：

| 字段 | 类型 | 约束 | 对应 PRD 字段 | 说明 |
|------|------|------|--------------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | — | 主键 |
| `ticket_no` | VARCHAR(32) | UK, NOT NULL | 工单编号 | 业务唯一编号，如 `T202401150001` |
| `user_id` | BIGINT | NOT NULL, FK | 客户 ID | 关联 sys_user 表 |
| `title` | VARCHAR(200) | NOT NULL | 工单标题 | — |
| `status` | TINYINT | NOT NULL, INDEX | 工单状态 | 1=待处理, 2=处理中, 3=已解决, 4=已关闭 |
| `priority` | TINYINT | NOT NULL | 优先级 | 1=紧急, 2=高, 3=中, 4=低 |
| `create_time` | DATETIME | NOT NULL | — | 审计字段（继承顶层定义） |
| `update_time` | DATETIME | NOT NULL | — | 审计字段（继承顶层定义） |

**索引设计**：

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|
| PRIMARY | 主键 | `id` | — |
| `uk_ticket_no` | 唯一 | `ticket_no` | 业务唯一编号 |
| `idx_user_status` | 普通 | `user_id`, `status` | 查询某用户的待处理工单 |
| `idx_create_time` | 普通 | `create_time` | 按时间范围查询 |

**设计说明**：
- 分片键选择 `user_id`（哈希），因为查询场景以用户维度为主
- `status` 字段使用 TINYINT 而非 ENUM，便于扩展状态值
- 预计单表数据量 15 万条（1000 万 / 64），InnoDB 单表性能良好

**常见陷阱**：
- ❌ 字段类型选择不合理（如用 VARCHAR 存日期）——导致查询性能差
- ❌ 缺少索引或索引设计不当——导致慢查询
- ❌ 分片键选择不当——导致数据倾斜
- ❌ 缺少审计字段——导致数据操作无法追溯

---

### 4.4 接口设计

**内容**：HTTP API 接口定义。包含接口汇总表 + 每个接口的详细定义。

**格式建议**：每个接口包含：
1. 接口基本信息（URL、方法、功能描述、对应功能点）
2. 请求参数（参数名 | 类型 | 必填 | 说明 | 校验规则）
3. 响应结构（成功响应 + 失败响应示例）
4. 错误码（错误码 | 触发场景 | 前端提示）

**规范**：
- HTTP 方法继承项目技术-顶层定义 §5.1 HTTP 方法规范
- 响应结构继承项目技术-顶层定义 §5.2 统一响应结构
- 接口版本继承项目技术-顶层定义 §5.5 版本管理与兼容性
- 错误码必须与 PRD §7（错误处理）建立双向映射

**边界**：不写业务逻辑实现细节。

**填写示例**（创建工单接口）：

**基本信息**：

| 项目 | 值 |
|------|-----|
| URL | `POST /api/v1/tickets` |
| 功能 | 创建工单 |
| 对应功能点 | TICKET-002 |
| 权限 | 登录 |

**请求参数**：

| 参数 | 类型 | 必填 | 说明 | 校验规则 |
|------|------|------|------|---------|
| `title` | String | 是 | 工单标题 | 长度 1-200 |
| `description` | String | 否 | 工单描述 | 长度 0-2000 |
| `priority` | Integer | 是 | 优先级 | 枚举：1/2/3/4 |
| `attachments` | Array | 否 | 附件列表 | 每项为文件 URL |

**成功响应**：

```json
{
  "code": "SUCCESS",
  "message": "工单创建成功",
  "data": {
    "ticketId": 12345,
    "ticketNo": "T202401150001",
    "status": 1,
    "createTime": "2024-01-15T10:30:00+08:00"
  },
  "traceId": "abc123",
  "timestamp": 1707985800000
}
```

**错误码**：

| 错误码 | 触发场景 | 前端提示 |
|--------|---------|---------|
| `PARAM_ERROR` | 标题为空或超过 200 字 | "标题长度需在 1-200 字之间" |
| `PRIORITY_INVALID` | 优先级不在枚举范围内 | "请选择有效的优先级" |
| `RATE_LIMIT` | 用户 1 分钟内创建超过 10 个工单 | "操作过于频繁，请稍后再试" |

**常见陷阱**：
- ❌ 接口 URL 风格不统一（有的用 `/api/tickets`，有的用 `/ticket/create`）——前端维护困难
- ❌ 请求参数缺少校验规则——导致非法数据进入系统
- ❌ 错误码与 PRD 不一致——导致前端提示与需求不符
- ❌ 缺少接口版本号——破坏性变更无法平滑过渡
- ❌ 接口文档与 PRD 功能点缺少映射覆盖——某些功能点无对应接口，或接口未标注对应的功能编号

### 4.5 状态机设计

**内容**：定义模块涉及的业务状态枚举、状态转换条件和状态图。PRD 中定义的业务状态机在此落地为技术实现。

**格式建议**：
1. 状态枚举表（状态值 | 状态名 | 说明）
2. 状态转换表（当前状态 | 事件/操作 | 目标状态 | 触发条件 | 前置校验）
3. 状态图（文本描述或 Mermaid 图）

**填写示例**（工单状态机）：

**状态枚举**：

| 状态值 | 状态名 | 说明 |
|--------|--------|------|
| 1 | `PENDING` | 待处理 |
| 2 | `PROCESSING` | 处理中 |
| 3 | `RESOLVED` | 已解决 |
| 4 | `CLOSED` | 已关闭 |

**状态转换**：

| 当前状态 | 事件/操作 | 目标状态 | 触发条件 | 前置校验 |
|---------|----------|---------|---------|---------|
| `PENDING` | 分配处理人 | `PROCESSING` | 客服领取工单 | 当前无处理人 |
| `PENDING` | 超时自动关闭 | `CLOSED` | 超过 7 天未处理 | — |
| `PROCESSING` | 标记已解决 | `RESOLVED` | 客服提交解决方案 | 必须填写解决说明 |
| `PROCESSING` | 转交他人 | `PENDING` | 当前处理人无法解决 | — |
| `RESOLVED` | 客户确认 | `CLOSED` | 客户点击"问题已解决" | — |
| `RESOLVED` | 客户拒绝 | `PROCESSING` | 客户点击"问题未解决" | 24 小时内可拒绝 |
| `CLOSED` | — | — | 终态，不可转换 | — |

**状态图**：

```
PENDING ──分配──> PROCESSING ──解决──> RESOLVED ──确认──> CLOSED
   │                    │                    │
   └──超时──> CLOSED    └──转交──> PENDING   └──拒绝──> PROCESSING
```

**常见陷阱**：
- ❌ 状态转换条件不明确——开发实现时随意添加转换路径
- ❌ 缺少终态定义——导致状态循环无法结束
- ❌ 状态值与 PRD 定义不一致——导致业务逻辑错误
- ❌ 缺少前置校验——导致非法状态转换

### 4.6 核心流程

**内容**：关键业务流程的技术实现步骤。纯 CRUD 单表操作可不画。

**格式建议**：时序图或流程图。标注：参与者、调用顺序、数据传递、异常分支。

**填写示例**（创建工单流程）：

```
用户 → 前端: 填写工单信息，点击提交
前端 → 后端: POST /api/v1/tickets (title, description, priority)
后端 → 校验服务: 校验参数合法性
校验服务 → 后端: 校验通过 / 参数错误（返回 PARAM_ERROR）
后端 → 数据库: INSERT ticket 表
数据库 → 后端: 返回 ticket_id
后端 → 消息队列: 发布 "ticket.created" 事件
后端 → 前端: 返回 SUCCESS + ticketId
消息队列 → 通知服务: 消费事件，发送通知
通知服务 → 用户: 推送/短信通知
```

**常见陷阱**：
- ❌ 流程图只有正常路径，没有异常分支——开发遗漏异常处理
- ❌ 流程与交互设计定义的流程不一致——导致实现与需求不符
- ❌ 缺少数据一致性说明——如分布式事务如何保证

---

### 4.7 异常处理

**内容**：模块专属的异常场景及技术处理方案。异常分类：业务异常、系统异常、第三方依赖异常。

**格式建议**：表格。列：异常编号 | 异常类型 | 场景 | 触发条件 | 技术处理 | 用户提示 | 错误码。

**填写示例**：

| 异常编号 | 异常类型 | 场景 | 触发条件 | 技术处理 | 用户提示 | 错误码 |
|---------|---------|------|---------|---------|---------|--------|
| EX-TICKET-001 | 业务异常 | 参数校验失败 | 标题为空 | 返回 PARAM_ERROR，不记录 ERROR 日志 | "标题不能为空" | `PARAM_ERROR` |
| EX-TICKET-002 | 业务异常 | 工单不存在 | 查询已删除工单 | 返回 NOT_FOUND | "工单不存在或已删除" | `NOT_FOUND` |
| EX-TICKET-003 | 系统异常 | 数据库连接超时 | MySQL 连接池耗尽 | 返回 SYSTEM_ERROR，记录 ERROR 日志，触发告警 | "系统繁忙，请稍后重试" | `SYSTEM_ERROR` |
| EX-TICKET-004 | 第三方异常 | 通知服务不可用 | RabbitMQ 宕机 | 写入本地补偿表，定时重试 | 无（异步通知，用户无感知）| — |

> 业务异常返回 HTTP 200 + 业务错误码（不触发熔断）；系统异常返回 HTTP 500 + 通用错误码（触发熔断告警）。

**常见陷阱**：
- ❌ 异常分类不清——导致业务异常触发熔断，或系统异常被静默处理
- ❌ 用户提示过于技术化（如"数据库连接超时"）——用户无法理解
- ❌ 缺少降级策略——第三方故障时服务雪崩

---

### 4.8 性能与扩展性

**内容**：性能目标、缓存策略、容量规划。

**格式建议**：
1. 性能目标表（指标 | 目标值 | 说明）
2. 缓存策略表（缓存数据 | 缓存类型 | Key 格式 | TTL | 失效策略）
3. 容量评估（当前量级 | 预测量级 | 扩容触发条件）

**填写示例**：

**性能目标**：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 创建工单接口 RT | < 200ms | P99 < 500ms |
| 工单列表查询 RT | < 100ms | 含分页，单页 20 条 |
| 并发创建工单 | 100 QPS | 单节点 |

**缓存策略**：

| 缓存数据 | 类型 | Key 格式 | TTL | 失效策略 | 一致性保障 |
|---------|------|---------|-----|---------|---------|
| 工单详情 | Redis Hash | `ticket:detail:{ticketId}` | 1 小时 | 更新时删除 | 先删缓存再写库，延迟双删 |
| 用户工单列表 | Redis ZSet | `ticket:list:{userId}` | 30 分钟 | 创建/更新时删除 | 先删缓存再写库 |
| 工单状态统计 | Redis String | `ticket:stat:{status}` | 5 分钟 | 定时刷新 | 定时任务刷新，允许短暂不一致 |

**容量评估**：

| 维度 | 当前 | 6 个月预测 | 扩容触发条件 |
|------|------|-----------|-------------|
| 数据量 | 0 | 500 万 | 单表 > 15 万条 |
| QPS | 0 | 50（峰值 200）| CPU > 70% |
| 存储 | 0 | 50GB | 磁盘 > 80% |

**常见陷阱**：
- ❌ 性能目标不切实际——如要求所有接口 < 10ms
- ❌ 缓存策略缺少失效机制——导致数据不一致
- ❌ 容量评估缺失——导致上线后容量不足

---

### 4.9 高可用设计

**内容**：熔断、降级、限流、重试、超时策略。

**格式建议**：表格。列：维度 | 策略 | 阈值 | 触发后的行为。

**填写示例**：

| 维度 | 策略 | 阈值 | 触发后的行为 |
|------|------|------|-------------|
| 熔断 | 基于错误率 | 10 秒内错误率 > 50% | 熔断 30 秒，期间返回降级数据 |
| 降级 | 返回缓存数据 | — | 数据库故障时返回 Redis 缓存（可能非最新） |
| 限流 | 基于用户 ID | 100 次/分钟 | 返回 `429 Too Many Requests` |
| 重试 | 指数退避 | 最多 3 次，间隔 1s/2s/4s | 仍失败则标记为系统异常 |
| 超时 | 接口调用 | 3 秒 | 超过则返回 SYSTEM_ERROR |

**常见陷阱**：
- ❌ 重试次数过多——导致故障放大
- ❌ 超时时间设置过长——导致线程池耗尽
- ❌ 缺少降级策略——依赖故障时服务完全不可用

---

### 4.10 安全设计

**内容**：模块级的鉴权、数据安全、输入校验、防攻击措施。

**格式建议**：列表或表格。

**填写示例**：

| 维度 | 措施 |
|------|------|
| 接口鉴权 | 所有接口校验 JWT Token，操作他人数据时校验 user_id 匹配 |
| 输入校验 | 标题长度 1-200、描述长度 0-2000、优先级枚举 1-4 |
| SQL 注入防护 | 使用预编译语句，禁止字符串拼接 SQL |
| XSS 防护 | 输入过滤 + 输出转义 |
| 敏感数据 | 工单中的手机号/邮箱在展示时脱敏 |
| 越权校验 | 查询工单时校验当前用户是否为工单创建者或管理员 |

**常见陷阱**：
- ❌ 缺少输入校验——导致非法数据进入系统
- ❌ 越权校验遗漏——导致用户 A 可查看/操作用户 B 的数据
- ❌ 敏感数据明文传输——导致信息泄露

---

### 4.11 监控与日志

**内容**：模块级的监控指标、日志规范、告警阈值。

**格式建议**：表格。列：监控维度 | 指标 | 告警阈值 | 通知方式 | 处理人。

**填写示例**：

| 监控维度 | 指标 | 告警阈值 | 通知方式 | 处理人 |
|---------|------|---------|---------|--------|
| 业务 | 工单创建成功率 | < 99% | 钉钉 | 后端负责人 |
| 业务 | 工单平均处理时长 | > 24h | 钉钉 | 运营负责人 |
| 性能 | 创建工单接口 P99 | > 500ms | 钉钉 | 后端负责人 |
| 错误 | 系统异常数 | > 10/分钟 | 钉钉+电话 | 后端负责人 |
| 资源 | 数据库连接池使用率 | > 80% | 钉钉 | DBA |

**日志规范**：

| 日志类型 | 内容 | 级别 |
|---------|------|------|
| 操作日志 | 用户创建/修改/删除工单 | INFO |
| 异常日志 | 系统异常、第三方调用失败 | ERROR |
| 性能日志 | 接口耗时 > 200ms | WARN |

**常见陷阱**：
- ❌ 监控指标与业务目标脱节——监控了 CPU 但没监控核心业务成功率
- ❌ 告警阈值设置不当——阈值太低导致告警轰炸，阈值太高导致问题发现滞后
- ❌ 日志中输出敏感信息——导致安全合规问题

---

### 4.12 灰度与回滚

**内容**：灰度发布策略、上线顺序、回滚方案。

**格式建议**：列表或表格。

**填写示例**：

| 项目 | 策略 |
|------|------|
| 灰度方式 | 按用户 ID 哈希灰度（先 1% → 10% → 50% → 100%） |
| 灰度观察期 | 每个阶段至少观察 30 分钟，监控核心业务指标 |
| 上线顺序 | ① 数据库迁移（兼容变更）② 后端服务 ③ 前端页面 |
| 回滚条件 | 错误率 > 1% 或 P99 > 1s 或用户投诉 > 10 条/小时 |
| 回滚方案 | 前端回滚到上一版本，后端回滚到上一镜像，数据库变更需提前设计回滚 SQL |
| 数据迁移 | 如需数据迁移，迁移脚本需可重复执行（幂等），且不影响线上服务 |

**常见陷阱**：
- ❌ 缺少灰度策略——全量上线导致故障影响面大
- ❌ 回滚方案不可行——如数据库变更不可逆导致无法回滚
- ❌ 上线顺序错误——如先上前端后上后端，导致接口不匹配

### 4.13 接口清单

**内容**：全部接口汇总。必须与 §4 接口设计中的接口一一对应。

**格式建议**：表格。列：序号 | 接口名 | 方法 | 路径 | 对应功能点 | 权限 | 版本。

**填写示例**：

| 序号 | 接口名 | 方法 | 路径 | 对应功能点 | 权限 | 版本 |
|------|--------|------|------|-----------|------|------|
| 1 | 创建工单 | POST | `/api/v1/tickets` | TICKET-002 | 登录 | v1 |
| 2 | 查询工单列表 | GET | `/api/v1/tickets` | TICKET-003 | 登录 | v1 |
| 3 | 查询工单详情 | GET | `/api/v1/tickets/{id}` | TICKET-001 | 登录 | v1 |
| 4 | 更新工单状态 | PATCH | `/api/v1/tickets/{id}/status` | TICKET-002 | 登录 | v1 |
| 5 | 删除工单 | DELETE | `/api/v1/tickets/{id}` | TICKET-002 | 管理员 | v1 |

**常见陷阱**：
- ❌ 接口清单与 §4 接口设计的接口数量/路径不一致——导致产物不一致
- ❌ 遗漏接口——导致功能遗漏

### 4.14 风险评估

**内容**：风险点、待跟进问题、缓解措施。

**格式建议**：表格。列：风险编号 | 风险描述 | 影响程度 | 发生概率 | 缓解措施 | 责任人 | 状态。

**填写示例**：

| 风险编号 | 风险描述 | 影响程度 | 发生概率 | 缓解措施 | 责任人 | 状态 |
|---------|---------|---------|---------|---------|--------|------|
| RISK-001 | 工单数据量增长超预期，单表性能下降 | 高 | 中 | 提前设计分表策略，监控单表数据量 | 后端负责人 | 已缓解 |
| RISK-002 | RabbitMQ 消息堆积导致通知延迟 | 中 | 中 | 设置消息 TTL，消费失败时转入死信队列 | 后端负责人 | 待跟进 |
| RISK-003 | 第三方用户服务不可用导致无法创建工单 | 高 | 低 | 设计降级策略（允许创建但延迟校验用户） | 后端负责人 | 待跟进 |

**常见陷阱**：
- ❌ 风险评估流于形式（"可能有风险"）——不具体、不可执行
- ❌ 缺少责任人——导致风险无人跟进
- ❌ 风险状态不更新——导致已缓解的风险仍被列为高风险

## §5 格式规范

### 5.1 SQL 规范

- 注释不写 `0:xxx 1:xxx`
- 布尔字段注释写语义描述
- 所有字段必须加 COMMENT
- 索引必须在表结构中显性标注

### 5.2 HTTP 规范

- HTTP 方法继承项目技术-顶层定义 §5.1 HTTP 方法规范
- 响应结构继承项目技术-顶层定义 §5.2 统一响应结构
- 接口版本继承项目技术-顶层定义 §5.5 版本管理与兼容性

### 5.3 错误码映射

**必须**与 PRD 错误处理章节建立完整双向映射。

### 5.4 缓存设计

缓存策略必须包含：缓存对象、缓存类型、Key 格式、TTL、失效策略、一致性保障。
