# UI 设计模板

> 定位：模块级视觉实现。
>
> **核心思想**：交互设计是骨架，UI 顶层定义是血肉。UI 设计稿就是把这两者拼起来，展示最终长什么样。
>
> **边界**：只写视觉（颜色、尺寸、间距、阴影、圆角、字体），不写行为（触发条件、交互流程、状态机）。

---

## §1 文档信息

```markdown
# UI-v{主版本号}-{设备}-{页面组}

**覆盖功能点**：{功能编号}, {功能编号}, ...
**覆盖页面**：{页面编号}, {页面编号}, ...
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
```

**上游文档**（实际产物中按上方格式写入表格）：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 交互设计-{页面组}（单业务端）或 交互设计-{业务端}-{页面组}（多业务端） | 结构输入 | SVG 线框图 → HTML 结构；状态矩阵 → 状态类/伪类名称映射（行为状态定义，不含视觉值） |
> 注：单业务端项目只有一个交互设计文件；多业务端项目交互设计按业务端分文件。UI 按业务端+设备分文件，一个 HTML 只展示一种设备的界面。
| 项目 UI-顶层定义 | 规范继承 | Token → CSS 变量；组件库 → HTML 元素；平台视觉规范（§12，多端项目适用）→ 平台状态差异、组件尺寸、安全区 |

### 设计系统引用

```css
:root {
  /* 来自项目 UI-顶层定义的 CSS 变量 */
}
```

## §2 HTML 原型

产物文件：`ui-{业务端}-{设备}-{页面组}.html`（统一格式，单业务端时业务端为 `DEFAULT`）
> 示例：`ui-DEFAULT-web-首页.html`（单业务端）、`ui-CONSUMER-web-首页.html`（多业务端）

```html
<!DOCTYPE html>
<html>
<head>
  <!-- upstream: 交互设计-{页面组}.md, 项目 UI-顶层定义.md -->
  <style>
    :root {
      /* CSS 变量来自项目 UI-顶层定义，如 --color-primary、--color-text-primary 等 */
    }
    .btn { background: var(--primary); }
    .btn:hover { opacity: 0.8; }
    /* 状态样式: hover / active / focus-visible / disabled / loading */
    /* 响应式: @media */
  </style>
</head>
<body>
  <!--
  状态映射表（关节层）：
  | 交互状态 | CSS 类/伪类 | Token 变量 | 技术方案引用 |
  | 默认态   | （无）       | —          | Button 组件 default 样式 |
  | 悬停态   | :hover       | --primary-hover | Button 组件 hover 样式 |
  | 按下态   | :active      | --primary-active | Button 组件 active 样式 |
  | 禁用态   | .disabled    | --disabled | Button 组件 disabled 属性 |
  | 加载态   | .loading     | --primary + opacity: 0.7 | Button 组件 loading 状态 |
  -->
  <section id="page-001"> <!-- 静态假数据，无 JS --> </section>
  <section id="page-002"> <!-- 静态假数据，无 JS --> </section>
</body>
</html>
```

原则：
- 一个页面组一个文件
- CSS 变量全部来自项目 UI-顶层定义
- 状态用伪类实现
- 静态原型，无 JS 业务逻辑

## §3 与交互设计对应

### 3.1 页面映射

| UI 页面 | 交互页面编号 | 对应 `<section id>` |
|---------|-------------|-------------------|
| 页面1 | PAGE-DEFAULT-WEB-PROFILE-001 | page-web-profile-001 |
| 页面2 | PAGE-DEFAULT-WEB-ORDER-001 | page-web-order-001 |

### 3.2 状态映射表（关节层）

> **核心作用**：显式声明"交互设计定义的状态名称 → UI 实现的 CSS 类/伪类 → UI-顶层定义 Token → 技术方案引用"的完整映射链。禁止让映射关系隐式散落在 HTML 代码中。

**格式**：每个页面一张状态映射表。列：**交互状态 | CSS 类/伪类 | Token 变量 | 技术方案引用**。

**填写示例**（PAGE-DEFAULT-WEB-PROFILE-001 个人资料页）：

| 交互状态 | CSS 类/伪类 | Token 变量 | 技术方案引用 |
|---------|------------|-----------|-------------|
| 默认态 | （无） | — | Button 组件 default 样式 |
| 悬停态 | `:hover` | `--color-primary-hover` | Button 组件 hover 样式 |
| 按下态 | `:active` | `--color-primary-active` | Button 组件 active 样式 |
| 聚焦态 | `:focus-visible` | `--focus-ring` | Button 组件 focus 样式 |
| 禁用态 | `.disabled` / `:disabled` | `--color-disabled` | Button 组件 disabled 属性 |
| 加载态 | `.loading` | `--color-primary` + `opacity: 0.7` | Button 组件 loading 状态 |
| 空状态 | `.empty` | `--color-text-secondary` | Empty 组件 |
| 错误状态 | `.error` | `--color-error` | Form 组件 error 样式 |
| 成功状态 | `.success` | `--color-success` | Toast 组件 success 样式 |
| 骨架态 | `.skeleton` | `--color-skeleton` | Skeleton 组件 |

**视觉差异判定标准**（供审计参考）：
- 禁用态：必须与默认态有可感知的视觉差异（如透明度变化、置灰、或降低对比度），禁止仅改变光标样式
- 错误态：必须使用语义错误色 Token（如 `--color-error`），配合边框变色、背景底色变化或图标提示，与默认态的差异必须可被普通用户识别
- 加载态：禁止完全空白或仅显示"加载中"文字而无布局占位，必须保留上下文结构（如骨架屏、旋转指示器在组件内）
- 空状态/错误状态/成功状态：必须有明确的视觉标识（图标、插画、颜色变化），与默认态的差异 ≥ 3:1 对比度

**常见陷阱**：
- ❌ 映射表只列状态名，不写 CSS 类名——开发人员无法确定用 `:hover` 还是 `.hover`
- ❌ 映射表不写 Token 变量——无法追溯视觉规范来源
- ❌ 映射表不写技术方案引用——下游技术方案无法对齐组件选型
- ❌ 交互设计更新了状态名（如"按下态"改为"激活态"），映射表未同步更新

> 基础状态：default（默认态）、hover（悬停态）、active（按下态）、focus-visible（聚焦态）、disabled（禁用态）、loading（加载态）
> 异步状态：empty（空状态）、error（错误状态）、success（成功状态）、skeleton（骨架态）

## §4 UI 设计不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 交互行为（触发条件、状态流转、页面跳转） | 行为规则不属于视觉规范 | `interaction-step.md` |
| 产品需求（功能点、业务规则、验收标准） | 需求定义不属于视觉实现 | `prd-step.md` |
| 技术实现（架构、接口、数据模型） | 实现方案不属于视觉设计 | `tech-step.md` |
| 测试策略（测试用例、测试范围） | 验证方法不属于视觉设计 | `test-step.md` |

**越界检测**：若 UI 设计中出现类似"点击后跳转到XX页面"的交互描述，或"调用XX接口"的技术描述，或"功能点F001的验收标准是..."的需求描述，应标记为**越界内容**，提示用户移至对应步骤。

## §5 检查清单

> 按 **6 维度 × 3 层级** 组织：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试的完整追溯链

### 5.1 结构维度

- [ ] **L1** `[脚本]` HTML 包含完整 DOCTYPE/html/head/body
- [ ] **L1** `[脚本]` HTML 声明 upstream 注释
- [ ] **L1** `[脚本]` 使用 CSS 变量（var(--*)），无硬编码色值
- [ ] **L1** `[脚本]` 状态映射表覆盖 10 种状态（6 基础 + 4 异步），每行 4 列
- [ ] **L1** `[脚本]` 状态映射表中的 CSS 类名在 HTML 中存在
- [ ] **L1** `[脚本]` 无 JS 业务逻辑/数据请求
- [ ] **L2** `[脚本]` 每个 `<section id>` 对应交互设计中的一个页面编号
- [ ] **L2** `[脚本]` CSS 变量命名与 UI-顶层定义 Token 命名一致
- [ ] **L2** `[脚本]` 状态映射表中的 Token 变量在 UI-顶层定义中声明

### 5.2 数据维度

- [ ] **L1** `[脚本]` 状态映射表每行包含交互状态/CSS类/Token/技术方案引用
- [ ] **L2** `[模型]` UI 展示的字段名与 PRD §5/技术方案 §3 一致
- [ ] **L2** `[模型]` 各状态视觉差异足够明显

### 5.3 状态维度

- [ ] **L1** `[脚本]` HTML 包含 9 种非默认状态样式（不含默认态；:hover/:active/:focus-visible/.disabled/.loading/.empty/.error/.success/.skeleton），对应 10 种交互状态中的非默认态
- [ ] **L2** `[脚本]` 交互设计定义的组件状态在 HTML 中有对应 CSS 类
- [ ] **L2** `[模型]` 禁用态有明确视觉区分
- [ ] **L2** `[模型]` 加载态保留上下文
- [ ] **L3** `[脚本]` 交互设计 §4.4 每个异步状态在 UI 中有对应状态类

### 5.4 错误维度

- [ ] **L1** `[模型]` 错误状态样式与默认态有明显区别
- [ ] **L2** `[脚本]` 错误状态类名在技术方案前端实现中有对应引用

### 5.5 性能维度

- [ ] **L1** `[脚本]` 如项目有响应式要求，HTML 包含 @media 查询
- [ ] **L2** `[脚本]` 响应式断点与 UI-顶层定义一致
- [ ] **L2** `[模型]` 响应式结构变更与交互设计 §4.9 一致

### 5.6 安全维度

- [ ] **L2** `[模型]` 敏感数据脱敏样式符合 PRD/技术方案要求

### 5.7 端到端链路

- [ ] **L3** `[脚本]` 交互设计 §4.3 的每个状态在 UI 中有对应 CSS 实现
- [ ] **L3** `[脚本]` 交互设计 §4.9 的响应式结构变更在 UI 中有 @media 实现
- [ ] **L3** `[模型]` 技术方案前端组件选型与 UI 状态类一致

### 5.8 版本与边界

- [ ] **L2** `[脚本]` 版本号与上游交互设计版本号一致
- [ ] **L2** `[模型]` 检查上游顶层定义在本产物及关联顶层定义中的完整映射，发现缺失标记为阻塞性差异
- [ ] **L1** `[模型]` 产物中无不属于 UI 范畴的行为/需求/技术描述
- [ ] **L2** `[模型]` 测试 §5 兼容性测试用例覆盖 UI-顶层定义/交互设计 §4.9 声明的所有响应式断点
