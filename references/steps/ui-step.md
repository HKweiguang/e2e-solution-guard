# UI 设计稿模板

> 定位：定义产品长什么样，以"页面→组件→状态→标注"为主线组织视觉设计。确保每个页面都有完整的组件覆盖、清晰的状态展示和精确的 Token 化标注，使设计稿可直接指导开发实现。

---

## §1 文档信息

```markdown

**覆盖功能点**：{功能编号}, {功能编号}, ...
**覆盖页面**：{页面编号}, {页面编号}, ...
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 交互设计-{页面组名} | 设计输入 | {页面编号范围}, 信息架构、组件交互、状态机、页面流程 |
| 项目 UI-顶层定义 | 规范继承 | 设计原则（§1）、设计令牌体系（§2）、色彩系统（§3）、字体系统（§4）、间距与布局系统（§5）、形状系统（§6）、阴影与海拔（§7）、图标规范（§9）、主题与暗黑模式（§10）、组件库（§11） |
```

## §2 章节结构

UI 设计稿文档以**页面**为主线组织。每个页面为一个一级章节。

**文档级章节**（整个文档只出现一次）：

| 章节 | 说明 |
|------|------|
| 设计系统引用 | 声明继承的顶层视觉规范（Token、色彩、字体、组件库） |

**页面级章节**（每个页面重复，标题格式：`## §N {页面编号} {页面名}`）：

| 子节 | 说明 |
|------|------|
| 页面布局 | 页面结构、栅格应用、响应式适配规则 |
| 组件样式 | 页面涉及的所有组件及其视觉规格（尺寸/颜色/字体/间距/阴影） |
| 状态展示 | 组件在所有交互状态下的视觉表现 |
| 与交互设计对应 | 交互页面编号、状态矩阵映射、交互流程视觉还原 |

## §3 各章节格式要求

### 3.1 设计系统引用（文档级）

**内容**：声明本模块 UI 稿继承的顶层视觉规范，包括 Token 体系、色彩系统、字体系统、间距与布局系统、形状系统、阴影与海拔、图标规范、主题、组件库。

**格式建议**：列表。每项说明继承的具体内容和适用范围。

> 此章节在文档中只出现一次，位于文件头部之后、第一个页面章节之前。

**填写示例**：

> - **Token 体系**：继承项目 UI-顶层定义 §2，使用 `primary`、`text-primary`、`space-md`、`radius-sm` 等语义化 Token
> - **色彩系统**：遵循项目 UI-顶层定义 §3 的色彩角色（primary/on-primary/surface/bg-page/text-primary/text-secondary/border-default/error）
> - **字体系统**：遵循项目 UI-顶层定义 §4 的字号层级（Display/H1-H3/Body/Caption）和字重（Regular/Medium/Bold）
> - **间距与布局系统**：遵循项目 UI-顶层定义 §5 的基础单位 4px、间距 Token（xs/sm/md/lg/xl）和栅格系统
> - **形状系统**：遵循项目 UI-顶层定义 §6 的圆角 Token（radius-sm/md/lg）和边框规则
> - **阴影与海拔**：遵循项目 UI-顶层定义 §7 的阴影层级（shadow-sm/md/lg）和 z-index 规范
> - **图标规范**：遵循项目 UI-顶层定义 §9 的图标尺寸、风格和颜色规则
> - **主题**：遵循项目 UI-顶层定义 §10 的亮色/暗色 Token 映射规则
> - **组件库**：从项目 UI-顶层定义 §11 的组件清单中选取 Button、Input、Table、Modal 等组件

**常见陷阱**：
- ❌ 只写"继承项目 UI-顶层定义"，不说明具体继承哪些章节和 Token
- ❌ 在设计稿中直接使用硬编码色值（如 `#1890ff`）而非 Token 名（如 `primary`）
- ❌ 使用项目 UI-顶层定义中未声明的 Token——导致开发时找不到对应变量

---

### 3.2 页面布局

**内容**：定义页面的整体结构、栅格应用和响应式适配规则。此子节只描述**结构框架**，不写具体组件样式。

**格式建议**：分三个子节。

#### 3.2.1 页面结构

**内容**：页面包含的区域及其排列关系。

**格式**：文本描述 + 简易框图或列表。

**边界**：不写具体像素值（用 Token），不写具体颜色，不写组件内部细节。示例中的数值仅为示意。

**常见陷阱**：
- ❌ 页面布局写到具体像素——应使用 Token 和栅格列数
- ❌ 布局结构与交互设计的信息架构不一致——设计稿结构应继承交互稿

**填写示例**：

> 页面采用左右两栏布局：
> - 左侧边栏（固定宽度 `size-sidebar`，`surface` 背景）：导航菜单
> - 右侧内容区（自适应，`bg-page` 背景）：
>   - 顶部操作栏（高度 `size-appbar-height`）：页面标题 + 操作按钮组
>   - 筛选区域（`space-md` 内边距）：条件筛选表单
>   - 数据表格区（自适应）：表格 + 分页器

#### 3.2.2 栅格应用

**内容**：页面如何使用项目 UI-顶层定义 §5 的栅格系统。

**格式建议**：文本说明。

**常见陷阱**：
- ❌ 栅格列数与组件宽度不兼容——如组件宽 350px，但列宽只能是 8 的倍数
- ❌ 栅格系统与项目 UI-顶层定义 §5 不一致——模块自行定义栅格

**填写示例**：

> 内容区采用 24 列栅格，页面边距 `space-lg` (24px)，槽宽 `space-md` (16px)。
> - 左侧边栏：固定 4 列（约 200px）
> - 数据表格：占满剩余 20 列

#### 3.2.3 响应式适配

**内容**：页面在不同断点下的布局调整规则。

**格式建议**：表格。列：断点 | 布局变化 | 组件调整。

**填写示例**：

| 断点 | 布局变化 | 组件调整 |
|------|---------|---------|
| 大屏（≥1440px）| 内容区最大宽度 1200px 居中，两侧留白 | 表格全列展示，操作栏展开 |
| 桌面（1024-1440px）| 左右两栏，左侧边栏固定 | 表格全列展示 |
| 平板（768-1024px）| 左侧边栏折叠为图标模式 | 表格隐藏次要列，横向滚动 |
| 手机（<768px）| 单栏布局，边栏收起到抽屉 | 表格改为卡片列表，筛选收起 |

**常见陷阱**：
- ❌ 页面布局写到具体像素——应使用 Token 和栅格列数
- ❌ 响应式只写"自适应"——无具体断点和调整规则
- ❌ 布局与交互设计的信息架构不一致——设计稿结构应继承交互稿

---

### 3.3 组件样式

**内容**：页面涉及的所有组件及其视觉规格。每个组件必须展示**完整的视觉定义**（尺寸、颜色、字体、间距、圆角、阴影）。

**格式建议**：分两种情况。

**全局组件库中已定义的组件**：列出名称和引用的 Token，不重复写样式数值。可选补充关键尺寸。

**模块特有组件/变体**：逐项定义样式，每张表列：**属性 | Token | 值 | 说明**。

**填写示例**（提交按钮——全局组件）：

> 继承项目 UI-顶层定义 §11 组件库中的 `Button` 组件，使用 `primary` 类型、`medium` 尺寸。
>
> 引用 Token：`primary`（背景）、`on-primary`（文字）、`font-size-body`、`font-weight-medium`、`radius-sm`、`space-lg`。

**填写示例**（工单状态标签——模块特有组件）：

| 属性 | Token | 值 | 说明 |
|------|-------|-----|------|
| 宽度 | — | 自适应 | — |
| 高度 | `size-tag-height` | 24px | — |
| 背景色 | `primary-container` | `#E6F7FF` | 待处理状态 |
| 文字色 | `on-primary-container` | `#096DD9` | — |
| 字号 | `font-size-caption` | 12px | — |
| 圆角 | `radius-full` | 999px | 胶囊形状 |
| 内边距 | `space-sm` | 8px | 水平方向 |

**常见陷阱**：
- ❌ 组件只写颜色不写尺寸——开发无法精确还原
- ❌ 使用项目 UI-顶层定义中未声明的 Token——导致开发无法映射
- ❌ 模块特有组件不定义完整样式——开发随意实现
- ❌ 全局组件重复定义样式数值——与顶层定义冲突时产生维护成本

---

### 3.4 状态展示

**内容**：组件在所有交互状态下的视觉表现。建立**交互状态 → 视觉变化**的映射。

**格式建议**：每个组件一张状态展示表。列：**状态 | 触发条件 | 视觉变化 | Token 变化**。

> 状态名称必须与项目交互-顶层定义 §3（交互状态规范）一致；状态视觉表现必须与项目 UI-顶层定义 §2.11.2 全局状态规范对齐。

**填写示例**（提交按钮）：

| 状态 | 触发条件 | 视觉变化 | Token 变化 |
|------|---------|---------|-----------|
| 默认态 | 无 | 主色背景，白色文字 | `primary` / `on-primary` |
| 悬停态 | 鼠标悬浮 | 背景色变深，光标变为手形 | `primary-600` / `on-primary` |
| 按下态 | 鼠标按下 | 背景色更深 | `primary-700` / `on-primary` |
| 聚焦态 | 键盘 Tab | 显示 2px 主色焦点环 | + `focus-ring-2` |
| 禁用态 | 条件不满足 | 置灰，透明度 50% | `text-disabled` / `surface`（50% 透明度）|
| 加载态 | 异步提交中 | 显示加载 Spinner，文字变为"提交中..." | `primary` / `on-primary` + `Spinner` 组件 |

**必须包含的状态**：
- 基础交互状态：默认态、悬停态、按下态、聚焦态、禁用态、加载态
- 异步/边界状态：空状态、错误状态、成功状态、骨架态（如页面级组件需展示）

**常见陷阱**：
- ❌ 只定义默认态，其他状态未定义——开发时随意实现
- ❌ 状态之间视觉差异不明显——用户无法区分
- ❌ 状态名称与交互-顶层定义不一致——导致跨文档追溯困难
- ❌ 禁用态无说明原因——用户困惑"为什么点不了"

---

### 3.5 与交互设计对应

**内容**：标注本模块所有 UI 页面对应的交互设计页面编号、组件清单、状态矩阵映射。页面内涉及的所有跳转/链接目标，需标注目标页面编号或外部文档引用。

**格式建议**：表格。列：UI 页面 | 交互页面编号 | 涉及组件 | 状态矩阵 | 交互流程节点 | 跳转目标。

**填写示例**：

| UI 页面 | 交互页面编号 | 涉及组件 | 状态矩阵 | 交互流程节点 | 跳转目标 |
|---------|-------------|---------|---------|-------------|---------|
| 工单列表页 | PAGE-TICKET-001 | Table、Button、Input、Tag | 默认态/悬停态/加载态 | 列表加载→筛选→查看详情→新建工单 | PAGE-TICKET-002（详情）、PAGE-TICKET-003（创建）|
| 工单详情页 | PAGE-TICKET-002 | Card、Button、Timeline、Dialog | 默认态/禁用态/加载态 | 详情加载→编辑→关闭工单 | PAGE-TICKET-001（返回列表）|
| 工单创建页 | PAGE-TICKET-003 | Form、Input、Select、Button | 默认态/聚焦态/禁用态/错误态/加载态 | 表单打开→填写→校验→提交 | PAGE-TICKET-001（提交成功后返回）|

**常见陷阱**：
- ❌ UI 页面与交互页面编号不一致——导致需求追溯困难
- ❌ 遗漏交互设计中定义的组件——导致功能遗漏
- ❌ 状态矩阵与交互设计的状态机不一致——如交互定义了加载态但 UI 未设计

## §4 格式规范

### 4.1 尺寸单位

- 使用 px（Web）或 pt（iOS）/ dp（Android）
- 同一文档内单位统一
- 所有尺寸优先使用项目 UI-顶层定义的 Token，Token 无法覆盖时写具体数值

**常见陷阱**：
- ❌ 同一文档混用 px 和 rem——导致开发换算困难
- ❌ 模块特有组件的尺寸也不写具体数值——开发无法还原

### 4.2 Token 引用

- 颜色必须使用 Token 名（如 `primary`、`text-secondary`），禁止直接写 HEX 值
- 间距必须使用 Token 名（如 `space-md`），禁止直接写 px 值
- 字号必须使用 Token 名（如 `font-size-body`），禁止直接写 pt/px 值
- 圆角必须使用 Token 名（如 `radius-sm`），禁止直接写数值
- 阴影必须使用 Token 名（如 `shadow-md`），禁止直接写阴影参数
- 例外：模块特有组件的尺寸（宽度/高度）如果无对应 Token，可写具体数值

**常见陷阱**：
- ❌ 使用项目 UI-顶层定义中未声明的 Token——开发找不到对应变量
- ❌ 同一 Token 在不同页面写不同值——如 `primary` 在 A 页面是 `#1890ff`，B 页面是 `#1677ff`
- ❌ 硬编码色值与 Token 混用——维护时无法通过修改 Token 统一调整

### 4.3 状态命名

- 状态名称必须与项目交互-顶层定义 §3 一致：默认态/悬停态/按下态/聚焦态/禁用态/加载态/空状态/错误状态/成功状态/骨架态
- 基础交互状态后缀统一为"态"
- 异步/边界状态后缀统一为"状态"（骨架态除外）

**常见陷阱**：
- ❌ 状态名称与交互-顶层定义不一致——如交互定义"按下态"，UI 写"点击态"
- ❌ 状态后缀不统一——如"悬停态"和"悬停状态"混用

### 4.4 文案规范

- 必须与交互设计逐字一致
- 禁止自行修改文案
- 发现不一致 → 停止生成，询问用户以哪个为准

**常见陷阱**：
- ❌ UI 设计稿中修改文案以"适配布局"——破坏需求一致性
- ❌ 发现文案不一致不标注来源——导致无法判断以谁为准

## §5 检查清单

### 5.1 一致性检验

- [ ] **结构一致性** `[脚本]`：每个页面包含 4 个子节（页面布局、组件样式、状态展示、与交互设计对应），且文档包含 1 个文档级章节（设计系统引用）
- [ ] **范围一致性** `[模型]`：所有页面编号在上游交互设计页面清单内；所有组件在上游交互设计的组件清单内
- [ ] **格式一致性** `[脚本]`：颜色/间距/字号/圆角/阴影使用项目 UI-顶层定义 Token；尺寸单位统一；文案与交互设计逐字一致
- [ ] **状态名称一致性** `[模型]`：组件状态名称与项目交互-顶层定义 §3 一致
- [ ] **状态视觉一致性** `[模型]`：组件状态视觉表现与项目 UI-顶层定义 §11.2 一致
- [ ] **Token 命名一致性** `[模型]`：Token 命名与项目 UI-顶层定义 §2 一致
- [ ] **内部自洽性** `[模型]`：本文档涉及的所有页面编号在「与交互设计对应」子节中都有标注；状态展示中的状态值与组件样式中的默认态不矛盾；同一页面内 Token 引用与设计系统引用声明一致
- [ ] **跨文档链路一致性** `[模型]`（如已有下游文档）：每个组件状态在下游技术方案中有对应 CSS/样式实现；每个 Token 在下游代码仓库中有对应变量定义；每个页面在下游测试报告中有视觉回归用例；每个组件状态在上游交互设计文档的组件交互子节中有对应触发条件

### 5.2 完整性检验

- [ ] 设计系统引用（文档级章节）已声明继承的顶层规范（Token、色彩、字体、间距、形状、阴影、图标、主题、组件库） `[模型]`
- [ ] 每个页面的页面布局子节包含页面结构、栅格应用、响应式适配 `[脚本]`
- [ ] 每个页面的组件样式子节包含所有涉及组件的视觉规格（尺寸/颜色/字体/间距/圆角/阴影） `[脚本]`
- [ ] 每个组件都引用了项目 UI-顶层定义中的 Token，模块特有组件的 Token 在顶层定义中存在 `[脚本]`
- [ ] 每个页面的状态展示子节覆盖至少 6 种基础交互状态（默认/悬停/按下/聚焦/禁用/加载） `[模型]`
- [ ] 页面级组件（如表格、列表）的状态展示覆盖异步状态（空/错误/成功/骨架） `[模型]`
- [ ] 每个异常状态有明确的视觉表现（颜色、文案、图标） `[模型]`
- [ ] 每个交互页面都对应交互设计中的一个页面编号和一个流程节点 `[脚本]`
- [ ] 所有跳转目标页面要么在本文档范围内，要么正确引用外部文档 `[脚本]`
- [ ] §4 格式规范已遵循（尺寸单位统一、无硬编码色值、状态命名符合规范、文案与交互设计逐字一致） `[脚本]`
