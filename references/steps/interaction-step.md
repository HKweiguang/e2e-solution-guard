# 交互设计模板

> 定位：定义用户怎么用产品，以"页面→组件→状态→反馈"为主线组织交互设计，确保每个页面都有完整的状态覆盖、清晰的操作路径和明确的反馈机制。

---

## §1 文档信息

```markdown
# 交互设计-v{主版本号}-{页面组名/流程名}（单业务端）
# 或 交互设计-v{主版本号}-{业务端}-{页面组名/流程名}（多业务端）

**覆盖功能点**：{功能编号}, {功能编号}, ...
**覆盖页面**：{页面编号}, {页面编号}, ...
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v{主版本号}-{模块名} | 需求输入 | {功能编号范围}, PRD §3 交互方式列中描述的页面名称 |
> 注：一个模块只有一个 PRD 文件，内部按业务端分章节声明功能点。交互设计按业务端分文件。同一业务端的不同设备（web/app）可合并或拆分。
| 项目交互-顶层定义 | 规范继承 | 交互令牌（§1）、交互状态规范（§2）、交互反馈规范（§3）、交互规范（§4）、无障碍设计（§5）、平台交互规范（§6）、页面编号规范（§7） |
```

## §2 章节结构

交互设计产物以**页面**为主线组织。每个页面为一个一级章节。

**产物级章节**（整个产物只出现一次）：

| 章节 | 说明 |
|------|------|
| 设计系统引用 | 声明继承的顶层设计令牌、状态规范、反馈规范、交互规范 |

**页面级章节**（每个页面重复，标题格式：`## §N {页面编号} {页面名}`）：

| 子节 | 说明 |
|------|------|
| 页面结构 | 页面页面结构图、信息架构、页面层级、路由规则、多端差异 |
| 组件交互 | 组件清单及每个组件的状态矩阵 |
| 状态机 | 页面级状态流转（默认/空/加载/错误/成功），与 PRD 业务状态映射 |
| 页面流程 | 用户操作路径 + 每个节点的反馈方式 |
| 异常处理 | 异常场景、系统响应、用户感知、恢复路径 |
| 与 PRD 对应 | 功能点、页面编号、业务规则、错误码的交叉引用 |
| 跨模块聚合页面（可选） | 首页、工作台、Dashboard 等聚合多个模块数据的页面 |
| 响应式适配（可选） | 不同断点下的结构变更（元素增减、布局变化、导航模式切换） |

> **章节稳定性**：产物级章节编号固定，页面级子节顺序固定。如无内容，保留标题声明"跳过"，不可删除导致编号变化，确保下游产物（UI 设计、测试）引用稳定。

## §3 交互不写的

交互设计只定义"用户怎么用"。以下内容的职责归属其他步骤：

- **视觉设计**（色彩、字体、间距、组件样式）→ `ui-step.md`
- **技术实现**（架构、接口、数据模型）→ `tech-step.md`
- **测试策略**（测试用例、测试范围）→ `test-step.md`

**越界检测**：若交互设计中出现视觉描述（"使用 XX 颜色"）、技术描述（"调用 XX 接口"），应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 设计系统引用（产物级）

**内容**：声明本模块交互设计继承的顶层规范，包括交互令牌、状态规范、反馈规范、交互规范、无障碍设计、平台交互规范、页面编号规范。

**格式建议**：列表。每项说明继承的具体内容和适用范围。

> ⚠️ **强制位置**：此章节在产物中只出现一次，必须位于文件头部之后、第一个页面章节（`## §N {页面编号}`）之前。禁止放在页面章节中间或末尾。


---

### 4.2 页面结构

**内容**：定义模块涉及的所有页面及其结构关系。

**格式建议**：分五个子节。

#### 4.2.1 页面页面结构图

**内容**：每个页面的**结构布局可视化**。用 SVG 代码绘制页面结构图，展示页面区域划分、元素位置和相对尺寸关系。

**核心原则：页面结构图必须体现页面独特功能，禁止复制粘贴通用模板。**

**格式**：`svg` 代码块。每个页面一张页面结构图。

**必须包含（按页面功能定制）**：
- 该页面的**核心功能区域**及其具体元素（如订单列表页必须有搜索栏、筛选器、订单卡片列表；个人中心页必须有头像、昵称、功能入口网格）
- 每个区域内部用文字标签说明**具体内容**，禁止用"内容区域"等笼统概括
- 页面特有的交互控件（如下拉菜单、日期选择器、 tabs 切换、弹窗触发按钮）
- 复杂交互元素用虚线框或注释标注交互方式

**边界**：
- ✅ 写：区域划分、元素位置、相对尺寸（用 Token 如 `size-sidebar`、`space-lg`）
- ❌ 不写：颜色、字体、阴影、圆角等视觉样式（这些在 UI 设计中定义）
- ❌ 不写：具体像素值（用 Token 或相对比例）
- ❌ 禁止：所有页面使用同一套"状态栏+导航栏+内容区域+Tab栏"的通用外壳，仅改文字标签

**页面结构图规范**：
- 使用中性灰色填充区分不同区域深浅，用灰色边框勾勒区域边界
- 区域内部用文字标签说明内容（如"导航菜单"、"数据表格"）
- 复杂交互元素（如下拉菜单、弹窗）用虚线框或注释标注
- 页面结构图仅使用中性灰色作为示意用途，不代表最终产品视觉色彩，最终色彩由 UI 设计定义

**反例（禁止）**：
```svg
<!-- 错误：所有页面都复制这个通用手机外壳 -->
<rect ... fill="#f5f5f5"/> <!-- 屏幕背景 -->
<rect .../> <!-- 状态栏 -->
<rect .../> <!-- 导航栏 -->
<rect .../> <!-- 内容区域（只写"内容区域"四个字） -->
<rect .../> <!-- Tab栏 -->
```

>

#### 4.2.2 信息架构

**内容**：页面信息层级、内容分组、导航关系。
**格式建议**：树状图或层级列表。

#### 4.2.3 页面层级

**内容**：当前页面在整体导航结构中的位置、父级/同级/子级页面关系。
**格式建议**：层级路径或面包屑结构。

#### 4.2.4 路由规则

**内容**：页面 URL 路径、路由参数、进入/离开条件。
**格式建议**：表格。列：路由路径 | 参数 | 进入条件 | 离开条件。

#### 4.2.5 多端差异

**内容**：同一页面在不同设备（Web/App/小程序）上的结构差异声明。
**格式建议**：表格或列表。如无差异则标注"无显著差异"。

---

### 4.3 组件交互

**内容**：模块涉及的所有交互组件及其状态矩阵。每个组件必须展示在所有**基础**交互状态下的表现。

> 本节只定义状态名称、触发条件和交互响应，**不定义视觉表现**（颜色、亮度、焦点环等）。视觉表现由 UI 设计稿（HTML）实现，基于项目 UI-顶层定义的 Token 体系。

**格式建议**：每个组件一张状态定义表。列：**状态 | 行为定义 | 交互响应 | 力学约束**。

> **力学约束**：定义该状态或状态转换中的时间、次数、防抖/节流等技术约束。例如超时时间、最大重试次数、输入防抖间隔、动画时长等。这些约束是骨架的"力学属性"，避免散落在 PRD 非功能需求和技术方案中无法追溯。
>
> 异步状态（空/错误/成功/骨架）在状态机子节中定义。如需多组件对比，可采用横表：组件/元素 | 默认态 | 悬停态 | 按下态 | 聚焦态 | 禁用态 | 加载态。


---

### 4.4 状态机

**内容**：页面在不同数据状态和交互状态下的展示与行为。建立**PRD 业务状态 → UI 交互状态 → 系统响应**的映射。

**格式建议**：每个页面一张状态映射表 + 状态流转图（可选）。

**必须包含**：默认态、空状态、加载态、错误状态、成功状态、骨架态。


---

### 4.5 页面流程

**内容**：用户从进入页面到完成核心任务的操作路径，以及每个节点的反馈方式。

**格式建议**：步骤列表。每个步骤包含：前置条件 | 用户操作 | 系统响应 | 反馈方式 | 后置状态。


---

### 4.6 异常处理

**内容**：页面可能遇到的异常情况及系统响应。至少覆盖 1 个异常，复杂页面建议 3 个。

**格式建议**：表格。列：异常场景 | 触发条件 | 系统响应 | 用户感知 | 恢复路径。


---

### 4.7 与 PRD 对应

**内容**：标注本模块所有交互页面对应的 PRD 功能点、用户故事、页面名称、业务规则和错误码。**本章节是"页面编号 → 功能点"的唯一真相源——PRD 只写页面名称不写编号，编号在此建立映射。**

**格式建议**：表格。列：交互页面 | PRD 页面名称 | 关联功能点 | 关联用户故事 | PRD 业务规则 | PRD 错误码。

> **PRD 页面名称**：填写 PRD §4.3「关联页面」列中的页面名称，确保与 PRD 完全一致。用于对齐 PRD 中的功能点与交互设计中的页面编号。
>
> **关联功能点**：**强制**。填写本页面涉及的 PRD §3 功能编号（如 `ORDER-DEFAULT-001`）。一个页面可能涉及多个功能，多个用逗号分隔。不允许为空——每个页面必须至少关联一个功能点。
>
> **关联用户故事**：填写本页面服务的 PRD §2 用户故事编号（如 `STORY-USER-DEFAULT-001`）。一个页面可能服务多个故事，多个用逗号分隔。


---

### 4.8 跨模块聚合页面

**内容**：处理首页、工作台、Dashboard 等聚合多个模块数据的页面。此类页面自身通常没有独立的业务规则，其结构和布局在本产物中定义，数据和规则引用上游 PRD。

**格式建议**：
- 在「与 PRD 对应」章节中，标注每个聚合区块的数据来源模块和功能点
- 不重复定义业务规则、数据模型、错误码——全部引用上游模块 PRD

**编号示例**：
- 首页：`PAGE-DEFAULT-WEB-HOME-001`
- 工作台：`PAGE-DEFAULT-WEB-WORKBENCH-001`
- Dashboard：`PAGE-DEFAULT-WEB-DASHBOARD-001`

> 聚合页面编号使用通用页面名标识（`HOME`/`WORKBENCH`/`DASHBOARD`），不绑定单一模块。各区块数据通过「与 PRD 对应」子节引用上游模块 PRD 的功能编号。

---

### 4.9 响应式适配（可选）

**内容**：当项目 UI-顶层定义要求响应式适配时，声明本页面在不同断点下的结构变更。将"什么时候结构变化"的决策权明确归属到交互设计，避免 UI 设计随意变更结构导致与交互设计不一致。

> **边界**：只写结构变更（元素增减、布局变化、导航模式切换），不写视觉变化（字体大小、间距调整——这些在 UI 设计中基于 Token 自动适配）。

**格式建议**：表格。列：断点 | 结构变更 | 触发条件 | 说明。


---

## §5 格式规范

### 5.1 章节层级

- 一级：`## §N {页面编号} {页面名}`
- 二级：`### N.M 子节名`
- 三级：`#### 细节描述（少用）`
- 禁止出现 `## §1.6` 这种同级但编号暗示子级的情况

### 5.2 页面编号

- 页面编号由交互设计根据项目交互-顶层定义 §7 分配，统一格式为 `PAGE-{业务端}-{平台}-{页面名}-{序号}`（如 `PAGE-DEFAULT-WEB-HOME-001`）。业务端继承 PRD-顶层定义 §2.2 业务端标识分配表
- 编号前缀独立于 PRD 功能编号前缀，避免混淆
- 页面名称参考 PRD §3 交互方式列中的描述
- 章节标题中标注：`## §1 {页面编号} {页面名}`

### 5.3 状态引用

- 交互状态名称必须与项目交互-顶层定义 §2（交互状态规范）一致（默认态/悬停态/按下态/聚焦态/禁用态/加载态/空状态/错误状态/成功状态/骨架态）
- 业务状态名称必须与项目 PRD-顶层定义 §5（状态值）一致

## §6 检查清单

> 按 **6 维度 × 3 层级** 组织：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试的完整追溯链

> 详细检查清单参见 [`references/workflow/audit-checklists.md`](../workflow/audit-checklists.md) **交互设计检查清单**部分。
>
> 以下仅列出本产物类型的高优先级 `[模型]` 检查项（需 AI 语义判断），`[脚本]` 项由 `doc-audit.py` 自动执行：

- [ ] **L1** `[模型]` 页面结构图反映了页面的核心功能结构，而非通用外壳模板（禁止所有页面复制同一套"状态栏+导航栏+内容区域+Tab栏"）
- [ ] **L1** `[模型]` 页面结构图中包含页面特有的交互元素和具体内容标签（禁止仅用"内容区域"等笼统概括）
- [ ] **L2** `[模型]` 每个页面对应上游 PRD §3 交互方式列中的一个描述
- [ ] **L2** `[模型]` 交互-顶层定义的主导航模块与 PRD-顶层定义的模块划分表在数量、名称、职责边界上一致
- [ ] **L3** `[模型]` 交互设计中的字段名与 PRD §5/技术方案 §3 一致
- [ ] **L1** `[模型]` 状态机与 PRD 业务状态建立了映射
- [ ] **L2** `[模型]` 状态机中的业务状态名称与 PRD §5 或 PRD-顶层定义 §5 一致
- [ ] **L3** `[模型]` PRD §7 每个错误码的触发场景在交互设计异常处理中有对应
- [ ] **L1** `[模型]` 简单页面≥1 异常场景，复杂页面≥3
- [ ] **L2** `[模型]` 响应式断点与项目 UI-顶层定义一致
- [ ] **L1** `[模型]` 无障碍设计声明（所有项目必须声明底线要求或后续迭代计划）
- [ ] **L2** `[模型]` 平台交互规范与项目交互-顶层定义一致
- [ ] **L2** `[模型]` 检查上游顶层定义在本产物及关联顶层定义中的完整映射，发现缺失标记为阻塞性差异
- [ ] **L1** `[模型]` 产物中无不属于交互范畴的视觉/技术描述
- [ ] **L1** `[模型]` 页面结构图不包含颜色/字体/阴影等视觉样式
