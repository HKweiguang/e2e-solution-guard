# 文档-代码一致性审计报告模板

> 定位：代码实现完成后，由独立只读 subagent 对比代码与技术方案/PRD 的一致性，产出审计结果。
>
> 本报告是文档与代码之间的最后一道锁。

---

## §1 文件头部格式

```markdown
# AUDIT-{服务名/模块名}-v{主版本号}

**审计对象**：TECH-v{主版本号}-{服务名}
**代码版本**：{Git commit hash 或版本标签}
**审计日期**：YYYY-MM-DD
**审计者**：只读 subagent

**上游文档**：
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| TECH-v{主版本号}-{服务名} | 审计依据 | 全部章节 |
| PRD-v{主版本号}-{模块名} | 审计依据 | §6 权限规则、§7 错误码 |
```

---

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 审计摘要 | 通过/不通过、差异总数、关键风险 |
| §2 接口一致性 | 逐项对比技术方案 §6 与代码路由/方法 |
| §3 数据模型一致性 | 逐项对比技术方案 §5 与代码表结构 |
| §4 错误码一致性 | 逐项对比 PRD §7 与代码抛出异常 |
| §5 权限一致性 | 逐项对比 PRD §6 与代码权限控制 |
| §6 业务规则一致性 | 逐项对比技术方案 §7 与代码核心流程 |
| §7 差异清单与建议 | 所有不一致项汇总、建议修正方向 |

---

## §3 各章节格式要求

### 3.1 审计摘要

**内容**：整体结论、差异统计、是否存在阻塞性差异。
**格式**：
- 结论：**通过** / **不通过**
- 差异总数：N 项（其中阻塞性 M 项，警告性 P 项）
- 关键风险：如存在阻塞性差异，列出最严重的前 3 项

**判定标准**：
- **通过**：五项检查均无阻塞性差异，警告性差异 ≤3 项且已标注
- **不通过**：存在任一阻塞性差异，或警告性差异 >3 项

**阻塞性差异定义**：
- 接口缺失或路径/方法不匹配
- 数据模型字段缺失或类型不兼容
- 错误码未按 PRD 定义实现
- 权限控制低于 PRD 要求（如应登录的接口未加登录校验）
- 业务规则关键步骤缺失或顺序错误

**警告性差异定义**：
- 字段顺序与文档不同
- 缺少非必填字段的默认值说明
- 注释与文档描述有细微偏差但不影响行为

### 3.2 接口一致性

**内容**：技术方案 §6 中定义的每个接口与代码实现的对照。
**格式**：表格，列：

| 接口编号 | 文档定义（URL/方法） | 代码实现（路由/方法） | 请求参数一致 | 响应结构一致 | 错误码一致 | 状态 |
|---------|---------------------|---------------------|-------------|-------------|-----------|------|
| API-001 | `POST /api/orders` | `POST /api/orders` | ✅ | ✅ | ✅ | 通过 |

**状态**：通过 / 警告 / 阻塞

### 3.3 数据模型一致性

**内容**：技术方案 §5 中定义的每张表与代码 entity/ORM 的对照。
**格式**：每张表一张对比表格，列：

| 字段名 | 文档类型 | 文档约束 | 代码字段名 | 代码类型 | 代码约束 | 一致 |
|--------|---------|---------|-----------|---------|---------|------|
| `order_id` | BIGINT | PK, NOT NULL | `orderId` | Long | `@Id` | ✅ |

**审计规则**：
- 字段名因语言命名规范不同（snake_case vs camelCase）不视为不一致
- 但业务含义必须一致，禁止文档中是 `user_id` 代码中是 `customer_id`
- 审计字段（创建时间/更新时间/创建人/更新人）必须存在

### 3.4 错误码一致性

**内容**：PRD §7 定义的错误码与代码中实际抛出的异常/返回码对照。
**格式**：表格，列：

| 错误码 | PRD 触发场景 | 代码中使用位置 | 使用场景与 PRD 一致 | 状态 |
|--------|-------------|---------------|-------------------|------|
| `E001` | 订单不存在 | `OrderService.java:45` | ✅ | 通过 |

**额外检查**：
- 代码中是否存在 PRD 未定义的新错误码？如有，列出并标注为阻塞
- PRD 定义的错误码是否在代码中都有使用？如无，列出并标注为警告

### 3.5 权限一致性

**内容**：PRD §6 权限规则与代码中的权限控制对照。
**格式**：表格，列：

| 功能点 | PRD 权限要求 | 代码实现（注解/拦截器） | 一致 | 状态 |
|--------|-------------|------------------------|------|------|
| `F001` | 登录用户 | `@RequireLogin` | ✅ | 通过 |

**阻塞性差异**：代码权限低于 PRD 要求（如 PRD 要求角色权限，代码只做了登录校验）

### 3.6 业务规则一致性

**内容**：技术方案 §7 核心流程与代码实现对照。
**格式**：按流程步骤列表对比：

| 步骤 | 文档定义 | 代码实现 | 一致 | 状态 |
|------|---------|---------|------|------|
| 1. 校验库存 | 查询库存表，判断 `stock > 0` | `InventoryService.checkStock()` | ✅ | 通过 |

**阻塞性差异**：关键步骤缺失、执行顺序错误、触发条件不匹配

### 3.7 差异清单与建议

**内容**：所有不一致项的汇总，按阻塞性/警告性分类。
**格式**：

#### 阻塞性差异
1. **{差异简述}**
   - 文档：{文档原文引用}
   - 代码：{代码原文引用}
   - 建议：改代码 / 改文档
   - 影响：{如果不修正会导致什么问题}

#### 警告性差异
1. **{差异简述}**
   - 说明：{偏差描述}
   - 建议：{可选修正}
