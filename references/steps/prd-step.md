# PRD 模板

> 定位：定义模块的产品功能，以"问题→用户→方案→验证"为主线组织需求，确保每个功能都有明确的用户目标、成功指标和验收标准。

---

## §1 文档信息

```markdown
# PRD-v{主版本号}-{模块名}

**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD-顶层定义 | 规范继承 | 全局规范（术语、状态值、编码规则、模块职责范围、业务端标识分配、权限级别定义等） |

```

---

> 各章节的填写示例与常见陷阱参见 [`references/examples/prd-examples.md`](../../references/examples/prd-examples.md)。

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 背景与目标 | 要解决的问题、产品目标、成功指标 |
| §2 用户与场景 | 目标用户、用户故事、用户旅程、使用场景 |
| §3 功能需求 | 以用户故事为索引的功能需求（含交互方式） |
| §4 非功能需求 | 性能、安全、可扩展、合规、可用性 |
| §5 数据模型 | 实体、字段、类型、约束 |
| §6 业务规则 | 状态流转、权限、计算逻辑、校验规则 |
| §7 错误处理 | 错误码、触发场景、前端提示 |
| §8 验收标准 | 可测试的条件+操作+预期结果 |
| §9 依赖与范围 | 上游依赖、技术约束、业务假设、明确不做 |
| §10 附件 | 参考资料、原型链接 |

> **章节稳定性**：以下章节编号必须固定，以确保下游产物（交互设计、技术方案、测试）的引用稳定性。如无内容，保留章节标题并声明"跳过"或"当前版本无此项"，**不可直接删除章节导致编号变化**。

## §3 PRD 不写的

PRD 只定义"做什么"和"为什么做"。以下内容的职责归属其他步骤：

- **交互设计**（页面结构、状态机、页面流转）→ `interaction-step.md`
- **视觉设计**（色彩、字体、间距、组件规范）→ `ui-step.md`
- **技术实现**（架构、接口、数据模型细节）→ `tech-step.md`
- **测试策略**（测试用例、测试范围）→ `test-step.md`

**越界检测**：若 PRD 中出现交互描述（"点击XX按钮后跳转到XX页面"）、视觉描述（"使用XX颜色"）、技术描述（"调用XX接口"），或独立的「页面概览/页面清单」章节，应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 背景与目标

**内容**：回答三个问题——**要解决什么问题**、**达成什么目标**、**如何衡量成功**。

**格式建议**：3-5 段自由文本，必须包含成功指标。

**成功指标**（必填）：
- 至少定义 1 个可量化的成功指标（如"工单平均响应时间从 4 小时降至 1 小时内"）
- 指标应覆盖业务价值（用户侧）和系统价值（运营侧）


---

### 4.2 用户与场景

**内容**：定义模块的目标用户、核心用户故事、关键使用场景。

**格式建议**：
- **目标用户**：1-2 段描述，包含用户角色（用户故事语法中的身份，非业务端）、职责、痛点
- **用户故事**：表格。列：故事编号 | 用户故事 | 优先级 | 对应功能编号。故事编号格式继承项目 PRD-顶层定义 §6（编码规则），统一格式为 `STORY-{模块缩写}-{业务端标识}-{序号}`（如 `STORY-USER-DEFAULT-001`）。**故事编号必须在项目 PRD-顶层定义 §6 编码规则中明确定义**。**多业务端项目应按业务端分开展示，每个业务端一个表格，禁止混排**

> **注意**：用户故事中的"角色"（如"As a 客服专员"）是具体用户身份，属于用户故事语法；业务端（如 CONSUMER/OPERATOR/ADMIN）是产物组织维度。两者不同，禁止混淆。
- **用户旅程**：流程图或步骤列表，标注痛点和机会点。**多业务端项目应按业务端分别描述旅程，避免混排**

**用户故事格式**（推荐）：
```
As a [用户角色], I want [目标], so that [收益]
```


---

### 4.3 功能需求

**内容**：以用户故事为索引，逐条定义功能需求。每个功能必须关联用户故事、业务规则和验收标准。

**格式建议**：表格。列：功能编号 | 故事编号 | 功能名称 | 优先级 | 关联页面 | 交互方式 | 业务规则 | 验收标准。

> 交互方式描述功能以什么形态呈现（如页面、弹窗、API、后台任务），不写具体布局和样式。功能的埋点事件、入口位置、通知触发等价值触达需求，在交互方式列简要说明或附件中补充。

**字段规范**：
- 功能编号：格式继承项目 PRD-顶层定义 §2（全局功能范围）中的业务端标识分配和 §6（编码规则），统一格式为 `{模块缩写}-{业务端标识}-{序号}`（如 `ORDER-DEFAULT-001`、`ORDER-CONSUMER-001`），如无定义则询问用户补充
- 功能名称：动宾结构，建议不超过 10 字
- 优先级：P0（必须）/ P1（重要）/ P2（加分）
- 关联页面：涉及的页面名称/描述（如"商品列表页"），多个页面用逗号分隔。一个功能可能跨越多个页面（如购买流程涉及商品列表页、商品详情页、购物车页、订单确认页）。**不写页面编号**——页面编号由交互设计分配，通过 interaction §4.7「与 PRD 对应」反向查询
- 故事编号：填 §2 中的故事编号（推荐，如 `STORY-ORDER-DEFAULT-001`），或用户故事摘要
- 业务规则：填写对应 §6 的规则编号（如 `RULE-ORDER-DEFAULT-001`）
- 验收标准：填写对应 §8 的验收项编号（如 `ACC-ORDER-DEFAULT-001`）


---

### 4.4 非功能需求

**内容**：定义功能需求之外的系统属性要求，包括性能、安全、可扩展性、合规、可用性。

**格式建议**：表格。列：类型 | 需求描述 | 衡量标准 | 优先级。


---

### 4.5 数据模型

**内容**：模块涉及的所有业务实体及其字段定义。

**格式**：每个实体一张表，列：字段 | 业务类型 | 约束 | 业务说明。

**字段规范**：
- 字段名：使用业务语义命名（如"下单时间"而非"created_at"），同一文档内统一风格
- 业务类型：字符串 / 整数 / 布尔 / 日期时间 / JSON / 金额 / 文本 / 枚举
- 约束（业务语义）：必填 / 可选 / 唯一 / 关联{实体名} / 有默认值：{值} / ≥0 / >0

**边界**：不写数据库实际类型、索引策略、分表策略、SQL、审计字段（如 created_at/updated_at）。


---

### 4.6 业务规则

**内容**：状态流转、权限规则、计算逻辑、校验规则。权限规则引用 PRD-顶层定义 §2 全局功能范围 — 权限级别定义，在模块内细化各权限级别的具体操作范围。

**格式**：表格（规则编号 | 规则描述 | 触发条件 | 处理逻辑）或状态机图。

**字段规范**：规则编号：格式继承项目 PRD-顶层定义 §6（编码规则），如无定义则询问用户补充。

**边界**：不写代码实现、不写前端交互。


---

### 4.7 错误处理

**内容**：模块内所有错误码的定义。

**格式**：表格，列：错误码 | 触发场景 | 前端提示。

**字段规范**：
- 错误码格式继承项目 PRD-顶层定义 §6（编码规则），如无定义则询问用户补充
- 前端提示：用户友好的中文，建议不超过 20 字

**边界**：不写 HTTP 状态码、后端日志级别、异常类名。


---

### 4.8 验收标准

**内容**：每个功能的可测试标准。

**格式建议**：表格。列：验收项编号 | 功能编号 | 验收项 | 条件 | 操作 | 预期结果。每条验收项必须用 `[条件] + [操作] + [预期结果]` 句式描述。

**字段规范**：验收项编号：项目自定义编号格式。

**边界**：不写测试步骤、不写测试数据、不写性能指标。


---

### 4.9 依赖与范围

**内容**：声明模块的外部依赖、技术约束、业务假设，以及当前版本明确不做的范围。

**格式建议**：4 个子节，每个子节用列表或表格。

**子节**：
- **上游依赖**：本模块依赖的其他模块/服务/数据源，以及依赖的具体内容
- **技术约束**：已知的技术限制（如必须兼容 IE11、必须使用现有权限体系）
- **业务假设**：产品决策基于的假设（如"假设所有客户已有统一客户ID"），假设不成立时的影响
- **明确不做**：本版本明确排除的功能/场景，每项附理由


---

### 4.10 附件

**内容**：参考资料、原型链接、相关文档链接。

**格式**：自由文本。

**边界**：只放链接和简要说明。

---

## §5 格式规范

- 编号（功能/故事/规则/错误码/验收项）继承项目 PRD-顶层定义 §6 编码规则
- 状态值继承项目 PRD-顶层定义 §5
- 表格使用标准 Markdown 语法，禁止合并单元格
