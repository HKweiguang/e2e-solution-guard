# PRD 模板

> 定位：定义模块的产品功能，包括功能范围、业务规则、数据模型、验收标准。

---

## §1 文件头部格式

```markdown
# PRD-v{主版本号}-{模块名}

**覆盖功能点**：F{起始编号}-F{结束编号}
**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD-顶层定义 | 规范继承 | 全局术语、状态值、错误码规则 |
```

---

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 模块概述 | 模块定位、用户价值、与其他模块关系 |
| §2 功能清单 | 功能编号、名称、优先级、说明 |
| §3 里程碑规划 | 当前版本范围、后续方向 |
| §4 页面清单 | 页面编号、名称、入口、核心功能 |
| §5 数据模型 | 实体、字段、类型、约束 |
| §6 业务规则 | 状态流转、权限、计算逻辑、校验规则 |
| §7 错误码 | 错误码、触发场景、前端提示 |
| §8 价值触达 | 埋点、入口、通知、引导 |
| §9 验收标准 | 可测试的条件+操作+预期结果 |
| §10 附件 | 参考资料、原型链接 |

## §3 各章节格式要求

### 3.1 模块概述

**内容**：模块定位、用户价值、与其他模块关系。
**格式**：自由文本，3-5 段。
**边界**：不写功能细节、不写页面列表。

### 3.2 功能清单

**内容**：模块内所有功能的编号、名称、优先级、简要说明。
**格式**：表格，列：功能编号 | 功能名称 | 优先级 | 说明。
**字段规范**：
- 功能编号：`F` + 3位序号，全局唯一，按模块分段分配
- 功能名称：动宾结构，不超过 10 字
- 优先级：P0（必须）/ P1（重要）/ P2（加分）
**边界**：不写实现细节、不写页面布局。

### 3.3 里程碑规划

**内容**：当前版本包含哪些功能、后续版本方向。
**格式**：自由文本。
**边界**：当前版本功能必须在 §2 中有对应；不写时间排期。

### 3.4 页面清单

**内容**：模块涉及的所有页面及其核心功能。
**格式**：表格，列：页面编号 | 页面名称 | 入口 | 核心功能。
**字段规范**：
- 页面编号：`P` + 3位序号，全局唯一，按模块分段分配
- 页面名称：名词，不超过 6 字
**边界**：不写页面布局、视觉样式、交互细节。

### 3.5 数据模型

**内容**：模块涉及的所有实体及其字段定义。
**格式**：每个实体一张表，列：字段 | 类型 | 约束 | 说明。
**字段规范**：
- 字段名：snake_case 或 camelCase，同一文档内统一
- 类型：业务类型（字符串/整数/布尔/日期时间/JSON/Decimal/文本）
- 约束：PK / FK / NOT NULL / DEFAULT / INDEX / UNIQUE
**边界**：不写数据库实际类型、索引类型、分表策略、SQL。

### 3.6 业务规则

**内容**：状态流转、权限规则、计算逻辑、校验规则。
**格式**：表格（规则编号 | 规则描述 | 触发条件 | 处理逻辑）或状态机图。
**字段规范**：规则编号：`R` + 3位序号。
**边界**：不写代码实现、不写前端交互。

### 3.7 错误码

**内容**：模块内所有错误码的定义。
**格式**：表格，列：错误码 | 触发场景 | 前端提示。
**字段规范**：
- 格式继承项目 PRD-顶层定义 §2.6：全局错误 `G{3位数字}`，模块错误 `E{模块缩写}{2位数字}`
- 前端提示：用户友好的中文，不超过 20 字
**边界**：不写 HTTP 状态码、后端日志级别、异常类名。

### 3.8 价值触达

**内容**：埋点、入口、通知、引导。
**格式**：固定 4 个子节：埋点事件、功能入口、通知触达、用户引导。
**边界**：不写埋点代码、不写入口样式。

### 3.9 验收标准

**内容**：每个功能的可测试标准。
**格式**：每条必须是 `[条件] + [操作] + [预期结果]`。
**格式**：表格，列：验收项编号 | 功能点 | 验收项 | 条件 | 操作 | 预期结果。
**字段规范**：验收项编号：`§9.` + 3位序号。
**边界**：不写测试步骤、不写测试数据、不写性能指标。

### 3.10 附件

**内容**：参考资料、原型链接、相关文档链接。
**格式**：自由文本。
**边界**：只放链接和简要说明。

## §4 格式规范

### 4.1 编号规范

| 编号类型 | 格式 | 连续性要求 |
|----------|------|-----------|
| 功能编号 | F + 3位数字 | 模块内连续，按顶层定义 §2.2 分配的编号段使用 |
| 页面编号 | P + 3位数字 | 模块内连续，按顶层定义 §2.2 分配的编号段使用 |
| 规则编号 | R + 3位数字 | 文档内连续 |
| 验收项编号 | §9. + 3位序号 | 文档内连续 |

### 4.2 状态值规范

- 状态字段：英文大写枚举字符串
- 开关字段：布尔类型
- 禁止数字魔法值

### 4.3 表格规范

- 表头使用 `| 字段 |` 格式
- 同一文档内表格列头必须一致
- 禁止合并单元格

## §5 检查清单

### 一致性检验

- [ ] **结构一致性** `[脚本]`：§1–§10 章节齐全，无缺漏
- [ ] **范围一致性** `[模型]`：§2 功能清单中的功能点在项目 PRD-顶层定义定义的模块职责范围内，功能编号按顶层定义分配；§4 页面清单中的页面在项目 PRD-顶层定义定义的模块页面范围内，页面编号按顶层定义分配
- [ ] **格式一致性** `[脚本]`：功能编号 `F` + 3位数字；页面编号 `P` + 3位数字；状态值为英文大写枚举，无数字魔法值；错误码格式符合项目 PRD-顶层定义
- [ ] **语义一致性** `[模型]`：§6 业务规则与上游 PRD 定义的状态流转一致；§7 错误码的触发场景与上游功能定义匹配；§5 数据模型字段与上游实体定义一致
- [ ] **内部自洽性**：§2 的功能编号在 §4/§5/§6/§7/§9 中引用时完全一致，无遗漏无多余 `[脚本]`；§4 的页面编号在 §5/§9 中出现时一致 `[脚本]`；§7 的每个错误码在 §6 业务规则中有对应触发场景 `[模型]`
- [ ] **跨文档链路一致性** `[模型]`（如已有下游文档）：§2 的每个功能点在下游交互设计/技术方案中有对应实现；§4 的每个页面编号在下游交互设计中被引用；§7 的每个错误码在下游技术方案接口设计中有映射；§9 的每条验收标准在下游测试报告中有对应用例

### 完整性检验

- [ ] §2 功能清单包含编号、名称、优先级、说明四列 `[脚本]`
- [ ] §2 功能编号无跳号、无重复 `[脚本]`
- [ ] §4 页面清单覆盖所有相关页面 `[模型]`，页面编号无跳号、无重复 `[脚本]`
- [ ] §4 的每个页面至少关联一个功能点（§2） `[脚本]`
- [ ] §5 数据模型每个实体包含字段、类型、约束、说明 `[脚本]`
- [ ] §6 业务规则覆盖所有状态流转和权限判断 `[模型]`
- [ ] §7 的每个错误码都有触发场景和前端提示 `[脚本]`
- [ ] §9 的每条验收标准都对应 §2 的一个功能点 `[脚本]`，且格式为「条件 + 操作 + 预期结果」 `[模型]`
