# PRD 模板

> 定位：定义模块的产品功能，以"问题→用户→方案→验证"为主线组织需求，确保每个功能都有明确的用户目标、成功指标和验收标准。

---

## §1 文档信息

```markdown
# PRD-v{主版本号}-{模块名}

**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
> 以下仅为该步骤常见依赖的示例，实际生成时需根据项目上下文动态判断，不要机械套用。

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD-顶层定义 | 规范继承 | 全局规范（术语、状态值、编码规则、模块职责范围、业务端标识分配、权限级别定义等） |

```

---

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 背景与目标 | 要解决的问题、产品目标、成功指标 |
| §2 用户与场景 | 目标用户、用户故事、用户旅程、使用场景 |
| §3 功能需求 | 以用户故事为索引的功能需求（含交互方式） |
| §4 非功能需求 | 性能、安全、可扩展、合规、可用性 |
| §5 数据模型 | 实体、字段、类型、约束 |
| §6 业务规则 | 状态流转、权限、计算逻辑、校验规则 |
| §7 错误处理 | 错误码、触发场景、前端提示 |
| §8 验收标准 | 可测试的条件+操作+预期结果 |
| §9 依赖与范围 | 上游依赖、技术约束、业务假设、明确不做 |
| §10 附件 | 参考资料、原型链接 |

---

## §3 PRD 不写的

| 不写 | 原因 | 应该在哪 |
|------|------|---------|
| 页面清单（页面范围、页面职责定义） | 页面是功能的组织结果，不属于需求 | `interaction-step.md` |
| 交互设计（页面结构、状态机、页面流转） | 行为规则不属于产品需求 | `interaction-step.md` |
| 视觉设计（色彩、字体、间距、组件规范） | 视觉表现不属于产品需求 | 项目 UI-顶层定义 + `ui-step.md`（HTML 原型） |
| 技术实现（架构、接口、数据模型细节） | 实现方案不属于产品需求 | `tech-step.md` |
| 测试策略（测试用例、测试范围） | 验证方法不属于产品需求 | `test-step.md` |

**越界检测**：若 PRD 中出现类似"点击XX按钮后跳转到XX页面"的交互描述，或"使用XX颜色"的视觉描述，或"调用XX接口"的技术描述，或独立的「页面概览」「页面清单」章节，应标记为**越界内容**，提示用户移至对应步骤。

## §4 各章节格式要求

### 4.1 背景与目标

**内容**：回答三个问题——**要解决什么问题**、**达成什么目标**、**如何衡量成功**。

**格式建议**：3-5 段自由文本，必须包含成功指标。

**成功指标**（必填）：
- 至少定义 1 个可量化的成功指标（如"工单平均响应时间从 4 小时降至 1 小时内"）
- 指标应覆盖业务价值（用户侧）和系统价值（运营侧）

> 填写示例见 `references/examples/prd-examples.md` §4.1 背景与目标

> 常见陷阱见 `references/examples/prd-examples.md` §4.1 背景与目标
---

### 4.2 用户与场景

**内容**：定义模块的目标用户、核心用户故事、关键使用场景。

**格式建议**：
- **目标用户**：1-2 段描述，包含用户角色（用户故事语法中的身份，非业务端）、职责、痛点
- **用户故事**：表格。列：故事编号 | 用户故事 | 优先级 | 对应功能编号。故事编号格式继承项目 PRD-顶层定义 §6（编码规则），统一格式为 `STORY-{模块缩写}-{业务端标识}-{序号}`（如 `STORY-USER-DEFAULT-001`）。**故事编号必须在项目 PRD-顶层定义 §6 编码规则中明确定义**。**多业务端项目应按业务端分开展示，每个业务端一个表格，禁止混排**

> **注意**：用户故事中的"角色"（如"As a 客服专员"）是具体用户身份，属于用户故事语法；业务端（如 CONSUMER/OPERATOR/ADMIN）是产物组织维度。两者不同，禁止混淆。
- **用户旅程**：流程图或步骤列表，标注痛点和机会点。**多业务端项目应按业务端分别描述旅程，避免混排**

**用户故事格式**（推荐）：
```
As a [用户角色], I want [目标], so that [收益]
```

> 填写示例见 `references/examples/prd-examples.md` §4.2 用户与场景

> 常见陷阱见 `references/examples/prd-examples.md` §4.2 用户与场景
---

### 4.3 功能需求

**内容**：以用户故事为索引，逐条定义功能需求。每个功能必须关联用户故事、业务规则和验收标准。

**格式建议**：表格。列：功能编号 | 故事编号 | 功能名称 | 优先级 | 交互方式 | 业务规则 | 验收标准。

> 交互方式描述功能以什么形态呈现（如页面、弹窗、API、后台任务），不写具体布局和样式。功能的埋点事件、入口位置、通知触发等价值触达需求，在交互方式列简要说明或附件中补充。

**字段规范**：
- 功能编号：格式继承项目 PRD-顶层定义 §2（全局功能范围）中的业务端标识分配和 §6（编码规则），统一格式为 `{模块缩写}-{业务端标识}-{序号}`（如 `ORDER-DEFAULT-001`、`ORDER-CONSUMER-001`），如无定义则询问用户补充
- 功能名称：动宾结构，建议不超过 10 字
- 优先级：P0（必须）/ P1（重要）/ P2（加分）
- 故事编号：填 §2 中的故事编号（推荐，如 `STORY-ORDER-DEFAULT-001`），或用户故事摘要
- 业务规则：填写对应 §6 的规则编号（如 `RULE-ORDER-DEFAULT-001`）
- 验收标准：填写对应 §8 的验收项编号（如 `ACC-ORDER-DEFAULT-001`）

> 填写示例见 `references/examples/prd-examples.md` §4.3 功能需求

> 常见陷阱见 `references/examples/prd-examples.md` §4.3 功能需求
---

### 4.4 非功能需求

**内容**：定义功能需求之外的系统属性要求，包括性能、安全、可扩展性、合规、可用性。

**格式建议**：表格。列：类型 | 需求描述 | 衡量标准 | 优先级。

> 填写示例见 `references/examples/prd-examples.md` §4.4 非功能需求

> 常见陷阱见 `references/examples/prd-examples.md` §4.4 非功能需求
---

### 4.5 数据模型

**内容**：模块涉及的所有业务实体及其字段定义。

**格式**：每个实体一张表，列：字段 | 业务类型 | 约束 | 业务说明。

**字段规范**：
- 字段名：使用业务语义命名（如"下单时间"而非"created_at"），同一文档内统一风格
- 业务类型：字符串 / 整数 / 布尔 / 日期时间 / JSON / 金额 / 文本 / 枚举
- 约束（业务语义）：必填 / 可选 / 唯一 / 关联{实体名} / 有默认值：{值} / ≥0 / >0

**边界**：不写数据库实际类型、索引策略、分表策略、SQL、审计字段（如 created_at/updated_at）。

> 填写示例见 `references/examples/prd-examples.md` §4.5 数据模型

> 常见陷阱见 `references/examples/prd-examples.md` §4.5 数据模型
---

### 4.6 业务规则

**内容**：状态流转、权限规则、计算逻辑、校验规则。权限规则引用 PRD-顶层定义 §2 全局功能范围 — 权限级别定义，在模块内细化各权限级别的具体操作范围。

**格式**：表格（规则编号 | 规则描述 | 触发条件 | 处理逻辑）或状态机图。

**字段规范**：规则编号：格式继承项目 PRD-顶层定义 §6（编码规则），如无定义则询问用户补充。

**边界**：不写代码实现、不写前端交互。

> 填写示例见 `references/examples/prd-examples.md` §4.6 业务规则

> 常见陷阱见 `references/examples/prd-examples.md` §4.6 业务规则
---

### 4.7 错误处理

**内容**：模块内所有错误码的定义。

**格式**：表格，列：错误码 | 触发场景 | 前端提示。

**字段规范**：
- 错误码格式继承项目 PRD-顶层定义 §6（编码规则），如无定义则询问用户补充
- 前端提示：用户友好的中文，建议不超过 20 字

**边界**：不写 HTTP 状态码、后端日志级别、异常类名。

> 填写示例见 `references/examples/prd-examples.md` §4.7 错误处理

> 常见陷阱见 `references/examples/prd-examples.md` §4.7 错误处理
---

### 4.8 验收标准

**内容**：每个功能的可测试标准。

**格式建议**：表格。列：验收项编号 | 功能编号 | 验收项 | 条件 | 操作 | 预期结果。每条验收项必须用 `[条件] + [操作] + [预期结果]` 句式描述。

**字段规范**：验收项编号：项目自定义编号格式。

**边界**：不写测试步骤、不写测试数据、不写性能指标。

> 填写示例见 `references/examples/prd-examples.md` §4.8 验收标准

> 常见陷阱见 `references/examples/prd-examples.md` §4.8 验收标准
---

### 4.9 依赖与范围

**内容**：声明模块的外部依赖、技术约束、业务假设，以及当前版本明确不做的范围。

**格式建议**：4 个子节，每个子节用列表或表格。

**子节**：
- **上游依赖**：本模块依赖的其他模块/服务/数据源，以及依赖的具体内容
- **技术约束**：已知的技术限制（如必须兼容 IE11、必须使用现有权限体系）
- **业务假设**：产品决策基于的假设（如"假设所有客户已有统一客户ID"），假设不成立时的影响
- **明确不做**：本版本明确排除的功能/场景，每项附理由

> 填写示例见 `references/examples/prd-examples.md` §4.9 依赖与范围

> 常见陷阱见 `references/examples/prd-examples.md` §4.9 依赖与范围
---

### 4.10 附件

**内容**：参考资料、原型链接、相关文档链接。

**格式**：自由文本。

**边界**：只放链接和简要说明。

> 常见陷阱见 `references/examples/prd-examples.md` §4.10 附件
---

## §5 格式规范

### 5.1 编号规范

功能编号、故事编号、规则编号、错误码、验收项编号统一继承项目 PRD-顶层定义 §2（全局功能范围）中的业务端标识分配和 §6（编码规则），统一格式为 `{模块缩写}-{业务端标识}-{序号}`。单业务端项目业务端标识固定为 `DEFAULT`，多业务端项目为实际业务端缩写（如 `CONSUMER`、`OPERATOR`）。如无定义，询问用户补充，禁止自行假设格式。

### 5.2 状态值规范

状态值格式继承项目 PRD-顶层定义 §5（状态值）。如无定义，询问用户补充，禁止自行假设格式。

### 5.3 表格规范

- 表头使用 `| 字段 |` 格式
- 同一文档内表格列头必须一致
- 禁止合并单元格

## §6 检查清单

> 按 **6 维度 × 3 层级** 组织：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试的完整追溯链

### 6.1 结构维度

- [ ] **L1** `[脚本]` §1–§10 章节齐全，无缺漏
- [ ] **L1** `[脚本]` 各章节表格格式正确（表头/分隔线/列数一致）
- [ ] **L1** `[脚本]` §3 功能需求表格列完整：编号/故事/名称/优先级/交互方式/技术单元/规则/验收
- [ ] **L1** `[脚本]` §5 数据模型每个实体包含字段/类型/约束/说明
- [ ] **L1** `[脚本]` §7 错误码表格每行包含错误码/触发场景/前端提示
- [ ] **L1** `[脚本]` 功能编号/故事编号/规则编号/错误码/验收项编号连续无重复
- [ ] **L1** `[脚本]` 上游文档声明完整，路径可解析
- [ ] **L2** `[脚本]` PRD 版本号与下游产物声明的 PRD 版本一致
- [ ] **L2** `[脚本]` PRD 覆盖功能点列表与 §3 实际功能点一致
- [ ] **L3** `[脚本]` 禁止同一编号在不同产物中定义不同含义

### 6.2 数据维度

- [ ] **L1** `[脚本]` §3 功能编号/故事编号前缀与 PRD-顶层定义 §6 编码规则一致
- [ ] **L1** `[模型]` §6 权限规则引用 PRD-顶层定义 §2 全局功能范围 — 权限级别定义，未将权限级别拆分为独立业务端
- [ ] **L1** `[脚本]` §5 枚举值在顶层定义或本模块中声明
- [ ] **L1** `[模型]` 字段名统一使用业务术语，不混用技术术语
- [ ] **L1** `[模型]` §3/§5/§6 中首次出现的业务术语标注来源
- [ ] **L2** `[模型]` §5 数据模型实体/字段名与技术方案 §3 表结构可映射
- [ ] **L2** `[模型]` §5 数据模型字段类型与技术方案 §3 字段数据库类型可映射
- [ ] **L2** `[脚本]` §5 枚举值与技术方案 §3 字段枚举值一致
- [ ] **L2** `[模型]` §5 字段约束在技术方案中有对应实现（NOT NULL/UNIQUE/INDEX 等）
- [ ] **L2** `[脚本]` PRD-顶层定义 §4 术语表中的每个术语在技术-顶层定义 §4 公共表定义 — 通用字段命名约定和交互-顶层定义 §4 交互规范 — 主导航结构中有对应使用
- [ ] **L3** `[脚本]` 同一术语在 PRD/交互/UI/技术/测试 中名称一致

### 6.3 状态维度

- [ ] **L1** `[脚本]` §6 业务规则中的状态值在 §5 或顶层定义中声明
- [ ] **L1** `[模型]` §6 状态流转有明确的触发条件和目标状态
- [ ] **L1** `[模型]` §6 每个规则覆盖至少一个状态流转或权限判断
- [ ] **L2** `[模型]` §6 状态值与交互设计 §4.4 状态机的业务状态列一致
- [ ] **L2** `[模型]` §6 状态值与技术方案 §5 状态机状态值一致

### 6.4 错误维度

- [ ] **L1** `[脚本]` §7 每个错误码在 §6 业务规则中有对应触发场景
- [ ] **L1** `[模型]` §7 错误码触发场景具体可执行（非"用户操作时"）
- [ ] **L2** `[脚本]` §7 错误码格式与 PRD-顶层定义 §6 编码规则一致
- [ ] **L2** `[脚本]` §7 每个错误码在技术方案 §7 异常处理中有对应映射
- [ ] **L2** `[脚本]` §7 每个错误码在测试 §2 异常测试用例中有对应验证
- [ ] **L3** `[脚本]` PRD §7 定义的错误码数量 = Tech §7 实现的错误码数量 = Test §2 验证的错误码数量

### 6.5 性能维度

- [ ] **L1** `[脚本]` §4 非功能需求覆盖性能/安全维度
- [ ] **L1** `[模型]` §4 性能指标可量化（有数字和阈值）
- [ ] **L1** `[模型]` §4 安全需求具体可执行（非"保证数据安全"）
- [ ] **L2** `[模型]` §4 性能指标在技术方案 §8 中有对应性能目标
- [ ] **L2** `[模型]` §4 性能指标在测试 §3 中有对应性能测试项
- [ ] **L2** `[模型]` §4 安全需求在技术方案 §10 中有对应安全措施
- [ ] **L2** `[模型]` §4 安全需求在测试 §4 中有对应安全测试项

### 6.6 安全维度

- [ ] **L1** `[模型]` §4 安全需求不过于笼统
- [ ] **L2** `[模型]` §4 安全需求→技术方案→测试 全链路可追溯

### 6.7 端到端链路

- [ ] **L3** `[模型]` §2 每个用户故事在 §3 中有至少一个功能点
- [ ] **L3** `[脚本]` §3 每个 P0 功能点有对应的验收标准（§8）
- [ ] **L3** `[脚本]` §3 每个 P0 功能点在下游有对应实现（交互页面/技术接口/测试用例）
- [ ] **L3** `[脚本]` §8 每条验收标准对应 §3 一个功能点，且在测试 §1/§6 中有对应用例

### 6.8 版本与边界

- [ ] **L2** `[脚本]` 版本号、制定日期、作者信息完整
- [ ] **L2** `[模型]` 检查上游顶层定义在本产物及关联顶层定义中的完整映射，发现缺失标记为阻塞性差异
- [ ] **L1** `[模型]` 产物中无不属于 PRD 范畴的交互/视觉/技术描述
