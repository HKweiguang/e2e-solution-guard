# 想法评估与方案融合分析

当用户提出一个新想法、需求变更或功能提议时，**不要直接开始输出产物或改代码**。先执行评估分析，判断这个想法与现有方案的兼容性、影响范围和可行性。

---

## 触发条件

- 用户提出"我想做一个..."、"能不能加..."、"如果改成..."
- 用户询问"这个想法可行吗"、"和现有方案冲突吗"
- 用户给出模糊需求，需要先分析再细化
- 用户要求"帮我看看这个需求怎么落地"

## 执行步骤

### 1. 加载现有方案

- 读取项目 AGENTS.md 了解项目背景
- 搜索项目内相关产物（PRD、技术方案、顶层定义等）
- 定位可能受影响的模块和上下游产物
- 如无现有产物，明确告知用户当前为"从 0 开始"，并建议先建立顶层定义

### 2. Gap 分析

将用户的想法与现有方案进行比对，识别：

- **信息缺口**：用户想法中缺失的关键信息（边界条件、权限规则、异常场景、数据模型、状态流转等）
- **功能重叠**：与现有功能是否重复、冲突或可以复用
- **范围蔓延**：是否超出了当前模块/产品的职责边界
- **技术风险**：是否需要引入新技术、新依赖，或涉及复杂架构调整

### 3. 影响范围评估

| 维度 | 分析内容 |
|------|---------|
| **直接影响** | 需要新建或修改哪些模块级产物（PRD、技术方案等） |
| **顶层影响** | 是否需要调整全局术语、状态值、错误码、模块划分 |
| **下游影响** | 对交互设计、UI 设计、技术方案、测试计划的连锁变更 |
| **依赖影响** | 是否需要其他团队/服务配合 |

### 4. 输出评估结论

评估报告必须包含：

- **可行性判断**：可行 / 需调整后可行 / 当前不可行（说明原因）
- **融合方式**：
  - **独立新增**：作为新模块，不影响现有功能
  - **扩展现有**：在现有模块上增加功能点
  - **替换现有**：废弃旧功能，用新方案替代（需评估迁移成本）
- **缺失信息清单**：必须补全的关键信息，列成问题清单询问用户
- **建议后续步骤**：
  - 信息完备 → 进入产物生成流程
  - 信息缺失 → 先补充需求，再进入产物流程
  - 技术不确定 → 建议先输出技术预研产物

## 关键原则

- **禁止直接输出完整方案**：未经 gap 分析确认，禁止基于模糊想法输出完整 PRD 或技术方案
- **必须引用现有方案**：评估结论必须与现有产物中的具体章节/功能点/规则进行比对，禁止脱离上下文的抽象判断
- **明确标注不确定性**：对于无法从现有产物中确认的部分，标注为「待确认」，不要假设
