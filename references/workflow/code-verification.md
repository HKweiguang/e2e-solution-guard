# 代码落地验证（产物→代码的闭环）

技术方案产物生成并通过检查（产物生成流程阶段 3）后，代码实现完成即触发本阶段。本阶段是产物与代码之间的最后一道锁，防止"产物完美、代码走样"。

---

## 触发条件

- 用户明确告知"代码已实现"或"请检查代码与产物是否一致"
- 主会话检测到代码文件发生变更且对应技术方案已存在

## 执行流程

| 顺序 | 动作 | 说明 |
|------|------|------|
| 12 | 定位代码文件 | 按项目技术-顶层定义 §2.2 的工程结构，定位当前服务/模块对应的代码文件（controller、service、entity/ORM、DAO、路由配置等） |
| 13 | 启动审计 | 加载：当前技术方案产物、技术方案 §1 声明的所有上游 PRD 产物（一个服务可能对应多个 PRD）、已定位的代码文件 |
| 14 | 执行七项对比 | 按下方「检查维度」逐项对比产物定义与代码实现 |
| 15 | 产出审计报告 | 生成《产物-代码一致性审计报告》（参见 `references/steps/code-audit-report.md`） |
| 16 | 判定结果 | **全部通过** → 阶段 5 完成，继续后续步骤<br>**存在差异** → 输出差异清单，向用户呈现，等待用户决定改代码或改产物 |
| 17 | 修正后复验 | 修正完成后**重新执行审计**从第 13 步执行，禁止主会话自行复判 |

## 检查维度

| 维度 | 产物侧 | 代码侧 | 判定标准 |
|------|--------|--------|---------|
| **功能点映射一致性** | PRD §3 功能需求（功能编号、技术实现单元）+ 技术方案 §1 覆盖功能点声明 + 技术方案 §4 接口的功能点标注 | 代码中的接口/方法注释或路由配置中的功能点标注 | PRD 中标注了该技术模块的每个功能点，在技术方案 §4 中都有对应接口，在代码中都有实现；无遗漏也无多余 |
| **接口一致性** | 技术方案接口设计章节（§4）及其接口汇总表（URL、方法、请求参数、响应结构、错误码） | controller/API 路由/方法签名/参数对象/响应对象 | 路径、方法、字段名、类型、必填性一致 |
| **数据模型一致性** | 技术方案数据模型章节（§3）（字段、类型、约束、审计字段） | entity/ORM/建表语句/迁移脚本 | 字段名、类型、约束、审计字段一致 |
| **错误码一致性** | PRD §7 错误码定义 | 代码中抛出的异常/返回的错误码 | 错误码编码、触发场景一致，无新增未定义错误码 |
| **权限一致性** | PRD §6 权限规则 | 代码中的权限注解/拦截器/角色判断 | 权限层级（公开/登录/角色）一致 |
| **业务规则一致性** | 技术方案核心流程章节（§6） | 代码中的核心业务流程实现 | 流程步骤、状态流转、触发条件一致 |
| **前端/UI 一致性** | UI 设计 HTML 原型（组件结构、状态类名、CSS 变量、响应式断点） | 前端代码（组件模板、样式文件、状态类实现） | 组件结构、状态类名（.empty/.error/.success/.skeleton）、CSS 变量引用、响应式断点一致 |

## 关键规则

- 审计执行 **只读**，禁止修改任何代码或产物
- 发现差异时必须明确标注：差异项、产物原文、代码原文、建议修正方向（改代码/改产物）
- 代码中的字段命名可能因语言规范与产物中的 snake_case/camelCase 不同，此类风格差异不视为不一致，但业务含义必须一致
- 代码中合理的内部辅助字段/方法（不暴露给外部）不纳入一致性检查范围
