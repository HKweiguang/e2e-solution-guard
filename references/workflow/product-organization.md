# 产物组织方式

本文件定义 e2e-solution-guard 的推荐目录结构、产物聚合单位、以及 PRD 与技术方案的映射关系。

> 适用场景：项目初始化时确定产物目录结构，或 AI 自动推断产物组织方式时参考。

---

## 聚合单位

各步骤的聚合单位不同，目录结构应与产物的天然边界一致：

| 产物 | 聚合单位 | 说明 |
|------|---------|------|
| PRD、测试报告 | **按业务模块聚合** | 同一模块的需求和验证放在一起，便于业务闭环追溯 |
| 交互设计 | **按业务端聚合**（推荐） | 同一业务端的交互体验是连贯的，跨模块。如"消费者"的所有交互放在同一目录下 |
| UI | **按业务端+设备聚合**（推荐） | 同一业务端+设备的视觉风格一致，跨模块。如"消费者-web"的所有页面放在同一目录下 |
| 技术方案 | **按技术实现边界聚合**（服务/模块/技术域） | 技术方案的边界是代码的物理边界，不是 PRD 的业务边界 |

> **注意**：交互设计和 UI 的聚合维度由项目实际情况决定。如果团队没有专职交互/UI 设计师，由产品经理兼任，也可以按模块聚合。Skill 提供参考，不强制。

---

## 目录示例

### 方案一：全部按模块聚合（简单项目）

所有产物放在模块目录下，最直观，适合小团队或单人项目：

```
用户管理/
├── {PRD 标题}.md                # 如：PRD-v1-用户管理.md
├── {交互设计标题}.md            # 如：交互设计-v1-消费者-用户管理.md
├── {UI 标题}.html               # 如：UI设计-v1-消费者-web-用户管理.html
└── {测试报告标题}.md            # 如：TEST-v1-用户管理.md
```

### 方案二：按产物类型分目录（推荐）

不同产物按各自的天然边界分目录存放：

**PRD 和测试按模块：**
```
需求/
├── {PRD 标题}.md               # 如：PRD-v1-用户管理.md
└── {PRD 标题}.md               # 如：PRD-v1-订单管理.md

测试/
├── {测试报告标题}.md           # 如：TEST-v1-用户管理.md
└── {测试报告标题}.md           # 如：TEST-v1-订单管理.md
```

**交互设计按业务端：**
```
交互设计/
├── 消费者/
│   ├── {交互设计标题}.md       # 如：交互设计-v1-用户管理.md
│   └── {交互设计标题}.md       # 如：交互设计-v1-订单管理.md
└── 商家/
    └── {交互设计标题}.md       # 如：交互设计-v1-订单管理.md
```

**UI 按业务端+设备：**
```
UI/
├── 消费者-web/
│   ├── {UI 标题}.html          # 如：UI设计-v1-消费者-web-用户管理.html
│   └── {UI 标题}.html          # 如：UI设计-v1-消费者-web-订单管理.html
└── 消费者-app/
    └── {UI 标题}.html          # 如：UI设计-v1-消费者-app-用户管理.html
```

**技术方案按服务：**
```
技术/
├── {技术方案标题}.md           # 如：TECH-v1-user-service.md
└── {技术方案标题}.md           # 如：TECH-v1-order-service.md
```

### 方案三：混合结构（业务按模块、交互/UI 按业务端、技术按服务）

最常用的结构，兼顾业务闭环追溯和各专业视角。文件名由文件自身的标题决定，以下为示例：

```
docs/
├── 顶层定义/
│   ├── {PRD-顶层定义}.md
│   ├── {交互-顶层定义}.md
│   ├── {UI-顶层定义}.md
│   └── {技术-顶层定义}.md
│
├── 需求/                        # PRD 按模块聚合
│   ├── {PRD 标题}.md
│   └── {PRD 标题}.md
│
├── 交互设计/                     # 按业务端聚合
│   ├── 消费者/
│   │   ├── {交互设计标题}.md
│   │   └── {交互设计标题}.md
│   └── 商家/
│       └── {交互设计标题}.md
│
├── UI/                          # 按业务端+设备聚合
│   ├── 消费者-web/
│   │   ├── {UI 标题}.html
│   │   └── {UI 标题}.html
│   └── 消费者-app/
│       └── {UI 标题}.html
│
├── 测试/                        # 按模块聚合
│   ├── {测试报告标题}.md
│   └── {测试报告标题}.md
│
└── 技术/                        # 按服务聚合
    ├── {技术方案标题}.md
    ├── {技术方案标题}.md
    └── {技术方案标题}.md
```

## PRD 与技术方案的映射关系

PRD（业务视角）与技术方案（实现视角）是**多对多映射**，通过功能点编号建立关联：

```
PRD 用户管理模块 ──┬──→  TECH-v1-user-service（技术方案）
                  │
PRD 订单管理模块 ──┼──→  TECH-v1-order-service
                  │        ↑
PRD 权限管理模块 ──┘        │
                     TECH-v1-payment-service
                     TECH-v1-notification-service（公共）
```

- **PRD → 技术方案：一对多**。一个 PRD 功能可能涉及多个服务（如"提交订单"涉及 order-service + payment-service + inventory-service）
- **技术方案 → PRD：一对多**。一个服务支撑多个 PRD 模块（如 user-service 被用户管理、权限管理、订单管理共用）
- 目录结构只负责聚合同类产物，真正的关联关系由产物内容中的"功能点编号"表达

## 组织原则

- **业务端由项目自定义**，Skill 提供框架，不预设具体业务端。所有项目统一使用带业务端标识的编号格式，只有一个业务端时标识为 `DEFAULT`
- **交互设计按业务端聚合**，UI 按业务端+设备聚合。只有一个业务端时，目录名可使用 `DEFAULT` 或省略业务端层级（由项目自定）
- **技术方案按技术实现边界聚合**（单体按代码模块，微服务按服务），通过上游文档表格声明该技术方案支撑了哪些业务模块的功能点
- **某个业务端/设备没有独立交互/UI 时，不产该文件**
- **目录名可自定义**：`需求/` 可改为 `prd/`、`product/`，`交互设计/` 可改为 `interaction/`，`技术/` 可改为 `tech/`，不强制

## 编号示例

| 编号类型 | 格式 | 示例 |
|---------|------|------|
| 功能编号 | `{模块缩写}-{业务端标识}-{序号}` | `USER-DEFAULT-001`、`ORDER-CONSUMER-001` |
| 页面编号 | `PAGE-{业务端标识}-{平台}-{页面名}-{序号}` | `PAGE-DEFAULT-WEB-HOME-001`、`PAGE-CONSUMER-APP-PROFILE-001` |
