# 产物组织方式

本文件定义 e2e-solution-guard 的推荐目录结构、产物聚合单位、以及 PRD 与技术方案的映射关系。

> 适用场景：项目初始化时确定产物目录结构，或 AI 自动推断产物组织方式时参考。

---

## 聚合单位

各步骤的聚合单位不同，目录结构应与产物的天然边界一致：

| 产物 | 聚合单位 | 说明 |
|------|---------|------|
| PRD、交互设计、UI、测试报告 | **按业务模块聚合**（推荐） | 同一模块的产物放在一起，便于业务闭环追溯 |
| 技术方案 | **按技术实现边界聚合**（服务/模块/技术域） | 技术方案的边界是代码的物理边界，不是 PRD 的业务边界。无论单体还是微服务，一个技术方案可支撑多个业务模块，一个业务模块也可涉及多个技术方案 |

---

## 目录示例

### 按模块聚合（PRD、交互、UI、测试）

**单角色项目**（最常见，如纯 C端）：
```
用户管理/
├── prd.md                  # 统一 PRD，包含所有功能和实体定义
├── 交互设计.md              # 交互设计
├── ui.html                 # UI 视觉稿
└── 测试报告.md
```

**多角色项目**（如 C端 + 管理后台）：
```
用户管理/
├── prd.md                  # 统一 PRD，包含所有角色的功能点
├── 交互设计-{角色A}.md      # 按角色分
├── 交互设计-{角色B}.md
├── ui-{角色A}-web.html     # 按角色+设备分
├── ui-{角色A}-app.html
├── ui-{角色B}-web.html
├── ui-{角色B}-app.html
└── 测试报告.md
```

### 按技术实现边界聚合（技术方案）

单体应用（按代码模块/技术域分）：
```
tech/
├── user-module.md           # 用户模块技术方案，支撑用户管理、权限管理等多个业务模块
├── trade-module.md          # 交易模块技术方案，包含订单+支付
└── message-module.md        # 消息模块技术方案
```

微服务架构（按服务分）：
```
tech/
├── user-service.md          # 用户服务，支撑用户管理、权限管理、订单管理等多个业务模块
├── order-service.md         # 订单服务
├── payment-service.md       # 支付服务
└── notification-service.md  # 通知服务
```

## 完整的项目目录示例

以下是一个**多角色 + 微服务**项目的完整产物目录：

```
docs/
├── top-level/
│   ├── prd-top-level.md           # 项目 PRD-顶层定义
│   ├── interaction-top-level.md   # 项目交互-顶层定义
│   ├── ui-top-level.md            # 项目 UI-顶层定义
│   └── tech-top-level.md          # 项目技术-顶层定义
│
├── modules/                        # 业务模块产物（PRD、交互、UI、测试）按模块聚合
│   ├── 用户管理/
│   │   ├── prd.md
│   │   ├── 交互设计-消费者.md
│   │   ├── 交互设计-商家.md
│   │   ├── ui-消费者-web.html
│   │   ├── ui-消费者-app.html
│   │   └── 测试报告.md
│   ├── 订单管理/
│   │   ├── prd.md
│   │   ├── 交互设计-消费者.md
│   │   ├── ui-消费者-web.html
│   │   └── 测试报告.md
│   └── 权限管理/
│       ├── prd.md
│       ├── 交互设计-管理后台.md
│       └── ui-管理后台-web.html
│
└── tech/                           # 技术方案按服务聚合
    ├── user-service.md             # 支撑用户管理、权限管理、订单管理
    ├── order-service.md            # 支撑订单管理
    ├── payment-service.md          # 支撑订单管理
    └── notification-service.md     # 公共，被多个模块共用
```

## PRD 与技术方案的映射关系

PRD（业务视角）与技术方案（实现视角）是**多对多映射**，通过功能点编号建立关联：

```
PRD 用户管理模块 ──┬──→  user-service（技术方案）
                  │
PRD 订单管理模块 ──┼──→  order-service
                  │        ↑
PRD 权限管理模块 ──┘        │
                     payment-service
                     notification-service（公共）
```

- **PRD → 技术方案：一对多**。一个 PRD 功能可能涉及多个服务（如"提交订单"涉及 order-service + payment-service + inventory-service）
- **技术方案 → PRD：一对多**。一个服务支撑多个 PRD 模块（如 user-service 被用户管理、权限管理、订单管理共用）
- 目录结构只负责聚合同类产物，真正的关联关系由产物内容中的"功能点编号"和"技术实现单元"表达

## 组织原则

- **角色由项目自定义**，Skill 提供框架，不预设具体角色
- **单角色项目保持简单**：无需角色后缀，编号不带角色标识
- **多角色项目**：一个模块一个 PRD，内部按角色分章节声明功能点
- **交互设计按角色分文件**，同一角色内不同设备可合并或拆分
- **UI 按角色+设备分**（一个 HTML 只展示一种设备的界面）
- **技术方案按技术实现边界聚合**（单体按代码模块，微服务按服务），通过上游文档表格声明该技术方案支撑了哪些业务模块的功能点
- 某个角色/设备没有独立交互/UI 时，不产该文件
