# 产物-代码一致性审计报告模板

> 定位：代码实现完成后，对比代码与技术方案/PRD 的一致性，产出审计结果。
>
> 本报告是产物与代码之间的最后一道锁。
>
> **注意**：本模板为审计报告专用，结构与步骤模板（§1–§6）不同，允许独立结构。

---

## §1 文件头部格式

```markdown
# AUDIT-{服务名/模块名}-v{主版本号}

**审计对象**：TECH-v{主版本号}-{服务名}
**代码版本**：{Git commit hash 或版本标签}
**审计日期**：YYYY-MM-DD
**审计者**：AI 审计（执行方式取决于模型上下文）

**上游文档**：
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| TECH-v{主版本号}-{服务名} | 审计依据 | 全部章节 |
| PRD-v{主版本号}-{模块名A} | 审计依据 | §3 功能需求（功能编号）、§6 业务规则、§7 错误处理 |
| PRD-v{主版本号}-{模块名B} | 审计依据 | §3 功能需求（功能编号）、§6 业务规则、§7 错误处理 |
```

## §2 章节结构

| 章节 | 说明 |
|------|------|
| §1 审计摘要 | 通过/不通过、差异总数、关键风险 |
| §2 功能点映射一致性 | 逐项对比 PRD §3 功能需求（功能编号）+ 技术方案 §1 覆盖功能点声明 + 技术方案 §4 接口的功能点标注与代码实现 |
| §3 接口一致性 | 逐项对比技术方案接口设计章节（§4）及其接口汇总表与代码路由/方法 |
| §4 数据模型一致性 | 逐项对比技术方案数据模型章节（§3）与代码表结构 |
| §5 错误码一致性 | 逐项对比 PRD 错误处理章节（§7）与代码抛出异常 |
| §6 业务规则一致性 | 逐项对比 PRD 业务规则章节（§6）与代码实现：状态流转、权限控制、计算逻辑、校验规则 |
| §7 核心流程一致性 | 逐项对比技术方案核心流程章节（§6）与代码核心流程 |
| §8 前端/UI 一致性 | 逐项对比 UI 设计 HTML 原型与前端代码实现 |
| §9 性能一致性 | 逐项对比技术方案 §8 性能目标与代码实际配置（缓存） |
| §10 安全一致性 | 逐项对比技术方案 §10 安全设计中的后端安全措施（SQL注入防护、输入过滤）与代码实现 |
| §11 高可用一致性 | 逐项对比技术方案 §9 高可用设计（熔断、降级、限流、重试、超时）与代码实现 |
| §12 差异清单与建议 | 所有不一致项汇总、建议修正方向 |

## §3 各章节格式要求

### 3.1 审计摘要

**内容**：整体结论、差异统计、是否存在阻塞性差异。
**格式**：
- 结论：**通过** / **不通过**
- 差异总数：N 项（其中阻塞性 M 项，警告性 P 项）
- 关键风险：如存在阻塞性差异，列出最严重的前 3 项

**判定标准**：
- **通过**：九项检查均无阻塞性差异，警告性差异 ≤3 项且已标注
- **不通过**：存在任一阻塞性差异，或警告性差异 >3 项

**阻塞性差异定义**：
- PRD 中标注了该技术模块的功能点，在技术方案 §4 中无对应接口，或在代码中无实现
- 技术方案 §1 声明覆盖的功能点在 PRD §3 中无定义
- 接口缺失或路径/方法不匹配
- 数据模型字段缺失或类型不兼容
- 错误码未按 PRD 定义实现
- 权限控制低于 PRD 要求（如应登录的接口未加登录校验）
- 代码中存在 PRD 未定义的新错误码
- 业务规则关键步骤缺失或顺序错误
- 前端组件状态类名与 UI 设计不一致
- CSS 变量未按 UI-顶层定义实现
- 接口超时配置未按技术方案 §9 落实
- 缓存 Key/TTL 未按技术方案 §8 落实
- 限流阈值未按技术方案 §9 落实
- SQL 注入防护措施未落实（如缺少预编译语句）
- 输入过滤/输出转义未实现
- 敏感数据接口未脱敏

**警告性差异定义**：
- 字段顺序与产物不同
- 缺少非必填字段的默认值说明
- 注释与产物描述有细微偏差但不影响行为

### 3.2 功能点映射一致性

**内容**：PRD §3 功能需求（功能编号）+ 技术方案 §1 覆盖功能点声明 + 技术方案 §4 接口的功能点标注与代码实现的对照。
**格式**：表格，列：

| 功能编号 | 技术方案接口 | 代码实现 | 状态 |
|---------|-------------|---------|------|
| `{功能编号}` | `{API-001}` | `{Controller.java:45}` | 通过 |

**状态**：通过 / 警告 / 阻塞

**阻塞性差异**：
- PRD 中标注了该技术模块的功能点，在技术方案 §4 中无对应接口
- 技术方案 §1 声明覆盖的功能点在 PRD §3 中无定义
- PRD 中标注了该技术模块的功能点，在代码中无实现

---

### 3.3 接口一致性

**内容**：技术方案接口设计章节（§4）及其接口汇总表中定义的每个接口与代码实现的对照。
**格式**：表格，列：

| 接口编号 | 产物定义（URL/方法） | 代码实现（路由/方法） | 请求参数一致 | 响应结构一致 | 错误码一致 | 状态 |
|---------|---------------------|---------------------|-------------|-------------|-----------|------|
| API-001 | `POST /api/orders` | `POST /api/orders` | ✅ | ✅ | ✅ | 通过 |

**状态**：通过 / 警告 / 阻塞

### 3.4 数据模型一致性

**内容**：技术方案数据模型章节（§3）中定义的每张表与代码 entity/ORM 的对照。
**格式**：每张表一张对比表格，列：

| 字段名 | 产物类型 | 产物约束 | 代码字段名 | 代码类型 | 代码约束 | 一致 |
|--------|---------|---------|-----------|---------|---------|------|
| `order_id` | BIGINT | PK, NOT NULL | `orderId` | Long | `@Id` | ✅ |

**审计规则**：
- 字段名因语言命名规范不同（snake_case vs camelCase）不视为不一致
- 但业务含义必须一致，禁止产物中是 `user_id` 代码中是 `customer_id`
- 审计字段（创建时间/更新时间/创建人/更新人）必须存在

### 3.5 错误码一致性

**内容**：PRD 错误处理章节（§7）定义的错误码与代码中实际抛出的异常/返回码对照。
**格式**：表格，列：

| 错误码 | PRD 触发场景 | 代码中使用位置 | 使用场景与 PRD 一致 | 状态 |
|--------|-------------|---------------|-------------------|------|
| `{错误码}` | 订单不存在 | `OrderService.java:45` | ✅ | 通过 |

**额外检查**：
- 代码中是否存在 PRD 未定义的新错误码？如有，列出并标注为阻塞
- PRD 定义的错误码是否在代码中都有使用？如无，列出并标注为警告

### 3.6 业务规则一致性

**内容**：PRD 业务规则章节（§6）中定义的全部规则与代码实现对照。覆盖四类规则：状态流转、权限控制、计算逻辑、校验规则。
**格式**：按规则类型分组，每组一个表格，列：

| 规则编号 | 规则类型 | PRD 规则描述 | 代码实现位置 | 代码实现摘要 | 一致 | 状态 |
|---------|---------|-------------|-------------|-------------|------|------|
| `RULE-ORDER-DEFAULT-001` | 状态流转 | 订单支付后状态变为「已支付」 | `OrderService.pay()` | 第 42 行更新状态为 `PAID` | ✅ | 通过 |
| `RULE-ORDER-DEFAULT-002` | 权限控制 | 仅管理员可取消订单 | `OrderController.cancel()` | `@RequireRole(ADMIN)` | ✅ | 通过 |
| `RULE-ORDER-DEFAULT-003` | 计算逻辑 | 订单满 100 免运费 | `OrderService.calcShipping()` | 第 28 行判断 `amount >= 100` | ✅ | 通过 |
| `RULE-ORDER-DEFAULT-004` | 校验规则 | 手机号格式必须为 11 位数字 | `UserService.validatePhone()` | 正则 `^1[3-9]\d{9}$` | ✅ | 通过 |

**阻塞性差异**：
- PRD 定义的任一规则在代码中无实现
- 代码实现与 PRD 规则语义矛盾（如 PRD 要求满100免运费，代码实现为满200）
- 权限控制低于 PRD 要求（如 PRD 要求角色权限，代码只做了登录校验）

### 3.7 核心流程一致性

**内容**：技术方案核心流程章节（§6）与代码实现对照。
**格式**：按流程步骤列表对比：

| 步骤 | 产物定义 | 代码实现 | 一致 | 状态 |
|------|---------|---------|------|------|
| 1. 校验库存 | 查询库存表，判断 `stock > 0` | `InventoryService.checkStock()` | ✅ | 通过 |

**阻塞性差异**：关键步骤缺失、执行顺序错误、触发条件不匹配

### 3.8 前端/UI 一致性

**内容**：UI 设计 HTML 原型（组件结构、状态类名、CSS 变量、响应式断点）与前端代码实现对照。
**格式**：表格，列：

| 组件/页面 | UI 设计定义 | 前端代码实现 | 一致 | 状态 |
|-----------|------------|-------------|------|------|
| 提交按钮 | `.btn` + `:hover` + `:disabled` + `.loading` | `Button.vue` 中对应类名和伪类 | ✅ | 通过 |

**阻塞性差异**：组件结构缺失、状态类名未实现、CSS 变量未引用 UI-顶层定义 Token

### 3.9 性能一致性

**内容**：逐项对比技术方案 §8 性能与扩展性中定义的性能目标（缓存策略、容量规划）与代码实际配置。
**格式**：表格，列：

| 性能项 | 技术方案目标 | 代码配置 | 一致 | 状态 |
|--------|-------------|---------|------|------|
| 缓存 TTL | `ticket:detail:{id} = 5分钟` | `Redis TTL=300s` | ✅ | 通过 |
| 容量规划 | 预计 1 年 1000 万条，分 64 表 | 单表无分表 | ❌ | 阻塞 |

**阻塞性差异**：缓存策略未落实、容量规划未按方案执行

### 3.10 安全一致性

**内容**：逐项对比技术方案 §10 安全设计中的后端安全措施与代码实现。
**格式**：表格，列：

| 安全措施 | 技术方案要求 | 代码实现 | 一致 | 状态 |
|---------|-------------|---------|------|------|
| SQL 注入防护 | 使用预编译语句 | `MyBatis #{}` | ✅ | 通过 |
| 输入过滤 | XSS 过滤 + 输出转义 | `HtmlUtils.escapeHtml()` | ✅ | 通过 |
| 敏感数据脱敏 | 手机号/邮箱脱敏 | `MaskUtils.maskPhone()` | ✅ | 通过 |

**阻塞性差异**：SQL 注入防护缺失、输入校验未落实、敏感数据未脱敏

### 3.11 高可用一致性

**内容**：逐项对比技术方案 §9 高可用设计（熔断、降级、限流、重试、超时）与代码实现。
**格式**：表格，列：

| 高可用项 | 技术方案要求 | 代码实现 | 一致 | 状态 |
|---------|-------------|---------|------|------|
| 接口超时 | `timeout=200ms` | `timeout=5000ms` | ❌ | 阻塞 |
| 限流 | `QPS=1000` | 未配置 | ❌ | 阻塞 |
| 熔断 | 错误率 > 50% 触发 | `CircuitBreaker` 已配置 | ✅ | 通过 |

**阻塞性差异**：超时配置未落实、限流阈值未配置、熔断/降级/重试策略缺失

### 3.12 差异清单与建议

**内容**：所有不一致项的汇总，按阻塞性/警告性分类。
**格式**：

#### 阻塞性差异
1. **{差异简述}**
   - 产物：{产物原文引用}
   - 代码：{代码原文引用}
   - 建议：改代码 / 改产物
   - 影响：{如果不修正会导致什么问题}

#### 警告性差异
1. **{差异简述}**
   - 说明：{偏差描述}
   - 建议：{可选修正}

## §4 检查清单

> 按 **6 维度 × 3 层级** 组织：
> - **L1 单产物内**：审计报告自身的结构和内容完整性
> - **L2 相邻产物间**：审计发现与技术方案/PRD/UI 的一致性
> - **L3 端到端链路**：从需求到代码的完整追溯链

### 4.1 结构维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §1 审计结论明确标注通过/不通过 |
| L1 | `[脚本]` §3.12 所有差异项都有明确建议修正方向（改代码/改产物） |
| L1 | `[脚本]` 阻塞性差异数量为 0 且警告性差异 ≤3 项已标注时方可判定为通过 |
| L2 | `[脚本]` §2 覆盖 PRD §3 中标注了该技术模块的全部功能点，以及技术方案 §1 声明的全部覆盖功能点 |
| L3 | `[脚本]` 所有阻塞性差异修复后均经过回归验证并重新审计 |

### 4.2 数据维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §4 覆盖技术方案数据模型章节（§3）的全部表结构 |
| L2 | `[模型]` §4 表结构与数据库实际 schema 一致 |
| L3 | `[模型]` §4 字段类型、约束、索引在技术方案和代码实现中一致 |

### 4.3 状态维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §8 覆盖 UI 设计 HTML 原型中的全部组件结构、状态类名、CSS 变量和响应式断点 |
| L2 | `[模型]` §8 组件状态类名（.loading/.disabled/.empty 等）在代码中有对应实现 |

### 4.4 错误维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §5 覆盖 PRD 错误处理章节（§7）的全部错误码 |
| L2 | `[脚本]` §5 错误码在代码中有对应的异常处理和返回逻辑 |
| L1 | `[脚本]` §6 覆盖 PRD 业务规则章节（§6）的全部规则编号（状态流转/权限/计算逻辑/校验规则） |
| L2 | `[脚本]` §6 每个规则在代码中有对应实现位置 |
| L2 | `[模型]` §6 代码实现与 PRD 规则语义一致（非仅同名） |
| L3 | `[脚本]` §6 权限控制不低于 PRD 要求 |

### 4.5 性能维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §9 覆盖技术方案 §8 性能目标与代码配置（缓存、容量规划） |
| L2 | `[模型]` §9 缓存策略、数据库查询优化在代码中有对应实现 |

### 4.6 安全维度

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §10 覆盖技术方案 §10 后端安全措施与代码实现 |
| L2 | `[脚本]` §10 输入校验、鉴权、数据脱敏在代码中实现 |
| L3 | `[脚本]` §11 覆盖技术方案 §9 高可用设计（熔断、降级、限流、重试、超时）与代码实现 |

### 4.7 端到端链路

| 层级 | 检查项 | 类型 |
|------|--------|------|
| L1 | `[脚本]` §3 覆盖技术方案接口设计章节（§4）及其接口汇总表的全部接口 |
| L2 | `[脚本]` §7 覆盖技术方案核心流程章节（§6）的全部核心流程 |
| L3 | `[模型]` 从 PRD 功能点 → 技术方案接口 → 代码实现，链路完整可追溯 |
