# 文档生成与修改的通用流程

执行任何文档相关步骤前，按以下顺序建立上下文并执行。

---

## 阶段 1：前置检查

| 顺序 | 动作 | 说明 |
|------|------|------|
| 1 | 读取 AGENTS.md | 获取项目信息（文档组织方式、项目背景）。**如 AGENTS.md 未定义文档组织方式，执行自动推断**（见下方「文档组织方式自动推断」） |
| 2 | 搜索现有文档 | 在项目目录内搜索当前步骤的现有文档（通过文件名特征或文件头部的 upstream-document 表格识别）。搜索结果不明确时，询问用户确认 |
| 3 | 确定模式 | **未找到** → 进入「新建模式」<br>**已找到** → 进入「修改模式」 |

## 阶段 2：加载上下文（模式分支）

**新建模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4a | 加载顶层定义 | **先查找项目目录**内是否有当前步骤类型对应的顶层定义。<br>• **有** → 加载项目顶层定义，作为生成约束；**同步加载该顶层定义上游文档中声明的依赖**（如交互-顶层定义需加载 PRD-顶层定义）。<br>• **无** → 加载 skill 中的顶层定义模板 `references/top-level/*-top-level-template.md`；**同步加载该顶层定义模板头部 upstream-document 中声明的上游顶层定义模板**；询问用户是否基于该模板生成项目顶层定义。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 5a | 解析上游文档 | 根据用户描述的任务范围推理需要哪些上游输入，或按用户明确指定加载。**上游不固定写死在步骤模板中** |
| 6a | 加载上游文档 | 在项目目录内搜索并读取步骤 5a 确定的上游文档 |
| 7a | 加载步骤模板 | 读取 `references/steps/*-step.md` 作为文档骨架，**同步加载对应示例文件 `references/examples/*-example.md` 作为参照** |

**修改模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4b | 加载现有文档 | 读取找到的现有文档内容 |
| 5b | 加载顶层定义 | **先查找项目目录**内对应类型的顶层定义，检查是否有更新；**同步加载该顶层定义上游文档中声明的依赖**（如交互-顶层定义需加载 PRD-顶层定义）。如无项目版本，则加载 skill 顶层定义模板 `references/top-level/*-top-level-template.md`，**同步加载该模板头部声明的上游顶层定义模板**，询问用户是否生成项目顶层定义。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 6b | 加载上游文档 | 根据现有文档头部的 upstream-document 表逐项读取，检查是否有更新 |
| 7b | 分析修改要求 | 明确用户要求修改的具体范围和内容；**同步加载对应示例文件作为参照** |

## 阶段 3：影响评估、执行与检查

| 顺序 | 动作 | 模式 | 说明 |
|------|------|------|------|
| 8 | 上下游影响扫描 | 仅修改模式 | 按文档关联原则分层扫描本次修改对下游文档的影响范围。如当前修改的是顶层定义，需分层扫描所有下游顶层定义和步骤文档的引用 |
| 9 | 生成/执行修改 | 两种模式 | 新建模式：按模板结构输出完整文档<br>修改模式：按用户要求执行具体修改 |
| 10 | 完整性与一致性检查 | 两种模式 | **必须由独立只读 subagent 执行，主会话禁止执行任何检查。**<br><br>执行流程：<br>1. 主会话准备审计素材：修改模式生成 diff；新建模式确认待审计全文。**主会话禁止运行 `doc-audit.py` 或做任何语义判断**<br>2. 主会话判断拆分策略：用 `wc -l` 统计当前文档行数，按 `references/workflow/audit-procedure.md`「拆分决策流程」决定启动几个 subagent、每组的审计范围和 timeout<br>3. 启动只读 subagent（可并行），由 subagent 自行加载分配的文档章节 + 必要的上游/下游引用。单 subagent 只审计一个明确的任务单元<br>4. subagent **在同一个任务内同时完成机械检查与语义审计**：调用 `doc-audit.py` 输出机械检查结果，同时由模型逐项执行 `[模型]` 标记的语义检查项（参见 `references/workflow/audit-procedure.md`「语义审计执行方法」）<br>5. 主会话汇总所有 subagent 结果：合并 blocking issue、warning/hint，如有冲突则仲裁或启动复核 subagent<br>6. 根据汇总结果处理：无异常→通过；有问题→主会话修正后重新启动对应 subagent 复验 |
| 11 | 写入/更新文档头部 | 两种模式 | 新建模式：在文件头部写入实际加载的上游文档表<br>修改模式：如上游文档有更新或引用范围有变化，同步更新 upstream-document 表 |

## 文档组织方式自动推断

当 AGENTS.md 未定义文档组织方式时，AI 通过以下启发式规则自动推断，**不停止流程**：

1. **搜索常见文档目录**：检查项目根目录是否存在 `docs/`、`doc/`、`specs/`、`documents/`、`design/` 等目录
2. **搜索文件特征**：在项目目录内搜索文件名匹配 `PRD-*.md`、`TECH-*.md`、`交互设计-*.md`、`UI设计-*.md`、`测试报告-*.md` 的文件
3. **推断逻辑**：
   - 发现 `docs/prd/`、`docs/tech/` 等子目录 → 按步骤聚合
   - 发现 `docs/订单模块/`、`docs/用户模块/` 等子目录 → 按模块聚合
   - 未发现任何文档目录或特征文件 → 询问用户期望的组织方式（只问一次，不阻塞），同时默认使用项目根目录下的 `docs/` 作为文档根目录
4. **告知用户**：自动推断完成后，向用户说明推断结果，如"未在 AGENTS.md 中找到文档组织方式，已自动推断为按步骤聚合：docs/prd/、docs/tech/、docs/interaction/"

## 关键规则

- `references/top-level/*-top-level-template.md` 是 skill 提供的**顶层定义模板（template）**，不是项目顶层定义本身。生成项目顶层定义时，必须严格区分两类内容：
  - **元内容（仅供 AI 参考，禁止输出到项目文档）**：文件中的 `§1 章节结构`、`§2 各章节写作要求`、`§3 内容边界`、`§4 与模块文档的关系`、`§5 检查清单` 等段落。这些内容属于 Skill 内部写作指南，说明"顶层定义应该怎么写"，严禁复制到项目文档中。
  - **实际规范内容（必须填充并输出）**：根据用户提供的项目信息，按照模板对应的章节编号和标题，输出具体的产品概述、模块划分表、术语定义、状态值枚举、错误码规则、技术栈选型、工程结构等实际内容。
  - **判定标准**：生成的项目顶层定义中**不得出现**"章节结构"、"写作要求"、"内容边界"、"检查清单"、"格式："、"表格："、"树形结构"等指导性文字。每个章节必须是可直接被下游文档引用的**实际定义**，而非"告诉 AI 怎么写"的说明。
- 没有项目顶层定义时，必须先基于该模板生成项目版本，禁止以 skill 模板代替项目顶层定义继续后续步骤。
- 步骤模板 §1 中的 `upstream-document` 表格仅为**常见依赖示例**，实际生成时必须根据项目上下文动态判断，禁止机械套用。
- 如关键上游文档缺失，暂停当前步骤，提示用户先完成上游。
- 顶层定义缺失时**先询问用户**是否基于模板生成项目版本。用户确认 → 生成后继续；**用户拒绝/延后时**：
  - 若当前任务是**生成/修改顶层定义本身** → 可继续，因为目标就是创建顶层定义
  - 若当前任务是**生成/修改模块级下游文档** → 如用户明确指定了任务范围且提供了足够上下文，可继续一次，但必须在生成文档的 upstream-document 表格中标注 `[顶层定义待补全]`，并提醒用户在模块文档定稿前补全顶层定义
- 生成的文档头部必须包含真实的 `upstream-document` 表，反映本次实际加载的所有上游。
