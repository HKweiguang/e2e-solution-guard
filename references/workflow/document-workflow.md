# 产物生成与修改的通用流程

执行任何产物相关步骤前，按以下顺序建立上下文并执行。

---

## 阶段 1：前置检查

| 顺序 | 动作 | 说明 |
|------|------|------|
| 1 | 读取 AGENTS.md | 获取项目信息（产物组织方式、项目背景）。**如 AGENTS.md 未定义产物组织方式，执行自动推断**（见下方「产物组织方式自动推断」） |
| 2 | 搜索现有产物 | 在项目目录内搜索当前步骤的现有产物（通过文件名特征或文件头部的 上游文档 表格识别）。搜索结果不明确时，询问用户确认 |
| 3 | 确定模式 | **未找到** → 进入「新建模式」<br>**已找到** → 进入「修改模式」 |

## 阶段 2：加载上下文（模式分支）

**新建模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4a | 加载顶层定义 | **先查找项目目录**内是否有当前步骤类型对应的顶层定义。<br>• **有** → 加载项目顶层定义，作为生成约束；**同步加载该顶层定义上游文档中声明的依赖**（如交互-顶层定义需加载 PRD-顶层定义）。<br>• **无** → 加载 skill 中的顶层定义模板 `references/top-level/*-top-level-template.md`；**同步加载该顶层定义模板头部 上游文档 中声明的上游顶层定义模板**；询问用户是否基于该模板生成项目顶层定义。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 5a | 解析上游文档 | 根据用户描述的任务范围推理需要哪些上游输入，或按用户明确指定加载。**上游不固定写死在步骤模板中** |
| 6a | 加载上游文档 | 在项目目录内搜索并读取步骤 5a 确定的上游文档 |
| 7a | 加载步骤模板 | 读取 `references/steps/*-step.md` 作为产物骨架 |

**修改模式**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 4b | 加载现有产物 | 读取找到的现有产物内容 |
| 5b | 加载顶层定义 | **先查找项目目录**内对应类型的顶层定义，检查是否有更新；**同步加载该顶层定义上游文档中声明的依赖**（如交互-顶层定义需加载 PRD-顶层定义）。如无项目版本，则加载 skill 顶层定义模板 `references/top-level/*-top-level-template.md`，**同步加载该模板头部声明的上游顶层定义模板**，询问用户是否生成项目顶层定义。用户确认 → 生成后继续。用户拒绝/延后时按下方「关键规则」处理 |
| 6b | 加载上游文档 | 根据现有产物头部的 上游文档 表逐项读取，检查是否有更新 |
| 7b | 分析修改要求 | 明确用户要求修改的具体范围和内容；参照 skill 步骤模板 `references/steps/{type}-step.md` 中的填写示例辅助理解 |

## 阶段 3：影响评估、执行与检查

| 顺序 | 动作 | 模式 | 说明 |
|------|------|------|------|
| 8 | 上下游影响扫描 | 仅修改模式 | 按产物关联原则分层扫描本次修改对下游产物的影响范围。如当前修改的是顶层定义，需分层扫描所有下游顶层定义和步骤产物的引用 |
| 8a | 技术方案拆分决策 | 仅技术方案新建模式 | 当当前步骤为技术方案时，先根据上游 PRD 决定产出几份技术方案及每份的边界。参见下方「技术方案拆分决策流程」 |
| 9 | 生成/执行修改 | 两种模式 | 新建模式：按模板结构输出完整产物（技术方案按 8a 的拆分决策逐份生成）<br>修改模式：按用户要求执行具体修改 |
| 9a | 生成测试报告（如适用） | 新建模式 | 技术方案生成并通过审计后，根据 PRD 验收标准、技术方案接口/异常场景、UI 设计视觉规范生成测试报告。测试报告是 UI 设计和技术方案的共同下游产物，须等两者完成后生成 |
| 10 | 审计 | 两种模式 | **一次只审计一份产物文件。** 执行方式参见 `references/workflow/audit-procedure.md`。审计分两个阶段串行执行：<br>1. **机械检查**：运行 `python3 "$HOME/.agents/skills/e2e-solution-guard/scripts/doc-audit.py"`（含 `--upstream` 和 `--top-level` 参数）。**机械未通过禁止进入语义检查。**<br>2. **语义检查**：仅在机械检查全部通过后启动。优先使用多 Agent 并行：脚本生成检查点（`--model-checklist`）→ 主会话按维度拆分 → 前台并行启动子 agent 逐项填空 → 验证覆盖率 → 合并底稿。<br>单 Agent 逐条填空仅在产物很短（<500 行）且 `[模型]` 项 ≤ 8 时使用。<br>**禁止**：一次加载多份产物文件混审；禁止在单产物审计中验证下游实现。
| 10a | 修复后复验 | 两种模式 | **审计发现问题并由主会话修复后，按修复类型分阶段复验，禁止混合处理。**<br>• **机械修复后**：必须对该产物文件重新执行完整机械检查（阶段一），确认通过后方可进入语义复验；**禁止**因"只修了编号"就直接跳过机械复验<br>• **语义修复后**：仅对该产物文件重新执行语义检查（阶段二）；但如修复涉及格式/编号/结构调整，必须追加完整机械检查<br>• **轮次上限**：单份产物审计+修复最多进行 **3 轮**（含初次审计）。若第3轮仍不通过，主会话停止自动修复，向用户呈现剩余问题并请求人工决策 |
| 11 | 写入/更新产物头部 | 两种模式 | 新建模式：在文件头部写入实际加载的上游文档表<br>修改模式：如上游文档有更新或引用范围有变化，同步更新 上游文档 表 |

## 阶段 4：代码实现

产物生成并通过审计后，进入代码实现阶段。本阶段由开发团队根据技术方案进行编码，不在产物生成流程的直接控制范围内，但产物维护者需关注以下事项：

- 代码实现过程中发现技术方案遗漏或矛盾 → 按变更传播流程处理（参见 `references/workflow/change-propagation.md`）
- 代码实现完成后 → 进入阶段 5 代码落地验证

## 阶段 5：代码落地验证

代码实现完成后，对比代码与技术方案/PRD 的一致性，产出审计结果。执行方式参见 `references/workflow/audit-procedure.md` §多 Agent 并行语义审计。本阶段是产物与代码之间的最后一道锁。

| 顺序 | 动作 | 说明 |
|------|------|------|
| 12 | 定位代码文件 | 按项目技术-顶层定义 §2.2 的工程结构，定位当前服务/模块对应的代码文件 |
| 13 | 启动审计 | 加载：当前技术方案产物、技术方案 §1 声明的所有上游 PRD 产物（一个服务可能对应多个 PRD）、已定位的代码文件 |
| 14 | 执行一致性审计 | 逐项对比功能点映射、接口、数据模型、错误码、权限、业务规则、前端/UI 一致性七项（参见 `references/workflow/code-verification.md`） |
| 15 | 产出审计报告 | 通过 / 不通过判定，差异清单与修正建议 |
| 16 | 判定结果 | **全部通过** → 阶段 5 完成，继续后续步骤<br>**存在差异** → 输出差异清单，向用户呈现，等待用户决定改代码或改产物 |
| 17 | 修正后复验 | 修正完成后**重新执行审计**从第 13 步执行，禁止主会话自行复判 |

**阶段 5 触发条件**：主会话检测到代码文件发生变更且对应技术方案已存在。

## 技术方案拆分决策流程

当当前步骤为**技术方案**且处于**新建模式**时，在执行生成前必须先完成拆分决策：一份 PRD 可能对应多份技术方案，拆分的边界由**服务/技术域**决定，而不是 PRD 的业务模块边界。

### 拆分判断标准（四问法）

| 问题 | 如果"是" | 结论 |
|------|---------|------|
| 多个功能点是否共用同一套核心数据模型（表结构）？ | 是 | 应放在同一份技术方案中 |
| 多个功能点是否通过同一组内部接口/RPC 紧密协作？ | 是 | 应放在同一份技术方案中 |
| 该功能是否需要独立部署、独立扩容或独立团队维护？ | 是 | 应拆分为独立技术方案 |
| 该功能是否被多个业务模块广泛复用（如用户认证、文件上传）？ | 是 | 应作为公共/基础设施技术方案独立存在 |

### 拆分执行步骤

1. **读取 PRD §3 功能需求表格**，提取所有功能点
2. **按服务/技术域初步分组**：基于 PRD §5 数据模型和 §6 业务规则推断服务边界，将功能点按服务归属归为一份技术方案的候选范围，用户确认后再继续
3. **交叉验证**：
   - 检查这些功能点是否共享核心数据模型（PRD §5 数据模型）
   - 检查它们之间的调用关系（PRD §6 业务规则中的跨功能引用）
4. **参考项目技术-顶层定义 §2.2 工程结构**：如果顶层定义已声明了服务/模块划分，直接按顶层定义拆分
5. **与用户确认拆分决策**：向用户呈现拟定的拆分方案（几份技术方案、每份覆盖哪些功能点、命名建议），用户确认后逐份生成

### 常见拆分模式

| 架构类型 | 拆分方式 | 示例 |
|---------|---------|------|
| **单体应用** | 按代码模块/分层拆分 | `user-module.md`（含用户+权限）、`trade-module.md`（含订单+支付） |
| **微服务** | 按服务拆分 | `user-service.md`、`order-service.md`、`payment-service.md` |
| **前后端分离** | 按端拆分 | `backend-user-service.md`、`frontend-user-web.md` |
| **公共基础设施** | 独立拆分 | `auth-service.md`、`file-service.md`、`notification-service.md` |

> **注意**：技术方案的拆分决策只需做一次。后续修改模式（增删功能点）时，按变更传播流程处理，无需重新拆分。

## 产物组织方式自动推断

当 AGENTS.md 未定义产物组织方式时，AI 通过以下启发式规则自动推断，**不停止流程**：

1. **搜索常见产物目录**：检查项目根目录是否存在 `docs/`、`doc/`、`specs/`、`documents/`、`design/` 等目录
2. **搜索文件特征**：在项目目录内搜索文件名匹配 `PRD-*.md`、`交互设计-*.md`、`ui-*.html`、`TECH-*.md`、`TEST-*.md` 的文件（也兼容 `prd-*.md`、`UI-*.html`、`技术方案.md`、`测试报告-*.md` 等旧格式）
3. **推断逻辑**：
   - 发现 `docs/modules/` 和 `docs/tech/` 并存 → 混合聚合（业务产物按模块、技术方案按服务）
   - 发现 `docs/prd/`、`docs/tech/` 等子目录 → 按步骤聚合
   - 发现 `docs/订单模块/`、`docs/用户模块/` 等子目录 → 按模块聚合（旧模式，可能包含模块内的技术方案）
   - 未发现任何产物目录或特征文件 → 询问用户期望的组织方式（只问一次，不阻塞），同时默认使用项目根目录下的 `docs/` 作为产物根目录
4. **告知用户**：自动推断完成后，向用户说明推断结果，如"未在 AGENTS.md 中找到产物组织方式，已自动推断为按步骤聚合：docs/prd/、docs/interaction/、docs/ui/、docs/tech/、docs/test/"

## 关键规则

- `references/top-level/*-top-level-template.md` 是 skill 提供的**顶层定义模板（template）**，不是项目顶层定义本身。生成项目顶层定义时，必须严格区分两类内容：
  - **元内容（仅供 AI 参考，禁止输出到项目产物）**：文件中的 `§1 章节结构`、`§2 各章节写作要求`、`§3 内容边界`、`§4 与模块{产物类型}的关系`（如"与模块 PRD 的关系""与模块交互设计的关系"）、`§5 检查清单` 等段落。这些内容属于 Skill 内部写作指南，说明"顶层定义应该怎么写"，严禁复制到项目产物中。
  - **实际规范内容（必须填充并输出）**：根据用户提供的项目信息，按照模板对应的章节编号和标题，输出具体的产品概述、模块划分表、术语定义、状态值枚举、错误码规则、技术栈选型、工程结构等实际内容。
  - **判定标准**：生成的项目顶层定义中**不得出现**"章节结构"、"写作要求"、"内容边界"、"检查清单"、"格式："、"表格："、"树形结构"等指导性文字。每个章节必须是可直接被下游产物引用的**实际定义**，而非"告诉 AI 怎么写"的说明。
- 没有项目顶层定义时，必须先基于该模板生成项目版本，禁止以 skill 模板代替项目顶层定义继续后续步骤。
- 步骤模板 §1 中的 `上游文档` 表格仅为**常见依赖示例**，实际生成时必须根据项目上下文动态判断，禁止机械套用。
- 如关键上游文档缺失，暂停当前步骤，提示用户先完成上游。
- 顶层定义缺失时**先询问用户**是否基于模板生成项目版本。用户确认 → 生成后继续；**用户拒绝/延后时**：
  - 若当前任务是**生成/修改顶层定义本身** → 可继续，因为目标就是创建顶层定义
  - 若当前任务是**生成/修改模块级下游产物** → 如用户明确指定了任务范围且提供了足够上下文，可继续一次，但必须在生成产物的 上游文档 表格中标注 `[顶层定义待补全]`，并提醒用户在模块产物定稿前补全顶层定义
- **模块级视觉实现**：交互设计提供 SVG 页面结构图（结构）+ 状态矩阵（行为），UI 设计产出 HTML 原型（视觉实现）。HTML 原型包含内联 CSS、静态假数据、交互状态样式，可在浏览器中直接预览。
- 生成的产物头部必须包含真实的 `上游文档` 表，反映本次实际加载的所有上游。
