# 审计检查清单索引

> 以下为所有产物检查的**统一组织方式**。各产物类型的具体检查清单见独立文件。
>
> 每个维度的检查按层级展开：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游产物的一致性（单向责任：产物只对自己的上游负责）
>
> 外加 2 个横向维度：
> - **版本维度**：各产物版本号、变更范围的一致性
> - **边界维度**：产物职责边界，越界内容检测
>
> ### 跨产物链路检查（独立任务）
>
> 全链路追溯（如"PRD 每个功能点是否在下游全部实现"）不在单产物审计中执行。此类检查作为独立的 `cross-reference-check` 任务，按需触发。参见本文档末尾"跨产物链路检查清单"。

---

## 维度总览

| 维度 | 核心问题 | 典型失败后果 |
|------|---------|-------------|
| **结构** | 章节齐吗？表格格式对吗？编号连续吗？ | 开发遗漏章节、联调发现接口缺失 |
| **数据** | 字段名全链路一致吗？枚举值对齐吗？ | 接口字段和 UI 对不上、状态判断错误 |
| **状态** | 状态定义完整吗？转换条件实现了吗？ | 空状态/错误状态未处理、状态机不一致 |
| **错误** | 错误码全链路覆盖吗？异常场景齐全吗？ | 用户看到"系统繁忙"、服务雪崩 |
| **性能** | 性能指标传递了吗？缓存策略落实了吗？ | 上线后 8s 响应、数据库拖垮 |
| **安全** | 安全需求实现了吗？测试覆盖了吗？ | 越权漏洞、敏感数据泄露 |

## 契约驱动编号追溯检查（全产物通用）

以下检查项**不由各产物类型清单枚举**，由 `doc-audit.py` 的 `ContractTraceabilityRule` 根据 PRD-顶层定义 §6 的编号引用契约矩阵自动推导执行：

| 检查项 | 说明 | 执行方 |
|--------|------|--------|
| 功能编号追溯 | 当前产物是否引用了上游 PRD 定义的全部功能编号（按契约矩阵要求） | `[脚本]` |
| 故事编号追溯 | 当前产物是否引用了上游 PRD 定义的全部故事编号（按契约矩阵要求） | `[脚本]` |
| 规则编号追溯 | 当前产物是否引用了上游 PRD 定义的全部规则编号（按契约矩阵要求） | `[脚本]` |
| 错误码追溯 | 当前产物是否引用了上游 PRD 定义的全部错误码（按契约矩阵要求） | `[脚本]` |
| 验收项编号追溯 | 当前产物是否引用了上游 PRD 定义的全部验收项编号（按契约矩阵要求） | `[脚本]` |

> **扩展方式**：新增编号类型时，只需在 PRD-顶层定义 §6 的编号引用契约矩阵中添加一行，`ContractTraceabilityRule` 自动在当前产物审计中覆盖，无需修改 audit-procedure.md。

## 跨产物链路检查清单（独立任务）

> 以下检查**不在单产物审计中执行**，作为独立的 `cross-reference-check` 任务，在需要全链路追溯时触发。
>
> 单向责任原则：单产物审计只检查"产物是否完整实现了上游定义"；跨产物链路检查才检查"上游定义是否在全部下游中都有对应实现"。

### 产物间一致性

| 检查项 | 说明 |
|--------|------|
| 编号全局唯一性 | 同一编号（功能编号/错误码/页面编号等）在不同产物中定义一致，无歧义 |
| 顶层定义间对齐 | PRD-顶层定义 §4 术语表 ↔ 交互-顶层定义 §4.1 主导航 ↔ 技术-顶层定义模块划分，数量/名称/职责边界一致 |

### 正向追溯（上游 → 下游）

| 上游 | 下游 | 检查项 |
|------|------|--------|
| PRD §3 | 交互设计 | 每个 P0 功能点在交互设计中有对应页面/组件 |
| PRD §5 | 技术方案 §3 | 每个实体在技术方案数据模型中有对应表 |
| PRD §5 | 技术方案 §3/§4 | 每个字段在技术方案中有对应实现 |
| PRD §6 | 交互设计 | 每个业务状态在交互设计状态机中有对应 |
| PRD §6 | 技术方案 §5 | 每个业务状态在技术方案状态机中有对应状态值 |
| PRD §7 | 技术方案 §4/§7 | 每个错误码在技术方案中有对应实现 |
| PRD §7 | 测试 §2 | 每个错误码在测试异常场景中有对应验证 |
| PRD §8 | 测试 §1/§6 | 每条验收标准在测试用例中有对应覆盖 |
| 交互设计 §4.3 | UI 设计 | 每个状态在 UI HTML 中有对应 CSS 实现 |
| 交互设计 §4.4 | UI 设计 | 每个异步状态在 UI 中有对应状态类 |
| 交互设计 §4.4 | 技术方案 §5 | 状态机中的每个状态在技术方案中有对应状态值 |
| 交互设计 §4.9 | UI 设计 | 响应式结构变更在 UI 中有 @media 实现 |
| 技术方案 §4 | 测试 §1/§2 | 每个接口在测试中有对应覆盖 |
| 技术方案 §7 | 测试 §2 | 每个异常场景在测试中有对应覆盖 |
| 技术方案 §8 | 测试 §3 | 每个性能指标在测试中有对应测试项 |
| 技术方案 §10 | 测试 §4 | 每个安全措施在安全测试中有对应验证 |

### 反向追溯（代码 → 产物）

| 起点 | 终点 | 检查项 |
|------|------|--------|
| 代码 §4.2 | 技术方案 §3 | 表结构与数据库 schema 一致 |
| 代码 §4.3 | UI 设计 | 组件状态类名在代码中有对应实现 |
| 代码 §4.4 | PRD §6 | 权限校验在代码中实现 |
| 代码 §4.5 | 技术方案 §8 | 缓存策略、查询优化在代码中有对应实现 |
| 代码 §4.6 | 技术方案 §9 | 高可用设计（熔断/降级/限流/重试/超时）在代码中实现 |

### 端到端链路完整性

| 检查项 | 说明 |
|--------|------|
| PRD → 交互 → UI → 代码 | 从 PRD 功能点 → 交互设计页面 → UI HTML → 代码实现，链路完整可追溯 |
| 核心流程覆盖 | PRD 关键业务场景在技术方案核心流程章节中有对应 |
| 接口覆盖 | 技术方案接口汇总表中的每个接口在代码中有对应实现 |
