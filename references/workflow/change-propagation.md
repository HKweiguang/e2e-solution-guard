# 回改触发与重新验证

在执行过程中发现当前产物的上游定义存在缺陷、遗漏或与需求矛盾时，触发回改。

---

## 执行流程

| 顺序 | 动作 | 说明 |
|------|------|------|
| R1 | 暂停当前步骤 | 立即停止当前产物的生成/修改，向用户报告冲突点 |
| R2 | 定位回改范围 | 明确需要修改的上游文档名称、章节、具体条目 |
| R3 | 评估下游影响 | 按产物关联原则分层扫描该上游文档的下游引用。使用 `scripts/doc-audit.py <上游文档路径> --type <类型> --scan-downstream <docs_dir>` 自动化列出受影响产物清单，按直接/间接下游分层展示。<br><br>**特殊情况**：如果回改的是 PRD §3 功能需求（尤其是增删功能点），下游影响不仅要扫描交互/UI/测试等产物，还要扫描所有引用了该 PRD 的技术方案产物。`--scan-downstream` 通过上游文档引用关系定位下游产物 |
| R4 | 用户确认 | 向用户呈现：回改内容 + 影响范围 + 建议方案。用户确认后执行上游修改 |
| R5 | 级联修改 | 同步修改所有受影响下游产物中引用的相关内容，或按产物关联原则标注 `[待同步]` |
| R6 | 重新验证 | 上游及下游全部修改完成后，回到产物生成流程阶段 3 第 10 步执行**全量审计**（禁止缩减维度，必须覆盖机械+语义+链路所有检查项），确认无异常后再继续。如回改涉及已落地代码，修正后还需触发代码落地验证 |

## 关键规则

- 禁止在发现上游缺陷时擅自绕过或修改下游来"适配"
- 回改完成后必须重新执行一致性检查，不能假设修改正确
- 涉及多个上游文档的回改，按依赖顺序从下往上逐层处理
