# 审计执行策略

产物生成或修改完成后，运行审计确认产物质量。审计分为机械检查（脚本执行）和语义检查（AI 执行）两部分。

## 常见任务类型

| 任务 | 加载内容 | 适用场景 |
|------|---------|---------|
| 单产物审计 | 当前产物 + 直接上游文档 | 新建/修改一份 PRD、技术方案等 |
| 单模块代码审计 | 当前技术方案 + 对应代码文件 + 上游 PRD | 代码落地验证阶段 |
| 跨产物链路检查 | 两份上下游产物的关键章节 | 验证编号引用、接口映射、错误码对应 |
| 下游影响扫描 | 上游文档 + 下游产物头部 | 修改上游后评估影响范围 |

---

## 加载优先级（同一任务内）

1. 当前产物（全文或变更 diff）
2. 直接上游文档（被引用的章节/段落）
3. 直接下游产物（上游文档 表格 + 受影响段落）
4. 间接上下游（按需，通常只在顶层定义变更时涉及）

## 脚本与 AI 的分工边界（架构决策）

> **这是一个必须遵守的架构决策**。禁止用脚本覆盖 AI 的职责，也禁止让 AI 做脚本该做的事。后续扩展检查项时，必须先对照此边界判断归属，不可凭直觉"一条路走到黑"。

### 脚本负责：结构化、确定性、可穷举的检查

特征：有明确的模式、规则、穷举空间，结果只有"通过/不通过"两种。

- **格式与模式**：ID 格式、连续性、前缀一致性；编号跳号/重复；表格列完整性
- **存在性验证**：链接有效性；必填字段/章节存在；双向引用存在（A 提到 B，B 是否提到 A）
- **与顶层定义一致性**：状态名、变量名、错误码前缀、缓存 Key 命名规范、枚举值等**格式层面**的对齐
- **跨产物编号映射存在性**：PRD 错误码列表中的每个 ID 是否在 Tech §7 中出现；PRD 功能点是否在下游产物中被引用
- **计数验证**：数量是否匹配（如 PRD 错误码数量 = Tech 异常数量 = Test 验证数量）

### AI 负责：语义、推理、业务逻辑判断

特征：需要理解业务含义、做推理判断、结果可能有"存疑"。

- **映射正确性**：字段映射是否正确（而非仅仅存在同名）；状态值是否语义一致（而非仅仅同名）
- **需求满足度**：需求是否被真正满足（而非仅仅有对应章节）；安全措施是否覆盖安全需求；力学约束是否有合理技术实现
- **具体可执行性**：描述是否具体（如错误码触发场景是否可执行，非"用户操作时"）
- **设计合理性**：状态机设计是否合理；边界是否清晰；是否过度复杂化
- **跨产物语义对齐**：PRD 性能指标是否在 Tech 中有**合理的**性能目标（而非仅仅有章节）

### 决策标准

| 标准 | 脚本 | AI |
|------|------|-----|
| 判断依据是模式/格式/存在性 | ✓ | |
| 判断依据是语义/正确性/合理性 | | ✓ |
| 可以用正则表达式或简单字符串匹配完成 | ✓ | |
| 需要理解业务上下文才能判断 | | ✓ |
| 结果是确定性的（无模糊地带） | ✓ | |
| 结果可能有"存疑" | | ✓ |
| 穷举空间明确且有限 | ✓ | |
| 需要推理或综合多段信息 | | ✓ |

### 扩展原则

当新增检查项时，按以下流程判断归属：

1. **先问**：这项检查能否用确定性规则描述？是否存在唯一的正确答案？
2. **如果能** → 交给脚本实现（放入 `doc-audit.py` 规则集）
3. **如果不能** → 交给 AI 语义审计（标记为 `[模型]`）
4. **禁止降级**：不可为了让脚本覆盖而放宽规则精度（如把"字段映射正确"降级为"字段名存在"后交给脚本），这会制造虚假安全感
5. **禁止越界**：脚本发现异常时只报"不通过"，不做语义解释；AI 发现格式问题时只报"发现格式异常"，不替代脚本做穷举检查

## 编排边界：谁决定"查什么"？（防偷懒核心机制）

> **这是架构的第三层边界，解决 AI 在语义审计中"自行决定检查范围"导致的漏检问题。**

### 问题根因

LLM 的注意力机制会随上下文长度增加而衰减。当 AI 同时承担：
1. **回忆**：从模板中提取全部 `[模型]` 检查项
2. **判断**：对每项做语义分析

"回忆"阶段必然被压缩，导致 AI 自行省略检查项（俗称"偷懒"）。Prompt 约束对此无效——模型不是故意违抗，而是注意力资源不足以覆盖长清单。

### 根本解法：穷举交给脚本，AI 只负责填空

| 职责 | 归属 | 执行方式 |
|------|------|---------|
| 从模板提取全部 `[模型]` 检查项 | **脚本** | `doc-audit.py --model-checklist` 输出完整清单 |
| 将检查项按维度/层级分组 | **脚本** | JSON 中 `dimension` + `level` 字段 |
| 为每个检查点分配唯一 ID | **脚本** | `id` 字段（如 `M-数据-L1-001`） |
| 按维度筛选分批执行 | **脚本** | `--scope` 参数 |
| 对每个检查点做语义判断 | **AI** | 输出固定格式表格，逐行填空 |
| 验证输出是否覆盖全部检查点 | **脚本/主会话** | 统计行数是否等于 `total` |

**禁止**：让 AI 自行从模板中"回忆"有哪些 `[模型]` 项需要执行。AI 收到的输入必须是脚本已经列好的检查点列表。

## 语义审计执行方法

审计任务分两轮执行，产出合并结果：
- **第一轮·机械检查**：运行 `doc-audit.py`（不含 `--model-checklist`）产出机械检查结果
- **第二轮·语义检查**：运行 `doc-audit.py --model-checklist` 生成检查点 → AI 逐条填空 → 脚本/主会话验证覆盖率

两者分工明确，不重复执行同一检查项。

### 调用方式

**机械检查**：

```bash
# 定位脚本
SCRIPT=$(find ~ -path "*/e2e-solution-guard/scripts/doc-audit.py" 2>/dev/null | head -1)

# 完整机械审计
python3 "$SCRIPT" <doc_path> --type <prd|interaction|ui|tech|test> \
  --upstream <upstream1.md> --upstream <upstream2.md> \
  --top-level <top-level1.md>
```

**语义检查点生成**（必须在 AI 执行语义审计前运行）：

```bash
# 生成全部 [模型] 检查点
python3 "$SCRIPT" --type <prd|interaction|ui|tech|test> --model-checklist

# 按维度拆分执行（推荐，降低单次认知负载）
python3 "$SCRIPT" --type prd --model-checklist --scope 数据
python3 "$SCRIPT" --type prd --model-checklist --scope 状态
python3 "$SCRIPT" --type prd --model-checklist --scope 错误
```

> `--scope` 按维度名筛选（如"数据"、"状态"、"错误"），仅在 `--model-checklist` 模式下有效。
> AI 单次只处理一个 scope，彻底消除"上下文太长导致压缩"的问题。

### 执行步骤（新）

1. **脚本生成检查点清单**（防偷懒关键步骤）
   
   运行 `doc-audit.py --model-checklist --type <type>`，输出 JSON 包含：
   - `total`：检查点总数（用于后续验证）
   - `checkpoints`：完整列表，每项含 `id` / `level` / `dimension` / `description`
   
   如上下文受限，加 `--scope <维度名>` 分批执行。

2. **加载必要上下文**
   
   按检查点列表中涉及的范围加载产物和上下游文档。不再让 AI 自行判断"需要查哪些"。

3. **AI 逐项填空**
   
   AI 对脚本生成的检查点列表逐条输出判断结果。**必须逐条输出，禁止合并行，禁止省略。**
   
   固定输出格式：
   
   | 检查点ID | 维度 | 层级 | 检查项 | 结果 | 证据（产物原文） | 建议 |
   |---------|------|------|--------|------|----------------|------|
   | M-数据-L1-001 | 数据 | L1 | §6 权限规则引用... | 通过/不通过/存疑 | "§6.3 中..." | ... |
   | ... | ... | ... | ... | ... | ... | ... |
   
   **末行必须输出**：`语义审计完成：{N}/{total} 项已检查，0 项跳过。`（`total` 来自脚本输出的 JSON）

4. **覆盖率验证**
   
   主会话或脚本统计 AI 输出的表格行数，与 `checkpoints` 数量比对。
   - 行数一致 → 进入汇总
   - 行数不一致 → **标记审计不完整**，要求 AI 补全遗漏项

5. **汇总结论**
   
   将语义检查结果与机械检查结果合并，输出最终审计结论。

### 关键原则

- **[模型] 检查项的"列出"归脚本，"判断"归 AI**。AI 无权决定查几项，只配决定每项的结果
- 语义审计输出必须是**固定行数的表格**，少一行即视为不完整
- 如因上下文不足无法判断某项，结果填「存疑」，证据填「缺少：需加载 XXX」——**该项仍然算作已检查**
- 修复后复验时，必须重新执行步骤 1（脚本重新生成检查点，防止修复引入新遗漏项），禁止沿用旧清单

## 多 Agent 并行语义审计（推荐·根本防偷懒机制）

> 利用 kimi code 的 `Agent` 工具**前台并行**启动多个子 agent，每个子 agent 独立处理一个维度的检查点。这是当前环境下最根本的防偷懒方案，因为：
> - 每个子 agent 上下文极小（3-6 个检查点 + 相关文档片段），不存在"太长导致压缩"
> - Agent 之间完全隔离，一个敷衍不影响其他
> - 输出写入独立文件，主会话可用脚本验证覆盖率和完整性
> - **前台并行**，避免后台任务超时被杀

### 执行流程

```
主会话（调度器）
├── 1. Bash: 运行 doc-audit.py --model-checklist → checkpoints.json
├── 2. Read: 按检查点涉及的章节，读取产物和上游文档的对应片段
├── 3. Agent×N: 前台并行启动子 agent（每维度一个，N ≤ 6）
│      每个子 agent 收到：维度检查点 + 相关文档片段 + 固定输出指令
│      每个子 agent 输出：写入独立临时文件（audit-{维度}.md）
├── 4. Bash: 验证所有临时文件存在、非空、行数正确
└── 5. 合并为最终语义审计底稿
```

### 主会话操作细节

**步骤 1：生成检查点**

```bash
SCRIPT=$(find ~ -path "*/e2e-solution-guard/scripts/doc-audit.py" 2>/dev/null | head -1)
CHECKPOINTS=$(python3 "$SCRIPT" --type <type> --model-checklist)
TOTAL=$(echo "$CHECKPOINTS" | python3 -c "import sys,json; print(json.load(sys.stdin)['total'])")
echo "共 $TOTAL 个 [模型] 检查点"
```

**步骤 2：读取相关章节**

主会话根据检查点 `description` 中提到的章节号（如 `§3`、`§5`、`§6`），用 `Read` 工具精确读取对应行范围。例如：
- 涉及 `§3` 的检查点 → `Read <doc_path> line_offset=<§3起始行>`
- 涉及 `§5` 的检查点 → `Read <doc_path> line_offset=<§5起始行>`

**禁止**：让子 agent 自己去读长文件——子 agent 的上下文必须已经是裁剪好的片段。

**步骤 3：前台并行启动子 agent**

主会话一次性发出多个 `Agent` 调用（`run_in_background=false`），每个处理一个维度。示例：

```
并行调用 1：Agent("审计 PRD 数据维度", prompt=...)
并行调用 2：Agent("审计 PRD 状态维度", prompt=...)
并行调用 3：Agent("审计 PRD 错误维度", prompt=...)
...（最多 6 个并行）
```

> **子 agent 不设置 `timeout`**，避免审计任务因超时时间过短被异常终止。如某个 agent 未完成，主会话单独重试该维度。

**步骤 4：验证覆盖率**

```bash
# 统计已完成的维度文件
DIMS=$(ls audit-*.md 2>/dev/null | wc -l)
echo "已完成维度: $DIMS"

# 粗略验证：每个文件非空且包含表格分隔线
for f in audit-*.md; do
  lines=$(grep -c '^|' "$f" || true)
  [ "$lines" -gt 2 ] || echo "警告: $f 表格行数不足"
done
```

**步骤 5：合并**

主会话读取所有 `audit-*.md`，按维度顺序合并为最终语义审计报告。

### 子 agent Prompt 模板

每个子 agent 收到的 prompt 必须包含**完整上下文**，禁止让子 agent 自行去文件系统查找：

````markdown
你是 e2e-solution-guard 的语义审计子 agent，负责单一维度的 [模型] 检查点审计。

## 任务
对以下 {N} 个检查点逐项判断，输出固定格式表格。

## 检查点清单
{逐条列出该维度的检查点，含 id / level / description}

## 产物相关章节（已裁剪）
```
{主会话读取的产物片段}
```

## 上游文档相关章节（如需要）
```
{主会话读取的上游片段}
```

## 输出格式（严格）

| 检查点ID | 层级 | 检查项 | 结果 | 证据（产物原文引用） | 建议 |
|---------|------|--------|------|-------------------|------|
| M-数据-L1-001 | L1 | §6 权限规则引用... | 通过/不通过/存疑 | "§6.3 原文..." | 如不通过则给修正方向 |
| ... | ... | ... | ... | ... | ... |

## 约束（违反视为审计失败）
1. 表格数据行必须等于检查点数量（{N} 行），禁止合并、禁止省略
2. 证据必须引用产物原文（章节号+具体段落），禁止笼统判断
3. 无法判断时结果填「存疑」，证据填「缺少：需加载 XXX」——该项仍然算作已检查
4. 最后一行输出：`维度审计完成：{N}/{N} 项已检查`
5. 将完整输出写入文件：{audit_维度.md 的绝对路径}
````

### 何时用单 Agent、何时用多 Agent

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 产物很短（<500 行）且 `[模型]` 项 ≤ 8 | 单 Agent 逐条填空 | 并行开销不划算 |
| 产物较长（>500 行）或 `[模型]` 项 > 8 | **多 Agent 并行** | 上下文隔离，防压缩 |
| 涉及多份上游文档（PRD + 交互 + 技术） | **多 Agent 并行** | 每维度的上游不同，拆分后各 agent 只加载需要的上游 |
| 修复后复验 | 单 Agent（仅变更维度） | 范围明确，不需要全量并行 |

## 检查清单框架（6 维度 × 3 层级 + 契约驱动）

> 以下框架是所有产物检查的**统一组织方式**。每个维度的检查按层级展开：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试/代码的完整追溯链
>
> 外加 2 个横向维度：
> - **版本维度**：各产物版本号、变更范围的一致性
> - **边界维度**：产物职责边界，越界内容检测
>
> ### 契约驱动原则
>
> 编号追溯检查**不再枚举**，而是从 PRD-顶层定义 §6 — 编号引用契约矩阵推导：
> - 脚本审计自动读取契约矩阵，对每一行"强制"的编号类型执行 L2/L3 追溯检查
> - 新增编号类型 → 只需在契约矩阵添加一行 → 全链路审计自动覆盖
> - 新增下游产物类型 → 只需在契约矩阵声明引用关系 → 审计自动检查
>
> 以下为各产物类型的**基础检查清单**（结构/格式/语义），编号追溯检查由 `doc-audit.py` 的 `ContractTraceabilityRule` 根据契约矩阵自动推导执行。

### 维度总览

| 维度 | 核心问题 | 典型失败后果 |
|------|---------|-------------|
| **结构** | 章节齐吗？表格格式对吗？编号连续吗？ | 开发遗漏章节、联调发现接口缺失 |
| **数据** | 字段名全链路一致吗？枚举值对齐吗？ | 接口字段和 UI 对不上、状态判断错误 |
| **状态** | 状态定义完整吗？转换条件实现了吗？ | 空状态/错误状态未处理、状态机不一致 |
| **错误** | 错误码全链路覆盖吗？异常场景齐全吗？ | 用户看到"系统繁忙"、服务雪崩 |
| **性能** | 性能指标传递了吗？缓存策略落实了吗？ | 上线后 8s 响应、数据库拖垮 |
| **安全** | 安全需求实现了吗？测试覆盖了吗？ | 越权漏洞、敏感数据泄露 |

### 契约驱动编号追溯检查（全产物通用）

以下检查项**不由各产物类型清单枚举**，由 `doc-audit.py` 的 `ContractTraceabilityRule` 根据 PRD-顶层定义 §6 的编号引用契约矩阵自动推导执行：

| 检查项 | 说明 | 执行方 |
|--------|------|--------|
| 功能编号追溯 | 契约矩阵中声明了功能编号必须引用的下游产物，逐个检查 | `[脚本]` |
| 故事编号追溯 | 契约矩阵中声明了故事编号必须引用的下游产物，逐个检查 | `[脚本]` |
| 规则编号追溯 | 契约矩阵中声明了规则编号必须引用的下游产物，逐个检查 | `[脚本]` |

## 常见遗漏项速查

以下是对上述检查清单的高风险项浓缩，执行审计时不得遗漏：

### 数据与功能
- 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术）
- 所有接口都标注了对应的功能点
- 所有表结构都包含审计字段（如项目技术-顶层定义有要求）
- 测试 §6 覆盖检查报告必须列出所有 PRD 验收标准

### 状态与错误
- 所有异常场景都对应了错误码、PRD §6 业务规则或模块特定异常
- 所有状态流转都有明确的触发条件和目标状态
- 交互设计每个页面状态机包含 6 种状态（默认/空/加载/错误/成功/骨架）

### 结构与完整性
- 所有 P0 用例都有明确的前置条件、测试步骤和预期结果
- 技术方案 §1–§13 章节齐全
- 所有阻塞性差异修复后都经过回归验证，测试 §7 回归范围与实际执行的测试用例匹配

### 性能与高可用
- PRD §4 性能指标可量化（有数字和阈值），安全需求具体可执行
- 技术方案 §8 性能目标明确（QPS/RT 有数字），缓存策略覆盖热点数据
- 所有第三方依赖都有不可用时的降级策略

### UI 与交互
- 所有交互设计定义的组件状态都有 UI 设计的 CSS 状态样式实现
- 产物中无不属于本产物范畴的描述
