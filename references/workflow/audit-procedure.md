# 审计执行策略

所有审计（机械检查 + 语义审计）必须由**独立只读 subagent**执行。主会话禁止运行 `doc-audit.py`，禁止做任何语义判断。
核心原则：**一个 subagent 只审计一个任务**。

---

## 常见任务类型

| 任务 | 加载内容 | 适用场景 |
|------|---------|---------|
| 单产物审计 | 当前产物 + 直接上游文档 | 新建/修改一份 PRD、技术方案等 |
| 单模块代码审计 | 当前技术方案 + 对应代码文件 | 代码落地验证阶段 |
| 跨产物链路检查 | 两份上下游产物的关键章节 | 验证编号引用、接口映射、错误码对应 |
| 下游影响扫描 | 上游文档 + 下游产物头部 | 修改上游后评估影响范围 |

---

## 拆分决策流程

主会话在启动审计前，**必须先用 `wc -l` 统计当前产物行数**，然后按以下规则决定拆分策略。

### 行数阈值与超时配置

| 总行数 | subagent 数量 | 每个 timeout | 策略 |
|--------|--------------|-------------|------|
| < 300 | 1 | 1800 | 单 subagent 全量审计（机械+语义） |
| 300-600 | 2 | 2400 | 按维度拆：机械检查 / 语义检查，并行执行 |
| > 600 | 3 | 3600 | 按维度拆：机械 / 语义前半 / 语义后半，并行执行 |

**核心原则：按检查维度拆，不按章节拆**。每个 subagent 都加载当前产物全文（机械 subagent 额外加载直接上游，语义 subagent 额外加载上下游），确保跨章节的检查项（如"§2 功能编号在 §6/§7/§8 中的引用一致性"）和跨产物的比对（如"页面编号前缀与 PRD 功能编号一致"）不会遗漏。

### 按产物类型的具体拆分策略

#### PRD

| subagent | 检查维度 | 加载内容 | 负责项 |
|----------|---------|---------|--------|
| #1 机械 | `[脚本]` 标记项 | 当前产物全文 + 项目 PRD-顶层定义 | 编号重复检测、编号跳号检测、表格格式、upstream引用、必填章节、术语一致性 |
| #2 语义 | `[模型]` 标记项 | 当前产物全文 + 上游 PRD + 项目 PRD-顶层定义 + 下游技术方案(如有) | 目标一致性（成功指标可量化、功能与用户故事对应）、范围一致性（模块职责边界）、业务规则一致性、错误码触发场景、跨章节引用一致性、验收标准格式 |
| #3 链路(可选) | 跨产物链路 | 当前产物 + 下游产物头部 | 功能点→下游、页面→交互设计、错误码→技术方案 |

#### 交互设计

| 页面数量 | 策略 |
|---------|------|
| < 5 | 单 subagent 全量（机械+语义） |
| ≥ 5 | 拆 2 组：#1 机械检查（全文+PRD §3）/ #2 语义检查（全文+上游PRD+项目交互-顶层定义，重点检查状态映射、页面流程覆盖度、视觉边界一致性） |

#### UI 设计

| 页面数量 | 策略 |
|------------|------|
| < 5 | 单 subagent 全量（机械+语义） |
| ≥ 5 | 拆 2 组：#1 机械检查（全文+交互设计+UI-顶层定义，重点检查 HTML 代码块结构、CSS 变量、状态样式、状态映射表）/ #2 语义检查（全文+上游交互设计+项目UI-顶层定义，重点检查状态映射、Token 一致性、行为边界一致性） |

#### 技术方案

| subagent | 检查维度 | 加载内容 | 负责项 |
|----------|---------|---------|--------|
| #1 机械 | `[脚本]` 标记项 | 当前产物全文 + PRD + 项目技术-顶层定义 | 表结构审计字段、接口清单与接口设计对应、接口功能点标注 |
| #2 语义 | `[模型]` 标记项 | 当前产物全文 + 上游 PRD + 项目技术-顶层定义 | 接口字段与上游一致、异常场景映射、缓存策略覆盖、权限规则一致性、状态一致性（状态机与PRD对齐）、Token一致性（缓存Key/权限与顶层定义对齐） |
| #3 代码(可选) | 代码落地验证 | 技术方案 + 代码文件 | 阶段5验证（参见 `references/workflow/code-verification.md`） |

#### 测试

测试产物模板 350 行左右，**默认不拆**。单 subagent 全量审计，加载内容：当前产物全文 + PRD + 交互设计 + UI + 技术方案。

### 修改模式的增量拆分

如果是**修改模式**，不按全文行数拆分，而是按变更范围拆分：

1. 主会话用 `diff` 生成变更范围，确定受影响的维度（机械项 or 语义项）
2. 启动对应维度的 subagent，但**仍然加载全文+上游**，只是检查重点聚焦在变更部分：
   - 只改了功能需求章节（影响编号和引用）→ 启动"机械"subagent，加载全文+上游，重点检查编号连续性和跨章节引用一致性
   - 改了 §6 业务规则 → 启动"语义"subagent，加载全文+上下游，重点检查业务规则一致性
3. 未受影响的维度可跳过（如只改表格格式，语义检查可跳过）

### 并行执行与结果汇总

主会话**同时启动多个只读 subagent**并行执行审计，每个 subagent 独立加载所需上下文，独立完成检查：

```
subagent-1 (机械): subagent_type="explore", prompt="加载全文+上游，执行所有[脚本]标记检查项...", timeout=600
subagent-2 (语义): subagent_type="explore", prompt="加载全文+上下游，按以下逐条拆表精读清单执行...", timeout=900
```

> **只读约束**：所有审计 subagent 必须使用 `explore` 类型，禁止写入任何文件。subagent 的审计结论通过返回结果传递，不由 subagent 自行修改产物。

待全部返回后，主会话汇总：
- 所有 blocking issue 合并输出
- 所有 warning/hint 合并输出
- 如有冲突（两个 subagent 对同一问题结论不同），主会话仲裁或启动第三个 subagent 复核

---

## 加载优先级（同一任务内）

1. 当前产物（全文或变更 diff）
2. 直接上游文档（被引用的章节/段落）
3. 直接下游产物（上游文档 表格 + 受影响段落）
4. 间接上下游（按需，通常只在顶层定义变更时涉及）

---

## subagent 配置建议

| 配置项 | 建议值                      | 说明 |
|--------|--------------------------|------|
| 类型 | `explore` | 审计任务禁止写入 |
| 超时 | 1800 秒 | 复杂产物审计需要充足时间 |
| 后台运行 | 多任务并行时可启用 | 减少主会话等待 |

## 语义审计执行方法

subagent 的审计任务必须同时产出**机械检查结果**和**语义审计结果**。机械检查通过调用 skill 根目录下的 `scripts/doc-audit.py` 完成，语义检查由模型直接分析完成，两者在同一个 subagent 会话内执行，缺一不可。

**注意**：如果当前工作目录不是 skill 根目录，需使用完整路径调用，例如：
```bash
python /path/to/e2e-solution-guard/scripts/doc-audit.py <doc> --type <type>
```

### 执行步骤

1. **定位检查清单**：根据当前产物类型（prd/interaction/ui/tech/test），读取 `references/steps/{type}-step.md` 的检查清单章节，提取所有标记为 `[模型]` 的检查项。注意：`ui-step.md` 模板的结构与其他步骤不同（§1-§5，无 §6 检查清单章节——检查清单位于 **§5**），UI 设计的语义审计以模板中声明的原则和边界为检查依据
2. **加载必要上下文**：除当前产物外，根据 `[模型]` 项的描述加载需要比对的上游或下游产物（如 PRD、交互设计、技术方案等）
3. **逐项分析并输出**：针对每个 `[模型]` 检查项，在当前产物与关联产物之间进行比对分析，输出结构化结果：

   | 字段 | 说明 |
   |------|------|
   | 检查项 | `[模型]` 项的原文描述 |
   | 结果 | **通过** / **不通过** / **存疑** |
   | 证据 | 引用的产物原文（章节号、段落、表格行），禁止笼统判断 |
   | 建议 | 如不通过或存疑，给出明确的修正方向 |

4. **汇总**：将所有 `[模型]` 项的结果汇总为语义审计结论，连同机械检查结果一并返回主会话

### 关键原则

- `[模型]` 检查项无法被脚本自动化，必须依靠模型的阅读理解能力逐条分析
- 语义审计的结论必须有产物原文支撑，禁止无依据的笼统判断
- 如果上下文不足以判断某项，标注为「存疑」并说明缺少的信息

## 逐条拆表精读检查清单（语义审计必须执行）

> 以下清单是对 `[模型]` 检查项的**具体执行指令**。subagent 不能笼统地说"检查验收标准格式"，必须按以下粒度逐条拆表、逐行比对。

### PRD 语义审计精读清单

| # | 检查项 | 精读方法 |
|---|--------|---------|
| P-S1 | §1 成功指标可量化、有基线 | 逐条检查成功指标是否包含数字和比较基准 |
| P-S2 | §2 用户故事对应 §3 功能 | 逐故事检查是否在 §3 有至少一个功能编号引用 |
| P-S3 | §3 功能在模块职责范围内 | 逐功能检查编号前缀是否与 PRD-顶层定义 §2 编码规则一致 |
| P-S4 | §3 功能编号格式符合顶层定义 | 逐行检查第一列编号格式（单角色/多角色格式） |
| P-S5 | §6 业务规则与 §3 规则引用一致 | 逐规则检查引用的功能编号是否存在于 §3 |
| P-S6 | §7 错误码与 §3 匹配 | 逐错误码检查是否在 §3 有对应功能点 |
| P-S7 | §8 验收标准格式 | **逐条验收标准逐字拆解**：是否为「条件 + 操作 + 预期结果」句式 |
| P-S8 | §8 每条验收标准对应 §3 功能点 | 逐条检查引用的功能编号是否存在于 §3 |

| # | 检查项 | 精读方法 |
|---|--------|---------|

## 12 项常见遗漏项责任分配

以下检查项来自 `references/rules/consistency-rules.md` §60-73，必须在每次审计中执行，不得遗漏。

| # | 遗漏项 | 负责 subagent | 检查方法 |
|---|--------|-------------|---------|
| L1 | 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术） | PRD 语义 #2 + 测试语义 | PRD 语义检查 §3 中标记为 P0 的功能是否在 §8 有验收标准；测试语义检查 §1 是否覆盖所有 P0 功能点 |
| L2 | 所有异常场景都对应了错误码或业务规则 | PRD 语义 #2 + 技术语义 #2 | PRD 语义检查 §7 每个错误码是否有触发场景；技术语义检查 §7 每个异常是否对应 PRD 错误码 |
| L3 | 所有状态流转都有明确的触发条件和目标状态 | 交互语义 #2 + 技术语义 #2 | 交互语义检查 §4.4 状态机；技术语义检查 §5 状态机 |
| L4 | 所有页面都有空状态和错误状态的定义 | 交互语义 #2 | 逐页面检查 §4.4 状态机是否包含"空"和"错误"状态 |
| L5 | 所有接口都标注了对应的功能点 | 技术语义 #2 | 逐接口检查 §4 是否标注了 PRD 功能编号 |
| L6 | 所有表结构都包含审计字段 | 技术机械 #1 | 逐表检查 §3 是否包含审计字段（created_at/updated_at 等） |
| L7 | 所有第三方依赖都有不可用时的降级策略 | 技术语义 #2 | 逐依赖检查 §2 是否有降级策略说明 |
| L8 | 所有 P0 用例都有明确的前置条件、测试步骤和预期结果 | 测试语义 | 逐 P0 用例拆表检查 |
| L9 | 所有阻塞性差异修复后都经过回归验证 | 审计报告产出时 | 审计结论中必须包含"建议回归验证的范围" |

| L12 | 所有 UI 设计页面都有响应式适配 | UI 语义 #2 | 如项目 UI-顶层定义有响应式要求，逐页面检查是否有 `@media` 或响应式结构变更 |

## 关键约束

- subagent **不可嵌套**（子 Agent 不能创建自己的子 Agent）
- subagent **只读**，禁止修改任何文件或代码
- 发现差异时必须明确标注：差异项、产物原文、代码原文、建议修正方向
