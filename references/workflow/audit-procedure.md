# 审计执行策略

所有审计（机械检查 + 语义审计）必须由**独立只读 subagent**执行。主会话禁止运行 `doc-audit.py`，禁止做任何语义判断。

---

## 核心原则

1. **单 agent 全量优先**：产物 < 1500 行时，一个 subagent 执行完整检查清单（机械+语义+遗漏），避免拆分导致的边界遗漏
2. **主会话预编排清单**：启动审计前，主会话先从本文档**复制对应产物的「完整检查清单」**写入 subagent prompt，禁止让 subagent 自行从模板中提取检查项
3. **强制逐项打勾**：subagent 必须对清单中的每一项给出明确结论（通过/不通过/存疑/不适用），禁止跳过，禁止笼统概括
4. **修复后最小回归**：用户修复问题后，执行精简回归清单，确认修复未引入新问题

---

## 任务类型与拆分策略

| 场景 | subagent 数量 | 策略 |
|------|--------------|------|
| 产物 < 1500 行 | **1** | 单 agent 全量审计（加载全文 + 直接上游 + 直接下游），执行完整检查清单 |
| 产物 ≥ 1500 行 | **2** | 按检查项编号对半拆分（如 #1 执行前半项 / #2 执行后半项），均加载全文+上下游 |
| 修改后回归 | **1** | 执行「修复后最小回归清单」 |

**密度回退**：产物包含大量 SVG 线框图、HTML 原型、长 SQL/接口代码块时，按 `字符数 × 1.5` 估算 token。单产物 + 上下游上下文合计可能超出 **150k tokens** 时，提前拆分。

**修改模式**：不按全文行数拆分，而是按变更范围聚焦——主会话用 `diff` 确定受影响章节，在完整检查清单中标记重点检查项（其余项仍需快速扫过）。

---

## 完整检查清单

> **核心分工**：
> - **内部完整性**：验证产物自身结构、格式、内部引用。脚本覆盖机械项，AI 覆盖语义项。
> - **跨步骤对齐**：验证产物与上下游产物的一致性。**主会话必须先预提取上游关键集合**（如功能编号列表、页面编号列表、状态列表）写入 prompt，subagent 逐项验证下游是否覆盖。
>
> 审计 subagent **只读本文档和产物文件**，不读取 `references/steps/*-step.md` 产物模板。

> **执行时机**：生成/修改任何**项目顶层定义**后，或模块产物审计前发现「顶层定义引用冲突」时。
> **审计对象**：项目 PRD-顶层定义、交互-顶层定义、UI-顶层定义、技术-顶层定义。
> **核心目标**：确认各顶层定义之间无冲突、无遗漏，且自身结构完整。模块产物与顶层定义的对齐留在各产物 B 部分检查。
>
> **主会话预提取**：加载所有顶层定义文件，提取模块划分表、术语表、状态值表、编码规则、Token 列表、组件状态列表，写入 prompt。

#### C1. 跨顶层定义一致性

| # | 检查项 | 负责方 | 精读方法 |
|---|--------|--------|---------|
| C-01 | PRD 模块划分 ↔ 技术工程结构映射 | 脚本 | 比对 PRD-顶层定义 §2 的模块标识/名称/职责 vs 技术-顶层定义 §2.2.2 的模块与代码包映射 |
| C-02 | PRD 术语表 ↔ 各顶层定义术语使用 | 脚本 | 提取 PRD-顶层定义 §4 术语关键词，检查其他顶层定义中是否出现「禁止别名」 |
| C-03 | PRD 状态值 ↔ 交互状态规范 | AI | 检查 PRD-顶层定义 §5 的状态值是否在交互-顶层定义 §2.2 中有对应交互状态定义 |
| C-04 | 交互 6+4 状态 ↔ UI 组件全局状态 | 脚本 | 比对交互-顶层定义 §2.2 的状态名称集合 vs UI-顶层定义 §2.11.2 的支持状态集合，报告差集 |
| C-05 | 交互令牌命名 ↔ UI Token 命名规范 | 脚本 | 检查交互-顶层定义 §2.1 的令牌命名模式是否符合 UI-顶层定义 §2.2 的 `category-usage-state` 规范 |
| C-06 | UI 响应式断点 ↔ 交互平台规范 | AI | 检查 UI-顶层定义 §2.5.5 的断点是否在交互-顶层定义 §2.6 平台交互规范中有对应适配规则 |
| C-07 | 技术-顶层定义接口约定 ↔ 交互反馈时效 | AI | 检查技术-顶层定义 §5.2 的响应结构字段是否与交互-顶层定义 §2.3.1 的反馈时效红线兼容（如 traceId 用于性能追踪） |

#### C2. 顶层定义自身完整性

| # | 检查项 | 负责方 | 精读方法 |
|---|--------|--------|---------|
| C-08 | 各顶层定义章节结构完整 | 脚本 | 检查各顶层定义是否包含模板要求的必填章节（无需求则声明"跳过"，不可直接删除） |
| C-09 | 各顶层定义 upstream-document 有效 | 脚本 | 检查各顶层定义头部 upstream-document 表格中的文档是否真实存在 |
| C-10 | 顶层定义间引用链完整 | 脚本 | 检查顶层定义正文中 `参见项目 X-顶层定义 §{章节}` 引用的章节是否存在于对应文件 |
| C-11 | 术语表无循环定义 | AI | 检查 PRD-顶层定义 §4 中术语 A 的定义是否引用术语 B，而 B 又引用 A |
| C-12 | 状态值有终态 | AI | 检查 PRD-顶层定义 §5 中每个状态字段是否有明确的终态（如 CLOSED） |
| C-13 | 编码规则覆盖所有产物编号类型 | AI | 检查 PRD-顶层定义 §6 是否定义了功能编号、页面编号、规则编号、错误码等所有需要的编码格式 |
| C-14 | 技术-顶层定义模块映射无遗漏 | AI | 检查技术-顶层定义 §2.2.2 是否覆盖了 PRD-顶层定义 §2 的所有模块 |
| C-15 | UI 组件库状态与交互状态规范对齐 | AI | 检查 UI-顶层定义 §2.11.2 的全局状态规范是否声明了交互-顶层定义 §2.2 的所有 6+4 种状态 |
| C-16 | 顶层定义间版本号一致 | 脚本 | 检查各顶层定义的版本号是否与项目当前版本一致（如 v1.0） |

## 审计输出格式（强制）

subagent 必须按以下格式返回审计报告。清单中的每一项都必须出现在「详细结果」表格中，**不得跳过、不得合并**。

```markdown
## 审计结论

**产物**：{文件名}  
**总项数**：{N} | **通过**：{N} | **不通过**：{N} | **存疑**：{N} | **不适用**：{N}

### 详细结果

| # | 检查项 | 结论 | 证据（章节/行号/原文） | 建议 |
|---|--------|------|----------------------|------|
| P-01 | §1 成功指标可量化 | 通过 | §1 第3行"平均响应时间从4h降至1h" | - |
| P-02 | §2 用户故事对应§3功能 | 不通过 | STORY-003 无 §3 引用 | 补充 TICKET-003 |
| ... | ... | ... | ... | ... |

### 完整性自检

- [x] 已完成清单中所有 {N} 项检查
- [ ] 有跳过项（如勾选此项，必须列出跳过的项及原因）
```

**主会话验收标准**：
- 如果「详细结果」表格的行数 ≠ 检查清单项数 → 打回重审，要求补全
- 如果存在「不通过」项 → 汇总输出给用户，等待修复后执行回归
- 如果所有项「通过」或「不适用」→ 本轮审计通过

## 修复后最小回归清单

用户修复上一轮问题后，启动单 agent 执行以下 5 项回归检查，确认修复未引入新问题：

| # | 检查项 | 说明 |
|---|--------|------|
| R-01 | 修改点引用完整性 | 检查修改处前后 3 行内的编号引用、链接是否断裂 |
| R-02 | 编号连续性 | 若修改涉及编号，检查是否出现跳号或重复 |
| R-03 | 术语一致性 | 修改处使用的术语与全文及上游文档是否一致 |
| R-04 | 跨章节引用同步 | 修改的内容是否在相关章节（如 §3 功能修改后检查 §6/§7/§8 引用）同步更新 |
| R-05 | 原始问题复核 | 确认上一轮标记为「不通过」的项已修复，且未引入新的「不通过」项 |

> **顶层定义修改的额外回归**：若修复涉及**项目顶层定义**（PRD-顶层定义/交互-顶层定义/UI-顶层定义/技术-顶层定义），增加以下两项：
>
> | # | 检查项 | 说明 |
> |---|--------|------|
> | R-T1 | 下游产物引用同步 | 检查所有声明继承该顶层定义的模块产物，其引用内容（术语/状态/Token/编码）是否与修改后的顶层定义一致 |
> | R-T2 | 跨顶层定义联动 | 检查与该顶层定义有引用关系的其他顶层定义是否需要同步更新（如 PRD 术语表修改后，交互/UI/技术顶层定义中的术语引用） |

回归清单的输出格式与全量审计相同，必须逐项打勾。

## 调用 doc-audit.py

subagent 的工作目录通常是项目目录而非 skill 根目录，执行前必须先定位脚本：

```bash
# 方式1：通过 find 定位（推荐）
SCRIPT=$(find ~ -path "*/e2e-solution-guard/scripts/doc-audit.py" 2>/dev/null | head -1)
python3 "$SCRIPT" <doc_path> --type <prd|interaction|ui|tech|test>

# 方式2：如果已知 skill 根目录
python3 <skill_root>/scripts/doc-audit.py <doc_path> --type <type>
```

## subagent 配置建议

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 类型 | `explore` | 审计任务禁止写入任何文件 |
| 并行 | foreground | 多 agent 时同时发起，等待全部结果返回后汇总 |
| 超时 | 不设硬上限 | 复杂产物审计需要充足时间，确保不会因超时而中断 |

## 关键约束

- subagent **不可嵌套**（子 Agent 不能创建自己的子 Agent）
- subagent **只读**，禁止修改任何文件或代码
- 发现差异时必须明确标注：差异项、产物原文、建议修正方向
