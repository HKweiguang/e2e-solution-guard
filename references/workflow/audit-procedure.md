# 审计执行策略

所有审计（机械检查 + 语义审计）必须由**独立只读 subagent**执行。主会话禁止运行 `doc-audit.py`，禁止做任何语义判断。
核心原则：**一个 subagent 只审计一个任务**。

---

## 常见任务类型

| 任务 | 加载内容 | 适用场景 |
|------|---------|---------|
| 单文档审计 | 当前文档 + 直接上游文档 | 新建/修改一份 PRD、技术方案等 |
| 单模块代码审计 | 当前技术方案 + 对应代码文件 | 代码落地验证阶段 |
| 跨文档链路检查 | 两份上下游文档的关键章节 | 验证编号引用、接口映射、错误码对应 |
| 下游影响扫描 | 上游文档 + 下游文档头部 | 修改上游后评估影响范围 |

## 拆分决策流程

主会话在启动审计前，**必须先用 `wc -l` 统计当前文档行数**，然后按以下规则决定拆分策略。

### 行数阈值与超时配置

| 总行数 | subagent 数量 | 每个 timeout | 策略 |
|--------|--------------|-------------|------|
| < 300 | 1 | 600 | 单 subagent 全量审计 |
| 300-600 | 2 | 600 | 按章节拆成 2 组，并行执行 |
| > 600 | 3 | 900 | 按章节拆成 2 组 + 跨文档链路单独 1 组，并行执行 |

### 按文档类型的具体拆分策略

#### PRD（§1-§10）

| subagent | 负责章节 | 检查重点 | 加载内容 |
|----------|---------|---------|---------|
| #1 机械 | §1-§5 | 编号连续性(F/P)、表格格式、upstream引用、必填章节 | 当前文档 + 顶层定义 |
| #2 语义 | §6-§10 | 业务规则一致性、错误码触发场景、验收标准格式、价值触达完整性 | 当前文档 + 上游 PRD + 下游技术方案(如有) |
| #3 链路(可选) | 跨文档 | §2功能点→下游引用、§4页面→交互设计、§7错误码→技术方案 | 当前文档 + 下游文档头部 |

#### 交互设计（每个页面为 §N Pxxx）

| 页面数量 | 拆分方式 |
|---------|---------|
| < 5 | 单 subagent 全量 |
| 5-10 | 拆 2 组：前半页面 / 后半页面 |
| > 10 | 拆 3 组，每组不超过 4 个页面 |

每组内部检查：5 个子节齐全、状态机完整、与 PRD 对应关系。

#### UI 设计（§1-§4，§4 按页面拆）

| §4 页面数量 | 拆分方式 |
|------------|---------|
| < 5 | 单 subagent 全量 |
| ≥ 5 | 拆 2 组：§1-§3 + 前半页面 / 后半页面 |

#### 技术方案（§1-§10）

| subagent | 负责章节 | 检查重点 |
|----------|---------|---------|
| #1 机械 | §1-§5 | 表结构审计字段、接口清单与§6对应、决策编号连续 |
| #2 语义 | §6-§10 | 接口字段与上游一致、异常场景映射、缓存策略覆盖 |
| #3 代码(可选) | 代码文件 | 阶段5代码落地验证（参见 `references/workflow/code-verification.md`） |

#### 测试（§1-§3）

测试文档通常较短（模板 90 行），**默认不拆**。单 subagent 全量审计。

### 修改模式的增量拆分

如果是**修改模式**，不按全文行数拆分，而是按变更范围拆分：

1. 主会话用 `diff` 生成变更范围，确定受影响的章节
2. 只启动审计受影响章节的 subagent：
   - 只改了 §2 功能清单 → 启动"机械"subagent，审计 §2 + 引用 §2 的 §4/§6/§7/§9
   - 改了 §6 业务规则 → 启动"语义"subagent，审计 §6 + 上游 PRD §6 + 下游技术方案 §7/§8
3. 未变更的章节不审计

### 并行执行与结果汇总

主会话使用 `run_in_background=true` 并行启动所有 subagent：

```
subagent-1 (机械): prompt="审计§1-§5，重点检查..." run_in_background=true timeout=600
subagent-2 (语义): prompt="审计§6-§10，重点检查..." run_in_background=true timeout=600
```

待全部返回后，主会话汇总：
- 所有 blocking issue 合并输出
- 所有 warning/hint 合并输出
- 如有冲突（两个 subagent 对同一问题结论不同），主会话仲裁或启动第三个 subagent 复核

## 加载优先级（同一任务内）

1. 当前文档（全文或变更 diff）
2. 直接上游文档（被引用的章节/段落）
3. 直接下游文档（upstream-document 表格 + 受影响段落）
4. 间接上下游（按需，通常只在顶层定义变更时涉及）

## subagent 配置建议

| 配置项 | 建议值                      | 说明 |
|--------|--------------------------|------|
| 类型 | `explore` | 审计任务禁止写入 |
| 超时 | 1800 秒以上                 | 复杂文档审计需要充足时间 |
| 后台运行 | 多任务并行时可启用                | 减少主会话等待 |

## 语义审计执行方法

subagent 的审计任务必须同时产出**机械检查结果**和**语义审计结果**。机械检查通过调用 `doc-audit.py` 完成，语义检查由模型直接分析完成，两者在同一个 subagent 会话内执行，缺一不可。

### 执行步骤

1. **定位检查清单**：根据当前文档类型（prd/interaction/ui/tech/test），读取 `references/steps/{type}-step.md` 的 §5 检查清单，提取所有标记为 `[模型]` 的检查项
2. **加载必要上下文**：除当前文档外，根据 `[模型]` 项的描述加载需要比对的上游或下游文档（如 PRD、交互设计、技术方案等）
3. **逐项分析并输出**：针对每个 `[模型]` 检查项，在当前文档与关联文档之间进行比对分析，输出结构化结果：

   | 字段 | 说明 |
   |------|------|
   | 检查项 | `[模型]` 项的原文描述 |
   | 结果 | **通过** / **不通过** / **存疑** |
   | 证据 | 引用的文档原文（章节号、段落、表格行），禁止笼统判断 |
   | 建议 | 如不通过或存疑，给出明确的修正方向 |

4. **汇总**：将所有 `[模型]` 项的结果汇总为语义审计结论，连同机械检查结果一并返回主会话

### 关键原则

- `[模型]` 检查项无法被脚本自动化，必须依靠模型的阅读理解能力逐条分析
- 语义审计的结论必须有文档原文支撑，禁止无依据的笼统判断
- 如果上下文不足以判断某项，标注为「存疑」并说明缺少的信息

## 关键约束

- subagent **不可嵌套**（子 Agent 不能创建自己的子 Agent）
- subagent **只读**，禁止修改任何文件或代码
- 发现差异时必须明确标注：差异项、文档原文、代码原文、建议修正方向
