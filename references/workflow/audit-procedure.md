# 审计执行策略

所有审计（机械检查 + 语义审计）必须由**独立只读 subagent**执行。主会话禁止运行 `doc-audit.py`，禁止做任何语义判断。
核心原则：**一个 subagent 只审计一个任务**。

---

## 常见任务类型

| 任务 | 加载内容 | 适用场景 |
|------|---------|---------|
| 单产物审计 | 当前产物 + 直接上游文档 | 新建/修改一份 PRD、技术方案等 |
| 单模块代码审计 | 当前技术方案 + 对应代码文件 | 代码落地验证阶段 |
| 跨产物链路检查 | 两份上下游产物的关键章节 | 验证编号引用、接口映射、错误码对应 |
| 下游影响扫描 | 上游文档 + 下游产物头部 | 修改上游后评估影响范围 |

---

## 拆分决策流程

主会话在启动审计前，**先用 `wc -l` 统计当前产物行数**作为拆分依据。基于 Kimi k2.6 的 256k 上下文能力，按以下规则决定拆分策略：

### 行数阈值与拆分策略

| 总行数 | subagent 数量 | 策略 |
|--------|--------------|------|
| < 1000 | 1 | 单 subagent 全量审计（机械+语义） |
| 1000-2000 | 2 | 按维度拆：机械检查 / 语义检查，并行执行 |
| > 2000 | 3 | 按维度拆：机械 / 语义前半 / 语义后半，并行执行 |

**密度回退**：如果产物包含大量 SVG 线框图、HTML 原型、长 SQL/接口代码块等高密度内容，按 `字符数 × 1.5` 粗略估算 token。若单产物 + 上下游上下文合计可能超出 **150k tokens**，主会话应下调阈值（如 `< 600` 即拆）。

**核心原则：按检查维度拆，不按章节拆**。每个 subagent 都加载当前产物全文（机械 subagent 额外加载直接上游，语义 subagent 额外加载上下游），确保跨章节的检查项（如"§2 功能编号在 §6/§7/§8 中的引用一致性"）和跨产物的比对（如"页面编号前缀与 PRD 功能编号一致"）不会遗漏。

### 按产物类型的具体拆分策略

#### PRD

| subagent | 检查维度 | 加载内容 | 负责项 |
|----------|---------|---------|--------|
| #1 机械 | `[脚本]` 标记项 + 脚本未覆盖项 | 当前产物全文 + 项目 PRD-顶层定义 | 编号重复/跳号检测、表格格式、upstream引用、必填章节、术语一致性、验收标准格式、功能编号对应关系 |
| #2 语义 | `[模型]` 标记项 | 当前产物全文 + 项目 PRD-顶层定义 + 下游技术方案(如有) | 目标一致性（成功指标可量化、功能与用户故事对应）、范围一致性（模块职责边界）、业务规则一致性、错误码触发场景、跨章节引用一致性、验收标准格式 |
| #3 链路(可选) | 跨产物链路 | 当前产物 + 下游产物头部 | 功能点→下游、页面→交互设计、错误码→技术方案 |

#### 交互设计

| 页面数量 | 策略 |
|---------|------|
| < 15 | 单 subagent 全量（机械+语义） |
| ≥ 15 | 拆 2 组：#1 机械检查（全文+PRD §3，含组件状态矩阵/力学约束/页面编号连续性）/ #2 语义检查（全文+上游PRD+项目交互-顶层定义，重点检查状态映射、页面流程覆盖度、视觉边界一致性） |

#### UI 设计

| 页面数量 | 策略 |
|------------|------|
| < 15 | 单 subagent 全量（机械+语义） |
| ≥ 15 | 拆 2 组：#1 机械检查（全文+交互设计+UI-顶层定义，重点检查 HTML 代码块结构、CSS 变量、状态样式、状态映射表4字段、CSS类名一致性）/ #2 语义检查（全文+上游交互设计+项目 UI-顶层定义，重点检查状态映射、Token 一致性、行为边界一致性） |

#### 技术方案

| subagent | 检查维度 | 加载内容 | 负责项 |
|----------|---------|---------|--------|
| #1 机械 | `[脚本]` 标记项 + 脚本未覆盖项 | 当前产物全文 + PRD + 项目技术-顶层定义 | 表结构审计字段、接口清单与接口设计对应、接口功能点标注、审计字段、接口5要素 |
| #2 语义 | `[模型]` 标记项 | 当前产物全文 + 上游 PRD（可能有多份）+ 项目技术-顶层定义 | 接口字段与上游一致、异常场景映射、缓存策略覆盖、权限规则一致性、状态一致性（状态机与PRD对齐）、Token一致性（缓存Key/权限与顶层定义对齐） |
| #3 代码(可选) | 代码落地验证 | 技术方案 + 代码文件 | 阶段5验证（参见 `references/workflow/code-verification.md`） |

#### 测试

测试产物模板 350 行左右，**默认不拆**。单 subagent 全量审计，加载内容：当前产物全文 + PRD + 交互设计 + UI + 技术方案。

### 修改模式的增量拆分

如果是**修改模式**，不按全文行数拆分，而是按变更范围拆分：

1. 主会话用 `diff` 生成变更范围，确定受影响的维度（机械项 or 语义项）
2. 启动对应维度的 subagent，但**仍然加载全文+上游**，只是检查重点聚焦在变更部分：
   - 只改了功能需求章节（影响编号和引用）→ 启动"机械"subagent，加载全文+上游，重点检查编号连续性和跨章节引用一致性
   - 改了 §6 业务规则 → 启动"语义"subagent，加载全文+上下游，重点检查业务规则一致性
3. 未受影响的维度可跳过（如只改表格格式，语义检查可跳过）

### 并行执行与结果汇总

主会话**同时启动多个只读 subagent**并行执行审计，每个 subagent 独立加载所需上下文，独立完成检查：

```
subagent-1 (机械): subagent_type="explore", run_in_background=false, prompt="加载全文+上游，执行所有[脚本]标记检查项..."
subagent-2 (语义): subagent_type="explore", run_in_background=false, prompt="加载全文+上下游，按以下逐条拆表精读清单执行..."
```

> **只读约束**：所有审计 subagent 必须使用 `explore` 类型，禁止写入任何文件。subagent 的审计结论通过返回结果传递，不由 subagent 自行修改产物。
>
> **并行方式**：主会话在一次响应中同时发起多个 **foreground** subagent 实现并行；审计任务严禁使用 `run_in_background=true`，必须等待全部结果返回后主会话方可汇总。

待全部返回后，主会话汇总：
- 所有 blocking issue 合并输出
- 所有 warning/hint 合并输出
- 如有冲突（两个 subagent 对同一问题结论不同），主会话仲裁或启动第三个 subagent 复核

## 加载优先级（同一任务内）

1. 当前产物（全文或变更 diff）
2. 直接上游文档（被引用的章节/段落）
3. 直接下游产物（上游文档 表格 + 受影响段落）
4. 间接上下游（按需，通常只在顶层定义变更时涉及）

## subagent 配置建议

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 类型 | `explore` | 审计任务禁止写入 |
| `run_in_background` | `false`（禁止） | 审计结果必须返回主会话汇总，**禁止后台执行** |

> 复杂产物审计需要充足时间，确保 subagent 不会因超时中断而遗漏检查项。

## 脚本与 AI 的分工边界（架构决策）

> **这是一个必须遵守的架构决策**。禁止用脚本覆盖 AI 的职责，也禁止让 AI 做脚本该做的事。后续扩展检查项时，必须先对照此边界判断归属，不可凭直觉"一条路走到黑"。

### 脚本负责：结构化、确定性、可穷举的检查

特征：有明确的模式、规则、穷举空间，结果只有"通过/不通过"两种。

- **格式与模式**：ID 格式、连续性、前缀一致性；编号跳号/重复；表格列完整性
- **存在性验证**：链接有效性；必填字段/章节存在；双向引用存在（A 提到 B，B 是否提到 A）
- **与顶层定义一致性**：状态名、变量名、错误码前缀、缓存 Key 命名规范、枚举值等**格式层面**的对齐
- **跨产物编号映射存在性**：PRD 错误码列表中的每个 ID 是否在 Tech §7 中出现；PRD 功能点是否在下游产物中被引用
- **计数验证**：数量是否匹配（如 PRD 错误码数量 = Tech 异常数量 = Test 验证数量）

### AI 负责：语义、推理、业务逻辑判断

特征：需要理解业务含义、做推理判断、结果可能有"存疑"。

- **映射正确性**：字段映射是否正确（而非仅仅存在同名）；状态值是否语义一致（而非仅仅同名）
- **需求满足度**：需求是否被真正满足（而非仅仅有对应章节）；安全措施是否覆盖安全需求；力学约束是否有合理技术实现
- **具体可执行性**：描述是否具体（如错误码触发场景是否可执行，非"用户操作时"）
- **设计合理性**：状态机设计是否合理；边界是否清晰；是否过度复杂化
- **跨产物语义对齐**：PRD 性能指标是否在 Tech 中有**合理的**性能目标（而非仅仅有章节）

### 决策标准

| 标准 | 脚本 | AI |
|------|------|-----|
| 判断依据是模式/格式/存在性 | ✓ | |
| 判断依据是语义/正确性/合理性 | | ✓ |
| 可以用正则表达式或简单字符串匹配完成 | ✓ | |
| 需要理解业务上下文才能判断 | | ✓ |
| 结果是确定性的（无模糊地带） | ✓ | |
| 结果可能有"存疑" | | ✓ |
| 穷举空间明确且有限 | ✓ | |
| 需要推理或综合多段信息 | | ✓ |

### 扩展原则

当新增检查项时，按以下流程判断归属：

1. **先问**：这项检查能否用确定性规则描述？是否存在唯一的正确答案？
2. **如果能** → 交给脚本实现（放入 `doc-audit.py` 规则集）
3. **如果不能** → 交给 AI 语义审计（标记为 `[模型]`，由 subagent 执行）
4. **禁止降级**：不可为了让脚本覆盖而放宽规则精度（如把"字段映射正确"降级为"字段名存在"后交给脚本），这会制造虚假安全感
5. **禁止越界**：脚本发现异常时只报"不通过"，不做语义解释；AI 发现格式问题时只报"发现格式异常"，不替代脚本做穷举检查

## 语义审计执行方法

subagent 的审计任务按负责维度产出结果：
- **机械 subagent**：调用 `doc-audit.py` 产出机械检查结果，同时对脚本尚未覆盖的 `[脚本]` 项执行补充检查
- **语义 subagent**：按逐条拆表精读清单产出语义审计结果

两者分工明确，不重复执行同一检查项。

**调用 `doc-audit.py` 的路径问题**：subagent 的工作目录通常是项目目录而非 skill 根目录，直接调用 `scripts/doc-audit.py` 会因找不到文件而失败。执行前必须先定位脚本的绝对路径：

```bash
# 方式1：通过 find 定位（推荐，适用于所有环境）
SCRIPT=$(find ~ -path "*/e2e-solution-guard/scripts/doc-audit.py" 2>/dev/null | head -1)
python3 "$SCRIPT" <doc_path> --type <prd|interaction|ui|tech|test>

# 方式2：如果已知 skill 根目录
python3 <skill_root>/scripts/doc-audit.py <doc_path> --type <type>
```

### 执行步骤

1. **定位检查清单**：根据当前产物类型（prd/interaction/ui/tech/test），读取 `references/steps/{type}-step.md` 的检查清单章节，提取所有标记为 `[模型]` 的检查项。注意：`ui-step.md` 模板的结构与其他步骤不同（§1-§5，无 §6 检查清单章节——检查清单位于 **§5**），UI 设计的语义审计以模板中声明的原则和边界为检查依据
2. **加载必要上下文**：除当前产物外，根据 `[模型]` 项的描述加载需要比对的上游或下游产物（如 PRD、交互设计、技术方案等）
3. **逐项分析并输出**：针对每个 `[模型]` 检查项，在当前产物与关联产物之间进行比对分析，输出结构化结果：

   | 字段 | 说明 |
   |------|------|
   | 检查项 | `[模型]` 项的原文描述 |
   | 结果 | **通过** / **不通过** / **存疑** |
   | 证据 | 引用的产物原文（章节号、段落、表格行），禁止笼统判断 |
   | 建议 | 如不通过或存疑，给出明确的修正方向 |

4. **汇总**：将所有 `[模型]` 项的结果汇总为语义审计结论，连同机械检查结果一并返回主会话

### 关键原则

- `[模型]` 检查项无法被脚本自动化，必须依靠模型的阅读理解能力逐条分析
- 语义审计的结论必须有产物原文支撑，禁止无依据的笼统判断
- 如果上下文不足以判断某项，标注为「存疑」并说明缺少的信息

## 检查清单框架（6 维度 × 3 层级）

> 以下框架是所有产物检查的**统一组织方式**。每个维度的检查按层级展开：
> - **L1 单产物内**：产物自身的结构和内容完整性
> - **L2 相邻产物间**：与直接上游/下游产物的一致性
> - **L3 端到端链路**：从需求到测试/代码的完整追溯链
>
> 外加 2 个横向维度：
> - **版本维度**：各产物版本号、变更范围的一致性
> - **边界维度**：产物职责边界，越界内容检测

### 维度总览

| 维度 | 核心问题 | 典型失败后果 |
|------|---------|-------------|
| **结构** | 章节齐吗？表格格式对吗？编号连续吗？ | 开发遗漏章节、联调发现接口缺失 |
| **数据** | 字段名全链路一致吗？枚举值对齐吗？ | 接口字段和 UI 对不上、状态判断错误 |
| **状态** | 状态定义完整吗？转换条件实现了吗？ | 空状态/错误状态未处理、状态机不一致 |
| **错误** | 错误码全链路覆盖吗？异常场景齐全吗？ | 用户看到"系统繁忙"、服务雪崩 |
| **性能** | 性能指标传递了吗？缓存策略落实了吗？ | 上线后 8s 响应、数据库拖垮 |
| **安全** | 安全需求实现了吗？测试覆盖了吗？ | 越权漏洞、敏感数据泄露 |
