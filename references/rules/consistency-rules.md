# 通用一致性硬规则

以下规则在所有步骤文档的生成和修改中强制适用，不随项目上下文变化。

---

## 编号连续性

- 同一模块/文档内的同类编号必须连续无跳号，编号格式以项目 PRD-顶层定义编码规则为准
- 用例编号在**同一文档内**必须连续无跳号
- 禁止同一编号在不同文档中定义不同含义
- 编号一旦分配即永久占用，删除功能/页面后编号留空不回收

## 双向引用完整性

- PRD §3 的功能点必须在 §6 业务规则 / §7 错误处理 / §8 验收标准中至少被一个章节引用（§5 数据模型通常不引用功能编号）
- 交互设计中的每个页面必须在「与 PRD 对应」子节中明确标注来源
- 技术方案接口清单章节必须与接口设计章节逐个接口一一对应（数量、路径、方法）
- 测试报告覆盖检查章节中的每个验收标准必须在 §1–§5 的对应测试用例中有覆盖（功能/异常/性能/安全/兼容性测试）

## 术语一致性

- 同一术语在同一文档中必须使用完全相同的名称（禁止「用户/客户/会员」混用）
- 术语首次出现时标注来源（项目顶层定义或本模块定义）
- 发现上下游术语定义冲突 → 立即停止，按回改流程处理

## 顶层定义交叉对齐

- PRD-顶层定义的模块划分表，必须在交互-顶层定义的主导航和技术-顶层定义的模块与代码包映射中有对应定义（模块数量、名称、职责边界一致）
- PRD-顶层定义的术语表，必须在技术-顶层定义的通用字段命名约定和交互-顶层定义的主导航名称中有对应使用
- PRD-顶层定义的状态值，必须在技术-顶层定义的公共表定义中有对应字段类型
- 交互-顶层定义的业务状态机，必须在 UI-顶层定义的色彩角色和组件状态中有对应视觉定义（如项目未建立 UI-顶层定义，此条豁免）
- 交互-顶层定义的交互状态规范（6+4种状态），必须在 UI-顶层定义的组件全局状态规范中有对应视觉定义（如禁用态、加载态、错误态的视觉表现）（如项目未建立 UI-顶层定义，此条豁免）
- 技术-顶层定义的技术栈选型，必须在各模块技术方案的 §1 技术决策中有对应引用或超出声明
- 技术-顶层定义的公共表审计字段规范，必须在各模块技术方案的数据模型中强制执行
- 技术-顶层定义的全局接口约定（HTTP方法/响应结构/分页/认证/版本），必须在各模块技术方案的接口设计中有对应遵循
- 禁止出现 PRD-顶层定义有定义但其他顶层定义无对应的情况，发现缺失 → 触发回改流程

## 模块文档交叉对齐

- PRD §5 数据模型中定义的每个实体，必须在技术方案数据模型章节中有对应表结构
- PRD §6 业务规则中定义的每个状态流转，必须在技术方案状态机设计章节中有对应状态转换
- PRD §7 错误处理中定义的每个错误码，必须在技术方案异常处理章节和测试报告异常测试用例中有对应映射
- PRD §8 验收标准中定义的每个验收项，必须在测试报告功能测试用例和覆盖检查报告中有对应用例
- 交互设计页面流程中定义的每个页面跳转，必须在技术方案接口设计中有对应接口支持
- 交互设计状态机中定义的每个状态，必须在技术方案状态机设计中有对应状态值
- 如模块生成了 UI 设计文档：UI 设计的 HTML 原型中实现的 CSS 状态样式（:hover/:active/:focus/.disabled/.loading），必须在交互设计的组件交互子节中有对应状态定义
- 如模块生成了 UI 设计文档：UI 设计的 HTML 原型中实现的异步状态样式（.empty/.error/.success/.skeleton），必须在交互设计的状态机子节中有对应触发条件
- 技术方案 §4 接口设计中定义的每个接口，必须在测试报告功能/异常测试用例中有对应测试覆盖
- 技术方案 §7 异常处理中定义的每个异常场景，必须在测试报告异常测试用例中有对应测试覆盖
- 技术方案 §10 安全设计中定义的每个安全措施，必须在测试报告安全测试中有对应验证

## 常见遗漏项（生成后必须逐条确认）

- [ ] 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术）
- [ ] 所有异常场景都对应了错误码或业务规则
- [ ] 所有状态流转都有明确的触发条件和目标状态
- [ ] 所有页面都有空状态和错误状态的定义
- [ ] 所有接口都标注了对应的功能点
- [ ] 所有表结构都包含审计字段（如项目技术-顶层定义有要求）
- [ ] 所有第三方依赖都有不可用时的降级策略（技术方案）
- [ ] 所有 P0 用例都有明确的前置条件、测试步骤和预期结果
- [ ] 所有阻塞性差异修复后都经过回归验证（审计报告）
