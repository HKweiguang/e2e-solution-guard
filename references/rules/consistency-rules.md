# 通用一致性硬规则

以下规则在所有步骤文档的生成和修改中强制适用，不随项目上下文变化。

---

## 编号连续性

- 同一模块/文档内的同类编号必须连续无跳号，编号格式以项目 PRD-顶层定义为准
- 用例编号在**同一文档内**必须连续无跳号
- 禁止同一编号在不同文档中定义不同含义
- 编号一旦分配即永久占用，删除功能/页面后编号留空不回收

## 双向引用完整性

- PRD §3 的功能点必须在 §5/§6/§7/§8 中至少被一个章节引用
- 交互设计中的每个页面必须在「与 PRD 对应」子节中明确标注来源
- 技术方案 §10 接口清单必须与 §6 逐个接口一一对应（数量、路径、方法）
- 测试报告 §3 覆盖检查中的每个验收标准必须在 §1 或 §2 中有对应用例

## 术语一致性

- 同一术语在同一文档中必须使用完全相同的名称（禁止「用户/客户/会员」混用）
- 术语首次出现时标注来源（项目顶层定义或本模块定义）
- 发现上下游术语定义冲突 → 立即停止，按回改流程处理

## 顶层定义交叉对齐

- PRD-顶层定义的模块划分表，必须在交互-顶层定义的主导航和技术-顶层定义的模块与代码包映射中有对应定义（模块数量、名称、职责边界一致）
- PRD-顶层定义的术语表，必须在技术-顶层定义的通用字段命名约定和交互-顶层定义的主导航名称中有对应使用
- PRD-顶层定义的状态值，必须在技术-顶层定义的公共表定义中有对应字段类型
- 交互-顶层定义的业务状态机，必须在 UI-顶层定义的语义色和组件状态中有对应视觉定义
- 禁止出现 PRD-顶层定义有定义但其他顶层定义无对应的情况，发现缺失 → 触发回改流程

## 常见遗漏项（生成后必须逐条确认）

- [ ] 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术）
- [ ] 所有异常场景都对应了错误码或业务规则
- [ ] 所有状态流转都有明确的触发条件和目标状态
- [ ] 所有页面都有空状态和错误状态的定义
- [ ] 所有接口都标注了对应的功能点和页面编号
- [ ] 所有表结构都包含审计字段（如项目技术-顶层定义有要求）
