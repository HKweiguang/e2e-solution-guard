# 通用一致性硬规则

以下规则在所有步骤产物的生成和修改中强制适用，不随项目上下文变化。

具体检查项和执行方式见 `references/workflow/audit-procedure.md`。

---

## 1. 编号连续性

- 同一模块/产物内的同类编号必须连续无跳号，编号格式以项目 PRD-顶层定义编码规则为准
- 用例编号在**同一产物内**必须连续无跳号
- 禁止同一编号在不同产物中定义不同含义
- 编号一旦分配即永久占用，删除功能/页面后编号留空不回收

## 2. 引用完整性

- 产物内部：定义项必须在产物内部被引用（如 PRD §3 功能编号必须在 §6/§7/§8 中出现）
- 产物之间：下游产物必须通过 upstream-document 表格声明上游来源，且引用的内容必须在对应上游产物中存在
- 跨产物关联必须通过显式标识符建立（如功能点编号、页面编号、错误码），禁止隐式关联

## 3. 术语一致性

- 同一术语在同一产物中必须使用完全相同的名称
- 术语首次出现时标注来源（项目顶层定义或本模块定义）
- 发现上下游术语定义冲突 → 立即停止，按回改流程处理

## 4. 顶层定义与模块产物的对齐

- 项目顶层定义（PRD/交互/UI/技术）之间必须相互对齐，禁止出现某顶层定义有定义但其他顶层定义无对应的情况
- 模块级产物必须继承并遵循对应顶层定义的规范（编码规则、状态值、术语、接口约定等）
- 模块级产物超出顶层定义范围时，必须在产物中明确声明超出部分及理由

## 5. 跨产物映射原则

- PRD 是需求源点，交互/UI/技术/测试产物必须与 PRD 建立可追溯的映射关系
- 交互设计是 UI 的上游，UI 必须与交互设计建立可追溯的映射关系
- 技术方案是测试的上游，测试必须与技术方案建立可追溯的映射关系
- 多对多映射是允许的，但必须显式标注（如一个功能点对应多个接口、一个接口支撑多个功能点）
