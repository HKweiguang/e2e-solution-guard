# 通用一致性硬规则

以下规则在所有步骤产物的生成和修改中强制适用，不随项目上下文变化。

组织方式：**6 维度 × 3 层级** 框架的跨产物规则集，外加 **2 个横向维度**（版本维度、边界维度）。

---

## 一、编号与术语基础规则

### 编号连续性

- 同一模块/产物内的同类编号新增时必须按序递增，不得跳号分配，编号格式以项目 PRD-顶层定义编码规则为准
- 用例编号在**同一产物内**新增时必须按序递增，不得跳号分配
- 禁止同一编号在不同产物中定义不同含义
- 编号一旦分配即永久占用，删除功能/页面后编号留空不回收

### 术语一致性

- 同一术语在同一产物中必须使用完全相同的名称（禁止「用户/客户/会员」混用）
- 术语首次出现时标注来源（项目顶层定义或本模块定义）
- 发现上下游术语定义冲突 → 立即停止，按回改流程处理
- **L3** 同一术语在 PRD/交互/UI/技术/测试 中名称一致

---

## 二、结构维度规则

### L1 单产物内

- 各产物必填章节齐全，表格格式正确，编号连续无重复
- PRD §3 功能需求表格列完整（编号/故事/名称/优先级/交互方式/技术单元/规则/验收）
- 交互设计每个页面包含 6 个必含子节（页面结构/组件交互/状态机/页面流程/异常处理/与PRD对应）
- UI 设计 HTML 包含完整 DOCTYPE/html/head/body，使用 CSS 变量，状态映射表覆盖 10 种状态
- 技术方案 §3 每张表包含审计字段，§4 每个接口包含 6 要素（URL/方法/请求参数/响应结构/错误码/版本号），§13 接口清单与 §4 对应
- 测试 §1–§7 章节齐全，§1 每个用例包含前置条件/测试步骤/预期结果

### L2 相邻产物间

- PRD-顶层定义的模块划分表，必须在交互-顶层定义的主导航和技术-顶层定义的模块与代码包映射中有对应定义（模块数量、名称、职责边界一致）
- 各产物版本号必须与上游产物版本号一致
- 各产物覆盖功能点/页面列表必须与产物实际内容一致

### L3 端到端链路

- 交互设计页面流程中定义的每个页面跳转，必须在技术方案接口设计中有对应接口支持
- UI 设计的响应式断点定义，必须在测试报告兼容性测试用例中有对应验证
- 技术方案 §4 接口设计中定义的每个接口，必须在测试报告功能/异常测试用例中有对应测试覆盖

---

## 三、数据维度规则

### L1 单产物内

- PRD §6 权限规则必须引用 PRD-顶层定义 §2.2.5 权限级别定义，禁止将通用权限级别拆分为独立业务域
- PRD §5/§6 中首次出现的业务术语须标注来源（项目顶层定义或本模块定义）

### L2 相邻产物间

- PRD-顶层定义的术语表，必须在技术-顶层定义的通用字段命名约定和交互-顶层定义的主导航名称中有对应使用
- PRD-顶层定义的状态值，必须在技术-顶层定义的公共表定义中有对应字段类型
- PRD §5 数据模型中定义的每个实体，必须在技术方案 §3 中有对应表结构
- PRD §5 数据模型字段名与技术方案 §3 字段名可映射，字段类型可映射
- PRD §5 枚举值与技术方案 §3 字段枚举值一致
- 技术方案 §3 表字段在技术方案 §4 接口请求/响应参数中有对应
- 技术方案 §4 接口响应字段与 UI 展示字段一致

### L3 端到端链路

- PRD §5 每个实体在技术方案 §3 中有对应表
- PRD §5 每个字段在技术方案 §3/§4 中有对应实现
- UI 设计的 §3.2 状态映射表必须显式声明：交互状态 → CSS 类/伪类 → Token 变量 → 技术方案引用的完整映射链，禁止让映射关系隐式散落在 HTML 代码中
- UI 设计的 HTML 原型中定义的每个组件和状态类，必须在技术方案前端实现中有对应引用（组件库选型、样式实现方案）

---

## 四、状态维度规则

### L1 单产物内

- PRD §6 业务规则中的状态值在 §5 或顶层定义中声明，每个状态流转有明确的触发条件和目标状态
- 交互设计每个页面状态机包含 6 种状态（默认/空/加载/错误/成功/骨架），与 PRD 业务状态建立了映射
- UI 设计 HTML 包含 9 种非默认状态样式，状态映射表每行包含交互状态/CSS类/Token/技术方案引用
- 技术方案 §5 状态机覆盖 PRD §6 所有业务状态，每个状态有转换路径，终态明确

### L2 相邻产物间

- 交互-顶层定义的交互状态规范，必须在 UI-顶层定义的色彩角色和组件状态中有对应视觉定义（如项目未建立 UI-顶层定义，此条豁免）
- 交互-顶层定义的交互状态规范（6+4种状态），必须在 UI-顶层定义的组件全局状态规范中有对应视觉定义（如禁用态、加载态、错误态的视觉表现）（如项目未建立 UI-顶层定义，此条豁免）
- PRD §6 业务规则中的状态值与交互设计 §4.4 状态机的业务状态列一致
- PRD §6 业务规则中的状态值与技术方案 §5 状态机状态值一致
- 技术方案 §5 异步操作处理（加载/空/错误/成功）与交互设计 §4.4 对应

### L3 端到端链路

- PRD §6 业务规则中定义的每个状态流转，必须在技术方案 §5 中有对应状态转换
- 交互设计 §4.4 状态机中定义的每个状态，必须在技术方案 §5 中有对应状态值
- 交互设计 §4.3 组件交互中定义的状态（默认/悬停/按下/聚焦/禁用/加载），必须在 UI 设计的 HTML 原型中有对应 CSS 状态样式（:hover/:active/:focus/.disabled/.loading）。交互设计**只定义状态名称、触发条件和行为含义**，禁止定义具体颜色值、亮度百分比、透明度数值等视觉属性
- 交互设计 §4.3 组件交互中定义的力学约束（超时时间、重试次数、防抖间隔），必须在技术方案中有对应实现（如接口超时配置、前端防抖逻辑）
- 交互设计 §4.4 状态机中定义的异步状态（空状态/错误状态/成功状态/骨架态），必须在 UI 设计的 HTML 原型中有对应状态类（.empty/.error/.success/.skeleton）。交互设计**只定义状态名称和业务含义**，禁止定义具体的视觉形态（如"骨架屏""插画"等）
- 交互设计 §4.9 响应式适配中定义的结构变更（元素增减、布局变化），必须在 UI 设计 HTML 原型中有对应实现（如 `@media` 查询中的结构变化）

## 五、错误维度规则

### L1 单产物内

- PRD §7 每个错误码必须在 §6 业务规则中有明确的触发场景或前置条件
- PRD §7 错误码触发场景必须具体可执行，禁止笼统描述（如"用户操作时"）
- 交互设计每个页面的异常处理子节必须包含异常场景、触发条件、系统响应、用户感知、恢复路径 5 个字段
- 交互设计简单页面至少定义 1 个异常场景，复杂页面至少定义 3 个

### L2 相邻产物间

- PRD §7 每个错误码在技术方案 §7 异常处理中有对应映射
- 技术方案 §7 异常处理必须覆盖业务异常、系统异常、第三方异常三类
- 技术方案 §7 中定义的每个异常场景必须在 §4 至少一个接口的错误码中有映射
- PRD §7 每个错误码在测试 §2 异常测试用例中有对应验证
- 技术方案 §7 异常处理中的错误码在测试 §2 中有对应验证

### L3 端到端链路

- PRD §7 错误处理中定义的每个错误码，必须在技术方案异常处理章节和测试报告异常测试用例中有对应映射
- PRD §7 定义的错误码数量 = 技术方案 §7 实现的错误码数量 = 测试 §2 验证的错误码数量
- 技术方案 §7 异常处理中定义的每个异常场景，必须在测试报告异常测试用例中有对应测试覆盖
- 测试 §1 每个功能测试用例必须关联 PRD §8 的一条验收标准
- 测试 §6 覆盖检查报告中标记为"已覆盖"的验收标准，必须在 §1 或 §2 中有对应用例存在

## 六、性能维度规则

### L1 单产物内

- PRD §4 性能指标可量化（有数字和阈值），安全需求具体可执行
- 技术方案 §8 性能目标明确（QPS/RT 有数字），缓存策略覆盖热点数据

### L2 相邻产物间

- PRD §4 性能指标在技术方案 §8 中有对应性能目标
- PRD §4 性能指标在测试 §3 中有对应性能测试项
- 技术方案 §8 缓存策略 Key 命名符合技术-顶层定义

### L3 端到端链路

- PRD §4 每个性能指标在技术方案 §8 中有对应目标
- PRD §4 每个性能指标在测试 §3 中有对应测试项
- 技术方案 §8 性能目标在测试 §3 中有对应测试项

---

## 七、安全维度规则

### L1 单产物内

- PRD §4 安全需求不过于笼统，必须具体可执行
- 技术方案 §10 安全设计覆盖鉴权/输入校验/越权校验/敏感数据处理
- 技术方案 §9 高可用设计覆盖熔断/降级/限流/重试/超时

### L2 相邻产物间

- PRD §4 安全需求在技术方案 §10 中有对应安全措施
- PRD §4 安全需求在测试 §4 中有对应安全测试项
- 技术方案 §10 安全措施在测试 §4 中有对应验证

### L3 端到端链路

- PRD §4 每个安全需求在技术方案 §10 中有对应措施
- 技术方案 §9 高可用设计必须覆盖熔断、降级、限流、重试、超时五类策略
- 技术方案 §10 安全设计中定义的每个安全措施，必须在测试报告安全测试中有对应验证

---

## 八、端到端功能追溯链

- PRD §3 功能需求中定义的每个 P0 功能点，必须在交互设计的「与 PRD 对应」子节中有对应页面或组件引用
- PRD §8 验收标准中定义的每个验收项，必须在测试报告功能测试用例和覆盖检查报告中有对应用例
- PRD §3 每个 P0 功能点有对应的验收标准（§8）/测试用例（测试）/接口实现（技术）

---

## 九、顶层定义强制对齐

- 技术-顶层定义的技术栈选型，必须在各模块技术方案的 §1 技术决策中有对应引用或超出声明
- 技术-顶层定义的公共表审计字段规范，必须在各模块技术方案的数据模型中强制执行
- 技术-顶层定义的全局接口约定（HTTP方法/响应结构/分页/认证/版本），必须在各模块技术方案的接口设计中有对应遵循
- 禁止出现 PRD-顶层定义有定义但其他顶层定义无对应的情况，发现缺失 → 触发回改流程

---

## 十、版本维度规则

### L2 相邻产物间

- 各产物版本号必须与上游产物版本号一致
- 版本号、制定日期、作者信息完整
- 检查上游顶层定义在本产物及关联顶层定义中的完整映射，发现缺失标记为阻塞性差异

## 十一、边界维度规则

### L1 单产物内

- 产物中无不属于本产物范畴的描述（PRD 不写交互/视觉/技术；交互不写视觉/技术/测试；UI 不写行为/需求/技术；技术不写产品价值/交互/视觉；测试不写需求/交互/视觉/技术）
- 交互设计线框图不包含颜色/字体/阴影等视觉样式
- UI 设计 HTML 中无 JS 业务逻辑/数据请求

## 十二、常见遗漏项（生成后必须逐条确认）

> 以下是对 6×3 框架的高风险检查点的浓缩提醒，执行审计时不得遗漏。

### 数据与功能

- [ ] 所有 P0 功能点都有对应的验收标准（PRD）/测试用例（测试）/接口实现（技术）
- [ ] 所有接口都标注了对应的功能点
- [ ] 所有表结构都包含审计字段（如项目技术-顶层定义有要求）
- [ ] 测试 §6 覆盖检查报告必须列出所有 PRD 验收标准
- [ ] 测试 §7 回归范围与实际执行的测试用例匹配

### 状态与错误

- [ ] 所有异常场景都对应了错误码、PRD §6 业务规则或模块特定异常
- [ ] 所有状态流转都有明确的触发条件和目标状态
- [ ] 所有页面都有空状态和错误状态的定义
- [ ] 交互设计每个页面状态机包含 6 种状态（默认/空/加载/错误/成功/骨架）

### 结构与完整性

- [ ] 所有 P0 用例都有明确的前置条件、测试步骤和预期结果
- [ ] 技术方案 §1–§14 章节齐全
- [ ] 所有阻塞性差异修复后都经过回归验证，测试 §7 回归范围与实际执行的测试用例匹配

### 性能维度

- [ ] PRD §4 性能指标可量化（有数字和阈值），安全需求具体可执行
- [ ] 技术方案 §8 性能目标明确（QPS/RT 有数字），缓存策略覆盖热点数据
- [ ] 技术方案 §8 缓存策略 Key 命名符合技术-顶层定义

### 高可用与安全

- [ ] 所有第三方依赖都有不可用时的降级策略（技术方案）

### UI 与交互

- [ ] 所有交互设计定义的组件状态都有 UI 设计的 CSS 状态样式实现
- [ ] 所有 UI 设计使用的 CSS 变量都已在项目 UI-顶层定义中声明
- [ ] 所有 UI 设计页面都有响应式适配（如项目 UI-顶层定义有要求）
- [ ] 产物中无不属于本产物范畴的描述
