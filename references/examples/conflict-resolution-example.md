# 示例：用户拒绝同步时的冲突标注与记录

> 本示例展示当上游发生变更、用户明确拒绝立即同步下游时，如何按 §2.2 要求标注 `[待同步]` 并记录冲突，防止上下游长期脱节。

---

## 场景描述

**变更来源**：产品经理在评审后决定，将 PRD-v1-订单模块中的「取消订单」功能（F006）从 P1 降级为 P2，并推迟到 v1.1 版本实现。

**用户决策**：用户（产品经理）明确要求：**本次迭代不实现 F006，也不修改已完成的下游文档**。

---

## 阶段 1：识别冲突

AI 在检查上游 PRD 变更时，发现以下差异：

| 变更项 | PRD 原定义 | PRD 新定义 | 影响范围 |
|--------|-----------|-----------|---------|
| F006 优先级 | P1 | P2 | 交互设计、技术方案、测试报告 |
| F006 里程碑 | v1.0 | v1.1 | 所有下游文档的功能点引用 |
| §7 错误码 E005 | 定义「订单状态不允许当前操作」 | **删除**（因功能推迟） | 技术方案 §6/§8、交互设计异常处理 |

---

## 阶段 2：用户确认与决策记录

AI 向用户呈现影响范围：

> **上游变更影响分析**：
> 1. 交互设计-订单交易流程.md：P006 订单详情中的「取消订单」按钮需标注 v1.1 实现
> 2. TECH-v1-订单服务.md：取消订单接口需标记为 v1.1，错误码 E005 需移除
> 3. 测试报告-订单功能测试.md：TC-F07、TC-E03 需标记为 v1.1
>
> 建议：同步修改所有下游文档，或在受影响文档中标注 `[待同步]`。

**用户回复**："先不改动代码和已写完的文档，下个迭代再说。"

**AI 决策**：执行 §2.2 「用户拒绝」分支 → 在所有受影响下游文档中标注 `[待同步]`，并记录冲突。

---

## 阶段 3：在下游文档中标注 `[待同步]`

### 3.1 交互设计文档标注

**文档**：`交互设计-订单交易流程.md`

**upstream-document 表格更新**：
```markdown
**上游文档**：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v1-订单模块 | 需求输入 | F001-F005, P001-P006 |
| PRD-v1-订单模块 | 需求输入（[待同步]） | F006（优先级 P1→P2，推迟至 v1.1） |
| 项目交互全局规范 | 规范继承 | 导航结构、手势规则、组件行为 |
```

**§6 P006 订单详情 的商家/用户操作栏中增加注释**：
```markdown
<!-- [待同步: PRD-v1-订单模块 §2 F006] 当前交互设计包含「取消订单」按钮，但上游已将 F006 推迟至 v1.1。v1.0 实现时可隐藏此按钮或置灰。 -->
```

### 3.2 技术方案文档标注

**文档**：`TECH-v1-订单服务.md`

**upstream-document 表格更新**：
```markdown
**上游文档**：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v1-订单模块 | 需求输入 | F001-F005 |
| PRD-v1-订单模块 | 需求输入（[待同步]） | F006（推迟至 v1.1）、E005（已删除） |
| 交互设计-订单交易流程 | 设计输入 | P001-P006 |
| 项目技术全局规范 | 规范继承 | 技术栈、公共表、接口约定 |
```

**§6 接口设计中增加待同步标注**：
```markdown

|---------|------|---------|---------|---------|
| EX005 | 并发取消 | ... | ... | — |

> **⚠️ [待同步]** PRD §7 错误码 E005「订单状态不允许当前操作」已随 F006 推迟而删除。
> 当前异常处理仍引用 E005，需在 F006 恢复或删除时同步清理。
```

### 3.3 测试报告文档标注

**文档**：`测试报告-订单功能测试.md`

**upstream-document 表格更新**：

## 阶段 4：冲突记录文件

AI 在项目文档目录中创建/更新冲突记录文件，统一追踪所有 `[待同步]` 项：

**文件**：`docs/CONFLICT-LOG.md`（或 `.doc-chain/conflicts.md`，由项目自定）

```markdown
# 文档冲突记录

> 本文件由 AI 在发现上下游不一致且用户拒绝立即同步时自动维护。
> 每项冲突必须包含：触发时间、上游文档、变更内容、受影响下游、预计解决版本、当前状态。

---

## 冲突 #2024-001

- **触发时间**：2024-01-20
- **上游文档**：`PRD-v1-订单模块.md`
- **变更内容**：
  - F006「取消订单」优先级 P1 → P2
  - F006 里程碑 v1.0 → v1.1
  - 删除错误码 E005（因功能推迟）
- **受影响下游**：
  | 下游文档 | 影响点 | 标注位置 |
  |---------|--------|---------|
  | 交互设计-订单交易流程.md | P006「取消订单」按钮状态未定义 | upstream-document 表格 + §6 注释 |
  | TECH-v1-订单服务.md | 取消订单接口、E005 错误码引用 | upstream-document 表格 + §6/§8 注释 |
  | 测试报告-订单功能测试.md | TC-F07/TC-F08 暂不执行 | upstream-document 表格 + §3 状态列 |
- **预计解决版本**：v1.1
- **当前状态**：🔴 待同步（用户确认推迟）
- **解决检查单**：
  - [ ] PRD 恢复 F006 为 P1 或决定删除
  - [ ] 交互设计删除或更新 P006 的「取消订单」按钮定义
  - [ ] 技术方案确认取消订单接口是否保留（如不保留则删除 §6/§8/§10 相关定义）
  - [ ] 测试报告恢复 TC-F07/TC-F08 为「已覆盖」或删除用例
  - [ ] 所有下游文档 upstream-document 表格删除 `[待同步]` 标记
  - [ ] 执行 doc-audit.py 全量验证

---

*最后更新：2024-01-20 by AI*
```

---

## 阶段 5：后续恢复流程

当 v1.1 迭代开始，产品经理决定恢复 F006 时：

1. AI 读取 `CONFLICT-LOG.md`，发现冲突 #2024-001 状态为「待同步」
2. AI 检查当前 PRD 是否已恢复 F006 为 P1：
   - **是** → 按 `upstream-change-example.md` 的流程执行级联修改，完成后在冲突记录中标记为「已解决」
   - **否** → 保持 `[待同步]` 标记，更新冲突记录的「最后检查时间」
3. 如 PRD 决定**彻底删除** F006（而非推迟），则按删除路径清理所有下游文档中的相关定义，并回收编号留空。

---

## 关键规则重申

- `[待同步]` 标注**必须**出现在 downstream 文档的 upstream-document 表格中，不能只写在正文注释里
- 冲突记录文件**必须**由 AI 自动维护，不能依赖人类记忆
- 禁止在未经用户确认的情况下，擅自删除已标注 `[待同步]` 的内容
- 任何涉及 `[待同步]` 的文档在后续修改时，AI 必须首先读取冲突记录，确认是否有未解决的依赖
