# 示例：上游 PRD 变更后的级联修改与重新验证

> 本示例展示当上游 PRD 发生变更时，如何按 §3.4 回改流程定位影响范围、级联修改下游文档，并重新验证一致性。

---

## 场景描述

**变更来源**：产品经理要求在订单模块中增加"订单改价"功能，允许商家在订单已支付但未发货前修改订单金额（补差价或优惠）。

**涉及的上游变更**：
- PRD-v1-订单模块：新增功能点 F007（订单改价）、页面 P007（改价确认）、错误码 E006
- 状态机变更：PAID 状态下新增允许 transitions → PRICE_CHANGED

---

## 阶段 1：触发回改

AI 在生成交互设计文档时，发现 PRD §6 状态机中 PAID → SHIPPED 的流转规则与新增需求矛盾：

> PRD §6.1 原文：
> ```
> PAID（已支付）
>     │ 商家发货
>     ▼
> SHIPPED（已发货）
> ```
>
> 新需求要求 PAID 状态下可改价，改价后状态变为 PRICE_CHANGED，再发货。

**AI 行为**：
1. 立即停止生成交互设计文档
2. 向用户报告冲突点："当前 PRD 状态机未定义 PRICE_CHANGED 状态，直接生成交互设计会导致下游与上游不一致"
3. 建议回改上游 PRD

---

## 阶段 2：回改上游 PRD

用户确认后，AI 修改 PRD-v1-订单模块：

### PRD 变更 diff

```diff
 ## §2 功能清单
 
 | 功能编号 | 功能名称 | 优先级 | 说明 |
 |---------|---------|--------|------|
 | F001 | 浏览商品 | P0 | ... |
 ...
+| F007 | 订单改价 | P1 | 商家在已支付未发货时修改订单金额，生成补差价链接 |

 ## §4 页面清单
 
 | 页面编号 | 页面名称 | 入口 | 核心功能 |
 |---------|---------|------|---------|
 | P001 | 商品列表 | ... | ... |
 ...
+| P007 | 改价确认 | P006 商家后台 | 输入新金额、确认改价、生成补差链接 |

 ## §6 业务规则
 
 ### 6.1 状态流转
 
 订单生命周期状态机：
 
 ```
 PENDING（待支付）
     │ 支付成功回调
     ▼
 PAID（已支付）
     │ 商家发货
+    │ 商家改价
+    ▼
+PRICE_CHANGED（已改价）
+    │ 商家发货
     ▼
 SHIPPED（已发货）
 ```

+### 6.4 改价规则
+
+| 规则编号 | 规则描述 | 触发条件 | 处理逻辑 |
+|---------|---------|---------|---------|
+| R007 | 改价仅限商家 | 商家点击「修改金额」 | 校验当前用户角色为商家，且订单状态为 PAID |
+| R008 | 改价后需补差 | 改价金额 > 原金额 | 生成补差价支付链接，用户支付后状态变为 PRICE_CHANGED |
+| R009 | 改价后需退款 | 改价金额 < 原金额 | 自动发起部分退款，退款成功后状态变为 PRICE_CHANGED |

 ## §7 错误码
 
 | 错误码 | 触发场景 | 前端提示 |
 |--------|---------|---------|
 | G001 | 系统内部异常 | ... |
 ...
+| E006 | 订单状态不允许改价 | 当前订单状态不允许改价 |

 ## §9 验收标准
 
 | 验收项编号 | 功能点 | 验收项 | 条件 | 操作 | 预期结果 |
 |-----------|--------|--------|------|------|---------|
 | §9.001 | F001 | ... | ... | ... | ... |
 ...
+| §9.009 | F007 | 改价成功 | 订单状态为 PAID，用户为商家 | 输入新金额 100 元，点击确认 | 订单状态变为 PRICE_CHANGED，生成补差链接 |
```

## 阶段 3：评估下游影响（分层扫描）

AI 执行下游影响扫描：

```bash
# 扫描 PRD 的直接下游
python scripts/doc-audit.py PRD-v1-订单模块.md --type prd --scan-downstream ./docs/
```

**扫描结果**：

```json
{

  "doc": "PRD-v1-订单模块.md",
  "downstream_count": 3,
  "downstream": [
    { "path": "interaction/交互设计-订单交易流程.md", "scope": "P001-P006" },
```

**影响分析**：
1. **交互设计**：需新增 P007 页面设计；P006 订单详情需增加「改价」按钮（商家视角）
2. **技术方案**：需新增改价接口、PRICE_CHANGED 状态处理、补差/退款流程
3. **测试报告**：需新增 F007 测试用例、§9.009 覆盖检查

## 阶段 4：级联修改下游文档

### 4.1 交互设计同步修改

**文档**：`交互设计-订单交易流程.md`

**变更**：
1. 文件头部的覆盖页面增加 P007
2. 新增 `## §7 P007 改价确认` 章节（含信息架构、状态机、页面流程、异常处理、与 PRD 对应）
3. `## §6 P006 订单详情` 的商家操作栏增加「修改金额」按钮

**upstream-document 表格更新**：
```markdown
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v1-订单模块 | 需求输入 | F001-F007, P001-P007 |
| 项目交互-顶层定义 | 规范继承 | C端导航结构、手势规则、组件行为 |
```

### 4.2 技术方案同步修改

**文档**：`TECH-v1-订单服务.md`

**变更**：
1. 文件头部的覆盖功能点增加 F007
2. §5 数据模型：t_order 表增加 `price_changed_amount` 字段，t_order_status_log 表支持 PRICE_CHANGED 状态
3. §6 接口设计：新增 `POST /api/v1/orders/{orderNo}/change-price` 接口
4. §7 核心流程：新增「商家改价流程」时序图
5. §8 异常处理：新增 EX006（改价权限不足）、EX007（改价金额非法）
6. §10 接口清单：新增改价接口

**upstream-document 表格更新**：
```markdown
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| PRD-v1-订单模块 | 需求输入 | F001-F007 |
| 交互设计-订单交易流程 | 设计输入 | P001-P007 |
| 项目技术-顶层定义 | 规范继承 | 技术栈、公共表、接口约定 |
```

### 4.3 测试报告同步修改

**文档**：`测试报告-订单功能测试.md`

**变更**：
1. §1 功能测试用例：新增 TC-F08（改价成功）、TC-F09（改价权限不足）
2. §2 异常测试用例：新增 TC-E08（改价金额非法）
3. §3 覆盖检查报告：增加 §9.009 对应 TC-F08

## 阶段 5：重新验证

所有上游及下游修改完成后，AI 按 §3.3 阶段 3 第 10 步执行重新验证：

### 5.1 机械检查（doc-audit.py）

```bash

python scripts/doc-audit.py PRD-v1-订单模块.md --type prd
# 结果：passed

python scripts/doc-audit.py 交互设计-订单交易流程.md --type interaction --delta P007
# 结果：passed

# 验证技术方案（增量）
python scripts/doc-audit.py TECH-v1-订单服务.md --type tech --delta F007 --upstream PRD-v1-订单模块.md
# 结果：passed

# 验证测试报告
python scripts/doc-audit.py 测试报告-订单功能测试.md --type test --upstream PRD-v1-订单模块.md TECH-v1-订单服务.md
# 结果：passed
```

### 5.2 语义检查（模型判断）

- PRD §6.1 状态机：PRICE_CHANGED 状态的入度（PAID）和出度（SHIPPED）均正确
- 交互设计 §7：P007 的「与 PRD 对应」子节正确标注 F007、§6.4
- 技术方案 §6：改价接口的请求参数与 PRD §5 的 `price_changed_amount` 字段对应
- 测试报告 §3：§9.009 与 TC-F08 的对应关系正确
