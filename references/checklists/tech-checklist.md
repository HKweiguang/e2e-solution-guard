# 技术方案检查清单

> 机械检查（格式、编号、表格列完整性）由 `doc-audit.py` 执行。以下为 AI 语义检查项。

---

## 输出格式要求（AI 必须遵守）

每项输出必须包含 5 个字段，缺一不可：

| 字段 | 要求 |
|------|------|
| **检查项** | 复制本清单中的检查项编号和名称 |
| **结论** | 通过 / 不通过 / 存疑（三选一） |
| **证据** | **必须引用技术方案原文片段**（接口定义、字段类型、状态值、异常处理描述） |
| **判定依据** | 为什么是这个结论，引用本清单中的判定标准 |
| **建议** | 结论为"不通过"或"存疑"时必须给出具体修正方向 |

**禁止**：跳过检查项、合并多项为一个结论、无证据写"符合"。

## 对上游（PRD）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| T-U-01 | 实体映射正确性 | **通过**：§3 每张表对应 PRD §5 一个业务实体，字段名与 PRD 业务字段名可映射（语义一致）。**不通过**：表中字段与 PRD 实体无法对应、或存在 PRD 实体在技术方案中无对应表 | 引用 PRD §5 实体定义 + 引用技术方案 §3 对应表结构 |
| T-U-02 | 字段类型映射 | **通过**：§3 字段的数据库类型与 PRD 业务类型可映射（如"金额"映射为 DECIMAL 而非 INT，"日期"映射为 DATETIME 而非 VARCHAR）。**不通过**：类型映射明显不合理（如将金额映射为 INT 丢失精度） | 引用 PRD §5 字段业务类型 + 引用技术方案 §3 对应数据库类型 |
| T-U-03 | 枚举值一致性 | **通过**：§3 枚举值与 PRD §5/§6 中声明的枚举值完全一致，无擅自增减。**不通过**：技术方案中枚举值与 PRD 不一致（如 PRD 定义 4 种状态，技术方案只有 3 种） | 引用 PRD 枚举值列表 + 引用技术方案 §3 对应枚举值列表 |
| T-U-04 | 接口与功能对齐 | **通过**：§4 每个接口对应 PRD §3 至少一个功能点，接口请求/响应字段覆盖该功能点所需全部数据。**不通过**：接口缺失功能点所需字段、或功能点在技术方案中无对应接口 | 引用 PRD §3 功能点数据需求 + 引用技术方案 §4 接口字段定义 |
| T-U-05 | 状态机与业务状态对齐 | **通过**：§5 状态机中的状态值与 PRD §6 业务状态一致，状态转换路径覆盖 PRD §6 中声明的全部流转条件。**不通过**：状态值名称不一致、转换路径遗漏 PRD 中声明的流转条件 | 引用 PRD §6 状态流转条件 + 引用技术方案 §5 状态转换路径 |
| T-U-06 | 错误码映射 | **通过**：§7 异常处理覆盖了 PRD §7 中定义的全部错误码，每个异常场景对应 PRD §7 一个错误码。**不通过**：存在 PRD 错误码在技术方案中无对应异常处理 | 引用 PRD §7 错误码列表 + 引用技术方案 §7 异常处理列表 |
| T-U-07 | 权限实现 | **通过**：§4 接口的权限控制与 PRD §6 业务规则中的权限级别定义一致（如 PRD 要求 ADMIN 才能取消订单，接口有对应鉴权）。**不通过**：权限控制低于 PRD 要求（如只做了登录校验未做角色校验） | 引用 PRD §6 权限规则 + 引用技术方案 §4 接口权限配置 |
| T-U-08 | 性能目标对齐 | **通过**：§8 性能目标（QPS/RT）与 PRD §4 非功能需求中的性能指标一致（数值不矛盾）。**不通过**：技术方案性能目标与 PRD 指标矛盾（如 PRD 要求 RT ≤ 200ms，技术方案目标为 500ms） | 引用 PRD §4 性能指标 + 引用技术方案 §8 性能目标 |
| T-U-09 | 安全需求对齐 | **通过**：§10 安全措施与 PRD §4 安全需求一一对应（每条安全需求都有具体技术措施）。**不通过**：PRD 安全需求在技术方案中无对应实现 | 引用 PRD §4 安全需求 + 引用技术方案 §10 对应措施 |

## 对上游（交互设计）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| T-U-10 | 异步状态对应 | **通过**：§5 异步操作处理（加载/空/错误/成功）与交互设计 §4.4 状态机对应，每个交互状态在技术方案中有对应实现。**不通过**：交互设计定义了但技术方案未实现的状态 | 引用交互设计状态机 + 引用技术方案 §5 异步处理描述 |

## 单产物内业务语义

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| T-S-01 | 接口粒度合理性 | **通过**：接口设计遵循高内聚原则（一个接口做一件事），不过度拆分也不过度聚合。**不通过**：一个接口做 5 件事、或 5 个接口其实只做了 1 件事 | 引用技术方案 §4 接口列表，分析每个接口的职责范围 |
| T-S-02 | 缓存策略合理性 | **通过**：§8 缓存策略针对真正的热点数据，TTL 与数据变更频率匹配（如用户详情缓存 5 分钟，配置表缓存 1 小时）。**不通过**：缓存策略明显不合理（如订单状态缓存 1 小时导致数据不一致） | 引用技术方案 §8 缓存策略描述，分析 TTL 与业务场景的匹配性 |
| T-S-03 | 降级策略完整性 | **通过**：第三方依赖故障时有明确的降级策略（如支付网关故障时进入人工审核流程，非直接抛异常）。**不通过**：第三方故障时直接抛异常或返回 500，无降级处理 | 引用技术方案 §7/§9 中第三方依赖的异常处理描述 |
| T-S-04 | 异常分类完整性 | **通过**：§7 异常覆盖了业务异常/系统异常/第三方异常三类，每类都有对应处理策略。**不通过**：只覆盖了其中 1-2 类，或某类异常无处理策略 | 引用技术方案 §7 异常分类列表，指出缺失类别 |
| T-S-05 | 高可用设计充分性 | **通过**：§9 熔断/降级/限流/重试/超时的阈值与业务场景匹配（如支付服务超时 3s vs 查询服务超时 10s）。**不通过**：阈值与业务场景明显不符（如支付接口超时设为 30s） | 引用技术方案 §9 中各高可用项的阈值配置，分析与业务场景的匹配性 |

## 边界维度

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| T-B-01 | 无越界内容 | **通过**：技术方案中无产品价值描述、交互细节、视觉样式。**不通过**：发现技术方案中包含 UI 颜色值、交互流程详细描述、产品卖点等不属于技术范畴的内容 | 引用越界内容原文 |
| T-B-02 | 技术决策合理性 | **通过**：§1 技术决策有明确的理由，技术栈选型与项目规模/团队能力/性能要求匹配。**不通过**：技术选型无理由（如小项目使用微服务架构无说明）、或选型与性能要求矛盾 | 引用技术方案 §1 技术决策描述，分析理由充分性 |
