# 代码审计检查清单

> 机械检查（格式）由 `doc-audit.py` 执行。以下为 AI 语义检查项。

---

## 输出格式要求（AI 必须遵守）

每项输出必须包含 5 个字段，缺一不可：

| 字段 | 要求 |
|------|------|
| **检查项** | 复制本清单中的检查项编号和名称 |
| **结论** | 通过 / 不通过 / 存疑（三选一） |
| **证据** | **必须引用代码片段 + 产物原文片段**，两者缺一不可 |
| **判定依据** | 为什么是这个结论，引用本清单中的判定标准 |
| **建议** | 结论为"不通过"或"存疑"时必须给出修正方向（改代码 / 改产物） |

**禁止**：跳过检查项、合并多项为一个结论、只引用代码不引用产物、只引用产物不引用代码。

## 对上游（PRD）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| C-U-01 | 功能实现覆盖 | **通过**：代码实现了 PRD §3 中声明的全部 P0 功能点，实现与 PRD 描述一致。**不通过**：存在 P0 功能点在代码中未实现、或实现与 PRD 描述矛盾 | 引用 PRD §3 功能点描述 + 引用代码中对应实现位置（文件:行号） |
| C-U-02 | 业务规则一致性 | **通过**：代码中的业务逻辑（状态判断、权限校验、数据校验、计算逻辑）与 PRD §6 业务规则一致。**不通过**：代码实现与 PRD 规则矛盾（如 PRD 要求满 100 免运费，代码实现为满 200） | 引用 PRD §6 规则原文 + 引用代码中对应实现片段 |
| C-U-03 | 错误码对应 | **通过**：代码中抛出的异常/返回的错误码与 PRD §7 错误码定义一致，错误触发场景与 PRD 一致。**不通过**：代码使用了 PRD 未定义的错误码、或 PRD 定义的错误码在代码中未实现 | 引用 PRD §7 错误码定义 + 引用代码中使用该错误码的位置 |
| C-U-04 | 权限实现 | **通过**：代码中的鉴权/越权校验与 PRD §6 权限规则一致（如 PRD 要求角色权限，代码实现了角色校验而非仅登录校验）。**不通过**：权限控制低于 PRD 要求 | 引用 PRD §6 权限规则 + 引用代码中权限校验实现 |

## 对上游（技术方案）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| C-U-05 | 接口实现一致性 | **通过**：代码中的接口实现（URL/方法/请求参数/响应结构/错误码）与技术方案 §4 接口设计一致。**不通过**：接口路径变更未同步、参数缺失/类型不符、响应结构不一致 | 引用技术方案 §4 接口定义 + 引用代码中对应路由/方法签名 |
| C-U-06 | 数据模型一致性 | **通过**：数据库 schema/表结构/字段类型与技术方案 §3 数据模型一致。**不通过**：字段缺失、类型不兼容、约束不符 | 引用技术方案 §3 字段定义 + 引用代码中 entity/ORM 对应字段 |
| C-U-07 | 状态机实现 | **通过**：代码中的状态流转逻辑与技术方案 §5 状态机设计一致。**不通过**：状态转换条件不符、转换路径缺失、或状态值不一致 | 引用技术方案 §5 状态转换路径 + 引用代码中状态更新逻辑 |
| C-U-08 | 异常处理覆盖 | **通过**：代码中的异常处理覆盖了技术方案 §7 中定义的全部异常场景。**不通过**：技术方案中定义的异常在代码中无对应处理 | 引用技术方案 §7 异常场景 + 引用代码中对应异常处理 |
| C-U-09 | 性能策略落实 | **通过**：代码中有技术方案 §8 声明的缓存策略、查询优化措施的实际实现。**不通过**：技术方案声明了缓存/优化但代码中未实现 | 引用技术方案 §8 性能策略 + 引用代码中对应实现 |
| C-U-10 | 安全措施落实 | **通过**：代码中实现了技术方案 §10 声明的安全措施（输入校验/鉴权/数据脱敏/SQL注入防护）。**不通过**：技术方案声明了但代码中未实现（如缺少预编译语句、输入未过滤） | 引用技术方案 §10 安全措施 + 引用代码中对应实现 |
| C-U-11 | 高可用实现 | **通过**：代码中实现了技术方案 §9 声明的高可用设计（熔断/降级/限流/重试/超时）。**不通过**：技术方案声明了但代码中未配置 | 引用技术方案 §9 高可用配置 + 引用代码中对应配置值 |

## 对上游（UI 设计）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| C-U-12 | 组件状态对应 | **通过**：代码中的组件状态类名（.loading/.disabled/.empty/.error 等）与 UI 设计 HTML 中的状态类名一致。**不通过**：UI 设计定义了状态类但代码中未实现、或类名不一致 | 引用 UI 设计 HTML 中状态类名 + 引用代码中对应组件状态类名 |
| C-U-13 | 响应式实现 | **通过**：代码中的响应式断点与 UI 设计声明的断点一致。**不通过**：断点设置与 UI 设计声明不符 | 引用 UI 设计中断点声明 + 引用代码中 @media 查询 |

## 单产物内业务语义

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| C-S-01 | 审计结论合理性 | **通过**：审计报告 §1 结论（通过/不通过）有充分证据支撑，每个阻塞性差异确实影响功能正确性。**不通过**：结论与证据矛盾（如标注"通过"但存在未修复的阻塞性差异） | 引用审计报告 §1 结论 + 引用 §4 差异清单中未修复的阻塞项 |
| C-S-02 | 差异修正方向 | **通过**：§4 每个差异项的建议修正方向合理（该改代码的改代码、该改产物的改产物）。**不通过**：明显应改产物但建议改代码、或反之 | 引用差异项描述 + 分析建议方向的合理性 |
| C-S-03 | 覆盖完整性 | **通过**：审计内容覆盖了声明的审计对象范围（模块/接口/功能点），无遗漏。**不通过**：声明了审计范围但实际审计内容有遗漏 | 引用审计报告头部声明的审计范围 + 引用实际审计内容列表，比对差异 |
