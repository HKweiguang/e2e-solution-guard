# PRD 检查清单

> 机械检查（格式、编号、表格列完整性）由 `doc-audit.py` 执行。以下为 AI 语义检查项。

---

## 输出格式要求（AI 必须遵守）

每项输出必须包含 5 个字段，缺一不可：

| 字段 | 要求 |
|------|------|
| **检查项** | 复制本清单中的检查项编号和名称 |
| **结论** | 通过 / 不通过 / 存疑（三选一，禁止写"基本通过""大致OK"） |
| **证据** | **必须引用产物原文片段**（功能编号、具体文本、表格行），禁止笼统写"符合要求" |
| **判定依据** | 为什么是这个结论，引用本清单中的判定标准 |
| **建议** | 结论为"不通过"或"存疑"时必须给出具体修正方向 |

**禁止**：跳过检查项、合并多项为一个结论、无证据写"✅"、用"整体看来"代替逐项判断。

## 对上游（PRD-顶层定义）

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| P-U-01 | 模块范围未越界 | **通过**：本 PRD 全部功能点的业务描述均落在顶层定义 §2 该模块的职责范围内。**不通过**：存在功能描述超出模块边界，或模块职责在顶层定义中未声明 | 引用顶层定义 §2 中该模块的职责描述 + 引用 PRD 中越界功能的原文 |
| P-U-02 | 目标用户一致性 | **通过**：PRD §1 目标用户描述与顶层定义 §1 产品概述中的目标用户群体一致（角色名称、职责描述无矛盾）。**不通过**：角色名称不同、职责范围矛盾、或目标用户群体不一致 | 引用顶层定义 §1 目标用户描述 + 引用 PRD §1 目标用户描述 |
| P-U-03 | 术语一致性 | **通过**：PRD 中所有业务术语与顶层定义 §4 术语表一致，同一概念无多个名称。**不通过**：发现术语别名混用（如"客户/用户/会员"），或术语定义与顶层定义矛盾 | 引用顶层定义 §4 术语定义 + 引用 PRD 中使用该术语的原文 |
| P-U-04 | 状态值一致性 | **通过**：PRD §5/§6 中使用的所有状态值均在顶层定义 §5 状态值表中已声明。**不通过**：发现 PRD 中使用了未在顶层定义中声明的状态值 | 引用顶层定义 §5 状态值列表 + 引用 PRD 中使用的状态值原文 |
| P-U-05 | 权限级别一致性 | **通过**：PRD §6 引用的权限级别在顶层定义 §2.2.5 范围内，未将权限级别拆分为独立业务端。**不通过**：发现 ADMIN/SUPER_ADMIN 等权限级别被当作独立业务端处理 | 引用顶层定义 §2.2.5 权限级别定义 + 引用 PRD §6 中权限相关规则原文 |
| P-U-06 | 编码规则一致性 | **通过**：PRD 中所有编号（功能/故事/规则/错误码/验收项）的格式、前缀与顶层定义 §6 编码规则一致。**不通过**：编号格式不符、前缀未在编码规则中定义、序号不连续 | 引用顶层定义 §6 编码规则 + 引用 PRD 中编号示例 |

## 单产物内业务语义

| 编号 | 检查项 | 判定标准 | 强制引用 |
|------|--------|---------|---------|
| P-S-01 | 用户故事覆盖完整性 | **通过**：每个用户故事在 §3 功能需求中至少有一个功能点对应；每个功能点至少关联一个用户故事。**不通过**：存在无对应功能点的用户故事，或存在无关联用户故事的功能点 | 引用 §2 用户故事表格 + 引用 §3 功能需求表格中对应行 |
| P-S-02 | 功能与痛点对齐 | **通过**：每个 P0 功能需求解决了其关联用户故事中声明的痛点（功能描述与痛点有直接因果关系）。**不通过**：功能描述与痛点脱节（如痛点是"系统切换耗时"，功能是"增加导出按钮"） | 引用 §2 用户故事中的痛点描述 + 引用 §3 对应功能的描述，说明因果关系 |
| P-S-03 | 业务规则可执行性 | **通过**：§6 每条规则都有明确的触发条件（何时触发）、判断逻辑（什么条件下做什么）、结果（执行后状态/数据变化）。**不通过**：规则描述为"根据实际情况处理""视情况而定"等不可执行的表述 | 引用 §6 中最模糊的一条规则原文，逐句分析其可执行性 |
| P-S-04 | 状态流转完整性 | **通过**：§6 中声明的状态流转覆盖了该业务实体的所有可能状态变化（包括正常路径和异常路径），每个状态都有明确的终态。**不通过**：存在状态无出口（死状态）、存在状态变化路径未定义、缺少异常回退路径 | 引用 §6 状态流转描述，列出所有状态和转换路径，指出遗漏 |
| P-S-05 | 错误码与规则对应 | **通过**：§7 每个错误码在 §6 中有明确的触发场景，触发条件包含具体的业务条件（非"用户操作时"）。**不通过**：错误码触发场景过于笼统，或存在 §6 中定义了但 §7 中未覆盖的错误场景 | 引用 §7 错误码表格 + 引用 §6 中对应触发场景的原文 |
| P-S-06 | 验收标准可验证 | **通过**：§8 每个验收标准给出可量化的预期结果（有明确的判断条件，如"响应时间 ≤ 2s""字段 A 显示值为 X"）。**不通过**：验收标准为"功能正常""用户体验良好"等不可量化的表述 | 引用 §8 中不可验证的验收标准原文，给出可量化的改写建议 |
| P-S-07 | 数据模型与功能对齐 | **通过**：§5 数据模型中的每个实体/字段至少被一个 §3 功能需求引用；§3 中涉及的数据在 §5 中有对应定义。**不通过**：存在功能需求引用了未在数据模型中定义的字段，或数据模型中存在功能未使用的孤儿字段 | 引用 §3 功能中涉及的数据 + 引用 §5 数据模型对应字段 |
| P-S-08 | 非功能需求量化 | **通过**：§4 中每个性能/安全指标都有具体数字和阈值（如"QPS ≥ 1000""响应时间 ≤ 200ms"），每个安全需求都有具体措施。**不通过**：性能指标无量化、安全需求为"保证数据安全"等笼统表述 | 引用 §4 中不可量化的指标原文，给出量化建议 |

|------|--------|---------|---------|
