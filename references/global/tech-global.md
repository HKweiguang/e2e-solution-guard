# TECH-全局规范

> 适用范围：定义全局技术架构、公共基础设施、跨模块共享的技术规则。模块级技术方案继承此全局规范。

**上游文档**：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD 全局规范 | 规范继承 | §2.2 模块划分、§2.4 全局术语、§2.5 全局状态值 |

---

## §1 章节结构

| 章节 | 必填 | 说明 |
|------|------|------|
| §1 技术栈 | 是 | 各层技术选型及版本 |
| §2 工程结构 | 是 | 项目目录结构、模块划分 |
| §3 公共模块 | 是 | 跨模块共享的代码层 |
| §4 公共表定义 | 是 | 跨模块引用的核心表 |
| §5 全局接口约定 | 是 | HTTP 方法、响应结构、分页、认证 |
| §6 全局安全规范 | 是 | 权限、数据安全、性能基线 |
| §7 基础设施 | 否 | 部署、日志、监控、缓存 |

---

## §2 各章节写作要求

### 2.1 技术栈

表格：层级 | 技术 | 版本 | 用途说明。版本号写主版本即可。无特定技术的层级写 `—` 并说明原因。

### 2.2 工程结构

#### 2.2.1 目录结构
树形结构，标注各目录职责。

#### 2.2.2 模块与代码包的映射
表格：业务模块 | 后端包名 | 前端页面路径。

### 2.3 公共模块

每个公共模块包含：所在位置、职责、关键类/函数、使用方式、禁止事项。常见公共模块：用户认证、文件上传、系统配置、通用响应。

### 2.4 公共表定义

#### 2.4.1 跨模块核心表
表格：字段 | 类型 | 约束 | 说明。

#### 2.4.2 审计字段
所有业务表必须包含创建时间/更新时间/创建人/更新人，以及项目定义的推荐项。

#### 2.4.3 通用字段命名约定
主键、外键、状态、标志、时间、操作人等字段的命名模式。

### 2.5 全局接口约定

#### 2.5.1 HTTP 方法规范
由项目统一定义，常见模式为 GET/POST only 或 RESTful。

#### 2.5.2 统一响应结构
code/message/data 三个字段。code 为字符串类型。

#### 2.5.3 分页规范
请求参数和响应结构由项目统一定义。

#### 2.5.4 认证方式
Token 传递位置、刷新机制、免登录白名单。

### 2.6 全局安全规范

#### 2.6.1 接口权限
公开/登录/角色三级。

#### 2.6.2 数据安全
敏感字段加密、SQL 注入防护、XSS 防护。

#### 2.6.3 限流与防刷
接口限流策略、验证码触发规则。

#### 2.6.4 性能基线
由项目统一定义。

### 2.7 基础设施

#### 2.7.1 部署
部署环境、配置环境变量规则。

#### 2.7.2 日志
日志级别、格式。异常日志必须打印堆栈。

#### 2.7.3 缓存
可用/禁用类型、key 命名规则。

## §3 内容边界

| 写在此文件中 | 写在模块技术方案中 |
|-------------|------------------|
| 技术栈选型 | 模块技术选型理由 |
| 公共表结构 | 模块私有表结构 |
| 公共模块定义 | 模块私有业务层/控制层 |
| 全局接口约定 | 模块具体接口定义 |
| 审计字段规范 | 表的具体业务字段 |
| 全局安全规则 | 模块特有的权限控制 |

**红线**：禁止出现具体业务逻辑、具体模块的接口 URL、具体模块的表字段（公共表除外）。

---

## §4 与模块技术方案的关系

- 模块技术方案开头声明：`继承项目技术全局规范的技术栈、公共表和接口约定`
- 引用格式：`参见本规范 §{章节}「{表名}」`
- 定义新表时必须包含本规范 2.4 规定的审计字段
- 接口必须遵循本规范 2.5 的 HTTP 方法和响应结构约定
- 发现公共定义不够用 → 反馈给本规范维护者，禁止模块擅自扩展
