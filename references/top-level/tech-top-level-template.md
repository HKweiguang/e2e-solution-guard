# 技术-顶层定义（Skill 模板）

> 适用范围：定义全局技术架构、公共基础设施、跨模块共享的技术规则。模块级技术方案继承此顶层定义。

**上游文档**：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD-顶层定义 | 规范继承 | §2.2 模块划分、§2.4 全局术语、§2.5 全局状态值 |

---

## §1 章节结构

| 章节 | 必填 | 说明 |
|------|------|------|
| §1 技术栈 | 是 | 各层技术选型及版本 |
| §2 工程结构 | 是 | 项目目录结构、模块划分 |
| §3 公共模块 | 是 | 跨模块共享的代码层 |
| §4 公共表定义 | 是 | 跨模块引用的核心表 |
| §5 全局接口约定 | 是 | HTTP 方法、响应结构、分页、认证 |
| §6 全局安全规范 | 是 | 权限、数据安全、性能基线 |
| §7 基础设施 | 否 | 部署、日志、监控、缓存 |

---

## §2 各章节写作要求

### 2.1 技术栈

表格：层级 | 技术 | 版本 | 用途说明。版本号写主版本即可。无特定技术的层级写 `—` 并说明原因。

### 2.2 工程结构

#### 2.2.1 目录结构
树形结构，标注各目录职责。

#### 2.2.2 模块与代码包的映射
表格：业务模块 | 代码位置（根据项目架构调整，如后端包名/前端页面路径/服务名等）。

### 2.3 公共模块

每个公共模块包含：所在位置、职责、关键类/函数、使用方式、禁止事项。常见公共模块：用户认证、文件上传、系统配置、通用响应。

### 2.4 公共表定义

#### 2.4.1 跨模块核心表
表格：字段 | 类型 | 约束 | 说明。

#### 2.4.2 审计字段

定义本项目所有业务表必须包含的审计字段清单。常见审计字段包括（由项目根据实际需求裁剪或扩充）：

| 字段 | 说明 | 是否强制 |
|------|------|---------|
| 创建时间 | 记录创建时间 | 建议强制 |
| 更新时间 | 记录最后更新时间 | 建议强制 |
| 创建人 | 记录创建者标识 | 按需 |
| 更新人 | 记录最后更新者标识 | 按需 |
| {项目自定义} | {说明} | {是否强制} |

> AI 生成项目顶层定义时：先基于常见实践列出建议清单，询问用户确认或调整，用户定稿后写入本表格。

#### 2.4.3 通用字段命名约定
主键、外键、状态、标志、时间、操作人等字段的命名模式。

### 2.5 全局接口约定

#### 2.5.1 HTTP 方法规范
由项目统一定义。不限于 RESTful（如项目使用 GraphQL/gRPC/WebSocket/tRPC，在此处声明）。

#### 2.5.2 统一响应结构
由项目统一定义统一响应结构。常见模式如 `code/message/data`（code 可为字符串或整数），或 `status/data/error` 等。AI 生成时应列出建议方案，询问用户确认。

#### 2.5.3 分页规范
请求参数和响应结构由项目统一定义。

#### 2.5.4 认证方式
Token 传递位置、刷新机制、免登录白名单。

### 2.6 全局安全规范

#### 2.6.1 接口权限
由项目统一定义。常见模式如三级（公开/登录/角色）或更细粒度（匿名/用户/管理员/超管/API-key/服务账号等）。AI 生成时应列出建议层级，询问用户确认。

#### 2.6.2 数据安全
项目定义的数据安全措施（如 Web 项目关注 SQL 注入/XSS/CSRF，移动端关注证书固定/签名验证，桌面端关注本地存储加密等）。

#### 2.6.3 限流与防刷
接口限流策略。如项目使用验证码，补充验证码触发规则；如使用其他验证方式（WebAuthn、OAuth、短信验证），在此处声明。

#### 2.6.4 性能基线
由项目统一定义。

### 2.7 基础设施

#### 2.7.1 部署
部署环境、配置环境变量规则。

#### 2.7.2 日志
日志级别、格式。异常日志处理策略由项目定义（如开发环境打印堆栈，生产环境脱敏后上报 APM）。

#### 2.7.3 缓存
可用/禁用类型、key 命名规则。

## §3 内容边界

| 写在此文件中 | 写在模块技术方案中 |
|-------------|------------------|
| 技术栈选型 | 模块技术选型理由 |
| 公共表结构 | 模块私有表结构 |
| 公共模块定义 | 模块私有业务层/控制层 |
| 全局接口约定 | 模块具体接口定义 |
| 审计字段规范 | 表的具体业务字段 |
| 全局安全规则 | 模块特有的权限控制 |

**红线**：禁止出现具体业务逻辑、具体模块的接口 URL、具体模块的表字段（公共表除外）。

---

## §4 与模块技术方案的关系

- 模块技术方案开头声明：`继承项目技术-顶层定义的技术栈、公共表和接口约定`
- 引用格式：`参见本规范 §{章节}「{表名}」`
- 定义新表时必须包含本规范 2.4.2 定义的审计字段清单
- 接口必须遵循本规范 2.5 的 HTTP 方法和响应结构约定
- 发现公共定义不够用 → 反馈给本规范维护者，禁止模块擅自扩展
