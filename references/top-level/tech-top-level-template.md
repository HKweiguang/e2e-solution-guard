# 技术-顶层定义（Skill 模板）

> 适用范围：定义全局技术架构、公共基础设施、跨模块共享的技术规则。模块级技术方案继承此顶层定义。

> **多端/多技术栈说明**：本节支持单技术栈和多技术栈项目（如前后端分离、微服务架构）。多技术栈时，技术栈按层声明，工程结构按层/服务组织。新增技术栈时不改动已有部分。

**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD

**上游文档**：

| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目 PRD-顶层定义 | 规范继承 | §2 全局功能范围（模块划分）、§4 术语表、§5 状态值、§6 编码规则 |

|------|------|------|

| §6 全局安全规范 | 是 | 权限模型、数据安全、限流防刷、性能基线 |
| §7 基础设施 | 否 | 部署、日志、监控告警、缓存。无则写"当前版本基础设施按后续迭代计划执行" |

## §2 各章节写作要求

### 2.1 技术栈

**内容要求**：定义项目各层的技术选型、版本号和选型理由。每层必须说明**为什么选这个技术**（至少一个理由），不能只列名字。

**格式建议**：表格。列：层级 | 技术 | 版本 | 用途说明 | 选型理由。

> 版本号写主版本即可（如 `Vue 3.x`、`Spring Boot 3.x`）。无特定技术的层级写 `—` 并说明原因。

**填写示例**：

| 层级 | 技术 | 版本 | 用途说明 | 选型理由 |
|------|------|------|---------|---------|
| 前端框架 | Vue | 3.x | 单页应用 | 团队熟悉，生态成熟，Composition API 逻辑复用性强 |
| UI 组件库 | Arco Design Vue | 2.x | 企业级组件 | 与 Vue 3 深度集成，主题定制能力强 |
| 状态管理 | Pinia | 2.x | 全局状态 | Vue 官方推荐，TypeScript 支持好 |
| 后端框架 | Spring Boot | 3.x | REST API 服务 | 团队熟悉，Spring Cloud 生态完善 |
| 数据库 | MySQL | 8.x | 关系型数据持久化 | 事务支持完善，主从复制成熟 |
| 缓存 | Redis | 7.x | 热点数据缓存、分布式锁 | 性能高，数据结构丰富 |
| 消息队列 | RabbitMQ | 3.x | 异步任务、事件驱动 | 可靠投递，支持延迟队列 |
| 搜索 | Elasticsearch | 8.x | 全文检索、日志分析 | 与 ELK 生态集成 |

**常见陷阱**：
- ❌ 只列技术名字和版本，不写选型理由——评审时无法判断合理性
- ❌ 选型理由过于主观（"这个技术很火"）——应写业务/技术层面的具体收益
- ❌ 版本号写得太具体（如 `Vue 3.2.47`）—— minor/patch 版本随项目迭代变化，顶层定义只锁定主版本

### 2.2 工程结构

**内容要求**：定义项目的目录结构，以及业务模块与代码位置的映射关系。必须与项目 PRD-顶层定义 §2（模块划分）对齐。

**格式建议**：分两个子节。

#### 2.2.1 目录结构

**内容要求**：用树形结构展示项目目录，标注各目录职责。

**填写示例**：

```
project-root/
├── frontend/               # 前端应用
│   ├── src/
│   │   ├── api/            # API 接口封装
│   │   ├── components/     # 公共组件
│   │   ├── views/          # 页面视图（按模块划分）
│   │   ├── stores/         # Pinia 状态管理
│   │   └── utils/          # 工具函数
│   └── public/
├── backend/                # 后端服务
│   ├── src/main/java/
│   │   ├── common/         # 公共模块（用户认证、文件上传等）
│   │   ├── modules/        # 业务模块（按 PRD 模块划分）
│   │   │   ├── ticket/     # 工单模块
│   │   │   └── customer/   # 客户模块
│   │   └── config/         # 全局配置
│   └── src/main/resources/
└── docs/                   # 项目文档
```

#### 2.2.2 模块与代码包的映射

**内容要求**：每个业务模块在代码库中的位置。必须与项目 PRD-顶层定义的模块划分一致。

**格式建议**：表格。列：业务模块 | 代码位置 | 服务名（微服务时）。

**填写示例**：

| 业务模块 | 代码位置 | 服务名 |
|---------|---------|--------|
| 工单管理 | `backend/src/main/java/modules/ticket/` | `ticket-service` |
| 客户管理 | `backend/src/main/java/modules/customer/` | `customer-service` |
| 系统配置 | `backend/src/main/java/common/config/` | `admin-service`（共享）|

**常见陷阱**：
- ❌ 目录结构与 PRD 模块划分不一致——导致开发人员找不到对应代码
- ❌ 模块间代码路径交叉引用——破坏模块边界

---

### 2.3 公共模块

**内容要求**：定义跨模块共享的代码层/服务。每个公共模块包含：所在位置、职责、关键类/函数、使用方式、禁止事项。

**格式建议**：每个公共模块一个子节或表格。

**填写示例**：

| 模块名 | 所在位置 | 职责 | 关键类/函数 | 使用方式 | 禁止事项 |
|--------|---------|------|------------|---------|---------|
| 用户认证 | `common/auth/` | JWT 签发/校验、权限解析 | `AuthUtil.verifyToken()` | 所有接口入口调用 | 禁止在业务代码中直接解析 Token 字符串 |
| 文件上传 | `common/file/` | 文件上传/下载/预览 | `FileService.upload()` | 通过注入使用 | 禁止上传可执行文件 |
| 统一响应 | `common/response/` | 统一响应体封装 | `Result.success()` / `Result.error()` | 所有 Controller 返回 | 禁止直接返回裸对象 |
| 分布式锁 | `common/lock/` | Redis 分布式锁 | `DistributedLock.acquire()` | 事务前加锁 | 禁止锁内执行远程调用 |

**常见陷阱**：
- ❌ 公共模块职责不清——导致什么都往里面放，变成"垃圾堆"
- ❌ 缺少"禁止事项"——导致公共模块被滥用
- ❌ 公共模块间循环依赖——导致编译/启动失败

---

### 2.4 公共表定义

**内容要求**：定义跨模块引用的核心表结构、审计字段、通用字段命名约定和索引设计规范。

**格式建议**：分四个子节。

#### 2.4.1 跨模块核心表

**内容要求**：列出所有跨模块共享的核心表。只定义表名和用途，不写具体字段（在模块技术方案中定义）。

**格式建议**：表格。列：表名 | 用途 | 被哪些模块引用 | 说明。

**填写示例**：

| 表名 | 用途 | 被引用模块 | 说明 |
|------|------|-----------|------|
| `sys_user` | 用户基础信息 | 所有模块 | 用户 ID、用户名、头像等 |
| `sys_role` | 角色定义 | 权限模块 | 角色编码、角色名称 |
| `sys_user_role` | 用户角色关联 | 权限模块 | 用户与角色的多对多关系 |
| `sys_config` | 系统配置 | 所有模块 | 配置键、配置值、配置类型 |

#### 2.4.2 审计字段

**内容要求**：定义本项目所有业务表必须包含的审计字段清单。

**填写示例**：

| 字段 | 类型 | 说明 | 是否强制 |
|------|------|------|---------|
| `id` | BIGINT / UUID | 主键 | 强制 |
| `create_time` | DATETIME | 记录创建时间 | 强制 |
| `update_time` | DATETIME | 记录最后更新时间 | 强制 |
| `create_by` | BIGINT | 创建者用户 ID | 强制 |
| `update_by` | BIGINT | 最后更新者用户 ID | 强制 |
| `is_deleted` | TINYINT | 逻辑删除标志（0=未删除，1=已删除）| 按需 |
| `tenant_id` | BIGINT | 多租户标识 | 按需 |

> 项目根据实际需求裁剪或扩充。AI 生成时先列出建议清单，询问用户确认。

#### 2.4.3 通用字段命名约定

**内容要求**：定义主键、外键、状态、标志、时间、操作人等字段的命名模式。

**填写示例**：

| 字段类型 | 命名模式 | 示例 |
|---------|---------|------|
| 主键 | `id` | `id` |
| 外键 | `{关联表}_{关联字段}` | `user_id`、`order_id` |
| 状态 | `{业务}_status` | `order_status`、`task_status` |
| 标志 | `is_{标志}` | `is_active`、`is_deleted` |
| 时间 | `{动作}_time` | `create_time`、`complete_time` |
| 操作人 | `{动作}_by` | `create_by`、`update_by` |
| 备注 | `remark` / `{业务}_remark` | `remark`、`reject_remark` |

> 统一使用小写+下划线风格（`snake_case`）。同一语义字段在不同表中保持命名一致。

#### 2.4.4 索引设计规范

**内容要求**：定义索引命名的统一规则和常用索引策略。

**填写示例**：

| 索引类型 | 命名规则 | 示例 |
|---------|---------|------|
| 主键索引 | `PRIMARY` | — |
| 唯一索引 | `uk_{表名}_{字段}` | `uk_user_phone` |
| 普通索引 | `idx_{表名}_{字段}` | `idx_order_status` |
| 联合索引 | `idx_{表名}_{字段1}_{字段2}` | `idx_order_user_status` |

> 高频查询字段优先建索引。索引字段在表结构中显性标注。新表索引需根据查询场景分析设计，不简单照搬旧表。

**常见陷阱**：
- ❌ 字段命名风格不统一（如 `createTime` 和 `create_time` 混用）——导致 SQL 混乱
- ❌ 外键命名不一致（如 `userId` 和 `creator_id` 指向同一字段）——导致 JOIN 困难
- ❌ 缺少审计字段——导致数据操作无法追溯
- ❌ 索引命名混乱——导致 DBA 维护困难

### 2.5 全局接口约定

**内容要求**：定义项目的接口规范。不限于 RESTful（如项目使用 GraphQL/gRPC/WebSocket/tRPC，在此处声明）。

**格式建议**：分五个子节。

#### 2.5.1 HTTP 方法规范

**内容要求**：定义各 HTTP 方法的使用场景。

**填写示例**：

| HTTP 方法 | 使用场景 | 幂等性 | 示例 |
|-----------|---------|--------|------|
| GET | 查询资源 | 幂等 | `GET /api/tickets` |
| POST | 创建资源 | 非幂等 | `POST /api/tickets` |
| PUT | 全量更新资源 | 幂等 | `PUT /api/tickets/{id}` |
| PATCH | 部分更新资源 | 非幂等 | `PATCH /api/tickets/{id}/status` |
| DELETE | 删除资源 | 幂等 | `DELETE /api/tickets/{id}` |

> 如使用 GraphQL，则声明所有操作通过 POST `/graphql` 发送， mutation 用于写操作，query 用于读操作。

#### 2.5.2 统一响应结构

**内容要求**：定义项目的统一响应结构。所有接口必须遵循此结构。

**填写示例**：

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": { ... },
  "traceId": "abc123def456",
  "timestamp": 1699123456789
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | String | 业务状态码，`SUCCESS` 表示成功，其他表示失败 |
| `message` | String | 用户可读提示，成功时可为空 |
| `data` | Object/Array | 业务数据，失败时可为 null |
| `traceId` | String | 链路追踪 ID，用于日志排查 |
| `timestamp` | Long | 响应时间戳（毫秒）|

#### 2.5.3 分页规范

**内容要求**：定义分页请求参数和响应结构。

**填写示例**：

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `pageNum` | Integer | 是 | 页码，从 1 开始 |
| `pageSize` | Integer | 是 | 每页条数，默认 20，最大 100 |

**响应结构**：

```json
{
  "code": "SUCCESS",
  "data": {
    "list": [ ... ],
    "total": 150,
    "pageNum": 1,
    "pageSize": 20,
    "pages": 8
  }
}
```

#### 2.5.4 认证方式

**内容要求**：定义 Token 传递位置、刷新机制、免登录白名单。

**填写示例**：

| 项目 | 规范 |
|------|------|
| Token 传递 | HTTP Header `Authorization: Bearer {token}` |
| Token 刷新 | Refresh Token 机制，有效期 7 天 |
| Access Token 有效期 | 2 小时 |
| 免登录白名单 | `/api/auth/login`、`/api/auth/register`、`/api/public/**` |
| Token 失效处理 | 返回 `401 Unauthorized`，前端引导重新登录 |

#### 2.5.5 版本管理与兼容性

**内容要求**：定义接口版本管理策略和兼容性规则。

**填写示例**：

| 项目 | 规范 |
|------|------|
| 版本路径 | `/api/v1/tickets`、`/api/v2/tickets` |
| 版本策略 | URL 路径版本（优先）或请求头 `X-API-Version: v1` |
| 兼容性规则 | 新增字段必须 optional；禁止修改已有字段类型；删除字段保留标签号 |
| 废弃流程 | 标记为 deprecated → 返回 `410 Gone` → 完全下线（至少保留 3 个版本）|

**常见陷阱**：
- ❌ 接口响应结构不一致（有的返回 `code/message/data`，有的返回 `status/data/error`）——前端处理困难
- ❌ 分页参数风格不统一（有的用 `page/pageSize`，有的用 `offset/limit`）——前端适配成本高
- ❌ 接口版本管理缺失——破坏性变更导致线上故障

### 2.6 全局安全规范

**内容要求**：定义项目的全局安全规则。

**格式建议**：分四个子节。

#### 2.6.1 接口权限

**内容要求**：定义项目的权限模型。

**填写示例**：

| 权限级别 | 说明 | 适用接口 |
|---------|------|---------|
| 公开 | 无需认证 | 登录、注册、公开数据查询 |
| 登录 | 需登录，无角色限制 | 个人中心、查看自己的数据 |
| 角色 | 需登录且拥有指定角色 | 管理员接口、操作他人数据 |
| API Key | 服务间调用 | 内部微服务通信 |

#### 2.6.2 数据安全

**内容要求**：定义数据安全措施。

**填写示例**：

| 维度 | 措施 |
|------|------|
| 传输加密 | 全站 HTTPS，TLS 1.3 |
| 敏感数据存储 | 手机号/身份证加密存储（AES-256） |
| 敏感数据展示 | 脱敏处理（手机号只展示后 4 位） |
| SQL 注入防护 | 预编译语句/ORM 参数绑定 |
| XSS 防护 | 输入过滤 + 输出转义 |
| CSRF 防护 | Token 验证 |

#### 2.6.3 限流与防刷

**内容要求**：定义接口限流策略。

**填写示例**：

| 限流维度 | 策略 | 阈值 | 超限处理 |
|---------|------|------|---------|
| 全局 | 基于 IP | 1000 次/分钟 | 返回 `429 Too Many Requests` |
| 用户 | 基于用户 ID | 100 次/分钟 | 返回 `429` + 提示"操作过于频繁" |
| 接口级 | 基于接口路径 | 10 次/秒 | 返回 `429` |
| 登录 | 基于 IP | 5 次/分钟 | 触发验证码或锁定账号 15 分钟 |

#### 2.6.4 性能基线

**内容要求**：定义项目的性能基线要求。

**填写示例**：

| 指标 | 基线 | 说明 |
|------|------|------|
| 接口平均响应时间 | < 200ms | P99 < 500ms |
| 数据库查询时间 | < 50ms | 单条查询 |
| 前端首屏加载时间 | < 3s | 包含资源加载和渲染 |
| 并发用户数 | 支持 1000+ | 无性能衰减 |
| 系统可用性 | ≥ 99.9% | 年度停机时间 < 8.76 小时 |

**常见陷阱**：
- ❌ 权限模型过于简单（只有"登录/未登录"）——无法满足复杂业务场景
- ❌ 限流策略缺失——容易被刷接口导致服务宕机
- ❌ 性能基线不切实际（如要求所有接口 < 10ms）——导致过度优化

### 2.7 基础设施

**内容要求**：定义项目的部署、日志、监控告警和缓存策略。

> 无基础设施需求则写：**当前版本基础设施按后续迭代计划执行。**

**格式建议**：分四个子节。

#### 2.7.1 部署

**内容要求**：部署环境、配置环境变量规则。

**填写示例**：

| 环境 | 说明 | 部署方式 |
|------|------|---------|
| 开发 | 本地开发 + Docker Compose | 本地启动 |
| 测试 | 独立测试环境 | CI/CD 自动部署 |
| 预发 | 生产镜像预验证 | 手动触发 |
| 生产 | 多节点集群 | 蓝绿部署/滚动更新 |

#### 2.7.2 日志

**内容要求**：日志级别、格式、异常日志处理策略。

**填写示例**：

| 项目 | 规范 |
|------|------|
| 日志级别 | DEBUG（开发）、INFO（生产）、WARN（异常但不影响服务）、ERROR（需立即处理） |
| 日志格式 | JSON，包含：timestamp、level、traceId、service、message、stackTrace（ERROR 时） |
| 日志保留 | 生产环境保留 30 天，ERROR 级别永久保留 |
| 敏感信息 | 日志中禁止输出密码、Token、身份证等敏感信息 |

#### 2.7.3 监控与告警

**内容要求**：监控维度、告警阈值、通知方式。

**填写示例**：

| 监控维度 | 指标 | 告警阈值 | 通知方式 |
|---------|------|---------|---------|
| 应用 | 接口错误率 | > 1% | 钉钉/企业微信 |
| 应用 | 接口 P99 延迟 | > 500ms | 钉钉/企业微信 |
| 数据库 | 连接池使用率 | > 80% | 钉钉/企业微信 |
| 服务器 | CPU 使用率 | > 80% | 钉钉/企业微信 |
| 服务器 | 内存使用率 | > 85% | 钉钉/企业微信 |
| 磁盘 | 磁盘使用率 | > 85% | 钉钉/企业微信 |

#### 2.7.4 缓存

**内容要求**：可用/禁用类型、key 命名规则。

**填写示例**：

| 项目 | 规范 |
|------|------|
| 可用类型 | Redis（String/Hash/Set/ZSet） |
| 禁用类型 | 本地缓存（多实例不一致） |
| Key 命名 | `{服务名}:{业务}:{标识}`，如 `ticket:status:{ticketId}` |
| TTL 策略 | 热点数据 1 小时，配置数据 24 小时，会话数据 2 小时 |
| 缓存穿透 | 空值缓存（设置短 TTL） |
| 缓存击穿 | 互斥锁（分布式锁） |
| 缓存雪崩 | 随机 TTL（基础值 ± 10% 偏移） |

**常见陷阱**：
- ❌ 日志中输出敏感信息——导致安全合规问题
- ❌ 缓存 Key 命名无规范——导致 Key 冲突、难以排查
- ❌ 缺少缓存失效策略——导致数据不一致

## §3 内容边界

| 写在此文件中 | 写在模块技术方案中 |
|-------------|------------------|
| 技术栈选型及版本 | 模块技术选型理由 |
| 工程目录结构 | 模块内部子目录结构 |
| 公共模块定义 | 模块私有业务层/控制层/领域服务 |
| 公共表结构（表名、审计字段） | 模块私有表结构（业务字段、索引） |
| 全局接口约定（响应结构、分页、认证） | 模块具体接口定义（URL、参数、错误码） |
| 审计字段规范 | 表的具体业务字段 |
| 全局安全规则（权限模型、数据安全基线） | 模块特有的权限控制、数据脱敏规则 |
| 性能基线 | 模块性能优化措施、压测结果 |
| 缓存 Key 命名规则 | 模块具体缓存 Key 设计和失效策略 |

**红线**：禁止出现具体业务逻辑、具体模块的接口 URL、具体模块的表字段（公共表除外）、具体业务场景的缓存策略。

## §4 与模块技术方案的关系

- 模块技术方案开头声明：`继承项目技术-顶层定义的技术栈、公共表和接口约定`
- 引用格式：`参见项目技术-顶层定义 §{章节}「{规则名}」`
- 章节编号可能随项目调整，引用时建议同时标注章节标题
- 定义新表时必须包含本规范 §4.2 定义的审计字段清单
- 接口必须遵循本规范 §5 的全局接口约定（HTTP 方法、响应结构和分页）
- 接口版本必须遵循本规范 §5.5 的版本管理与兼容性策略
- 发现公共定义不够用 → 反馈给本规范维护者，禁止模块擅自扩展
- 模块技术方案中的技术选型超出本规范 §1 技术栈范围 → 上升为架构决策（ADR），经全局评审后方可使用
