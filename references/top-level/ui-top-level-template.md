# UI-顶层定义（Skill 模板）

> 适用范围：定义跨模块、跨产物线共享的视觉系统。交互设计产物直接引用此顶层定义中的 Token 和组件规范作为模块级视觉规范来源。模块级视觉表现（组件在各状态下的颜色、亮度、焦点环等）在 UI 设计稿（HTML）中实现。

> **多端说明**：本节支持单产物线和多产物线项目。多产物线项目时，Token 体系统一，各产物线的差异在 §5 响应式断点（C 端）、§11 组件库和 §12 {产物线名} 视觉规范中声明。新增产物线时不改动已有产物线的 Token。

## §2 各章节写作要求

### 2.1 设计原则

**内容要求**：用 3-5 句话回答三个问题——**这套视觉系统要传达什么感受**、**为谁而设计**、**核心设计关键词是什么**。必须包含至少 2 个可指导具体设计决策的关键词（如"清晰""专业""温暖"）。不要展开 Token 列表或技术方案。

**格式建议**：自由文本，建议控制在 100 字以内 + 设计关键词列表。

**填写示例**：

> 面向企业级后台管理系统，视觉系统以"清晰、专业、高效"为核心关键词。通过克制的色彩、明确的层级对比和一致的间距节奏，降低用户认知负荷，帮助用户快速定位关键信息。
>
> **设计关键词**：
> - **清晰**：信息层级明确，一眼可辨主次
> - **专业**：色彩克制，避免过度装饰
> - **高效**：间距统一，减少视觉噪音

**常见陷阱**：
- ❌ 设计原则写成功能清单（"支持暗黑模式、支持响应式"）——这些是能力，不是原则
- ❌ 关键词过于抽象（"美观""好用"）——无法指导具体设计决策
- ❌ 关键词之间互相矛盾（如"极简"和"丰富"未说明适用场景）

---

### 2.2 设计令牌体系

**内容要求**：定义项目的 Design Token 命名规范和层级体系。Token 是设计系统的"原子"——所有视觉属性（颜色、字体、间距、圆角、阴影等）都必须通过 Token 管理，禁止在模块 UI 稿中直接使用硬编码值。

**格式建议**：分三个子节。

#### 2.2.1 命名规范

**内容要求**：定义 Token 的命名格式。命名应包含**类别-用途-状态**三级信息，确保自描述性。

**填写示例**：

| 命名模式 | 示例 | 说明 |
|---------|------|------|
| 颜色 | `{用途}-{层级/状态}` | `primary`、`primary-hover`、`text-secondary`、`bg-page` |
| 字体 | `{属性}-{层级}` | `font-size-h1`、`font-weight-bold`、`line-height-body` |
| 间距 | `{规模}` | `space-xs`、`space-sm`、`space-md`、`space-lg` |
| 圆角 | `{规模}` | `radius-sm`、`radius-md`、`radius-lg`、`radius-full` |
| 阴影 | `{层级}` | `shadow-sm`、`shadow-md`、`shadow-lg` |

> **状态命名约定**：默认态省略状态后缀（`primary` 而非 `primary-default`），变体态加后缀（`primary-hover`、`primary-disabled`）。

#### 2.2.2 Token 层级映射

**内容要求**：定义 Token 的分层体系。主流设计系统采用 2-3 级映射，项目根据规模选择。

**填写示例**（三级映射）：

| 层级 | 作用 | 示例 |
|------|------|------|
| **Primitive（基础值）** | 原始色值、字号、间距数值，不直接用于组件 | `#1890ff`、`16px`、`4px` |
| **Semantic（语义化）** | 与用途绑定的 Token，描述"是什么" | `primary`、`text-primary`、`space-md` |
| **Component（组件级）** | 与组件状态绑定的 Token，描述"在哪里用" | `button-primary-bg`、`input-border-focus` |

> Primitive 值（如 `#1890ff`）必须在 §3 色彩系统色阶体系、§4 字体系统字号层级、§5 间距与布局系统基础单位中明确定义，禁止仅在示例中写出而不在规范表格中声明。

> **小规模项目**可简化为两级（Semantic→Component），**大型项目**建议三级。禁止 Primitive 值直接出现在组件样式中。

#### 2.2.3 Token 类型清单

**内容要求**：列出项目支持的所有 Token 类型及其命名空间。

**填写示例**：

| Token 类型 | 命名前缀 | 说明 |
|-----------|---------|------|
| 颜色 | （可选前缀）| 如 `primary`、`text-secondary`、`bg-page`、`border-default` |
| 字体 | `font-` | 包含字号、字重、行高、字体家族 |
| 间距 | `space-` | 包含内边距、外边距、间隙 |
| 尺寸 | `size-` | 包含元素高度、宽度、图标尺寸 |
| 圆角 | `radius-` | 圆角大小 |
| 阴影 | `shadow-` | 阴影层级 |
| 层级 | `z-` | z-index 层级 |
| 动效 | `motion-` | 过渡时长、缓动曲线（与交互层对齐） |

**常见陷阱**：
- ❌ Token 命名不包含用途信息（如 `color-1`、`color-2`）——无法自描述
- ❌ 同一语义在不同场景使用不同 Token 名（如按钮主色叫 `primary`，卡片主色叫 `main`）
- ❌ Token 层级混乱（Primitive 值直接用于组件样式）

### 2.3 色彩系统

**内容要求**：定义项目的完整色彩体系。色彩是视觉系统中最复杂的部分，必须系统化、可扩展、无障碍友好。

**格式建议**：分五个子节。

#### 2.3.1 调色板

**内容要求**：定义项目的所有颜色类别及其用途。

**填写示例**：

| 颜色类别 | 用途 | 常见色阶数 |
|---------|------|-----------|
| 主色（Brand） | 品牌识别、主要行动按钮、关键元素 | 8-10 级 |
| 语义色（Semantic） | 成功/警告/错误/信息等功能状态 | 每色 4-6 级 |
| 中性色（Neutral） | 文字、背景、分割线、边框 | 10-14 级灰度 |
| 数据色（Data） | 图表、数据可视化 | 8-12 色 |
| 扩展色（Extended） | 标签、分类、辅助标识 | 按需定义 |

> 色阶命名建议：数字权重体系（50/100/200/.../900）或语义命名（lightest/lighter/light/default/dark/darker/darkest）。

#### 2.3.2 色阶体系

**内容要求**：定义色阶的生成规则。色阶不是简单的线性渐变，而应基于感知均匀的色彩空间生成。

**格式建议**：文本说明 + **带具体色值的示例表格**。这是 Primitive 层 Token 的唯一定义处，必须包含具体 HEX/RGB 值。

**填写示例**：

> 色阶基于 OKLCH 色彩空间生成，确保相邻色阶的感知亮度差均匀。主色生成 10 级色阶（50-950），中性色生成 14 级灰度（0-1300）。

| 色阶 | 命名 | 值 | 用途示例 |
|------|------|-----|---------|
| 50 | `primary-50` | `#E6F7FF` | 最浅背景、悬停底色 |
| 100 | `primary-100` | `#BAE7FF` | 轻量背景、选中态 |
| 500 | `primary-500` | `#1890ff` | 默认主色（主按钮、链接）|
| 600 | `primary-600` | `#40a9ff` | 悬停态 |
| 700 | `primary-700` | `#096dd9` | 按下态、深主色文字 |

> **注意**：表格中的"值"列是**强制要求**，不可省略。下游 Semantic Token（如 `primary`）通过映射到此处的 Primitive Token 获得具体色值。如果缺少"值"列，开发人员无法将 Token 映射到实际 CSS 变量值。

#### 2.3.3 色彩角色

**内容要求**：定义每种颜色在 UI 中的职责分配。参照 Material Design 3 的色彩角色模型，确保每种 UI 元素都有明确的色彩来源。

**格式建议**：表格。列：色彩角色 | Token 名 | 默认映射 | 用途。

**填写示例**：

| 色彩角色 | Token 名 | 默认映射 | 用途 |
|---------|---------|---------|------|
| 主色 | `primary` | `primary-500` | 主按钮、链接、关键图标 |
| 主色-文字 | `on-primary` | `neutral-0` | 主色背景上的文字（与primary形成对比）|
| 主色-容器 | `primary-container` | `primary-100` | 轻量主色背景（标签、徽章）|
| 容器-文字 | `on-primary-container` | `primary-700` | 主色容器上的文字 |
| 表面色 | `surface` | `neutral-50` | 卡片、弹窗背景 |
| 背景色 | `bg-page` | `neutral-0` | 页面底层背景 |
| 文字-主 | `text-primary` | `neutral-900` | 标题、正文 |
| 文字-次 | `text-secondary` | `neutral-600` | 辅助说明、描述 |
| 文字-禁用 | `text-disabled` | `neutral-400` | 禁用态文字 |
| 边框 | `border-default` | `neutral-200` | 输入框、卡片边框 |
| 边框-轻 | `border-light` | `neutral-100` | 分割线、次要分隔 |
| 错误 | `error` | `semantic-red-500` | 错误提示、校验失败 |
| 错误-文字 | `on-error` | `neutral-0` | 错误色背景上的文字 |

> 项目根据实际需求增减色彩角色，但至少覆盖：主色/表面/背景/文字（3级）/边框/错误。

#### 2.3.4 色彩使用规则

**内容要求**：定义颜色与 UI 场景之间的映射关系，防止色彩滥用。

**格式建议**：表格。列：UI 场景 | 背景色 | 文字色 | 边框色 | 备注。

**填写示例**：

| UI 场景 | 背景色 | 文字色 | 边框色 | 备注 |
|---------|--------|--------|--------|------|
| 主按钮-默认 | `primary` | `on-primary` | 无 | — |
| 主按钮-悬停 | `primary-600` | `on-primary` | 无 | — |
| 次按钮-默认 | `surface` | `primary` | `primary` | — |
| 次按钮-禁用 | `surface` | `text-disabled` | `border-default` | — |
| 输入框-默认 | `bg-page` | `text-primary` | `border-default` | — |
| 输入框-聚焦 | `bg-page` | `text-primary` | `primary` | — |
| 输入框-错误 | `semantic-red-50` | `text-primary` | `error` | — |
| 卡片 | `surface` | `text-primary` | 无 | 可带阴影 |

#### 2.3.5 无障碍色彩

**内容要求**：定义色彩的无障碍要求，确保色盲用户和弱视力用户可正常使用。

**格式建议**：列表。

**填写示例**：

- 正文文字与背景对比度 ≥ 4.5:1（WCAG AA 级）
- 大文字（≥18pt 或 ≥14pt 粗体）对比度 ≥ 3:1
- 语义色（成功/警告/错误）不单独依赖颜色传达信息，必须配合图标或文字说明
- 避免红-绿组合作为唯一区分方式（红绿色盲最常见）
- 提供高对比度主题选项（可选）

**常见陷阱**：
- ❌ 只用颜色表示状态（如红色文字表示错误，但没有错误图标）——色盲用户无法区分
- ❌ 主色和语义色混用（如主按钮和成功按钮使用同一种绿色）——语义模糊
- ❌ 色阶体系中缺少"值"列——开发人员无法知道 Token 对应的具体色值
- ❌ 色阶数量过少（如主色只有 3 级）——无法满足 Hover/Focus/Disabled 等多种状态

### 2.4 字体系统

**内容要求**：定义项目的字体家族、字号层级、字重、行高和字体使用规则。

**格式建议**：分四个子节。

#### 2.4.1 字体家族

**内容要求**：定义项目中文字体、西文字体和等宽字体的选择。

**填写示例**：

| 用途 | 字体家族 | 回退栈 |
|------|---------|--------|
| 正文/标题（中文）| 系统默认 | `-apple-system, "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif` |
| 正文/标题（西文）| 系统默认 | `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` |
| 代码/数据 | 等宽字体 | `"SF Mono", "Fira Code", "Consolas", monospace` |

#### 2.4.2 字号层级

**内容要求**：定义项目的字号层级体系。相邻层级需有明显视觉区分（建议最小差异 ≥2px 或 ≥1 个倍数单位）。

**格式建议**：表格。列：层级名 | Token | 字号 | 行高 | 用途。

**填写示例**：

| 层级名 | Token | 字号 | 行高 | 用途 |
|--------|-------|------|------|------|
| Display | `font-size-display` | 36px | 44px | 数据大屏标题、空状态 |
| H1 | `font-size-h1` | 28px | 36px | 页面标题 |
| H2 | `font-size-h2` | 22px | 30px | 模块标题 |
| H3 | `font-size-h3` | 18px | 26px | 卡片标题 |
| Body | `font-size-body` | 14px | 22px | 正文 |
| Caption | `font-size-caption` | 12px | 20px | 辅助文字、时间戳 |

> 项目根据实际需求定义层级数量（常见 5-8 级）。移动端项目建议字号整体下调 1-2px。

#### 2.4.3 字重与行高

**内容要求**：定义可用的字重和默认行高倍数。

**填写示例**：

| 字重名 | 数值 | 用途 |
|--------|------|------|
| Regular | 400 | 正文、描述 |
| Medium | 500 | 按钮文字、标签 |
| Bold | 700 | 标题、强调 |

> 行高倍数：标题 1.2-1.4 倍，正文 1.5-1.6 倍，紧凑场景 1.25 倍。

#### 2.4.4 字体使用规则

**内容要求**：定义字体层级与 UI 场景之间的映射关系。

**格式建议**：表格。列：UI 场景 | 字号 | 字重 | 颜色 | 行高。

**填写示例**：

| UI 场景 | 字号 | 字重 | 颜色 | 行高 |
|---------|------|------|------|------|
| 页面标题 | `font-size-h1` | Bold | `text-primary` | 1.3 |
| 模块标题 | `font-size-h2` | Bold | `text-primary` | 1.3 |
| 正文 | `font-size-body` | Regular | `text-primary` | 1.5 |
| 辅助说明 | `font-size-caption` | Regular | `text-secondary` | 1.5 |
| 按钮文字 | `font-size-body` | Medium | `on-primary` | 1.25 |

**常见陷阱**：
- ❌ 字号层级过少（如只有大/中/小三级）——无法满足复杂信息层级
- ❌ 相邻层级视觉差异不明显（如 14px 和 15px）——用户无法区分
- ❌ 同一页面使用超过 4 种字号——视觉混乱

### 2.5 间距与布局系统

**内容要求**：定义项目的基础间距单位、间距 Token、栅格系统、页面边距和响应式断点。

**格式建议**：分五个子节。

#### 2.5.1 基础单位

**内容要求**：定义所有间距和尺寸的基础单位。所有视觉尺寸必须是基础单位的整数倍。

**填写示例**：

> 基础单位：4px。所有间距、尺寸、圆角必须是 4px 的整数倍。禁止使用 1px、2px、6px 等非 4 的倍数（边框线宽和阴影偏移/模糊半径除外）。

#### 2.5.2 间距 Token

**内容要求**：定义项目的间距 Token 体系。间距用于控制元素之间的距离（内边距、外边距、间隙）。

**格式建议**：表格。列：Token | 数值 | 用途。

**填写示例**：

| Token | 数值 | 用途 |
|-------|------|------|
| `space-xs` | 4px | 图标与文字间隙、紧凑内边距 |
| `space-sm` | 8px | 表单标签与输入框间隙 |
| `space-md` | 16px | 卡片内边距、列表项间隙 |
| `space-lg` | 24px | 模块间间距 |
| `space-xl` | 32px | 页面区域间距 |
| `space-2xl` | 48px | 大模块间距 |
| `space-3xl` | 64px | 页面级间距 |

> 项目根据实际需求定义数量（常见 5-8 级）。命名建议：xs/sm/md/lg/xl/2xl/3xl 或 4/8/16/24/32/48/64。

#### 2.5.3 栅格系统

**内容要求**：定义项目的页面栅格体系，确保布局的一致性和响应式适配。

**格式建议**：文本说明 + 表格。

**填写示例**：

> 采用 24 列栅格系统。页面内容区最大宽度 1200px，居中显示。列宽自适应，槽宽（gutter）固定 24px，边距（margin）固定 24px。

| 参数 | 值 | 说明 |
|------|-----|------|
| 列数 | 24 | — |
| 最大内容宽度 | 1200px | 超过此宽度时内容居中，两侧留白 |
| 槽宽（gutter）| 24px | 列与列之间的固定间距 |
| 边距（margin）| 24px | 内容区与视口边缘的最小间距 |
| 列宽 | 自适应 | `(内容宽度 - 23 × gutter) / 24` |

#### 2.5.4 页面边距

**内容要求**：定义不同页面区域的内边距标准。

**填写示例**：

| 区域 | 水平边距 | 垂直边距 | 说明 |
|------|---------|---------|------|
| 页面级容器 | `space-lg` (24px) | `space-lg` (24px) | 最外层包裹 |
| 卡片 | `space-md` (16px) | `space-md` (16px) | 卡片内部 |
| 表单区块 | `space-lg` (24px) | `space-md` (16px) | 表单分组 |
| 列表项 | `space-md` (16px) | `space-sm` (8px) | 列表行 |

#### 2.5.5 响应式断点（C 端产物线适用）

**内容要求**：定义 C 端产物线（App / Web / 小程序）的响应式断点，以及各断点下的布局调整规则。**管理后台、车载、手表等异构产物线不使用响应式断点，其适配规则在 §12 {产物线名} 视觉规范中定义。**

**格式建议**：表格。列：断点名 | 范围 | 布局调整 | 典型设备。

**填写示例**（C 端）：

| 断点名 | 范围 | 布局调整 | 典型设备 |
|--------|------|---------|---------|
| 手机 | < 768px | 单列布局，导航收起到抽屉，字号下调 1-2px | 手机 |
| 平板 | 768px - 1024px | 双列或单列，侧边栏可折叠 | iPad / 小平板 |
| 桌面 | 1024px - 1440px | 标准 24 列栅格，全功能导航 | 笔记本 / 显示器 |
| 大屏 | ≥ 1440px | 最大宽度限制 1200px 居中，两侧留白 | 大屏显示器 |

> 断点数量和范围由项目根据目标设备定义。禁止在断点之间出现布局"断裂"（如 767px 和 769px 差异过大）。
> **异构产物线（车载/手表）注意**：这些产物线不是 C 端的"缩小版"，不套用响应式断点。车载有固定屏幕尺寸和安全区规范，手表有圆形屏幕和低功耗模式，均在 §12 中独立定义。

**常见陷阱**：
- ❌ 间距没有统一的基础单位——导致视觉节奏混乱
- ❌ 栅格列数与组件宽度不兼容（如组件宽 350px，但列宽只能是 8 的倍数）
- ❌ 响应式断点过多（超过 5 个）——开发和维护成本剧增

### 2.6 形状系统

**内容要求**：定义项目的圆角和边框规范。

**格式建议**：分两个子节。

#### 2.6.1 圆角 Token

**内容要求**：定义项目中使用的圆角大小。不同圆角传达不同的视觉感受（尖锐 vs 柔和）。

**格式建议**：表格。列：Token | 数值 | 用途。

**填写示例**：

| Token | 数值 | 用途 |
|-------|------|------|
| `radius-none` | 0px | 表格、数据密集区域 |
| `radius-sm` | 4px | 按钮、标签、小卡片 |
| `radius-md` | 8px | 输入框、中等卡片 |
| `radius-lg` | 12px | 大卡片、弹窗 |
| `radius-xl` | 16px | 页面级容器、浮层 |
| `radius-full` | 999px | 胶囊按钮、头像 |

#### 2.6.2 边框规则

**内容要求**：定义边框的默认宽度、颜色和风格。

**填写示例**：

| 场景 | 宽度 | 颜色 | 风格 | 说明 |
|------|------|------|------|------|
| 输入框-默认 | 1px | `border-default` | solid | — |
| 输入框-聚焦 | 2px | `primary` | solid | 加粗并变色 |
| 输入框-错误 | 1px | `error` | solid | — |
| 卡片 | 1px | `border-default` | solid | 可选，也可只用阴影区分 |
| 分割线 | 1px | `border-light` | solid | 水平分隔 |

**常见陷阱**：
- ❌ 圆角大小不统一（有的按钮 2px，有的 4px，无规则）
- ❌ 边框颜色与背景色对比度过低——用户看不清边界

### 2.7 阴影与海拔

**内容要求**：定义项目的阴影层级和 z-index 层叠规范。阴影不仅用于装饰，更用于表达元素的海拔高度和层级关系。

**格式建议**：分两个子节。

#### 2.7.1 阴影层级

**内容要求**：定义不同层级的阴影样式。层级越高，阴影越大越模糊。

**格式建议**：表格。列：Token | 水平偏移 | 垂直偏移 | 模糊半径 | 扩展半径 | 颜色 | 用途。

**填写示例**：

| Token | X | Y | 模糊 | 扩展 | 颜色 | 用途 |
|-------|---|---|------|------|------|------|
| `shadow-sm` | 0 | 1px | 2px | 0 | `rgba(0,0,0,0.05)` | 按钮悬停、轻微提升 |
| `shadow-md` | 0 | 4px | 8px | 0 | `rgba(0,0,0,0.08)` | 卡片、下拉菜单 |
| `shadow-lg` | 0 | 8px | 16px | 0 | `rgba(0,0,0,0.12)` | 弹窗、抽屉 |
| `shadow-xl` | 0 | 16px | 32px | 0 | `rgba(0,0,0,0.16)` | 模态框、全屏浮层 |

> 暗色模式下阴影应减弱或改用表面变亮（surface brightening）代替阴影。

#### 2.7.2 海拔与层叠（z-index）

**内容要求**：定义项目的 z-index 层级体系，防止层叠混乱。

**格式建议**：表格。列：层级 | z-index 范围 | 典型元素 |

**填写示例**：

| 层级 | z-index | 典型元素 |
|------|---------|---------|
| 基础内容 | 0 - 10 | 页面内容、普通组件 |
| 局部浮层 | 100 - 200 | Dropdown、Tooltip、Popover |
| 固定元素 | 500 - 600 | 固定导航栏、回到顶部按钮 |
| 模态层 | 1000 - 1100 | Modal、Dialog、Drawer |
| 全局通知 | 2000 - 2100 | Toast、全局消息、加载遮罩 |

> 每个层级预留 100 的跨度，方便模块内细分。禁止直接使用 `z-index: 9999`。

**常见陷阱**：
- ❌ 阴影层级只有 1-2 级——无法满足弹窗/卡片/按钮的区分需求
- ❌ z-index 随意赋值（如 99、999、9999）——层叠关系不可维护

### 2.8 动效与过渡

**内容要求**：定义 UI 层的动效 Token。UI 层只定义**具体的视觉动效参数**（时长、缓动曲线），动效的**行为规则**（分级、红线、适用场景）在交互-顶层定义中管理。

> **边界声明**：UI 层定义"动效长什么样"（时长多少、用什么曲线），交互层定义"什么时候用哪种动效"（快/标准/慢三级的适用场景、反馈时效红线）。模块 UI 稿引用 UI 层的 Token 值，引用交互层的规则。

**格式建议**：分三个子节。

#### 2.8.1 过渡时长

**内容要求**：定义项目中使用的过渡时长 Token。与交互-顶层定义的动效分级对齐。

**格式建议**：表格。列：Token | 时长 | 对应交互分级 | 用途。

**填写示例**：

| Token | 时长 | 对应交互分级 | 用途 |
|-------|------|-------------|------|
| `motion-instant` | 0ms | — | 无过渡（状态切换）|
| `motion-fast` | 100ms | 快 | 按钮状态切换、微交互 |
| `motion-normal` | 200ms | 标准 | 弹窗出现/消失、页面切换 |
| `motion-slow` | 400ms | 慢 | 复杂展开、引导动画 |

> 时长值由项目根据品牌调性定义，需与交互-顶层定义的动效分级对应。

#### 2.8.2 缓动曲线

**内容要求**：定义项目中使用的缓动曲线 Token。

**格式建议**：表格。列：Token | 曲线 | 用途。

**填写示例**：

| Token | 曲线 | 用途 |
|-------|------|------|
| `ease-out` | `cubic-bezier(0, 0, 0.2, 1)` | 元素进入视野（弹窗打开）|
| `ease-in-out` | `cubic-bezier(0.4, 0, 0.2, 1)` | 状态切换、展开收起 |
| `linear` | `linear` | 进度条、循环动画 |

#### 2.8.3 与交互层边界

**内容要求**：明确 UI 层和交互层在动效上的分工，避免重复定义或遗漏。

**填写示例**：

| 维度 | UI-顶层定义 | 交互-顶层定义 |
|------|------------|--------------|
| 时长值 | 定义具体毫秒数（0ms/100ms/250ms/400ms）| 定义分级（快/标准/慢）和适用场景；instant 为无过渡/即时响应 |
| 缓动曲线 | 定义具体曲线值 | 定义曲线类型（ease-out/ease-in-out/linear）和适用场景 |
| 反馈时效 | — | 定义红线（点击 50ms/弹窗 200ms/页面切换 300ms）|
| 动效分级 | — | 定义快/标准/慢三级的使用规则 |

**常见陷阱**：
- ❌ UI 层和交互层都定义"弹窗出现用 250ms"——重复定义，修改时容易遗漏
- ❌ UI 层定义了时长但交互层没有对应分级——开发不知道何时用哪个时长

### 2.9 图标规范

**内容要求**：定义项目中图标的使用规范。图标风格必须与整体视觉调性一致。

**格式建议**：表格或列表。

**填写示例**：

| 维度 | 规范 |
|------|------|
| 尺寸 | 小 16px / 中 20px / 大 24px / 超大 32px / 空状态 48px |
| 风格 | 线性（stroke 1.5px），圆角端点，禁止填充风格 |
| 颜色 | 默认继承文字色（`currentColor`），可显式指定 `primary` / `text-secondary` / `error` |
| 使用场景 | 操作按钮内（中 20px）、列表行内（小 16px）、空状态（大 48px）|
| 禁止场景 | 禁止用图标代替文字按钮（无 Tooltip 时）、禁止在同一页面混用线性/填充风格 |

**常见陷阱**：
- ❌ 图标尺寸不统一（同一页面有 14px、16px、18px 的图标）
- ❌ 图标风格与品牌调性冲突（如严肃后台用卡通风格图标）

### 2.10 主题与暗黑模式

**内容要求**：定义项目的主题变量体系和亮色/暗色模式下的 Token 映射。通过 Token 映射实现主题切换，不是简单的颜色替换。

**格式建议**：分两个子节。

#### 2.10.1 主题变量体系

**内容要求**：定义哪些 Token 是"主题敏感的"（随主题变化），哪些是"主题不变的"。

**填写示例**：

> 主题敏感 Token：所有色彩角色（`primary`、`surface`、`bg-page`、`text-primary` 等）。
> 主题不变 Token：间距、圆角、阴影模糊半径、字体家族、字号层级。

#### 2.10.2 亮色/暗色 Token 映射

**内容要求**：定义亮色模式和暗色模式下主题敏感 Token 的映射关系。

**格式建议**：表格。列：Token | 亮色模式 | 暗色模式 | 说明。

**填写示例**：

| Token | 亮色模式 | 暗色模式 | 说明 |
|-------|---------|---------|------|
| `bg-page` | `neutral-0`（#FFFFFF）| `neutral-950`（#1F1F1F）| 页面背景 |
| `surface` | `neutral-50`（#FAFAFA）| `neutral-900`（#2D2D2D）| 卡片背景 |
| `text-primary` | `neutral-900`（#1F1F1F）| `neutral-0`（#FFFFFF）| 主文字 |
| `text-secondary` | `neutral-600`（#666666）| `neutral-400`（#999999）| 辅助文字 |
| `border-default` | `neutral-200`（#E5E5E5）| `neutral-800`（#4D4D4D）| 边框 |
| `primary` | `primary-500` | `primary-400` | 主色在暗色下适当提亮 |
| `error` | `semantic-red-500` | `semantic-red-400` | 错误提示，暗色下适当提亮 |

> 暗色模式不是简单的"反色"——需要单独校准每种颜色的暗色映射，确保对比度和品牌一致性。

**常见陷阱**：
- ❌ 暗色模式只是简单反色（白变黑、黑变白）——导致色彩饱和度过高、视觉疲劳
- ❌ 阴影在暗色模式下仍然使用黑色阴影——应改用表面变亮表示海拔

### 2.11 组件库

**内容要求**：定义项目组件库的分类体系、全局状态规范和组件清单。此处**不写具体组件的样式细节**（尺寸/颜色/间距），只定义分类、状态支持和通用规则。具体样式在模块 UI 稿或独立组件产物中定义。

**内容要求**：定义组件库中所有组件必须支持的状态及其视觉规范。状态定义必须与交互-顶层定义 §3（交互状态规范）对齐。

**格式建议**：表格。列：状态 | 视觉要求 | 对应交互-顶层定义 |

| 组件名 | 分类 | 支持状态 | 备注 |

- ❌ 组件状态与交互-顶层定义不一致——如交互定义了 6 种基础状态，但 UI 只实现了 4 种
- ❌ 缺少组件分类体系——导致组件查找困难、命名混乱

### 2.12 {产物线名} 视觉规范（可选）

> **适用场景**：项目包含异构产物线（管理后台、车载端、手表端等）时使用。C 端产物线（App/Web/小程序）的差异在 §5 响应式断点中处理，**不需要**单独成节。
>
> **单产物线项目**：本节标题替换为实际产物线名（如"管理后台视觉规范"），写一次即可。
> **多产物线项目**：重复本节，每产物线顺延编号（如"§12 管理后台视觉规范"、"§13 车载端视觉规范"、"§14 手表端视觉规范"）。新增产物线时不改动已有产物线的内容。

**内容要求**：定义该产物线特有的视觉规范，包括但不限于：
- **屏幕适配**：固定屏幕尺寸、安全区、异形屏处理（刘海/挖孔/圆形屏）
- **字体映射**：该产物线的字体家族与 §4 字体层级的映射关系（如车载端使用 DIN 字体替代系统字体）
- **色彩差异**：该产物线特有的色彩角色或 Token 覆盖（如车载端高对比度模式、手表端低功耗黑白模式）
- **组件差异**：该产物线特有的组件规范或状态差异（如车载端按钮最小触摸区域 ≥ 80×80px、手表端列表项纵向排列）
- **动效限制**：该产物线的动效限制（如车载端禁止复杂动画防眩晕、手表端禁用长时长过渡）

**格式建议**：每个产物线一节，结构参照 §2.5-§2.11 的章节框架，只写与 C 端**不同**的内容，相同内容引用 C 端规范即可。

**填写示例**（管理后台视觉规范）：

| 规范项 | C 端规范 | 管理后台差异 | 说明 |
|-------|---------|-------------|------|
| 页面边距 | 16-24px | 12-16px | 信息密度更高 |
| 表格行高 | — | 48px | 数据表格默认行高 |
| 按钮最小高度 | 40px | 32px | 操作密度更高 |
| 侧边导航宽度 | — | 200px | 固定侧边栏 |

**填写示例**（车载端视觉规范）：

| 规范项 | C 端规范 | 车载端差异 | 说明 |
|-------|---------|-----------|------|
| 色彩对比度 | ≥ 4.5:1 | ≥ 7:1 | 夜间驾驶安全 |
| 按钮触摸区域 | ≥ 44×44px | ≥ 80×80px | 驾驶中盲操作 |
| 动画时长 | 200-300ms | 禁止复杂动画 | 防眩晕 |
| 字体字号 | 14-20px | 18-32px | 远距离阅读 |
| 夜间模式 | 可选 | 强制 | 自动切换 |

**常见陷阱**：
- ❌ 异构产物线套用 C 端响应式断点——车载屏幕尺寸固定，手表屏幕为圆形，不适用断点
- ❌ 重复定义与 C 端相同的规范——应引用 C 端规范，只写差异
- ❌ 未声明安全区或硬件限制——导致内容被系统 UI 遮挡或不符合法规要求

## §3 内容边界

| 写在此文件中 | 写在模块 UI 稿中 |
|-------------|----------------|
| 设计原则、设计关键词 | 模块级视觉调性的具体应用 |

| 间距 Token、栅格系统、断点 | 具体页面的布局间距 |
| 圆角、边框、阴影、z-index 层级 | 具体元素的视觉效果 |

**红线**：禁止出现具体页面的布局结构、具体页面的文案内容、具体组件的像素级样式定义（尺寸/颜色/间距的硬编码值）。

## §4 与模块 UI 稿的关系

- 模块 UI 稿开头声明：`继承项目 UI-顶层定义的 Token 体系、色彩系统和组件规范`
- 使用全局 Token 时直接引用 Token 名，不重复写色值或像素值
- 组件必须从项目 UI-顶层定义 §11 组件库的组件清单中选取，模块特有组件需在模块 UI 稿中注明并申请纳入全局组件库
- 布局尺寸必须使用 §5 的间距 Token 和栅格系统
- 发现全局 Token 不够用 → 反馈给本规范维护者，禁止模块擅自新增 Token
- 发现全局组件不满足需求 → 申请扩展组件库或提交模块级组件设计方案
