e2e-solution-guard 机制解析报告

端到端产物一致性管理系统 · 从想法到代码的链路约束机制

1. 系统定位

e2e-solution-guard 是一套面向软件工程全流程的产物一致性管理系统。其核心目标并非生成文档模板，而是建立并维护一条从想法到代码的可追溯链路，确保链路中的每一个事实（功能定义、交互规则、视觉规范、技术实现、验证标准）在各环节之间保持对齐，防止因信息传递失真或变更不同步导致的工程失控。

该系统适用于以下场景：

  • 中大型软件项目，涉及多人、多角色协作
  • 需求变更频繁，需要评估变更影响范围的业务
  • 多端/多产物线项目（如同时面向 C 端用户、管理后台运营人员、车载系统等）
  • 对产物一致性和可维护性有明确要求的长期项目

2. 核心理念

2.1 产物依赖网络

系统将软件工程过程抽象为一个有向无环图（DAG），节点为各阶段产物，边为依赖关系。每一份产物必须通过文件头部的 upstream-document 表格显式声明其上游输入。依赖关系构成一个可遍历的网络，使得：

  • 修改上游产物时，可以分层扫描所有下游产物，评估影响范围
  • 修改下游产物时，可以逐条比对是否超出上游定义范围
  • 任何代码实现都可以追溯到最初的功能定义来源

2.2 边界控制

系统对每一份产物划定明确的职责边界，并通过审计机制强制执行：

  • 交互设计 = 结构（SVG 线框图）+ 行为（状态名、触发条件、交互响应、力学约束）。禁止写入视觉属性（颜色、色值、字体、阴影、圆角、精确像素值）。
  • UI 设计 = 视觉（颜色、尺寸、间距、阴影、圆角、字体）+ 显式状态映射表（交互状态 → CSS 类/伪类 → Token 变量 → 技术方案引用）。禁止写入行为（触发条件、交互流程、页面跳转、状态流转、校验规则）。
  • 技术方案 = 实现方案（架构、接口、数据模型、状态机、异常处理）。禁止写入需求定义或视觉规范。

2.3 产物线拆分

当项目面向多个异构用户群体时（如 C 端消费者、管理后台运营人员、车载系统驾驶员），系统支持按产物线（Product Line）拆分产物链。同一业务域在不同产物线中可能具有完全不同的功能集、交互范式和视觉规范。每条产物线拥有独立的产物链（PRD → 交互设计 → UI 设计 → 技术方案 → 测试报告），顶层定义按产物线划分章节，新增产物线不影响已有产物线的规范。

项目级顶层定义（所有产物线共享）
├── 项目 PRD-顶层定义（模块划分、术语表、编码规则）
├── 项目交互-顶层定义（状态规范、反馈规范、各产物线交互规范）
├── 项目 UI-顶层定义（Token 体系、组件库、各产物线视觉规范）
└── 项目技术-顶层定义（技术栈、公共表、接口约定）
         │
         ▲ upstream（所有模块级产物引用）
         │
模块级产物链（按产物线独立，按模块拆分）

3. 产物体系

3.1 产物类型与形式

产物类型 文件形式 核心职责 边界
PRD Markdown 定义"做什么"：功能需求、业务规则、数据模型、错误处理、验收标准 不写实现方案、不写视觉规范
交互设计 Markdown（含 SVG 线框图） 定义骨架：页面结构、组件状态矩阵、状态机、页面流程、异常处理、力学约束 不写视觉属性
UI 设计 独立 HTML 文件 定义血肉：视觉实现、显式状态映射表、CSS 变量、响应式结构 不写行为、无 JS 业务逻辑
技术方案 Markdown 定义"怎么做"：技术决策、架构、接口、数据模型、状态机实现、异常处理 不写需求、不写视觉
测试报告 Markdown 定义验证方案：测试用例、覆盖检查、缺陷汇总、测试结论 不写需求、不写实现

3.2 关节层

骨架（交互设计）与血肉（UI 设计）之间的映射关系，传统模式下隐式散落在 HTML 代码中，由开发人员自行推断。该系统要求将映射关系显式声明为产物的一部分：

  • 状态映射表：每页一张表格，列：交互状态 | CSS 类/伪类 | Token 变量 | 技术方案引用。确保交互设计中的"禁用态"与 UI 设计中的 .disabled 和 --color-disabled 之间有明确的一一对应关系。
  • 力学约束：在交互设计的组件交互表格中增设"力学约束"列，记录超时时间、重试次数、防抖间隔等用户感知层面的技术约束。这些约束需在技术方案中有对应实现。
  • 响应式适配：在交互设计中增设可选的"响应式适配"子节，声明断点下的结构变更（元素增减、布局变化），将结构变更的决策权从 UI 设计回收到交互设计。

4. 引用机制

4.1 上游声明

每一份产物文件头部必须包含 upstream-document 表格，声明引用的上游产物名称、类型和引用范围。Markdown 产物使用表格形式；HTML 产物使用 <!-- upstream: ... --> 注释形式。

示例（Markdown 产物）：

| 产物 | 类型 | 引用范围 |
|------|------|---------|
| 用户管理-PRD.md | PRD | USER-001~USER-003 |
| 项目交互-顶层定义.md | 规范继承 | §3 状态规范、§5 C端交互规范 |

4.2 双向引用完整性

系统通过一致性硬规则强制维护双向引用的完整性：

  • PRD 中的功能点必须在交互设计的「与 PRD 对应」子节中有对应页面或组件引用
  • 交互设计中的每个组件状态必须在 UI 设计的 HTML 原型中有对应 CSS 状态样式
  • UI 设计中的每个组件和状态类必须在技术方案前端实现中有对应引用
  • 技术方案接口清单章节必须与接口设计章节逐个接口一一对应
  • 测试报告中的每个验收标准必须在对应测试用例中有覆盖

5. 审计体系

5.1 三层审计模型

审计层 执行方 检查内容 输出
Mechanical 脚本自动化 编号连续性、upstream 存在性、表格格式、接口清单一致性、HTML 结构完整性、跨产物编号对齐 结构化 JSON（blocking / warning / hint）
Semantic 模型/人工 功能与用户故事对应、业务规则一致性、状态机与 PRD 对齐、术语一致性、边界一致性 审计报告（含证据引用和建议修正方向）
Code Verification 只读 subagent 代码与技术方案/PRD 的一致性对比 差异清单（代码原文 vs 产物原文）

5.2 自动化审计能力矩阵

doc-audit.py 为标准库 only 的审计脚本，支持 9 种审计类型：

审计类型 检查范围
prd 必填章节、功能编号重复、§3→§6/§7/§8 引用一致性、验收标准关联、错误码触发场景
interaction 页面章节格式、6 个必含子节、SVG 线框图、硬编码色值边界、力学约束列、响应式适配
ui HTML 完整性（DOCTYPE/upstream/CSS 变量）、硬编码色值、状态样式完整性、JS 业务逻辑、行为事件、状态映射表注释
tech 必填章节、§4 与 §13 接口一一对应、异常场景映射、错误码格式（--error-prefixes）
test 必填章节、验收标准覆盖、术语一致性
global-* 顶层定义的必填章节、Token 层级、状态规范、组件库分类

5.3 跨产物对齐

当 --upstream 参数提供上游产物时，脚本自动提取上游编号集合（如 USER-001、PAGE-TICKET-001），与当前产物中的编号集合做差集比对，报告缺失项。编号提取器支持大小写不敏感匹配，可处理 Markdown 大写编号与 HTML 小写 section id 的跨格式对齐。

6. 变更管理

6.1 上游变更的传播

修改上游产物时，必须执行以下流程：

  1. 使用 --scan-downstream 扫描所有下游产物，输出受影响清单（支持 Markdown 和 HTML 格式）
  2. 按直接/间接下游分层展示影响范围，按优先级排序
  3. 询问用户是否级联修改
  4. 用户确认 → 同步修改所有下游产物
  5. 用户拒绝 → 在下游产物中标注 [待同步]，禁止留下不一致而不标注

6.2 下游变更的约束

修改下游产物时，必须执行以下流程：

  1. 读取所有上游产物
  2. 逐条比对修改内容是否在上游定义范围内
  3. 未超出范围 → 直接修改
  4. 超出范围 → 立即停止，先改上游

6.3 增量审计

支持 --delta 参数进行增量审计，只检查变更范围内的标识符。适用于大产物的小范围修改场景，减少审计开销。

7. 适用边界

7.1 适合场景

  • 中大型项目，多人、多角色协作
  • 需求变更频繁，需要评估变更影响范围
  • 多端/多产物线项目，需要隔离不同用户群体的功能差异
  • 对产物一致性和可维护性有明确要求的长期项目
  • 团队新人比例高，需要标准化产物模板和检查清单

7.2 不适合场景

  • 原型验证、MVP 阶段，追求极致速度
  • 个人 side project，产物维护成本高于收益
  • 探索性技术预研，需求尚未收敛
  • 纯车载/纯手表等单一极端异构端项目（产物链独立，维护成本较高）

8. 系统代价

采用该系统的显性成本包括：

  • 前期投入：建立顶层定义、产物模板、产物依赖网络需要数人日的初始投入
  • 持续维护：产物网络需要随项目演进持续维护，变更时必须同步上下游
  • 学习成本：团队成员需要理解上下游关系、产物线拆分、边界规则
  • 工具依赖：审计脚本和模板体系是约束力的载体，脱离工具链后依赖团队自觉

收益与成本的权衡取决于项目的规模、生命周期和变更频率。对于需求稳定、规模小、生命周期短的项目，该系统的投入产出比可能不理想。
