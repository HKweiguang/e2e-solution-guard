e2e-solution-guard

让"从想法到代码"的每一步都有迹可循、有据可依

💡 它解决什么问题

做软件时，你有没有遇到过这些情况：

  • PRD 改了功能，技术方案和测试用例没人同步更新
  • 设计师画的交互状态，前端不知道怎么实现，凭感觉写 CSS
  • 同一功能在 App、Web、管理后台长得不一样，用户困惑
  • 代码写完了，回头找不到这个功能最初的需求来源

e2e-solution-guard 就是用来防止这些断裂的。它不是帮你写文档，而是帮你管理从想法到代码的完整链路——让每个环节都知道自己从哪来、该去哪、边界在哪。

🗺️ 一张图看懂全貌

想象你要盖一栋楼。你不能直接让工人搬砖，得先有图纸、有结构图、有材料清单、有验收标准。软件工程也一样：

💡 想法 → 📋 PRD → 🎨 交互 → 🖌️ UI → ⚡ 技术 → 🧪 测试 → 💻 代码

每一步的产物都明确声明"我从哪来"（上游文档），系统能自动追踪这些关系。改前面的图纸，后面的施工队会收到通知。

📦 5 种产物长什么样

产物 形式 打个比方 不能写什么
PRD Markdown 建筑的功能说明书：有几个房间、每个房间干什么用 不写怎么施工、不装修方案
交互设计 Markdown + SVG 图 结构图：门在哪、楼梯怎么摆、开关按下去灯怎么亮 不写墙刷什么颜色、灯什么造型
UI 设计 独立 HTML 文件 装修效果图：墙面颜色、家具尺寸、灯光氛围 不写开关怎么触发、电路怎么走
技术方案 Markdown 施工方案：用什么材料、电路怎么走、承重怎么算 不写房间功能、不装修效果
测试报告 Markdown 验收单：每个功能测了没、有没有 bug、能不能交付 不写功能定义、不施工方案

🔗 "我从哪来"怎么声明

每份产物开头都有一张"家谱表"，写清楚它引用了哪些上游文档。就像论文的参考文献：

📋 用户管理-交互设计.md

上游文档：
| 产物名称          | 类型     | 引用范围               |
|------------------|---------|-----------------------|
| 用户管理-PRD.md   | PRD     | USER-001 ~ USER-003   |
| 项目交互-顶层定义  | 规范继承 | §2 交互状态规范、§4 交互规范 |

--- 正文开始 ---

HTML 产物（UI 设计）也用类似的方式，只不过写在注释里：

<!-- upstream: 用户管理-交互设计.md, 项目 UI-顶层定义.md -->

有了这张"家谱表"，系统就能回答两个问题：

  • 改上游时：自动扫描所有"子孙后代"，告诉你哪些产物受影响
  • 改下游时：检查你的修改有没有"超出祖辈规定的范围"

🔧 关节层：以前没人管的地方

传统流程里，交互设计说"这个按钮有禁用态"，UI 设计画了灰色，前端开发时凭经验写了个 .disabled。三个人心里想的可能是同一件事，但没有一张纸把对应关系写下来。

🦴 关节层（v2.8 新增）

现在要求显式声明这些对应关系，写在产物里，而不是散落在各人的脑子里：

  • 状态映射表：交互设计中的"禁用态" → UI 中的 .disabled → Token --color-disabled → 技术方案中的 Button 组件 disabled 属性
  • 力学约束：按钮按下去多久要反馈？加载超时多久转错误态？以前散落在各处，现在统一写在交互设计里
  • 响应式适配：手机端表格变成卡片列表，这个决策权以前模糊，现在明确归属交互设计

📱 多端支持：同一模块，不同面孔

一个模块可能需要服务不同的端或用户群体。同一套底层能力，在不同端呈现不同交互：

📱 C 端用户

注册、登录、编辑头像、修改密码

🖥️ 管理后台

用户表格、批量禁用、权限分配、数据导出

🚗 车载系统

人脸识别自动登录、语音控制、极简大按钮

⌚ 手表

查看信息、接收通知、旋转表冠操作

按模块聚合：同一模块的所有步骤产物（PRD→交互→UI→技术→测试）放在一起。模块内的多端差异在交互/UI 产物中声明，变更可追溯。

🤖 自动审计：机器能查的，不让人来

系统内置一个 Python 脚本（标准库 only，无依赖），能自动查出这些低级错误：

检查项 什么意思 举个例子
编号连续性 功能编号有没有跳号或重复 USER-001 后面直接 USER-003，中间漏了 002
上游缺失 产物开头有没有声明引用了谁 交互设计忘了写"我引用了哪个 PRD"
跨产物对齐 上游定义的编号，下游有没有漏掉 PRD 定义了 USER-002，交互设计里没提到
接口一致性 技术方案的接口设计和清单是否对得上 §4 写了 5 个接口，§13 清单只列了 4 个
HTML 完整性 UI 设计稿是不是合格的原型 没写 DOCTYPE、没声明 upstream、没 CSS 变量、写了 onclick
边界违规 有没有越界写了不该写的东西 交互设计里写了"按钮用 #ff0000 红色"

机器查完低级错误后，剩下的内容质量（比如业务逻辑对不对、状态机是否合理）由人工或模型来审。

🚦 两条红线

红线 1：改上游必须通知下游

你改了建筑的功能说明书（PRD），所有拿着这份说明书施工的人（交互、UI、技术、测试）都必须知道。要么一起改，要么在图纸上贴个便利贴写"这里待同步"。禁止改了上游，下游还停留在旧版本。

红线 2：改下游不能超出上游范围

施工队不能擅自给房子加个阳台，除非先回去改图纸（上游）。想在下游加新功能？先改 PRD，否则直接叫停。

⚖️ 用它值不值

✅ 什么时候值得用

  • 项目人多、协作复杂
  • 需求经常变，怕改漏
  • 同时做 App、Web、管理后台等多个端
  • 代码要长期维护，不能写完就扔
  • 团队新人多，需要统一标准

⚠️ 什么时候不太适合

  • 做个原型验证，越快越好
  • 个人 side project
  • 探索性尝试，需求还没想清楚
  • 需求极其简单，维护成本高于收益

📋 一句话总结

类比： e2e-solution-guard 就像建筑行业的图纸管理系统。它不是帮你画图纸，而是确保每份图纸都有编号、知道从哪份总图派生、改了总图能自动通知所有相关图纸、施工队拿到的永远是最新版。它的价值不在"写得多快"，而在"改的时候不慌"。
