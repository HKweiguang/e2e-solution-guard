🛡️ e2e-solution-guard

从"拍脑袋需求"到"代码落地"的全链路守护者——用可追溯的产物依赖网络，把变更失控扼杀在摇篮里

📌 一句话概括

这不是一个"写文档工具"，而是一个事实管理器。它管理从想法到代码的完整链路中的一切事实：任何想法必须经过评估，任何变更必须同步到所有相关产物，任何代码必须与约定保持一致。

⚙️ 主要机制（通俗版）

1. 产物形式

PRD → Markdown 交互设计 → Markdown（含 SVG 线框图） UI 设计 → 独立 HTML 文件 技术方案 → Markdown 测试报告 → Markdown

UI 设计产物是独立 HTML 文件，不是 Markdown 中的代码块。所有产物通过文件头部的 upstream-document 表格建立引用关系。

2. 全链路流水线（按产物线拆分）

💡 想法/需求 → 📋 PRD → 🎨 交互设计 → 🖌️ UI 设计 → ⚡ 技术方案 → 🧪 测试 → 💻 代码
↺ 发现问题？随时回改上游，并自动级联同步到所有下游产物

📦 产物线拆分

同一条业务线在不同产物线中功能不同。例如"用户管理"：

  • C 端（App/Web/小程序）：用户注册、登录、编辑资料
  • 管理后台：表格操作、权限分配、批量禁用、数据导出
  • 车载端：人脸识别登录、语音控制、极简交互
  • 手表端：信息查看、通知接收、旋转表冠操作

每条产物线有独立的产物链（PRD→交互→UI→技术→测试），变更互不影响。

3. 骨架 + 血肉 + 关节

🔧 关节层（v2.8 新增）

骨架（交互设计）和血肉（UI 顶层定义）之间的连接方式，以前隐式散落在代码中，现在显式声明：

  • 状态映射表：交互状态 → CSS 类/伪类 → Token 变量 → 技术方案引用
  • 力学约束：超时时间、重试次数、防抖间隔（写在交互设计中）
  • 响应式适配：断点下的结构变更（元素增减、布局变化）

4. 四大核心机制

🔍

Gap 分析（事前评估）

动手写方案之前，先问自己：现有方案是什么？缺什么信息？会不会和已有的冲突？把问题列清楚再问用户，而不是直接开干。

🔗

上下游追踪（变更不遗漏）

每份产物开头必须写明"我引用了哪些上游产物"。改上游时自动扫描所有下游，支持 Markdown 和 HTML 格式。要么一起改，要么标注"待同步"，禁止留不一致。

🤖

自动审计（机械检查）

内置 doc-audit.py 脚本，标准库 only，支持 9 种审计类型。自动检查编号连续性、双向引用、术语一致性、跨产物编号对齐、HTML 完整性（CSS 变量、状态样式、upstream 注释）、边界违规（交互写视觉 / UI 写行为）等。

🚫

红线拦截（防擅自扩展）

下游产物的功能范围、页面、状态值、错误码等，必须在上游有定义。想扩展？先改上游，否则直接叫停。交互设计不写视觉，UI 设计不写行为。

5. 五个"禁止"行为准则

禁止脑补 禁止过度复杂化 禁止模糊继续 禁止缺前置定义继续 禁止批量输出

翻译成人话：没搞清楚就不动手，一步一步来，别自作聪明加需求。

👍 优点 vs 👎 缺点

👍 优点

  • 需求不丢：改了 PRD 会自动提醒改技术方案和测试用例，告别"需求改了代码没改"
  • 口径一致：术语、错误码、状态值全项目统一，减少沟通成本
  • 质量兜底：自动审计抓低级错误（编号重复、上游缺失、跨产物编号不对齐），人工专注业务逻辑
  • 关节显式：状态映射表让"交互状态 → CSS 类 → Token"的关系不再隐式散落，新人能快速理解
  • 多端可控：按产物线拆分后，C 端、管理后台、车载端的变更互不影响
  • 可追溯：任何代码都能找到对应的需求来源，排查问题有依据

👎 缺点与代价

  • 前期重：小需求也要走评估和产物流程，对"快点上线"的急性子不友好
  • 维护成本：产物网络需要持续维护，项目忙时容易沦为形式主义
  • 学习曲线：需要理解上下游关系、产物线拆分、模板规范，团队需要培训
  • 过度防御："遇冲突必停"可能导致简单需求被过度分析
  • 异构端成本高：车载、手表等极端异构端需要独立的产物链，产物数量倍增
  • 灵活度受限：严格的红线拦截在探索性、实验性项目中可能成为阻力

🚀 对工程的实际增益

1

减少"隐性返工"（最大的成本杀手）

很多项目的返工不是因为代码写得差，而是需求理解错了、上下游没对齐。这个 skill 在"动手写代码前"就强制对齐，把返工成本从"代码阶段"前置到"产物阶段"，成本降低 10 倍以上。

2

建立"团队共享内存"

产物不是写给领导的，是写给"两周后的自己"和"新加入的同事"的。上下游追踪 + 显式状态映射让任何人都能快速理解"这个功能为什么这样设计"、"改了会影响哪些地方"。

3

把"人治"变成"法治"

不依赖某个老员工的记性，而是靠产物网络和自动化审计来保证质量。人员流动、上下文切换时，项目不会瞬间失控。

4

提升代码审查效率

代码审查时可以直接对照技术方案和 UI 规范，争论从"我觉得应该这样"变成"产物里怎么约定的"，减少主观扯皮。

5

保护技术债不扩散

"禁止缺前置定义继续"这条规则，直接堵死了"先随便写一版后面再补"的借口。临时方案如果要转正，必须先补产物。

🎯 适合谁用？

✅ 非常适合

  • 中大型项目，多人协作
  • 需求变更频繁的业务
  • 对质量有明确要求的 B 端系统
  • 需要长期维护的核心模块
  • 多端/多产物线项目（App + Web + 管理后台）
  • 新人较多的团队

⚠️ 需要权衡

  • 原型验证、MVP 阶段
  • 个人 side project
  • 探索性技术预研
  • 极度追求速度的营销活动页
  • 纯车载/纯手表等极端异构端（产物链独立，成本较高）

📋 总结

e2e-solution-guard 本质上是一套"工程纪律"。它不会让你的代码写得更快，但会让你少做无用功、少踩重复的坑、少在需求变更时手忙脚乱。v2.8 的核心升级是产物线拆分（C端/管理后台/车载/手表各走各的链路）和关节层显式化（状态映射表、力学约束、响应式适配），让"从想法到代码"的链路在复杂多端场景下依然可控。它的成本是前期的产物投入，收益是整个项目周期的可控性和可维护性。如果你受够了"需求一改全项目爆炸"，这是一个值得认真考虑的方案。
