e2e-solution-guard v4.0

端到端方案一致性守护：从想法到代码，变更可追溯、落地不跑偏

支持多业务端 · 统一编号规范 · 全链路审计

💡 一句话

从想法评估、产物生成、跨模块对齐到代码验证，全程辅助——让每一步都有据可依、变更可追溯、落地不跑偏。

📐 核心原则

🎯项目产物优先：已有定义 > Skill 模板，未定义部分才参考模板
💬对话式提问：关键决策点主动引导用户确认，不一次性抛长问卷
🔍先 gap 分析：识别缺失信息和冲突后再输出，禁止基于模糊想法直接写
🚫禁止脑补：未定义的功能、字段、规则禁止自行补充
🛑遇冲突必停：需求不合理或与现有方案矛盾时明确提出质疑
📏逐步推进：一次只推进一个单元，确认无异常后再开始下一个

🔥 解决什么问题

📝PRD 改了，技术方案和代码没同步
🔗一个服务支撑多个模块，不知道影响范围
🎨交互设计的状态，前端凭感觉写 CSS
🔍代码写完了，找不到最初的需求来源

🗺️ 全链路：8 步产物 + 1 层定义 + 1 道审计

想法 → PRD → 交互 → UI → 技术方案 → 测试 → 代码 → 代码审计。顶层定义横向支撑，每步产出后独立审计。

🔢 统一编号规范：天生支持多业务端

所有产物使用同一种格式，新增业务端时仅需分配新的业务端标识，编号格式不变。

⚡ 核心：PRD 与技术方案不是一一对应

传统做法是一份 PRD 配一份技术方案。实际工程中，一个服务支撑多个模块，一个功能涉及多个服务。

PRD §3 功能需求以「功能编号」为索引，技术方案 §1 声明「覆盖功能点」，双向验证确保无遗漏。

🏗️ 产物组织

业务产物按模块聚合，技术方案按服务聚合。目录结构反映天然边界，关联关系由内容表达。

🤖 审计：机械 + 语义

所有审计由独立只读 subagent执行，主会话只做汇总。禁止后台执行（`run_in_background=false`），必须等待结果返回。

📊产物审计：编号连续性、表格格式、上游引用、必填章节
🧠语义审计：目标一致性、业务规则、跨章节引用、验收标准
🔍代码审计：目录结构、数据流、状态管理、异常处理、性能、安全、端到端
🔗一致性审计：功能点编号 ↔ 接口编号 ↔ 用例编号 双向追踪

🚦 两条红线

⛔ 改上游必须通知下游

改了 PRD → 扫描所有涉及的技术方案、交互、UI、测试，同步更新或标注 [待同步]

⛔ 改下游不能超出上游

技术方案想加新接口 → 先确认 PRD 中有对应功能点，禁止擅自扩展

📋 一句话

类比：e2e-solution-guard 像建筑行业的图纸管理系统——总图派生各专业图纸，改了结构图自动通知机电和装修，施工队按图施工，竣工时逐层核对。价值不在"画得有多快"，而在"改的时候不慌、查的时候有据"。
